## 3. JIT (Just-In-Time) 置換の根拠

### 3.1 由来と現状

- **由来**: 1970 年代の Toyota Production System (在庫管理) → 1980 年代以降のコンピュータ科学 (JIT コンパイル・JVM/V8)
- **AI/LLM 文脈**: 一部研究論文で「Just-In-Time retrieval」として登場するが、**業界標準用語ではない**
- **問題**: 受講者ペルソナ (社会人 2-3 年目・IT 用語ある程度) は JIT を「在庫管理の方法」または「JVM のコンパイル」と誤認する可能性

### 3.2 置換候補の比較

| 候補 | メリット | デメリット |
|------|---------|-----------|
| **必要時取得 (Selective Retrieval)** | 日本語で意味が直接伝わる・LLM 業界での「必要時に context を引き出す」設計と整合 | 「Selective Retrieval」自体は確立用語ではないが、意図が明確 |
| **RAG (Retrieval-Augmented Generation)** | ⭐5 業界標準 (Lewis et al. 2020) | RAG は外部 KB からの retrieval が主・本講座の memory + handoff + 必要時読込 + PreCompact の総称とは厳密一致しない |
| **オンデマンド検索** | 日本語で平易 | 「on-demand search」と混同される |

### 3.3 採用案

**必要時取得 (Selective Retrieval)** + 関連用語として **RAG** に言及

理由:
1. 日本語で意味が直接伝わる (受講者ペルソナへの配慮)
2. RAG の発想と通底することを ch04 で明示 (`必要時取得 概念スライド` で「RAG 系の発想」と注記)
3. 本講座の 4 道具 (claude-mem/handoff/必要時読込/PreCompact) を包含する総称として機能

### 3.4 置換実施件数

**16 件 → 0 件** (NG リスト掲載 1 件のみ意図的残置)

## 4. Lazy Load 置換の根拠

### 4.1 由来と現状

- **由来**: 2000 年代後半の Web 開発 (画像・JavaScript モジュールの遅延読込)
- **AI/LLM 文脈**: Hugging Face Transformers 等で `lazy loading` は使われるが、**「memory ファイルをタスク該当時のみ Read する」という本講座の意図とは異なる**
- **問題**: 受講者は「Web ページの画像遅延読込」のイメージで理解し、本講座の context 節約という意図が伝わらない可能性

### 4.2 採用案

**必要時読込 (On-Demand Loading)**

理由:
1. 日本語で「必要なときだけ読み込む」という意図が直接伝わる
2. 「On-Demand Loading」は Web/CS 一般用語として確立 (⭐4)
3. 「Lazy Load」より意図が明確 (lazy = 怠惰 のニュアンスを排除)

### 4.3 置換実施件数

**8 件 → 0 件** (NG リスト掲載 1 件のみ意図的残置)
