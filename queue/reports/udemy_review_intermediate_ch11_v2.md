## A. わからない箇所 (具体・行番号 or 引用つき)

- L95-116 「宣言型検証 Gate — 設計思想」: "宣言型検証 = AIが事前に完了条件をYAMLに書いておき、Gateがそれを機械的にチェックする"とあるが、「宣言型」が何を意味するのか分からない。普通に言えば「事前に条件を書いておく」だけでいいのでは? わざわざ「宣言型」という言葉を使う理由が説明されていない。
- L102-116 YAMLコードブロック: `post_steps:` `verify:` `command:` `timeout_seconds:` が一気に出てくる。YAMLの説明として「インデントで階層を表す」とL101にあるが、インデントの階層が何を意味するのか(親子関係?)が分からない。コードを見ても「何がどこに属しているのか」が読み取れない。
- L125-147 「hook による自動検証」: 4ステップのフロー説明は分かりやすいが、L142-146 で突然「設定ファイル (settings.json)」と「JSONは { "キー": "値" } という形式」と出てくる。YAMLの次はJSON? 2つの設定ファイル形式が連続して登場し、頭が混乱する。
- L174-195 YAMLの再現例: `status: done` と `verify_result: pending` が同時に存在する矛盾例を「AIが書いた状態」として提示しているが、AIがなぜこんな矛盾した状態を作るのかの説明(L192)が「事後手続きを忘れている」と一言だけ。もう一声、「AIは作業が終わった瞬間に done と書き込むが、テスト実行やファイル保存は後回しになるため、このズレが起きる」という具体がないと「自分でpendingと書いて怒られている」ようにしか見えない。

## B. 論理破綻・飛躍

- L66 「どちらも『人間のあいまいな判断』を『機械的な合格/不合格』に置き換えるのがポイントです」: 直前まで「Supervisorはシェフの味見チェック」という人間の比喩で説明していた。人間の味見チェックは「人間のあいまいな判断」の代表例のはず。それを機械に置き換えるというなら、味見の比喩自体がこの章の本筋と矛盾している。比喩のレベルと本来の仕組みのレベルが混在している。
- L93-94 「これらを毎回人間が確認するのは非現実的 → 自動化が必須」: 前のスライド(L51-66)で「シェフが味見する」を正しい仕組みとして説明したばかりなのに、ここで「人間が確認するのは非現実的」と否定されている。シェフの味見は人間の確認のはずだが、その矛盾に戸惑う。

## C. 詰まる箇所

- L95-124 (宣言型検証 + 3つの設計原則): ここが最大の壁。「宣言型」「opt-in」「自動連鎖」が3つ連続で出てくる上に、それぞれの説明が1行ずつしかない。L120-124の3行は抽象度が高すぎて、ペルソナとしてはここで「理解を追うのを諦めて、なんとなく次に進む」モードに切り替える。以降のスライドも「なんとなく読む」状態になり、確認問題は消去法で解くことになる。
- L194-195 BLOCK メッセージの例: メッセージの形式が YAML と混ざっており、どこがメッセージでどこがYAMLの構造なのか判別できない。

## 改善提案 (ペルソナ目線 3点)

1. L95-124 の「宣言型検証」スライドを2枚に分割する。1枚目で「事前条件をYAMLに書く = 宣言型」を料理の比喩で説明し(例: レシピに「盛り付け済みであること」と書いておく)、2枚目で「opt-in」と「自動連鎖」を説明する。3つの概念を1枚に詰め込むと思考停止する。
2. YAMLコードブロック(L102-116)の前に、インデントの読み方を1枚挟む。「左端に書かれた post_steps が大項目、その下の - path/to/output_file.csv が小項目」のように、2段階だけでも階層構造を図解してほしい。
3. L142-146 のJSON(settings.json)の説明を「詳細は上級編・ここでは概念のみ」に留め、hookの仕組みは「設定ファイルに『YAMLが編集されたら検証を実行』と書いておくと自動で動く」という1文だけで伝える。YAML→JSONと設定ファイル形式が2つ連続すると、初学者は「この2つの違いは何?」で止まる。
