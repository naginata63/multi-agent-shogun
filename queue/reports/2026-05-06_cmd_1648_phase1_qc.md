## 2. 5 件 CRITICAL 抜け穴対策の検証 (cmd_1646 軍師指摘との突合)

cmd_1646 §0 で軍師が指摘した 5 件 CRITICAL 抜け穴がすべて実装に反映されていることを確認:

### CRITICAL #1: stdlib HTTP server 認識 (Flask 不要訂正) → ✅ 反映済

- **指摘**: タスク notes が「Flask 使用」と誤認したが、実機は `http.server.ThreadingHTTPServer` + `BaseHTTPRequestHandler`
- **実装**: `BaseHTTPRequestHandler.do_GET` 内に SSE endpoint 追加 (L2471-2540)
- **追加対策**: `protocol_version = 'HTTP/1.1'` を class 属性で明示 (L1743) → keep-alive 有効化

### CRITICAL #2: server.py 再起動で in-memory queue 全消失対策 → ✅ 反映済

- **指摘**: `dashboard_lifecycle.sh` が 1 時間ごとに server.py 再起動 → in-memory queue ロスト
- **実装**: SSE 接続初期化時に SQLite から未読 (`read=0`) を先 push (L2496-2521)
- **設計**: queue は通知トリガーのみ・SQLite が Single Source of Truth
- **コード**:
  ```python
  # L2502-2512
  rows = conn.execute(
      'SELECT id, from_agent, type, content, timestamp FROM inbox_messages '
      'WHERE agent=? AND read=0 AND id > ? ORDER BY timestamp',
      (agent, last_event_id)
  ).fetchall()
  ```

### CRITICAL #3: Last-Event-ID 未実装 → ✅ 反映済

- **指摘**: SSE 仕様の `Last-Event-ID` ヘッダー対応がないと再接続時に取りこぼし
- **実装**: L2497 で `last_event_id = self.headers.get('Last-Event-ID', '')` 取得・L2501-2506 で SQL `id > last_event_id` クエリ・L2520 で `id: <msg_id>` をすべての event に付与
- **副次効果**: SQLite 接続初期化フェーズと統合され、再接続時に確実に未配送メッセージのみ流れる

### CRITICAL #4: feature flag → ✅ 反映済

- **指摘**: 段階導入のために `ENABLE_SSE_INBOX` 環境変数で切替必須
- **実装**: L43 `ENABLE_SSE_INBOX = os.environ.get('ENABLE_SSE_INBOX', 'false').lower() == 'true'`
- **default**: **false** (本番環境への即時影響を防ぐ)
- **動作**: ENABLE_SSE_INBOX=false の時 /api/inbox_stream が 404 (L2473-2478)・既存 /api/inbox_write も queue push を skip (L3074: `if ENABLE_SSE_INBOX:`)

### CRITICAL #5: SPOF 対策 (server.py 落ちで通知断絶) → ✅ 反映済

- **指摘**: server.py 落ちで SSE のみ依存だと通知完全断絶
- **実装**: 既存 inbox_watcher.sh (10 プロセス) を **完全並走稼働継続** で safety net 確保
- **検証**: inbox_watcher.sh は 4 月 28 日のタイムスタンプから不変 (cmd_1648 で touch されていない)
- **次フェーズ**: systemd 化と健全性 cron は cmd_1651 (Phase 4) で対応 (本 cmd では割切り)


→ task_id が **空文字** で含まれる。`_push_to_inbox_queue` (L48-50) の呼出元 (L3074-3075) で task_id をどう渡しているか要確認。

**軍師判断**: テストが test_runner (task_id 未指定) で実施されたため空文字なのは妥当だが、本番運用で task_id が必要な場合に確実に伝搬されるか Phase 2 で検証必要。**本 QC では NON-BLOCKING**。

### 6.3 SSE event format の一貫性

L2520 (SQLite 初期 push): `id: {row["id"]}\ndata: {evt}\n\n`
L2532 (queue loop): `id: {msg_id}\ndata: {evt}\n\n`

→ 両者で format 一致 ✅。Last-Event-ID 仕様準拠。

### 6.4 wfile.flush() 即時配信の検証

cmd_1646 §2 軸 1 で「`wfile.flush()` が即時配信を保証しない可能性」を指摘した。ash1 Test 5 で SSE event 1 件受信が確認されたため、本 Phase 1 の単体テストレベルでは問題なし。**Phase 2 (1 agent 24h 連続接続) で実運用負荷下の挙動検証** 必要。
