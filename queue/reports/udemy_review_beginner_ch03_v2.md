## B. 論理破綻・飛躍

- L204-206: 「commandsの review.md は『レビューして』の1回きり / skillsの deploy.md は『テスト→ビルド→デプロイ』の流れを丸ごと自動化」 — この比較が不正確に感じる。review.mdの中身（L173-176）を見ると「セキュリティ確認・パフォーマンス指摘・可読性コメント」の3ステップを指示しているので、これも「複数ステップ」に見える。commandsとskillsの違いが「1回 vs 複数ステップ」という説明だと、review.mdの時点で既に矛盾している。条件分岐（成功/失敗で次を変える）が本当の違いなら、そこをもっと前に強調すべき。

- L348: Q3の回答で「imports または global CLAUDE.md」と2つの正解があるが、使い分け基準として「importsはチーム共有向け、globalは1人で使う設定」とある。しかしimportsの説明（L242-247）ではチームという言葉が一切出てこず、「共通設定をプロジェクト間で共有」としか書いていない。1人で複数プロジェクトを跨ぐ場合のimports使い方とglobalの違いが、Q3の回答で初めて語られている。

- L287: レイヤー表で agents.md と imports を同じ「Role」レイヤーに分類しているが、importsは「外部ファイルの取り込み」であって「役割」ではない。分類根拠が飛躍している。

## 改善提案 (1-3個)

1. **`@path` と `~` に1行ずつ補足を入れる**: 「`@` は『このファイルを読み込んで』という意味の記号です」「`~` はホームディレクトリ（あなたのPCの基本フォルダ）を表す記号です」という一文を追加するだけで、L234とL259の離脱ポイントが解消する。社会人2年目の多くはチルダを知らない。

2. **commands と skills の違いを「条件分岐」で説明し直す**: 現在の「1回 vs 複数ステップ」という対比はreview.mdの内容と矛盾している。「commands = 毎回同じ指示を出す / skills = 結果を見て次の行動を変える（条件分岐）」という差に絞ると、具体例とも辻褄が合う。

3. **「エージェント」という用語の初出時に1行補足**: L52かL82で、「エージェントとは、この講座では『AIに特定の役割を与えたもの』くらいの意味です」と注記すると、未定義用語の連発による思考停止を防げる。
