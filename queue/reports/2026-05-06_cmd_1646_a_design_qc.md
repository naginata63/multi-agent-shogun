## 2. 8 軸 QC 評価

### 軸 1: SSE 実装 (stdlib HTTP server での SSE)

#### 抜け穴 (3 件)

1. **CRITICAL — `wfile.write()` の即時配信保証なし**:
   - `wfile` は `socket._fileobject` ベースで内部バッファあり
   - `wfile.flush()` を呼んでも、TCP 送信バッファに書き込まれるだけで client 即時受信は保証されない
   - 受信遅延発生時に SSE 通知が遅延・体感が悪い可能性

2. **HIGH — `Connection: keep-alive` の HTTP/1.0 動作リスク**:
   - stdlib HTTP server はデフォルトで HTTP/1.0 互換モードに falling back する場合あり
   - `protocol_version = 'HTTP/1.1'` を明示しないと keep-alive が機能せず、各メッセージで接続切断・再接続が発生

3. **MEDIUM — chunked encoding の自動付与**:
   - `Content-Length` ヘッダー未指定の場合 stdlib が自動で chunked encoding を付ける
   - `text/event-stream` と chunked encoding の組合せは仕様上 OK だが、curl 等の client 実装によっては挙動が異なる

#### 改善提案

```python
# server.py に追加すべき設定
class DashboardHandler(http.server.BaseHTTPRequestHandler):
    protocol_version = 'HTTP/1.1'  # ← 追加 (keep-alive 有効化)

# SSE generator 内で明示的 flush
self.wfile.write(f'data: {data}\n\n'.encode('utf-8'))
self.wfile.flush()
# socket level の low-level 配送確認は OS 任せで OK (TCP_NODELAY を設定する手段もあるが過剰最適化)
```

### 軸 2: queue 設計 (in-memory queue + SQLite 連携)

#### 抜け穴 (3 件)

1. **CRITICAL — server.py 再起動で queue 全消失**:
   - cmd_1643 設計の `_INBOX_QUEUES = defaultdict(queue.Queue)` は in-memory 限定
   - server.py プロセスが落ちる/再起動すると **全エージェント宛の未配送メッセージがロスト**
   - `dashboard_lifecycle.sh` で 1 時間ごとに server.py が再起動されるため (本プロジェクトの運用実態)・本問題は確実に発生する

2. **HIGH — `maxsize` 未指定で in-memory 無制限肥大化**:
   - cmd_1643 注意事項で「maxsize=1000 推奨」と記載されたが、実装コードには反映漏れの可能性
   - Monitor 接続が切れた agent 宛 queue が無限蓄積 → メモリ枯渇

3. **MEDIUM — queue.put() ブロック挙動**:
   - maxsize 設定後、queue が full になると `put()` がブロック → /api/inbox_write 全体がハングする可能性
   - `put_nowait()` + `queue.Full` 例外処理が必要

#### 改善提案

```python
# 1) 起動時 queue maxsize=1000 で初期化
_INBOX_QUEUES = defaultdict(lambda: queue.Queue(maxsize=1000))

# 2) server.py 再起動対策: SSE 接続時に SQLite から未読を先に push
def do_GET_inbox_stream(self, agent):
    self.send_response(200)
    self.send_header('Content-Type', 'text/event-stream')
    ...
    self.end_headers()

    # 初期化フェーズ: SQLite の未読を先に流す
    last_event_id = self.headers.get('Last-Event-ID', '0')
    conn = sqlite3.connect(DB_PATH, timeout=5)
    cur = conn.execute(
        'SELECT id, type, content, timestamp FROM inbox_messages '
        'WHERE agent=? AND read=0 AND id > ? ORDER BY timestamp',
        (agent, last_event_id)
    )
    for row in cur.fetchall():
        msg = {'msg_id': row[0], 'type': row[1], 'content': row[2], 'timestamp': row[3]}
        self.wfile.write(f'id: {row[0]}\ndata: {json.dumps(msg)}\n\n'.encode())
        self.wfile.flush()
    conn.close()

    # 通常フェーズ: in-memory queue から配信
    q = _INBOX_QUEUES[agent]
    while True:
        try:
            msg = q.get(timeout=30)
            self.wfile.write(f'id: {msg["msg_id"]}\ndata: {json.dumps(msg)}\n\n'.encode())
            self.wfile.flush()
        except queue.Empty:
            self.wfile.write(b': keepalive\n\n')
            self.wfile.flush()

# 3) put_nowait + Full 例外処理
def _push_to_inbox_queue(target_agent, message_dict):
    try:
        _INBOX_QUEUES[target_agent].put_nowait(message_dict)
    except queue.Full:
        # SQLite に書込済なので queue からあふれても OK・Monitor 再接続時に SQLite から再生される
        print(f"[inbox_queue] WARN: queue full for {target_agent}, message dropped from in-memory (still in SQLite)", file=sys.stderr)
```

### 軸 3: 10 agent 同時接続負荷

#### 抜け穴 (3 件)

1. **HIGH — ThreadingHTTPServer の thread 制限**:
   - Python `socketserver.ThreadingMixIn` のデフォルト `daemon_threads = False`
   - SSE は long-lived 接続 = thread が永続化 → 既存 endpoint 用の thread と競合
   - 10 agent + 既存 API 利用 = 数十 thread の常時稼働

2. **MEDIUM — request_queue_size の上限**:
   - `socketserver.TCPServer.request_queue_size = 5` (デフォルト)
   - 同時接続超過時に listen queue が overflow して TCP RST が返る可能性
   - 通常運用では問題ないが、再起動直後に 10 agent + cron 系が同時接続するとリスク

3. **MEDIUM — GIL によるスループット低下**:
   - SSE generator は `time.sleep` 系で待つため GIL 影響は小さい
   - ただし `wfile.write()` 中の Python バイトコード実行は GIL を握る → 大量同時通知時に遅延

#### 改善提案

```python
# server.py の ThreadingHTTPServer 設定強化
class _ThreadedServer(http.server.ThreadingHTTPServer):
    daemon_threads = True   # ← 追加 (server 終了時に thread を巻き取る)
    request_queue_size = 32 # ← 5 → 32 に拡大
    allow_reuse_address = True  # 既に設定済の可能性高

# 接続上限の監視: SSE 接続数を _SSE_CONNECTION_COUNT で追跡
_SSE_CONNECTION_COUNT = {}  # agent_id -> count
_SSE_CONN_LOCK = threading.Lock()

def do_GET_inbox_stream(self, agent):
    with _SSE_CONN_LOCK:
        if _SSE_CONNECTION_COUNT.get(agent, 0) >= 2:
            # 同一 agent から 2 接続以上は拒否 (冪等性確保)
            self.send_response(429)
            self.end_headers()
            return
        _SSE_CONNECTION_COUNT[agent] = _SSE_CONNECTION_COUNT.get(agent, 0) + 1
    try:
        # ... SSE generator ...
    finally:
        with _SSE_CONN_LOCK:
            _SSE_CONNECTION_COUNT[agent] -= 1
```

### 軸 4: SSE 信頼性 (取りこぼし保証)

#### 抜け穴 (3 件)

1. **CRITICAL — Last-Event-ID 未実装**:
   - cmd_1643 設計には `id:` フィールド書込が無い
   - SSE 仕様では client 切断時に再接続 request の `Last-Event-ID` ヘッダーで再生位置を指定する
   - 未実装だと client 再接続時に「どこから再開すれば良いか分からない」 → 取りこぼし発生

2. **HIGH — at-most-once 配信**:
   - `q.get()` で Monitor が受信 → SSE 経由で送信 → ack 応答なし
   - Monitor が受信中に死亡すると、queue から取り出された分はロスト

3. **MEDIUM — TCP keep-alive の検出遅延**:
   - クライアント側 (curl) で TCP 接続が切れても、server 側はすぐに気付かない (デフォルト数十分)
   - `socket.SO_KEEPALIVE` 設定で OS レベル keep-alive を有効化推奨

#### 改善提案

```python
# 1) Last-Event-ID 対応: SSE generator 内で id: msg_id を必ず書込
self.wfile.write(f'id: {msg["msg_id"]}\ndata: {json.dumps(msg)}\n\n'.encode())

# 2) at-most-once → at-least-once: SQLite を Source of Truth として、
#    Monitor が `inbox_mark_read` API を叩くまで `read=0` で残す
#    SSE は「通知トリガー」と割り切り、メッセージ取得は SQLite query で agent 側が確実に取得

# 3) TCP keep-alive
import socket as _socket
class _KeepAliveServer(_ThreadedServer):
    def server_bind(self):
        super().server_bind()
        self.socket.setsockopt(_socket.SOL_SOCKET, _socket.SO_KEEPALIVE, 1)
```

3. **MEDIUM — wfile が close されない場合の socket leak**:

# (subprocess.run の bash 呼出を完全廃止して直接 Python で処理)
def _api_inbox_write(self, body):

    conn = sqlite3.connect(DB_PATH, timeout=5)
    msg_id = f"msg_{datetime.now().strftime('%Y%m%d_%H%M%S')}_{uuid.uuid4().hex[:8]}"
    conn.execute(
        'INSERT INTO inbox_messages (id, agent, from_agent, type, content, read, timestamp) VALUES (?, ?, ?, ?, ?, 0, ?)',
        (msg_id, target, from_agent, msg_type, message, datetime.now().isoformat())
    )
    conn.commit()
    conn.close()
    # in-memory queue に push
    _push_to_inbox_queue(target, {'msg_id': msg_id, 'type': msg_type, ...})
    # YAML 書込は別 thread で非同期 (取りこぼし耐性のため、まず SQLite + queue を確実に)

# 2) socket leak 対策: SSE generator を contextmanager で wrap
@contextmanager
def _sse_session(self, agent):
    try:
        yield
    finally:
        try:
            self.wfile.close()
        except Exception:
            pass
        with _SSE_CONN_LOCK:
            _SSE_CONNECTION_COUNT[agent] = max(0, _SSE_CONNECTION_COUNT.get(agent, 0) - 1)
