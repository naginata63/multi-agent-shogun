# Gemini画像生成プロンプト設計ルール

## North Star

Geminiが「描くべきもの」と「理解すべき背景情報」を混同しないプロンプト構造を徹底すること。

## 核心ルール: contents vs system_instruction

Gemini画像生成は `contents`（台本）に書いたテキストを**画像内に文字として描画しやすい**。
`system_instruction`（舞台監督メモ）に書いたテキストは**AIの理解のみに使われ、画像内に描画されにくい**。

### system_instruction に入れるもの（舞台監督メモ = 描画しない情報）

| 情報の種類 | 例 | 理由 |
|-----------|-----|------|
| キャラ性格・演出方針 | 「天然ボケ・マイペース」→「ゆったりした佇まい、急がない慌てない」 | 内面情報。テキスト描画されると崩壊 |
| situation（場面状況） | 「能力発動でドズルがぼんの元へワープ」 | 背景理解用。描画するとメタテキストが出る |
| 禁止ルール（メタ） | 「YouTubeロゴ描画禁止」「指定外キャラ描画禁止」 | ルール文は絵に不要 |
| 顔立ち維持指示 | 「リファレンス画像の顔立ちを80%維持」 | 制御パラメータ |

### contents に入れるもの（台本 = 描画する情報）

| 情報の種類 | 例 | 理由 |
|-----------|-----|------|
| シーンの視覚描写 | 「ぼんの隣にワープ着地したドズル。余裕の表情」 | 実際に描く絵の指示 |
| セリフ吹き出し | 「あーそうゆうことか」 | 画像内に文字として描画してほしい |
| スタイル指示 | 「日本の漫画・アニメ風カラーイラスト」 | 絵の画風指定 |
| リファレンス画像 | selected/の全身PNG、背景スクショ | 視覚的参照 |

## 性格の伝え方: 抽象→具体変換

**絶対禁止**: 「天然ボケ」「クール」等の抽象的性格語をcontentsに入れること。

**正しい方法**: system_instructionで具体的な仕草に翻訳する。

```
× contents: 「天然ボケな女の子が走っている」
○ system_instruction: 「常にゆったりした佇まい。急がない、慌てない。笑顔は穏やかでのんびり」
○ contents: 「森をのんびり走っている。髪が少し乱れている」
```

## チェックリスト（新しいpanels JSONフィールド追加時）

新フィールドをpanels JSONに追加する際:

1. **そのテキストは画像内に文字として表示したいか？**
   - YES → contents に入れる（例: セリフ、テロップ）
   - NO → system_instruction に入れる（例: 状況説明、性格、ルール）
2. **抽象的な概念か、具体的な視覚描写か？**
   - 抽象（「天然ボケ」「能力発動」）→ system_instruction
   - 具体（「口を大きく開けて笑う」「ワープ着地」）→ contents
3. **Geminiがそのまま描画しても問題ないか？**
   - 問題ある → system_instruction
   - 問題ない → contents

## scene_descの書き方ルール

scene_descには**純粋な視覚描写のみ**書け。以下はcontentsに入るため、演出メタ語はGeminiが文字として描画する。

| NG（演出メタ語） | OK（視覚描写） |
|-----------------|---------------|
| 「テロップ+ドズルが飛ばされる絵」 | 「ドズルが光に包まれて宙に浮き、飛んでいく」 |
| 「同パネル別コマ漫画技法」 | 「上コマ: MENがニヤニヤ。下コマ: ドズルが顔を赤くする」 |
| 「最大パネル。落差が笑いの核」 | 「大きなコマ。おんりーが威圧的に睨みつける」 |
| 「オチ。苦笑い」 | 「肩をすくめて苦笑い。呆れた表情」 |
| 「P1と同デザインで対比」 | 「panel_01と同じ構図」 |

**禁止ワード**: テロップ、パネル技法、漫画技法、最大パネル、オチ、対比、落差、演出

## 過去の事故

- **situation文字描画問題（cmd_1029）**: `situation`フィールドがcontentsに入っていたため、「能力発動でドズルがぼんの元へワープ」等のメタテキストが画像内に文字として描画された。system_instructionに移動して解決。
