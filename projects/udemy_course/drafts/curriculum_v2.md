## 0. マーケティング基盤

### 0.1 ペルソナ定義 (5タイプ)

本講座の主ターゲット層を5つのペルソナで定義する。

#### P1: 中堅バックエンドエンジニア「田中」
- **年代**: 32歳
- **職種**: Web系バックエンドエンジニア (Ruby/Python)
- **業界**: SaaSスタートアップ (従業員30名)
- **年収**: 650万円
- **具体困りごと**: 自社プロダクトにClaude Codeを導入したが、プロンプトが長すぎてコンテキスト爆発を起こす。何度も同じ失敗をAIが繰り返す。社内の他エンジニアとAI利用ナレッジを共有する仕組みがない。
- **購買動機**: 「AI開発の生産性を上げたい」ではなく「AIの暴走を制御する仕組みを作りたい」。プロンプトの書き方ではなく、システムとしてAIを組み込みたい。

#### P2: フリーランスAI活用コンサルタント「佐藤」
- **年代**: 28歳
- **職種**: フリーランス / AI導入支援
- **業界**: 中小企業DX支援
- **年収**: 480万円 (不安定)
- **具体困りごと**: クライアント先でClaude Codeを導入する案件が増えているが、導入後の「運用設計」ができず、1ヶ月で使われなくなる。差別化できるスキルセットが欲しい。
- **購買動機**: 「ハーネスエンジニアリング」という固有スキルで競合コンサルと差別化。クライアントに「導入後も自分で改善できる仕組み」を納品したい。

#### P3: 情報発信者・テックブロガー「林」
- **年代**: 35歳
- **職種**: テック系YouTuber / Zenn Qiita同人
- **業界**: 個人メディア運営
- **年収**: 420万円 (広告+note+YouTube)
- **具体困りごと**: Claude Codeで動画パイプラインを自動化しているが、セッションを跨ぐたびに設定をやり直している。毎回同じようなプロンプトを書いている時間が無駄。
- **購買動機**: コンテキストの永続化で「設定ゼロの作業再開」を実現したい。自身の情報発信ネタとしても「3階層統合」は独自性が高い。

#### P4: SIerのプロジェクトマネージャー「木村」
- **年代**: 40歳
- **職種**: プロジェクトマネージャー (元SE)
- **業界**: 中堅SIer (従業員500名)
- **年収**: 800万円
- **具体困りごと**: 顧客からのAI開発案件が急増しているが、社内の若手エンジニアが「プロンプトを頑張る」止まりで、本番運用で事故る。AI開発の品質管理フレームワークが欲しい。
- **購買動機**: AI開発品質を「個人のプロンプトスキル」から「組織のシステム」に引き上げたい。done_gate や silent_fail_watcher のような品質ハーネスを社内標準にしたい。

#### P5: AIスタートアップCTO「陳」
- **年代**: 30歳
- **職種**: CTO兼フルスタックエンジニア
- **業界**: AIエージェント開発スタートアップ (従業員8名・シリーズA)
- **年収**: 600万円 (+ストックオプション)
- **具体困りごと**: 自社プロダクトでマルチエージェント構成を採用したが、エージェント間のコンテキスト同期が壊れやすい。デバッグに毎日2-3時間取られている。
- **購買動機**: エージェント間通信の設計パターンとハーネスによる自動検知で、デバッグ時間を半分以下に削減したい。本講座の9体運用事例は自社の参考になる。

```
AI開発で、こんな経験ありませんか？

☑ 複数のAIエージェントを使いたいが、連携の仕方が分からない

その原因は「プロンプト」ではありません。
3つの階層（プロンプト / コンテキスト / ハーネス）を
統合設計することで、AI開発は予測可能で再現性のあるものになります。

本講座は、9体のAIエージェントを並列実運用する著者が、
18本の実コードと共に教える「AI開発のシステム化」実践ガイドです。
```

```
本講座だけにある5つの要素:

1. 「3階層統合」は市場唯一（2026年4月・Udemy日本語圏60コース調査で確認）
```

## 2. 章別構成 (ハンズオン具体化 + ストーリー設計)

### 序章: AI開発 3階層モデルとは (L0: Remember + Understand)

**所要**: 30 min / **無料プレビュー**

#### ストーリー (Emotional Hook)
> 「あなたが今やっているAI開発は、『壁に絵を描いては消す』を繰り返している状態かもしれません。本講座は、その壁に『窓』を開け、さらに『配管』まで通す方法を教えます。」

- 受講者の「これだ！」ポイント: 序章の3分タイムラプスで「before: 無秩序なプロンプト → after: 3層システム稼働」を見せる。このギャップが購買決断のトリガー。

#### 学習目標
- 3階層の定義を自分の言葉で説明できる
- 単独層では解決しない理由を切り分けられる
- 本講座の終着点 (L6 Create) を把握する

#### アウトライン
1. 3階層モデル図 (1枚スライド・全講座通貫)
2. 各層の failure mode 実例 3本ずつ
3. 既存ベストセラー講座マップと本講座の位置づけ
4. 最終成果物ツアー (受講後のリポジトリを3分で見せる)

### 第1章 プロンプトを「関数」として設計する

**所要**: 45 min / Bloom: L3 Apply / **無料プレビュー**

#### ストーリー (Emotional Hook)
> 「『AIに伝わればいい』と思っていませんか？ 伝わるプロンプトと、**再利用できる**プロンプトは別物です。この章では、1回書いて終わる指示を、何度でも使える『関数』に変えます。」

- 受講者の「これだ！」ポイント: 自分の日常業務タスクが「口頭指示 → 関数プロンプト」に変わる瞬間。before/after をその場で体験。

#### ハンズオン具体化
- **何を作る**: `prompts/task_XX_v1.md` (業務タスク1件の関数プロンプト)
- **どう作る**: (1) 日常の口頭指示を1つ選ぶ → (2) 入力/処理/出力の3パートに分解 → (3) XMLテンプレートに当てはめる → (4) Claude Codeで実行して結果を確認
- **完成品見た目**: 30-50行の構造化マークダウン。`<input>`/`<task>`/`<output_format>`タグ付き。
- **所要時間**: 演習15min + 解説30min

#### 成果物
- `prompts/task_XX_v1.md`

### 第2章 構造化プロンプトと失敗パターン

**所要**: 45 min / Bloom: L4 Analyze

#### ストーリー (Emotional Hook)
> 「プロンプトは『上手く書く』だけでは足りません。**どこが壊れているかを診断する力** が必要です。医者が病気を見分けるように、あなたはプロンプトの病気を見分けられるようになります。」

- 受講者の「これだ！」ポイント: 自分が書いたプロンプトを診断チェックリストに当てはめ、「あ、ここが原因だった」と発見する瞬間。

#### ハンズオン具体化
- **何を作る**: `prompts/diagnoses/case_1-3.md` (3件の失敗プロンプト診断レポート)
- **どう作る**: (1) 用意されたfailureプロンプト3本を読む → (2) 診断checklist(10項目)を適用 → (3) 修正案を作成 → (4) 再実行して改善を確認
- **完成品見た目**: 各ケース「症状 → 原因 → 修正 → 結果」の4セクション構成
- **所要時間**: 演習15min + 解説30min

#### 成果物
- `prompts/diagnoses/case_1-3.md`

- **完成品見た目**: CLAUDE.md 30行以内 + instructions/role.md + instructions/project.md の構成
- **所要時間**: 演習20min + 解説25min
