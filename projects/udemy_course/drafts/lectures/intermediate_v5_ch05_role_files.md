## 現象 — こんな経験ありませんか?

**ケース A**: 営業の AI に「メール返信」「提案書作成」「競合分析」を全部 `CLAUDE.md` に書いている。指示が長すぎてメールのトーン指定が無視される

**ケース B**: PM の AI に「タスク分解して」と頼んだのに、勝手に進捗確認まで始めてしまった

**ケース C**: 100 ページの企画書を 1 つの AI にチェックさせたら、中盤の重要な指摘を見落とした (ch03 LitM)

---

## 対策の方向性 — 「全部入り」からの脱却

```
Before:  1つのAIに全部頼む → 指示が膨大 → 抜け漏れ + LitM発生

After:   役割別にファイルを分ける + 長文は別AIに分担
         → 指示が短く明確 → LitMも起きない
```

**核心**: 1 つの AI に「何でも屋」をやらせない。役割を分けて、それぞれに最小限の指示を渡す

# 柱①: 3 段重ねの役割別ファイル
## 「全部入り」をやめて、役割ごとに整理する

---

## 「全部入り」の罠

よくある `CLAUDE.md`:

```markdown
# 私のAIルール

## 基本方針
- 丁寧な言葉遣い
- 日本語で回答

## メール返信のルール
- 宛先は CC に営業部長を入れる
- 署名は自社テンプレートを使う
- 返信は24時間以内

## 提案書作成のルール
- 競合比較表を必ず含める
- 価格は標準プランから提示
- 図表は3枚以上

## データ分析のルール
- 欠損値は除外してから分析
- 可視化は棒グラフと折れ線を併用
- 統計的有意差も確認する

... (まだ 60 行続く)
```

**問題**: ch03 で学んだ通り、100 行の指示は後半が読み飛ばされる

---

## なぜ「全部入り」が失敗するのか?

| 問題 | 原因 |
|------|------|
| 指示の最後のルールが無視される | ch03 Lost in the Middle — 長い指示の中央〜末尾が読み飛ばされる |
| 無関係なルールが混在 | メール返信時にデータ分析ルールまで読む無駄 |
| 役割が曖昧 | 「この AI は何担当?」が不明確 |
| 更新が怖い | 100 行の中に重要なルールが埋もれていて、編集すると壊れそう |

**ch04 の「長文を作らない」設計思想と同じ問題** — 指示ファイルが長文になっている

---

## 解決策: 3 段重ねに分ける

```
┌────────────────────────────────────────┐
│  第1層: CLAUDE.md (常駐ルール)          │
│  全役割で共通する最低限のルール          │
│  例: 「日本語で回答」「丁寧な言葉遣い」  │
│  → 常に読み込まれる (短く保つ)          │
├────────────────────────────────────────┤
│  第2層: 役割別ファイル                   │
│  各役割に固有のルール                    │
│  例: instructions/email.md (メール担当)  │
│       instructions/proposal.md (提案担当)│
│  → 必要な役割のファイルだけ読み込む      │
├────────────────────────────────────────┤
│  第3層: 個別指示 (タスク)               │
│  その都度の具体的な指示                  │
│  例: 「このメールに返信して」            │
│  → そのタスク限定の指示                  │
└────────────────────────────────────────┘
```

---

## 分類の基準 — 変更頻度 × 適用範囲

| | 全役割で共通 | 特定の役割のみ |
|---|---|---|
| **変更頻度: 低** | CLAUDE.md (第1層) | 役割別ファイル (第2層) |
| **変更頻度: 高** | (CLAUDE.md に入れない) | 個別指示 (第3層) |

**コツ**:
- CLAUDE.md は **30 行以内** に保つ (= ch03 配置技術の実践)
- 役割別ファイルは **その役割の AI だけが読む** (= ch04 必要時読込の実践)
- 個別指示は **使い捨て** で書く (常駐ルールに混ぜない)

---

## Before / After 実例

**Before** (全部入り 100 行):
```
CLAUDE.md (100行)
├── 基本方針 (5行)
├── メールルール (20行)
├── 提案書ルール (25行)
├── データ分析ルール (30行)
└── その他 (20行)
```

**After** (3 段重ね):
```
CLAUDE.md (15行) — 基本方針だけ
instructions/email.md (20行) — メール担当のルール
instructions/proposal.md (25行) — 提案担当のルール
instructions/analysis.md (30行) — 分析担当のルール

個別指示: 「このメールに返信して」(その都度)
```

---

## 役割別ファイルはどう読み込まれるのか?

**仕組み**:

```
CLAUDE.md → AI 起動時に自動で全て読み込まれる (常時有効)

instructions/ のファイル:
  → ユーザーが「〇〇の役割でやって」と指示すると
  → AI が該当する役割のファイルを自動で探して読み込む

個別指示:
  → その都度プロンプトに直接書く
```

**ポイント**: ファイルを置けば AI が勝手に読むわけではない。ユーザーの指示に応じて AI が適切なファイルを選んで読み込む。

---

## 3 段重ねが機能する理由

```
メール返信を頼む:
  1. CLAUDE.md → 「日本語で回答」「丁寧な言葉遣い」(常に読む)
  2. instructions/email.md → 「CC に営業部長を入れる」「署名テンプレ使用」(メール担当のみ)
  3. 個別指示 → 「この顧客メールに返信して」(今回のみ)

提案書作成を頼む:
  1. CLAUDE.md → 「日本語で回答」「丁寧な言葉遣い」(常に読む)
  2. instructions/proposal.md → 「競合比較表を含める」「図表3枚以上」(提案担当のみ)
  3. 個別指示 → 「A社向けの提案書を作成して」(今回のみ)
```

**無関係なルールが混入しない** → 指示が短く明確 → 抜け漏れ激減

# 柱②: サブエージェントで長文を物理分割
## ch03 LitM のもう一つの根本対策

---

## 長文レビューの問題

**シナリオ**: 100 ページの企画書を AI にチェックしてほしい。

```
あなた: 「この 100 ページの企画書をレビューして」

AI:  (100 ページを読み込む)
     → ch03 LitM が発生
     → 中盤の重要なリスク指摘を見落とす
     → 「問題ありません」と回答 ← 嘘
```

**ch04 の chunk + RAG も一つの解決策**ですが、別のアプローチがあります。

---

## サブエージェント = 「別の AI に丸投げ」

<div class="vocab-box">

### 用語: サブエージェント

**メインの AI** から **別の AI** に作業を依頼する仕組み。

ざっくりいうと: 「AI が別の AI に『この部分お願い』と仕事を振る」

</div>

```
メインAI (あなたが話しているAI)
  │
  ├──「企画書のレビュー、100ページあるから別AIにお願いして」
  │
  └──→ サブエージェント (別のAI)
         │
         ├── 100ページを独立して読む
         ├── レビュー結果をまとめる
         │
         └──→ メインAIに報告: 「リスク指摘が3件見つかりました」
```

---

## なぜ LitM が防げるのか?

| | メイン AI が全部やる | サブエージェントに分担 |
|---|---|---|
| **メインの文脈** | 100 ページの企画書 + 会話履歴 | サマリだけ (3-5 行) |
| **LitM リスク** | 高い (長文が混入) | なし (メインは短いまま) |
| **レビュー品質** | 中盤が読み飛ばされる | サブエージェントは 100 ページだけに集中 |

**核心**: 長文をメインの文脈に**持ち込まない**。別の AI に読んでもらって、結果だけを受け取る。

---

## サブエージェントが効く場面

| 場面 | 何を分担するか | 効果 |
|------|--------------|------|
| 企画書レビュー | 100 ページの企画書全体 | メインは結果だけ受け取る |
| 複数資料の調査 | 10 個の PDF レポート | 各資料の要約だけ受け取る |
| 大規模な文書作成 | 対象ファイル 20 個 | 変更提案だけ受け取る |
| テスト一括確認 | 100 個のテストケース | 結果サマリだけ受け取る |

**共通パターン**: 「長文を読む作業」を別 AI に丸投げ → メインは短い結果だけ処理

---

## ch04 との使い分け

| アプローチ | 手法 | 適している場面 |
|-----------|------|--------------|
| **ch04: chunk + RAG** | 長文を小片に分けて検索 | 「特定の箇所だけ知りたい」場合 |
| **ch05: サブエージェント** | 長文を別 AI に丸投げ | 「全体をちゃんと見てほしい」場合 |
| **組み合わせ** | chunk → サブエージェント | まず chunk で関連部分を特定 → その部分の深掘りを別 AI に依頼 |

**どちらも ch03 LitM の根本対策** — 「長文をメインの文脈に持ち込まない」という共通の哲学

---

## 実践: サブエージェントの使い方

Claude Code では、ユーザーが「長文のレビューをして」と頼むと、AI が自動的にサブエージェントを使って分担します。

```
あなた: 「この 100 ページの企画書をレビューして」

AI:
1. サブエージェントを起動
2. サブエージェントに企画書全体を渡す
3. サブエージェントが独立してレビュー
4. レビュー結果を受け取る
5. あなたに報告: 「リスク指摘が 3 件見つかりました」
```

**メイン AI の文脈には、レビュー結果のサマリだけが残る** → LitM 発生なし

---

## サブエージェントの注意点

> **⚠️ サブエージェントは万能ではありません**

- **指示は明確に**: 「レビューして」より「セキュリティ上の問題がないか確認して」
- **結果は検証する**: サブエージェントも AI。ch06 で学ぶ第三者レビューで検証を
- **使いどころを見極める**: 短い文章ならメイン AI で十分。長文 (数十ページ以上) でこそ本領発揮

---

## 実例①: 営業担当の役割分業

**現象**: 営業の AI に「顧客メール返信」「提案書作成」「競合分析」を全部 `CLAUDE.md` に書いている。指示が長すぎて、メール返信のトーン指定が無視される

**3 段重ね解決**:

```
CLAUDE.md (15行):
  - 自社製品の基本情報
  - 丁寧な言葉遣い

instructions/email.md:
  - 顧客メール返信のトーン・テンプレート
  - CC ルール・署名

instructions/proposal.md:
  - 提案書のフォーマット
  - 競合比較表の必須項目
```

各タスクで必要なファイルだけ読み込む → 無関係なルールが混入しない

**置換**: 自社の業務区分に変えて即応用

---

## 実例②: PM サポートの役割分業

**現象**: PM の AI に「タスク分解」「進捗確認」「ステークホルダー報告」を全部頼んでいる。「タスク分解して」と頼んだのに進捗確認まで始めてしまう

**3 段重ね解決**:

```
CLAUDE.md (10行):
  - プロジェクトの基本情報

instructions/task_decompose.md:
  - タスク分解のフォーマット (大→中→小に細分化)
  - 依存関係の明記ルール

instructions/progress_report.md:
  - 進捗レポートのフォーマット
  - リスク記載ルール
```

指示時にどの役割を呼ぶか明示 → 抜けも余計な動作も防げる

**置換**: 自社の PM 業務フローに変えて即応用

---

## 実例③: データ分析の役割分業 + サブエージェント

**現象**: 分析 AI に「データ前処理」「統計分析」「レポート作成」を 1 つのプロンプトで頼む。前処理で型変換ミスに気づかず、間違った統計結果をレポートに書いてしまう

**3 段重ね + サブエージェント解決**:

```
CLAUDE.md (10行):
  - データソースの基本情報

instructions/preprocess.md:
  - 前処理のルール (欠損値の扱い・型変換)

instructions/analysis.md:
  - 分析手法の選択基準

instructions/report.md:
  - レポートのフォーマット

+ サブエージェント:
  前処理結果の検証を別 AI に依頼 → 型変換ミスを検出
```

各工程の品質が独立して担保される

**置換**: 自社のデータパイプライン (抽出/変換/分析/可視化) に変えて即応用
