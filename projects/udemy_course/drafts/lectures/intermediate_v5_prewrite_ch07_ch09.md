## ch07. 自動で動かす — 4 種類の hook + Skill 連携 [ハーネス層] ★ 検索 [自動]

### [1] 対象問題

**担当する困りごと (①-⑪)**:
- ⑧ **API 変更後にテストせず壊れる** — ハーネス層欠如 (`PostToolUse` hook なし)
- ⑨ **大事なドキュメントを確認しない** — コンテキスト層 + ハーネス層欠如 (関連 doc 自動引用なし)

**3 層欠如マッピング**:
- ⑧: AI が外部 API を呼び出した直後に「本当に動いたか」を自動検証する仕組み (ハーネス層) が未構築
- ⑨: プロンプト送信時に「その業務に関連する資料」を自動で探して読み込む仕組み (ハーネス層 + コンテキスト層) が未構築

**ch04 との回収 (橋渡しスライド②)**:
ch04 で学んだ「必要時取得」は人間が手で `mem-search` を叩く手動版。ch07 では同じ検索を **hook で自動発動** させる。ch04 の技術がハーネス層に昇華される瞬間がこの章の核心。

**Skills 縦糸の中継地点 (ch01→ch07→ch11)**:
- ch01: Skill という概念の「布石」を提示 (`/command` の上位互換として紹介)
- **ch07: hook と Skill が連携する仕組み** を具体化 (hook = 引き金 / Skill = 業務知識セット)
- ch11: 自分専用の Skill を作る (到達点)

### [2] 持ち帰り

**言える 1 文**:
「hook で AI の動作の前後に処理を差し込めば、テスト忘れも資料確認忘れも防げる。さらに Skill と連携させれば、AI が自律的に必要な道具を呼び出す」

**できる 1 動作**:
`UserPromptSubmit` hook を 1 つ作って、プロンプト送信時に自動で関連ドキュメントを grep する仕組みを動かせる

### [3] ストーリー (現象 → 対策)

**現象 (導入)**:
「ファイルを編集したあと、毎回手でテストを走らせるのを忘れて、後で本番でエラーに気づく」「prompt を送るたびに関連する仕様書をコピペしている。面倒だから貼らなくなって、AI が古い仕様でコードを書いて壊れた」「ch04 で `mem-search` を覚えたけど、1 日に 30 回も手で叩くのは無理」

**対策 (展開)**:
- **Step 1 — 橋渡しスライド②で ch04 を回収**: ch04 の検索コマンドを「人が叩かなくても勝手に動く」に昇華する概念提示
- **Step 2 — hook の概念導入**: 防犯カメラの動体検知センサーのように、AI の特定タイミングで自動発動するプログラム
- **Step 3 — 4 種類の hook 実演**:
  - `SessionStart`: セッション開始時にルールを読み込む
  - `PreToolUse`: ツール実行前に「本当に実行していいか」を検査 (禁止操作リストと連動 → ch08 への布石)
  - `PostToolUse`: ツール実行後に「本当に成功したか」を検証 (⑧ の直接解決)
  - `UserPromptSubmit`: プロンプト送信時に自動 grep → ch04 の手動検索の自動化版 (⑨ の直接解決)
- **Step 4 — Skill 連携スライド**: hook 単体は「いつでも同じ処理」だが、Skill と連携させると「文脈に応じて AI が自律判断で呼ぶ道具」に進化する → ch11 への布石完成

**物語の縦軸**:
同じ「検索技術」が ch04 (手動) → ch07 (hook 自動化) → ch11 (Skill 連携で自律化) と 3 段階で進化する。ch07 はその中継地点。

### [4] 実例 (受講者の現場に置換可能)

**例① — 営業の顧客対応 (UserPromptSubmit hook)**:
- 現象: 営業が「A 社の提案書を書いて」と AI に頼むが、A 社の過去の議事録や要望を毎回コピペしている
- hook 解決: `UserPromptSubmit` で「A 社」が含まれるプロンプトを検知 → 自動で顧客フォルダを grep → 関連議事録を context に注入
- 置換: 自社の顧客名・案件名に置き換えて即応用

**例② — PM の仕様変更確認 (PostToolUse hook)**:
- 現象: API の仕様書が更新されたのに、AI が古い仕様でコードを書いてテストもせず commit → 本番で 500 エラー
- hook 解決: `PostToolUse` でコード編集後にテストを自動実行 → 失敗すれば即座に通知
- 置換: 自社の API・テストコマンドに置き換えて即応用

**例③ — データ分析の前処理 (PreToolUse hook)**:
- 現象: 分析用 SQL を実行する前に「本番 DB じゃないか」「WHERE 句はあるか」の確認を毎回手動で行っている
- hook 解決: `PreToolUse` で SQL 実行前に安全チェック → 本番 DB への接続や WHERE なし DELETE を阻止
- 置換: 自社の DB 名・禁止パターンに置き換えて即応用

**Skill 連携の実例 (章末スライド用)**:
- hook (引き金) が「プロンプト送信時」に発火 → Skill が「この場面では議事録要約 Skill を呼ぶべきか」と自律判断 → 業務知識を自動実行

## ch08. 失敗を見逃さない仕組み [ハーネス層]

### [1] 対象問題

**担当する困りごと (①-⑪)**:
- ④ **失敗を fallback で隠す** — ハーネス層欠如 (検知機構なし)
- ⑤ **「止めろ」が AI に伝わらない** — ハーネス層欠如 (ガード機構なし)

**3 層欠如マッピング**:
- ④: `try/except` で例外を握り潰すコードや、エラー時に代替出力 (fallback) で誤魔化す AI の挙動を検知する仕組みがない。失敗が「見えない」ので直しようがない
- ⑤: 「このコマンドは使うな」「このパスは触るな」という禁止事項を AI に伝えても、実行時に強制阻止する仕組みがない (ch07 の `PreToolUse` hook が解決手段 → ch07 と ch08 の連動)

**ch07 との連動**:
ch07 で学んだ `PreToolUse` hook は「実行前に止める」技術。ch08 ではこれを **完了 Gate (段階確認)** と **禁止操作リスト** に発展させ、失敗を体系的に検知・阻止する包括的な仕組みを構築する。

### [2] 持ち帰り

**言える 1 文**:
「黙って失敗する AI を能動的に検知し、止めるべき操作を宣言的に阻止する仕組みを作れる」

**できる 1 動作**:
自分の業務に「段階確認」(各工程の終わりにチェックポイントを置く) を 1 つ仕込める

### [3] ストーリー (現象 → 対策)

**現象 (導入)**:
「AI が `try/except` で例外を握り潰して、『完了しました』と報告してきた。実際にはデータが壊れていた」「エラーが出たとき fallback で代替値を返すようにしたら、何が起きたか全く痕跡が残らない。後日同じエラーが起きたときに原因調査ができない」「『そのコマンド使うな』と指示したのに、AI が普通に実行して本番環境を壊した」

**対策 (展開)**:
- **Step 1 — 「黙って失敗」の 3 パターン分類**:
  1. 例外の握り潰し (`try/except pass`)
  2. fallback による代替値返却 (エラー隠蔽)
  3. 条件分岐の漏れ (想定外入力で無言の不正動作)
- **Step 2 — 段階確認 (完了 Gate)**:
  各工程の終わりにチェックポイントを置く設計。「データ取得 → [検証] → 前処理 → [検証] → 出力」のように、工程間に Gate を挟む。Gate で異常を検知したら即座に停止・報告
- **Step 3 — 検知の仕組み (watcher)**:
  cron や hook で定期的に異常を検知する仕組み。AI の実行ログを監視し、失敗パターン (exit code 非 0 / 出力ファイル不在 / 容量異常) を自動検出
- **Step 4 — `--dry-run` / `--no-confirm` フラグ**:
  安全に試す技術。「本番実行の前に dry-run で結果を確認」→「OK なら本番実行」の 2 段階で確実性を担保
- **Step 5 — 禁止操作リスト (D001-D008)**:
  「絶対に実行してはいけない操作」を宣言的に定義。ch07 の `PreToolUse` hook と連動して、禁止操作を AI が実行しようとした瞬間に阻止する (⑤ の直接解決)
- **Step 6 — ch07 連動の再確認**: `PreToolUse` hook が禁止操作リストの「実行時強制力」を担保する → ch07 と ch08 を繋ぐ

**物語の縦軸**:
ch07 で「自動で動かす」を学んだ直後に、その自動化が「黙って失敗」しないための仕組みを学ぶ。自動化 + 検知 が揃って初めてハーネス層が完成する。

### [4] 実例 (受講者の現場に置換可能)

**例① — 営業レポート生成 (段階確認)**:
- 現象: AI に週次レポートを生成させたら、過去データの取得に失敗したのに fallback でダミーデータを埋めて「完了しました」と報告。気づかずに上司に提出
- 解決: 「データ取得 → [Gate: 取得件数 > 0 を確認] → レポート生成 → [Gate: 前週差分が妥当か確認] → 出力」の段階確認で、異常時に即 STOP
- 置換: 自社のレポート・KPI 名前に置き換えて即応用

**例② — PM の本番デプロイ (禁止操作リスト)**:
- 現象: 「本番 DB の DROP TABLE は禁止」と指示したのに、AI が開発環境だと思って本番で実行。指示は文字で書いたが実行時の強制力がなかった
- 解決: `PreToolUse` hook で `DROP` / `TRUNCATE` / 本番 DB ホスト名を含む SQL を即座にブロック。指示ではなく「仕組み」で止める
- 置換: 自社の禁止コマンド・本番ホスト名に置き換えて即応用

**例③ — データ分析のパイプライン (dry-run + watcher)**:
- 現象: 毎朝自動で前日データを取り込むバッチが、CSV フォーマット変更に気づかず 1 週間不正データを蓄積
- 解決: dry-run で「取り込む予定の件数・フォーマット」を事前確認 → watcher で「前日比 50% 以上の変動」を異常検知 → 即座に通知
- 置換: 自社のバッチ処理・監視条件に置き換えて即応用

## ch09. 3 層を統合して業務に組み込む — 営業/PM/データ分析 [応用]

### [1] 対象問題

**担当する困りごと**:
3 層統合の応用 (特定の困りごと①-⑪ではなく、全 11 件を 3 層で体系化した後に「自分の現場でどう組み合わせるか」を解決する)

**3 層欠如マッピング**:
- この章自体は「特定の層が欠けている」のではなく、「3 層の知識はあるが、現場への適用方法が見えない」という実務ギャップを解決する
- 営業ケース: プロンプト層 (顧客メール作成) + コンテキスト層 (過去の顧客対応履歴検索)
- PM ケース: ハーネス層 (task 分解 + `/advisor`) + ハーネス層 (進捗 hook)
- データ分析ケース: ハーネス層 (役割別分業) + ハーネス層 (段階確認) + コンテキスト層 (memory)

**完全新規章 (v4 にない)**:
v4 では各技術を個別に学ぶ構成だったが、v5 の 3 層世界観を活かし「3 層を組み合わせる型」として 3 ケースを提示する統合章が新設された。

### [2] 持ち帰り

**言える 1 文**:
「自分の業務に 3 層の組み合わせを 1 つ適用する構想図を書き出せる」

**できる 1 動作**:
自分の業務 1 件について「どの層から始めるか」「どの道具を組み合わせるか」の構想図 (3 層組合せ) を書き出せる

### [3] ストーリー (現象 → 対策)

**現象 (導入)**:
「ch01-ch08 で 3 層の道具を学んだ。でも、自分の現場に戻ったら何から始めればいいか分からない」「営業・PM・データ分析で、どの層から手をつけるべきか迷う」「道具は覚えたが、自分の業務にどう組み合わせるかの『型』が見えない」

**対策 (展開)**:
- **Step 1 — 3 ケースで「組み合わせの型」を具体化**:
  - ケース① 営業: `/command` (プロンプト層) で顧客メール定型化 + 議事録要約 + 顧客メモ semantic 検索 (コンテキスト層)
  - ケース② PM サポート: task 分解 + `/advisor` レビュー (ハーネス層) + 進捗 hook (ハーネス層)
  - ケース③ データ分析: 役割別分業 (ハーネス層) + 段階確認 (ハーネス層) + memory (コンテキスト層)
- **Step 2 — 着手順序の判断基準**:
  - 「毎回同じプロンプトを書いている」→ プロンプト層から始める
  - 「過去の知識を AI が知らない」→ コンテキスト層から始める
  - 「AI に任せきりで品質が見えない」→ ハーネス層から始める
- **Step 3 — 自分の業務への当てはめ演習**:
  各受講者が自分の業務 1 件を選び、「どの層から始めるか」「どの道具を組み合わせるか」を書き出す

**物語の縦軸**:
ch00 で「3 層の地図」を描き、ch01-ch08 で各層を順に強化してきた。ch09 はその 3 本の糸を 1 つに撚る章。ch10 (逆引き辞典) と ch11 (Skill 作成) への橋渡しでもある。

### [4] 実例 (受講者の現場に置換可能)

**例① — 営業の 3 層組み合わせ**:
- プロンプト層: `/command` で「顧客提案メール作成」プロンプトを資産化 (件名・トーン・締めの一言を毎回ゼロから書かない)
- コンテキスト層: `claude-mem` で過去の顧客対応履歴を意味検索 → 前回の要望・クレームを自動で参照
- ハーネス層: `/advisor` で提案内容の第三者レビュー (念押し)
- 置換: 自社の顧客名・製品名・メールテンプレートに置き換えて即応用

**例② — PM サポートの 3 層組み合わせ**:
- プロンプト層: task を構造化して書く技術 (ch02 の診断 10 項目を適用)
- コンテキスト層: `/handoff` でセッション跨ぎの進捗引き継ぎ
- ハーネス層: `/advisor` で WBS の妥当性レビュー + `PostToolUse` hook でコード変更時のテスト自動実行
- 置換: 自社のプロジェクト管理ツール・WBS フォーマットに置き換えて即応用

**例③ — データ分析の 3 層組み合わせ**:
- プロンプト層: 分析要件を XML タグで構造化して AI に渡す (ch02 の確認指示の応用)
- コンテキスト層: `memory` で過去の分析結果・注意事項を記憶 → 同じミスを繰り返さない
- ハーネス層: 役割別分業 (データ取得担当 / 分析担当 / レポート担当) + 段階確認 (前処理後のデータ件数 Gate)
- 置換: 自社の分析対象データ・レポート形式に置き換えて即応用
