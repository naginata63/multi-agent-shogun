# プロンプトの3層構造

> AIへの指示は **3つのパート** に分けると精度が劇的に上がる

| パート | タグ | 役割 |
|--------|------|------|
| **入力** | `<input>` | AIに与えるデータ・素材 |
| **処理** | `<task>` | AIにやってほしいこと |
| **出力** | `<output_format>` | 結果をどんな形で返すか |

**なぜ分けるのか**: AIは「どこまでがデータで、何をしてほしいのか」を混同しやすい。タグで区切ると誤解が減る。

<!--
プロンプトの基本は3層構造です。入力タグには素材を、処理タグにはやってほしいことを、出力タグには結果の形式を書きます。人間なら「これを使って、ああして、こう返して」と文脈で分かりますが、AIは境界が曖昧だと混乱します。XMLタグで明確に区切ることで、AIの誤解を防ぎます。
-->

---

# Before: 非構造化プロンプト

```
以下の文章を要約して。日本語で3行で。

Quantum computing is a rapidly emerging technology
that leverages the principles of quantum mechanics...
（長い英語の記事本文）
```

**問題点**:
- 「以下の文章」の範囲が曖昧
- 要約の基準（文字数? 行数?）が不明確
- 出力形式の指定が雑

<!--
まずは悪い例を見てください。これでもAIは要約を出力しますが、「3行」の解釈が毎回ブレます。3文の場合もあれば、3段落の場合もある。文章の範囲も、指示部分と記事本文の境界が曖昧です。これを構造化してみましょう。
-->

---

# After: 構造化プロンプト

```xml
<input>
Quantum computing is a rapidly emerging
technology that leverages...
（記事全文）
</input>

<task>
input内の英語記事を日本語で要約せよ。
専門用語は可能な限り平易な表現に置き換えること。
</task>

<output_format>
3つの箇条書しで出力。各項目は1文・50字以内。
</output_format>
```

**改善点**: 入力・処理・出力の **境界が明確**

<!--
構造化したプロンプトです。inputタグに記事本文、taskタグに要約の指示、output_formatタグに出力形式を書きました。境界が明確なので、AIは「どこまでが記事本文で、何をしてほしいのか」を誤解しません。さらに、この形式なら同じプロンプトのinputだけ差し替えて別の記事に使い回せます。
-->

---

# XMLタグ解説 — `<input>`

**役割**: AIに処理させる **データ・素材** を入れる

```xml
<input>
（ここにデータを入れる）
</input>
```

**ポイント**:
- 生データをそのまま入れる（指示は書かない）
- データの種類: テキスト / コード / ログ / 表 など
- 長いデータもそのまま入れてよい（AIは長文が得意）

<!--
inputタグには、AIに処理してほしいデータをそのまま入れます。指示は書きません。ここは「素材」を置く場所。テキストでもコードでもログでも何でもOKです。長くても構いません。Claudeは長文処理が得意なので、数千行のログをそのまま投げても問題ありません。
-->

---

# XMLタグ解説 — `<task>`

**役割**: AIに **やってほしいこと** を具体的に書く

```xml
<task>
input内のコードから、セキュリティ脆弱性を
3つ特定せよ。
各脆弱性について、深刻度（高/中/低）と
修正方法を併記すること。
</task>
```

**ポイント**:
- 動詞で始める（「特定せよ」「変換せよ」「比較せよ」）
- 条件・制約を明示する
- 曖昧な言葉を避ける（「いい感じに」は禁止）

<!--
taskタグには、AIにやってほしいことを書きます。コツは動詞で始めること。「〜してください」より「〜せよ」「〜しろ」という直接的な表現の方がAIには伝わりやすい。あと「いい感じに」とか「適当に」は絶対にNGです。AIは親切なので適当に解釈してくれますが、それが毎回同じ解釈になる保証はありません。
-->

---

# XMLタグ解説 — `<output_format>`

**役割**: 結果を **どんな形** で返すか指定する

```xml
<output_format>
Markdown表形式で出力。
列: 番号 / 脆弱性名 / 深刻度 / 修正方法
深刻度高のものから上位に並べること。
</output_format>
```

**ポイント**:
- 形式: 箇条書し / 表 / JSON / コード など
- 並び順・件数も指定できる
- **ここを省略すると出力が毎回バラバラになる**

<!--
出力形式の指定は再現性の鍵です。「表にして」「JSONで」と指定するだけで、結果が安定します。並び順や件数も指定可能。ここを省略すると、同じプロンプトでも1回目は箇条書き、2回目は段落、3回目は表とバラバラになります。再利用するプロンプトほど、ここを丁寧に書くことが重要です。
-->

---

# Before / After — 出力の違い

**Before（非構造化）**: 毎回違う形式で出力
```
1回目: 段落形式で要約
2回目: 箇条書きで要約
3回目: 表形式で要約
```

**After（構造化）**: 毎回同じ形式で安定出力
```
1回目: 箇条書き3項目 ✓
2回目: 箇条書き3項目 ✓
3回目: 箇条書き3項目 ✓
```

> **再現性** = 同じプロンプトで同じ品質の結果を得ること

<!--
BeforeとAfterの違いを可視化しました。非構造化プロンプトは、実行するたびに出力形式がブレます。段落だったり箇条書きだったり表だったり。構造化プロンプトは、output_formatで指定しているので毎回同じ形式。これが「再現性」です。実務では、プロンプトをチームで共有して使い回す場面が多くあります。その時に再現性がなければ意味がありません。
-->

---

# 再現性が重要な理由

**プロンプトは「1回限り」ではない**:

- **使い回し**: 同じ処理を別のデータに適用
- **共有**: チームメンバーにプロンプトを渡す
- **自動化**: スクリプトの中で何度も実行

> **再現性のないプロンプト = 使い捨て**
> **再現性のあるプロンプト = 資産**

<!--
再現性がなぜ重要か。3つの理由があります。まず使い回し。inputのデータだけ差し替えて同じ処理を繰り返す場面は多いです。次に共有。チームメンバーに「このプロンプトを使って」と渡す時に、相手が実行しても同じ結果が出なければ困ります。そして自動化。スクリプトの中でプロンプトを何百回と実行する時に、結果がブレると自動化の意味がありません。再現性のあるプロンプトは、あなたのチームの資産になります。
-->

---

# よくある失敗パターン 3選

| # | 失敗パターン | 例 | 修正方法 |
|---|-------------|-----|---------|
| 1 | **曖昧な指示** | 「いい感じに要約して」 | 具体的な条件をtaskに書く |
| 2 | **出力指定なし** | taskだけでformat未指定 | output_formatを必ず書く |
| 3 | **コンテキスト不足** | inputに必要な情報が欠落 | 背景情報をinputに追加 |

> この3つを避けるだけで、プロンプトの品質が **劇的に向上** する

<!--
プロンプトで一番よくある失敗を3つ紹介します。1つ目は曖昧な指示。「いい感じに」「適当に」はAIには伝わりません。2つ目は出力指定なし。これが一番多い失敗で、output_formatを省略すると結果が不安定になります。3つ目はコンテキスト不足。AIはあなたの文脈を知らないので、必要な背景情報はinputに入れてください。この3つを意識するだけで、プロンプトの品質が飛躍的に向上します。
-->

# 📚 まとめ — プロンプト3層構造の復習

```
<input>        → 素材・データ
<task>         → やってほしいこと
<output_format>→ 結果の形式
```

**この章で学んだこと**:
1. XMLタグで指示を3パートに分割する
2. 各タグの役割（入力/処理/出力）を明確にする
3. output_formatで再現性を担保する

<!--
まとめです。プロンプトはinput、task、output_formatの3層構造。素材と指示と形式を分けることで、AIの誤解を防ぎ、再現性を確保します。この3層構造は、この後の章でもずっと使います。CLAUDE.mdを書く時も、hookを設計する時も、基本は「AIに確実に伝える」こと。その基礎がこの3層構造です。
