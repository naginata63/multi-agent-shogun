# 「同じ失敗を繰り返している」 — 3層で解決する

> 「タスクを起票したのに、過去に同じ失敗をしたことを検索し忘れて、また同じ罠に落ちた」
> — この現象は、**3つの層がバラバラ** だから起きる。

**各層の役割のおさらい**:
- **L1（プロンプト層）**: AIに「何をしてほしいか」を指示する
- **L2（コンテキスト層）**: AIに「事前に知っておくべき情報」を渡す
- **L3（ハーネス層）**: AIの動きを「自動で制御する仕組み」を作る

<!--
つかみです。AIにタスクを依頼したのに、過去に同じ失敗をしたことを検索し忘れて、また同じ罠に落ちた経験はありませんか。これは3つの層がバラバラに動いているから起きる現象です。各層の役割を確認しましょう。L1は指示、L2は事前情報、L3は自動制御です。
-->

---

## なぜ1つの層だけではダメなのか

役割を確認した上で、それぞれ「単独だと何が足りないか」を見てみましょう:

- **L1だけ**: 「過去事例を検索して」と毎回指示する → **人間が忘れる**
- **L2だけ**: 過去事例は記録されている → **検索クエリを自動生成しない**
- **L3だけ**: hookは発火する → **何を検索すべきか分からない**

> **3層統合** だけが、この問題を解決する。

<!--
各層の役割を踏まえた上で、単独だと何が足りないかを確認します。L1だけでは人間が指示を忘れます。L2だけでは記録はあっても検索しません。L3だけではhookが発火しても検索内容が分かりません。3つの層が連携して初めて解決します。
-->

---

# 解決の全体像 — hookスクリプト5ファイル構成

**5つのファイルが連携する3層統合hook**:

| ファイル | 担当層 | 役割 |
|---------|--------|------|
| `cmd_intake_hook.sh` | **L3** | hook本体・全体制御 |
| `mem_search.sh` | **L1** | クエリ生成・名詞抽出・検索実行 |
| `auto_record.sh` | **L2** | 4系統で結果を自動記録 |
| `audit_log.sh` | **fallback** | 実行ログの監査証跡 |
| `session_ingest.sh` | **fallback** | セッション開始時に未処理キューを消化 |

- hook発火 → L1がクエリ生成 → L2が結果を記録 → fallbackが安全網

<!--
解決の全体像です。5つのファイルで構成されるhookスクリプトが3層を統合します。L3のhook本体が全体を制御し、L1がクエリを生成し、L2が結果を記録します。さらに2つのfallbackファイルが監査証跡と未処理キューの消化を担当します。
-->

# L1層: 検索クエリの生成ルール

**mem_search.sh** — AIの「検索意図」をクエリに変換:

1. **入力**: タスク説明文のテキスト
2. **名詞抽出**: テキストからキーワードを抽出
3. **クエリ生成**: 抽出したキーワードで検索APIを呼ぶ

```
# クエリ生成の核心 (概念コード)
# extract_nouns は「テキストから名詞だけを抜き出す」関数
keywords=$(echo "$task_text" | extract_nouns)
results=$(search-api "$keywords")
```

- **ポイント**: L1（プロンプト層）が「何を検索するか」を決める
- 名詞抽出は `awk`（テキスト処理ツール）ベース — 重い依存なし

<!--
L1層の解説です。mem_search.shがタスク説明文からキーワードを抽出し、検索APIへのクエリを生成します。extract_nounsはテキストから名詞だけを抜き出す関数です。awkベースの軽量なルールで、MeCabのような重い依存はありません。L1プロンプト層の役割は「何を検索するか」を決めることです。
-->

---

# L2層: 4系統自動記録設計

**auto_record.sh** — 検索結果を永続的に保存:

| 系統 | 記録内容 | タイミング |
|------|---------|-----------|
| **系統1** | タスク起票内容 | hook発火時 |
| **系統2** | 検索結果 | 検索完了時 |
| **系統3** | 過去事例との類似度判定 | 類似事例検出時 |
| **系統4** | 実行結果の要約 | タスク完了時 |

- **重要**: 記録は API経由でL2（コンテキスト層）に自動保存される
- 4系統すべてが **自動実行** — 手動操作なし

<!--
L2層の解説です。auto_record.shが4つの系統で情報を自動記録します。系統1はタスク起票内容、系統2は検索結果、系統3は類似度判定、系統4は実行結果の要約です。すべてAPI経由でL2コンテキスト層に自動保存され、手動操作は不要です。
-->

---

# L3層: hookの登録と発火仕組み

**cmd_intake_hook.sh** — hookを設定ファイルに登録:

```
# 設定ファイル (settings.json) への登録
{
  "hooks": {
    "PreToolUse": [{
      "command": "bash cmd_intake_hook.sh",
      "matcher": "Write|Edit"
    }]
  }
}
```

- **PreToolUse**: ツールが動く**直前**に自動で走るhookの種類
- **matcher**: 正規表現で「どのツールの前に発火するか」を指定
  - `Write|Edit` = Write **または** Edit ツールの前に発火（`|`は「または」の意味）
- **発火の流れ**: ツール呼出検知 → タスク文取得 → 過去事例検索 → 結果記録 → 類似事例があれば警告

<!--
L3層の解説です。設定ファイルにhookを登録します。PreToolUseはツールが動く直前に走るhookです。matcherは正規表現で、パイプ記号は「または」を意味します。WriteまたはEditツールが呼ばれる直前にhookが発火し、自動的に過去事例を検索して警告を出します。
-->

---

# 3層連携の流れ — 6ステップの実行シーケンス

**hook発火から記録完了まで**:

1. **[L3]** Write/Edit 呼出検知 → hook発火
2. **[L1]** タスク説明文 → 名詞抽出 → クエリ生成
3. **[L1]** 検索API実行 → 過去事例を取得
4. **[L2]** 4系統で結果を自動記録
5. **[L3]** 類似事例あり → 警告表示 / なし → 通常続行
6. **[fallback]** 監査ログに実行証跡を記録

> 6ステップが **自動実行** — 開発者は何もしなくてよい

<!--
3層が連携する6ステップの実行シーケンスです。L3のhookが発火し、L1がクエリを生成して検索し、L2が結果を自動記録し、L3が警告の有無を判断し、fallbackが監査ログに証跡を残します。この6ステップがすべて自動で実行されます。開発者は何も意識する必要がありません。
-->

---

# fallback設計 3点 — 安全網の仕組み

### 1. 監査ログ (`audit_log.sh`)
- すべてのhook実行をJSON形式で記録
- 後から「何が起きたか」を追跡可能

### 2. 保留キュー (pending queue)
- 検索APIが応答できない場合、キューに積んで後で再実行
- ネットワーク障害時にデータを消失しない

### 3. セッション開始時の取り込み (`session_ingest.sh`)
- 新しいセッション開始時に未処理の保留キューを消化
- セッションを跨いでも記録漏れなし

> **fallback = 「最悪のケース」に備える設計** — 正常系だけでなく異常系も考える

<!--
3つのfallback設計の解説です。1つ目の監査ログはすべてのhook実行を記録し追跡可能にします。2つ目の保留キューはAPIが応答できない場合に後で再実行します。3つ目のセッション開始時の取り込みは、セッションをまたいでも記録漏れを防ぎます。fallbackは最悪のケースに備える設計です。
-->

---

# 設計原則 — 再利用可能なhook設計

1. **単一責任の原則**: 1ファイル = 1層の役割に分割
   - 変更容易性: L1のクエリルールだけ変更可能

2. **冪等性（べきとうせい）**: 同じ操作を何度やっても同じ結果になる性質
   - 2回hookが発火しても重複記録しない

3. **段階的縮退（graceful degradation）**: 一部が壊れても全体は止まらない
   - 検索APIが落ちてもfallbackが記録を保証

> この3原則は **どのようなhook設計にも適用** できる普遍パターン

<!--
3つの設計原則です。1つ目の単一責任は1ファイルが1層を担当することです。2つ目の冪等性（べきとうせい）は、何度実行しても同じ結果になる性質です。3つ目の段階的縮退は、一部が壊れても全体が止まらない設計です。この3原則はどんなhook設計にも適用できます。
-->

# この章のまとめ

- **3層統合** は「過去事例検索忘れ」をゼロにする — 単独層では解決不能
- **L1** (検索クエリ生成): 名詞抽出が「検索意図」を形にする
- **L2** (4系統自動記録): 検索結果を「永続的な情報」として保存する
- **L3** (hook発火): ツール呼出時に「安全装置」として自動で動く
- **fallback 3点**: 監査ログ + 保留キュー + セッション開始時取り込み で異常系をカバー
- **設計原則**: 単一責任・冪等性（べきとうせい）・段階的縮退

<!--
この章のまとめです。3層統合によって過去事例の検索忘れをゼロにできます。L1が検索意図を形にし、L2が結果を永続保存し、L3が安全装置として発火します。fallbackの3点で異常系もカバーします。設計原則はどんなhook設計にも使える普遍パターンです。
-->

- B: mem_search.sh
- C: auto_record.sh

**Q2**: 保留キュー（pending queue）はどのような問題を解決するか？
- A: 検索結果をキャッシュして高速化する
- B: 検索APIが応答できない場合に、後で再実行してデータ消失を防ぐ
- C: セッションをまたいで会話を継続する

- A: L1（プロンプト層）— 保存先のルールを指示として書く
- B: L2（コンテキスト層）— 過去の保存先を記録して参照する
- C: L3（ハーネス層）— hookとして発火させる
