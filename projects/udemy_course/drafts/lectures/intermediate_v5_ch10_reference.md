# 3層インデックス — 「どの層が足りないか」から引く

> 症状から引くだけでも便利ですが、**「どの層が欠けているか」** を特定できれば、もっと早く解決できます。

11の困りごとはすべて、**3つの層のいずれかが足りない** から起きています。

<!--
もう一つの逆引き方法です。「症状」から探すのではなく、「どの層が足りないか」から探します。
-->

---

## 3層別分類 — 全11件を層で整理

| 不足している層 | 該当する困りごと | 対策章 |
|--------------|-----------------|-------|
| **プロンプト層不足** | ①既存を無視 ②手動繰返 ③推測で突進 | ch01 / ch02 |
| **コンテキスト層不足** | ④中央が読まれない ⑤完了を忘れる | ch03 / ch04 |
| **ハーネス層不足** | ⑥言ったのに違う ⑦盲信 ⑧仕様変更で壊れる ⑩失敗を隠す ⑪止まらない | ch05 / ch06 / ch07 / ch08 |
| **複合 (要組み合わせ)** | ⑨資料未確認 (コンテキスト + ハーネスの両方) | ch04 + ch07 |

<!--
3つの層で11件を整理しました。ハーネス層が5件と最多です。「何かが変だ」と感じたら、まず「ハーネス層が足りないのでは?」と疑うのがコツです。
-->

---

## 使い方 — 現場で詰まったら

**ステップ1**: 症状を確認する
- 「AIが何かおかしい」と感じたら、具体的に何が起きているかを言葉にする

**ステップ2**: 3段早見表の「症状」列から該当行を探す
- 似た症状が複数あれば、「現象」列で見分ける

**ステップ3**: 「対策」列の道具を使う
- 該当章を見直せば、具体的な手順が載っている

**所要時間**: 慣れれば **30秒** で該当箇所にたどり着ける

<!--
実務での使い方です。3ステップで解決策にたどり着けます。最初は表を行き来するかもしれませんが、2〜3回使えば「これは〇〇層だ」と瞬時に判断できるようになります。
-->

# 現場の具体例で練習

> 3つのパターンを実際の業務で追ってみましょう。

---

## パターン1: 営業担当のケース

**症状**: AIに「先週の商談メモを要約して」と頼んだら、AIが勝手に「案件はクローズ」と出力した

**分析**:
1. AIが**確認せず推測で出力**した = 困りごと③
2. どの層? → **プロンプト層不足**
3. 対策 → ch02 の診断10項目で「確認問いかけ」を追加

**改善後のプロンプト例**:
```
先週の商談メモを要約して。
ただし、不明な点は推測せず「不明」と明記すること。
```

<!--
営業現場での例です。「案件はクローズ」とAIが勝手に判断したのは、プロンプトに「推測するな」という指示がなかったから。ch02の診断10項目の1つ「確認を求める指示があるか」に該当します。
-->

---

## パターン2: PMサポートのケース

**症状**: 仕様変更メールをAIに読ませたら、古い仕様の箇所を無視して回答した

**分析**:
1. 長文メールの**中央にある変更箇所が読み飛ばされた** = 困りごと④
2. どの層? → **コンテキスト層不足**
3. 対策 → ch03 の TL;DR 配置 + ch04 の必要時取得

**改善策**:
- メールの**先頭に「変更点サマリ」を3行で書く** (ch03)
- 過去の仕様変更履歴を記憶から検索できるようにする (ch04)

<!--
PMサポートの例です。長いメールの中ほどにある変更点が無視されたのは、Lost in the Middle現象。ch03で学んだ「重要情報は冒頭か末尾に」を適用すれば防げます。
-->

---

## パターン3: データ分析担当のケース

**症状**: 毎週同じ前処理プロンプトをコピペしている

**分析**:
1. 同じ作業を**毎回手動で繰り返している** = 困りごと②
2. どの層? → **プロンプト層不足**
3. 対策 → ch01 の `/command` → ch11 の Skill 化へ

**改善ステップ**:
1. まず `/command` でプロンプトを資産化 (ch01)
2. 慣れたらより高度な自動化へ (詳しくはch11で解説)

<!--
データ分析の例です。「毎週コピペ」はもったいない。ch01で学んだ`/command`で資産化すれば、1回の呼び出しで済みます。さらにch11でSkill化すれば、AIが文脈で自律的に呼び出すようになります。
-->

# 似た症状の見分け方

> 困りごとによっては症状が似ています。ここで間違えると「違う章を見直す」ことになります。
> 2つの見分け方を押さえましょう。

---

## 見分け方①: 「推測で突進」と「盲信して失敗」

| | ③ 推測で突進 (AIが確認せず勝手に進める) | ⑦ 盲信して失敗 (AI出力を信じ込む) |
|--|----------------------|----------------------|
| **いつ起きる?** | AIが**出力する前** | AIの**出力を見た後** |
| **誰が悪い?** | AI (指示不足) | 人 (確認不足) |
| **どの層?** | プロンプト層 | ハーネス層 |
| **対策章** | ch02 | ch06 |
| **対策** | プロンプトに確認指示を追加 | `/advisor` で第三者レビュー |

> **判断のコツ**: AIが**出力する前**に起きるなら③、**出力を見た後**に気づくなら⑦

<!--
一番間違えやすい2つの困りごとです。「AIが勝手に進めた」のか「AIの出力を自分が信じ込んだ」のか。タイミングで判別します。
-->

---

## 見分け方②: 「忘れている」と「読み飛ばしている」

| | ⑤ 完了を忘れる (過去の作業を覚えていない) | ④ 中央が読まれない (長文の真ん中が抜け落ちる) |
|--|------------------------|---------------------------|
| **症状** | 過去にやったことをAIが知らない | 長文の中ほどが無視される |
| **原因** | 記憶が引き継がれていない | 配置の問題 (Lost in the Middle) |
| **どの層?** | コンテキスト層 | コンテキスト層 |
| **対策章** | ch04 (記憶・引き継ぎ) | ch03 (配置技術) |

> **判断のコツ**: 「前回やったこと」を覚えていない → ch04 / 長文の特定箇所が無視される → ch03

<!--
もう一つの間違えやすいペアです。どちらもコンテキスト層ですが、対策が違います。「忘れている」のか「見落としている」のかで該当章が変わります。
-->

---

## 見分け方③: 「言ったのに違う」と「止まらない」

| | ⑥ 言ったのに違う (指示を部分的に無視) | ⑪ 止まらない (止めても実行が続く) |
|--|--------------------------|---------------------|
| **症状** | AIが指示を部分的に無視する | 「止めて」と言っても実行が続く |
| **原因** | 役割・指示が1人に詰め込まれている | 実行を止める仕組みがない |
| **どの層?** | ハーネス層 (役割分業) | ハーネス層 (ガード機構) |
| **対策章** | ch05 (役割別ファイル) | ch08 (禁止操作リスト) |

> **判断のコツ**: AIが**部分的に間違える** → ch05 / AIが**止まらない** → ch08
