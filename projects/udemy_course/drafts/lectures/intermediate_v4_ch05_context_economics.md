# 「AIが急にボケた」 — その原因は財布が空だった

> 「さっきまでちゃんと動いていたのに、急に前提を忘れて変な回答が返ってきた」
> — この現象の正体は、**コンテキストウィンドウの枯渇**です。

- コンテキストウィンドウ = AIが **一度に認識できる情報の上限**
- この上限を超えると、**古い情報から順に忘れていく**
- 多くの開発者が気づかないうちに上限に達している

<!--
このスライドがつかみです。AIを使っていて「急にボケた」「前提を忘れた」と感じたことはありませんか？ その原因の多くはコンテキストウィンドウの枯渇です。コンテキストウィンドウとはAIが一度に認識できる情報量の上限で、この上限を超えると古い情報から順に忘れていきます。CLIツールを使っていると特に気づきにくいのですが、裏で確実にこの現象が起きています。この章では、この「見えない財布」の仕組みを徹底的に理解します。
-->

# トークン — AIの「通貨」

- **トークン** = AIがテキストを処理する最小単位
  - 日本語: 1文字 ≒ 1〜2トークン
  - 英語: 1単語 ≒ 1〜2トークン
- **コンテキストウィンドウ** = 一度に扱えるトークン数の上限
  - 例: 200Kトークン ≒ 日本語で約10万字分

| 規模 | トークン数 | ざっくり換算 |
|------|----------|-------------|
| 小規模プロジェクト | 〜50K | 短い会話 + 数ファイル |
| 中規模プロジェクト | 50K〜150K | 長い会話 + 多数ファイル |
| 大規模プロジェクト | 150K〜200K | 全ファイル読込 + 長セッション |

<!--
トークンはAIの通貨です。AIはテキストをトークンという単位で処理します。日本語では1文字が1〜2トークン、英語では1単語が1〜2トークンに相当します。そしてコンテキストウィンドウは、一度に扱えるトークン数の上限です。例えば200Kトークンのモデルなら、日本語で約10万字分の情報を一度に認識できます。プロジェクトの規模によって消費量は大きく変わります。小規模なら50K以下でも済みますが、大規模プロジェクトで全ファイルを読み込むと200Kに近づきます。
-->

# 「見えている」範囲 — AIはどこまで覚えているか

**コンテキストウィンドウの構造 (200K)**:

| 層 | 内容 | 性質 |
|---|------|------|
| **固定層** | システム指示 + CLAUDE.md | 常時ロード |
| **作業層** | ツール結果 + ファイル読込 | タスクに応じて増減 |
| **会話層** | ユーザー発言 + AI回答 | セッション経過で増加 |
| **忘却帯** | 枠を超えた古い情報 | 自動的に認識外に |

- **重要**: 新しい情報を読み込むと、古い情報が押し出される
- 「忘れた」のではなく「見えなくなった」のが正しい理解

<!--
AIが認識できる範囲を図で表しました。コンテキストウィンドウの中に、システム指示、CLAUDE.md、会話履歴、ツール結果、ファイル読込などが詰め込まれます。AIはこの枠の中しか見えません。枠を超えた古い会話は自動的に忘却されます。「AIが忘れた」というより「AIから見えなくなった」と理解するのが正確です。新しいファイルを読み込むと、その分古い情報が押し出されるイメージです。
-->

# 消費源3カテゴリ — どこでトークンが消えているか

| カテゴリ | 内容 | 典型的な割合 |
|---------|------|-------------|
| **ファイル読込** | CLAUDE.md / instructions / ソースコード | **50〜70%** |
| **ツール結果** | Read / Grep / API応答 | **15〜30%** |
| **会話履歴** | ユーザー発言 + AI回答 | **10〜25%** |

> **発見のポイント**: 多くの場合、**ファイル読込が7割を占めている** — ここを最適化すれば効果が大きい

<!--
トークンの消費源を3つのカテゴリに分けました。1番目のファイル読込が最も多く、典型的なセッションでは50〜70%を占めます。CLAUDE.mdやinstructionsファイル、ソースコードの読込がここに含まれます。2番目のツール結果は15〜30%で、ReadコマンドやGrepの結果などです。3番目の会話履歴は10〜25%です。多くの開発者が会話履歴を気にしますが、実際にはファイル読込が圧倒的な消費源です。ここを最適化するのが最も効果的です。
-->

# 消費源マトリクス — どこを削るか見える化

| 操作 | トークン消費 | 頻度 | 削減余地 |
|------|------------|------|---------|
| `Read` 大ファイル全体 | 🔴 高 | 高 | ★★★ 分割読込で対応 |
| `Grep` 結果表示 | 🟡 中 | 高 | ★★ `-n` 行番号付きで必要行のみ |
| `Read` instructions/*.md | 🟡 中 | 毎回 | ★ 共通化で重複排除 |
| `Read` CLAUDE.md | 🟢 低〜中 | 毎回 | ★ @import で物理分離 |
| 会話の長文化 | 🟡 中 | セッション次第 | ★★ /clear でリセット |

> **即効性のある対策**: 大ファイルは `offset`/`limit` で必要部分だけ読む

<!--
消費源をマトリクス形式で見える化しました。最も影響が大きいのは大ファイルの全体読込です。数百行あるファイルをReadすると、それだけで数千トークンを消費します。対策は、offsetとlimitパラメータで必要な部分だけを読むことです。Grepも-nオプションで行番号付きにすれば、結果のトークン消費を抑えられます。instructionsファイルの共通化も効果的で、重複するセクションを1つにまとめれば、毎回のセッション開始時の消費を削減できます。
-->

# 予算配分の考え方 — プロジェクト規模別

**考え方**: コンテキストウィンドウを「予算」として扱い、配分を決める

| プロジェクト | CLAUDE.md | instructions | ソース読込 | 会話 | 余裕 |
|------------|----------|-------------|----------|------|------|
| **小規模** (1-5ファイル) | 5% | 5% | 30% | 30% | 30% |
| **中規模** (5-30ファイル) | 3% | 3% | 50% | 20% | 24% |
| **大規模** (30+ファイル) | 2% | 2% | 60% | 15% | 21% |

- **余裕は必ず20%以上確保** — ギリギリまで使うとcompactが頻発
- 大規模プロジェクトほど **ソース読込の割合を上げ、会話を短く保つ**

<!--
コンテキストウィンドウを予算として考えます。プロジェクトの規模によって配分を変える必要があります。小規模プロジェクトならソース読込が30%で十分ですが、大規模プロジェクトでは60%近くをソース読込に割く必要があります。どの規模でも共通するのは、余裕を20%以上確保することです。ギリギリまで使うと、auto-compactが頻発して作業効率が落ちます。大規模プロジェクトほど、会話を短く保ち、/clearで定期的にリセットするのが重要です。
-->

# auto-compact — 自動圧縮の仕組み

**auto-compact** = コンテキストウィンドウが閾値に達したとき、自動的に要約圧縮する機能

- **圧縮の流れ**:
  1. 閾値到達 → 古い会話履歴を要約に圧縮
  2. 要約は「要点」のみ残す — 詳細は失われる
  3. 新しい会話スペースが確保される
- **設定例**:
  ```
  # settings.json
  "contextCompressionThreshold": 0.8  # 80%到達で圧縮
  ```

> **注意**: compact後は「詳細」が失われる — 重要な情報はファイルに保存済みであることが前提

<!--
auto-compactは、コンテキストウィンドウが一定の閾値に達すると自動的に会話履歴を圧縮する機能です。古い会話を要約に変え、新しい会話のためのスペースを確保します。ただし、要約では詳細が失われます。そのため、重要な情報は会話の中に残すのではなく、ファイルに保存しておくことが前提になります。CLAUDE.mdやinstructionsに書かれた情報は圧縮の対象外なので、ここに重要なルールを書いておくことがL2コンテキスト層の基本です。
-->

# コスト爆発 事例3選

### 事例1: 「全ファイル一括読込」爆発
- **症状**: `Read` で src/ 全ファイルを読み込んだ瞬間にコンテキスト枯渇
- **対策**: 必要なファイルだけ、必要な部分だけ読む

### 事例2: 「無限デバッグループ」爆発
- **症状**: エラー→修正→再実行→エラー のループで会話履歴が膨張
- **対策**: 3回連続失敗したら /clear で履歴をリセット

### 事例3: 「巨大ログ貼り付け」爆発
- **症状**: ユーザーが数千行のログをそのまま貼り付け
- **対策**: エラー行周辺だけ抽出してからAIに渡す

<!--
コスト爆発の典型的な事例を3つ紹介します。1つ目は全ファイル一括読込です。ソースコードディレクトリの全ファイルを一度にReadすると、それだけでコンテキストが埋まってしまいます。2つ目は無限デバッグループです。エラーと修正を繰り返すうちに会話履歴が膨張し、いつの間にか上限に達します。3つ目は巨大ログの貼り付けです。数千行のログをそのままAIに渡すと、それだけで数万トークンを消費します。いずれも「必要な分だけ扱う」という原則で防げます。
-->

---

# 爆発対策 — 5つのルール

1. **分割読込**: 大ファイルは `offset` / `limit` で必要部分だけ
2. **定期的なリセット**: タスク境界で `/clear` を活用
3. **情報はファイルに保存**: 会話に残さず、CLAUDE.md や memory に書く
4. **共通ファイルの最適化**: 重複セクションを1つにまとめる
5. **監視**: セッション残量を意識する

> これらはすべて **L2 コンテキスト層** の設計に直結する考え方です。

<!--
コスト爆発を防ぐ5つのルールです。1つ目は分割読込で、大ファイルは必要な部分だけ読みます。2つ目は定期的なリセットで、タスクの区切りで/clearを使います。3つ目は情報をファイルに保存することで、会話に残さずCLAUDE.mdやmemoryに書きます。4つ目は共通ファイルの最適化で、重複するセクションを1つにまとめて読込量を削減します。5つ目は監視で、セッションの残量を意識します。これらはすべてL2コンテキスト層の設計に直結します。ファイルに何を書くか、どう読み込むかが、AI開発の効率を大きく左右するのです。
-->
