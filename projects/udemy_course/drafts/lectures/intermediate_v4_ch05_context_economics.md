# LangChain コンテキスト4戦略 — 全体像

LangChainはコンテキスト管理を **4つの戦略** に体系化している:

| 戦略 | 目的 | 対応章 |
|------|------|--------|
| **Write** | AIに正しい情報を書き込む | 本章 (第5章) |
| **Select** | 必要な情報だけを選択して取り出す | 第6章 |
| **Compress** | 情報を圧縮して容量を節約する | 第7章 |
| **Isolate** | 情報を分離して干渉を防ぐ | 第8章 |

> **この章のテーマ**: Write戦略 — AIが正しく動くために「何を」「どう」書き込むか

<!--
LangChainはAIエージェントのコンテキスト管理を4つの戦略に体系化しています。WriteはAIに情報を書き込む戦略、Selectは必要な情報だけを選択する戦略、Compressは情報を圧縮する戦略、Isolateは情報を分離する戦略です。本章ではこの中のWrite戦略を深く解説します。AIに正しい情報をどう書き込むかが、AI開発の品質を左右する第一歩だからです。
-->

# 「AIが急にボケた」 — 正体は Context Rot

> 「さっきまでちゃんと動いていたのに、急に前提を忘れて変な回答が返ってきた」

- **Context Rot** = コンテキストウィンドウ内の情報が劣化する現象
  - 情報量が増えるほど、古い情報の認識精度が低下する
  - 統計的に200Kトークンの70%到達時点で品質低下が顕著化
- コンテキストウィンドウ = AIが **一度に認識できる情報の上限**
- 多くの開発者が **気づかないうちにRotに巻き込まれている**

<!--
このスライドがつかみです。Context Rotは、コンテキストウィンドウ内の情報が劣化する現象です。会話が長くなるほど、AIは古い情報を正確に認識できなくなります。200Kトークンのモデルで70%到達時、つまり140Kトークン付近で品質低下が統計的に顕著になります。多くの開発者が「AIが急にボケた」と感じる正体はこれです。本章では、このRotを防ぎながらAIに正しい情報を書き込むWrite戦略を学びます。
-->

# トークン — AIの「通貨」とコンテキスト構造

- **トークン** = AIがテキストを処理する最小単位
  - 日本語: 1文字 ≒ 1〜2トークン / 英語: 1単語 ≒ 1〜2トークン
- **コンテキストウィンドウの層構造**:

| 層 | 内容 | 性質 |
|---|------|------|
| **固定層** | システム指示 + CLAUDE.md | 常時ロード |
| **作業層** | ツール結果 + ファイル読込 | タスクに応じて増減 |
| **会話層** | ユーザー発言 + AI回答 | セッション経過で増加 |
| **忘却帯** | 枠を超えた古い情報 | 自動的に認識外に |

> 「忘れた」のではなく「見えなくなった」 — 新しい情報が古い情報を押し出す

<!--
トークンはAIの通貨です。コンテキストウィンドウの中には4つの層があります。システム指示とCLAUDE.mdが入る固定層、ツール結果とファイル読込が入る作業層、会話履歴が入る会話層、そして枠を超えた古い情報が入る忘却帯です。新しい情報を読み込むと、古い情報が押し出されて見えなくなります。この構造を理解することがWrite戦略の出発点です。
-->

# Goldilocks Zone — 多すぎず、少なすぎず

**Goldilocks Zone** = コンテキスト量が「ちょうどいい」範囲

| 状態 | トークン使用率 | AIの振る舞い |
|------|--------------|-------------|
| **少なすぎる** | 〜30% | 文脈不足で的外れな回答 |
| **Goldilocks Zone** | 30%〜70% | 安定した高品質な出力 |
| **多すぎる** | 70%〜 | Context Rotによる品質低下 |
| **枯渇** | 90%〜 | compact強制・実質停止 |

> **Write戦略の目標**: 常にGoldilocks Zone (30-70%) を維持すること

<!--
Goldilocks Zoneは童話の「3匹のくま」に由来する概念で、多すぎず少なすぎないちょうどいい範囲を指します。コンテキスト量が30%未満だと情報不足でAIの回答品質が下がり、70%を超えるとContext Rotで品質が低下します。30%から70%の間がGoldilocks Zoneです。Write戦略の目標は、常にこの範囲を維持することです。CLAUDE.mdに何を書くか、instructionsに何を書くか、すべてこの範囲を維持するための設計になります。
-->

# 消費源3カテゴリ — どこでトークンが消えているか

| カテゴリ | 内容 | 典型的な割合 |
|---------|------|-------------|
| **ファイル読込** | CLAUDE.md / instructions / ソースコード | **50〜70%** |
| **ツール結果** | Read / Grep / API応答 | **15〜30%** |
| **会話履歴** | ユーザー発言 + AI回答 | **10〜25%** |

> **Write戦略の最重要ポイント**: ファイル読込が7割 — **ここを最適化すれば効果が最大**

<!--
トークンの消費源を3つのカテゴリに分けました。1番目のファイル読込が最も多く、典型的なセッションでは50〜70%を占めます。Write戦略で最も重要なのは、このファイル読込の最適化です。CLAUDE.mdやinstructionsに何を書き、何を書かないか。この設計がコンテキストウィンドウの効率を決定します。2番目のツール結果は15〜30%、3番目の会話履歴は10〜25%です。ファイル読込が最大のボトルネックであることを覚えておきましょう。
-->

# Write戦略の実践 — プロジェクト規模別予算配分

コンテキストウィンドウを「予算」として扱い、Writeする量を制御する:

| プロジェクト | CLAUDE.md | instructions | ソース読込 | 会話 | 余裕 |
|------------|----------|-------------|----------|------|------|
| **小規模** (1-5ファイル) | 5% | 5% | 30% | 30% | 30% |
| **中規模** (5-30ファイル) | 3% | 3% | 50% | 20% | 24% |
| **大規模** (30+ファイル) | 2% | 2% | 60% | 15% | 21% |

- **余裕は必ず20%以上確保** — ギリギリまで使うとcompactが頻発
- 大規模ほど **CLAUDE.mdは簡潔に** — 詳細はinstructionsに分離

<!--
Write戦略の実践として、コンテキストウィンドウを予算として扱い、プロジェクト規模別に配分を設計します。小規模ならCLAUDE.mdに5%割いても余裕がありますが、大規模プロジェクトでは2%に抑える必要があります。いずれの規模でも余裕を20%以上確保することが鉄則です。大規模プロジェクトほど、CLAUDE.mdは簡潔にし、詳細は役割別のinstructionsファイルに分離するのがWrite戦略の基本です。
-->

# Write戦略の実際 — 本講座の実装例

> 本講座サンプル実装

本講座の制作システムでは、Write戦略を以下のように実践している:

| 書込先 | Write内容 | トークン節約の工夫 |
|--------|----------|-----------------|
| `CLAUDE.md` | プロジェクト全体ルール (1ファイル) | 共通ルールを1箇所に集約 |
| `instructions/*.md` | 役割別ルール (役割ごとに1ファイル) | 全員読む必要のないものを分離 |
| `memory/*.md` | ユーザー判断・教訓 (必要時のみ) | 200行制限で肥大化防止 |

- **効果**: セッション開始時の固定読込を **約3,000トークン** に抑制
- 各エージェントは自分のinstructions **1ファイルだけ** を読み込む設計

<!--
本講座の制作システムでのWrite戦略の実装例を紹介します。CLAUDE.mdにプロジェクト全体の共通ルールを集約し、instructionsディレクトリに役割別のルールファイルを分離しています。これにより、各エージェントは自分に関連するinstructionsファイル1つだけを読み込めば済む設計になっています。memoryにはユーザーの判断や教訓だけを書き、200行制限で肥大化を防いでいます。結果として、セッション開始時の固定読込を約3,000トークンに抑えています。
-->

# auto-compact — 自動圧縮の仕組み

**auto-compact** = コンテキストが閾値に達したとき、自動的に要約圧縮する機能

- **圧縮の流れ**:
  1. 閾値到達 → 古い会話履歴を要約に圧縮
  2. 要約は「要点」のみ残す — 詳細は失われる
  3. 新しい会話スペースが確保される
- **設定例**:
  ```
  # settings.json
  "contextCompressionThreshold": 0.8  # 80%到達で圧縮
  ```

> **Write戦略との関係**: 重要情報はcompact対象外の **ファイルに書いておく** ことが前提

<!--
auto-compactは、コンテキストウィンドウが一定の閾値に達すると自動的に会話履歴を圧縮する機能です。compact後は詳細が失われますが、CLAUDE.mdやinstructionsに書かれた情報は圧縮対象外です。つまり、Write戦略でファイルに正しく情報を書き込んでおけば、compactが走っても重要な情報は保持されます。これが「重要な情報はファイルに書く」という原則の技術的根拠です。
-->

# コスト爆発 事例3選 — Write戦略の失敗パターン

### 事例1: 「全ファイル一括Write」爆発
- **症状**: `Read` で src/ 全ファイルを読み込んだ瞬間にコンテキスト枯渇
- **対策**: 必要なファイルだけ、`offset`/`limit` で必要部分だけ読む

### 事例2: 「無限デバッグループ」爆発
- **症状**: エラー→修正→再実行のループで会話層が膨張 → Context Rot
- **対策**: 3回連続失敗したら `/clear` でリセット

### 事例3: 「巨大ログ貼り付け」爆発
- **症状**: ユーザーが数千行のログをそのまま貼り付け
- **対策**: エラー行周辺だけ抽出してからAIに渡す

<!--
コスト爆発の典型的な事例3つです。1つ目は全ファイル一括読込です。一度に大量のファイルを読み込むと、それだけでGoldilocks Zoneを超えてしまいます。2つ目は無限デバッグループで、エラーと修正の繰り返しで会話層が膨張し、Context Rotを引き起こします。3つ目は巨大ログの貼り付けで、数千行をそのまま渡すと数万トークンを消費します。いずれもGoldilocks Zoneを維持する意識で防げます。
-->
