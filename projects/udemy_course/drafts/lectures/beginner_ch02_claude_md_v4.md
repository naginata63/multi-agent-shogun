# CLAUDE.mdとは — AIに読ませるREADME

> 🪞 「新しいプロジェクトに入った時、最初に何を読みますか？」

- 新しいプロジェクト → まず **README.md** を読む
- AIも同じ → **CLAUDE.md** を読む
- **CLAUDE.md = AIに読ませるプロジェクトの説明書**

> Claude Codeは起動時に自動的にCLAUDE.mdを読み込み、プロジェクトのルールを理解する

<!--
人間が新しいプロジェクトに入った時、まずREADME.mdを読んで「このプロジェクトは何で、どう動いているのか」を理解しますよね。Claude Codeも同じことをします。起動すると自動的にCLAUDE.mdというファイルを探して読み込みます。このファイルにプロジェクトのルールを書いておけば、AIは毎回そのルールを守って作業してくれます。これがCLAUDE.mdの基本的な役割です。
-->

# なぜCLAUDE.mdが必要なのか

**CLAUDE.mdなしの会話**:
- 「このプロジェクトはPythonです」→ 次の会話で忘れる
- 「テストはpytestで実行して」→ 毎回言わないといけない
- 「TypeScriptのstrictモードです」→ また忘れる

**CLAUDE.mdありの会話**:
- 一度書けば **毎セッション自動で読み込まれる**
- プロジェクトのルールを **AIが自動で守る**
- 同じ前提説明の **繰り返しがゼロに**

<!--
CLAUDE.mdがないと、毎回AIに「このプロジェクトはPythonです」「テストはpytestで」「TypeScriptのstrictモードです」と説明し直す必要があります。会話が長くなると、AIが前の方を忘れてしまうこともあります。CLAUDE.mdがあれば、プロジェクトのルールを一度だけ書けば、AIは起動のたびに自動で読み込み、ルールを守り続けます。この「繰り返しのゼロ」が最大のメリットです。
-->

# Before / After — 効果の実感

**Before (CLAUDE.md なし)**:
- AI「テストを書きました」 → **Jestで書いてる** (プロジェクトはpytest)
- AI「コードを修正しました」 → **2スペースインデント** (プロジェクトは4スペース)
- 毎回指示を繰り返す必要がある

**After (CLAUDE.md あり)**:
- AI「pytestでテストを書きました」 → 自動で適切なツールを選択
- AI「4スペースインデントで修正しました」 → コーディング規約を遵守
- **指示なしで正しい振る舞い**

<!--
この違いを見てください。CLAUDE.mdがないと、AIはプロジェクトの事情を知らないので、デフォルトの振る舞いをします。PythonプロジェクトなのにJestでテストを書いたり、4スペースのプロジェクトなのに2スペースでコードを書いたり。CLAUDE.mdがあれば、AIは最初からプロジェクトのルールを理解しているので、指示しなくても正しいツールと規約を使います。この違いが、CLAUDE.mdの価値です。
-->

# 5セクション構成 — 30行で書ける

CLAUDE.mdは以下の5セクションで構成するのが基本:

| # | セクション | 役割 | 例 |
|---|----------|------|-----|
| 1 | 使用言語・FW | 何を使っているか | Python 3.12 / FastAPI |
| 2 | コーディング規約 | スタイルの統一 | 4スペース / 型ヒント必須 |
| 3 | よく使うコマンド | ビルド・テスト・実行 | `pytest` / `uvicorn` |
| 4 | 禁止事項 | やってはいけないこと | `rm -rf` 禁止 / `main`にpush禁止 |
| 5 | プロジェクト構造 | ディレクトリの意味 | `src/` = 本体 / `tests/` = テスト |

> これだけで **約30行**。最初はこれだけで十分

# 30行ミニマム版の例

> 以下はPythonの例ですが、**中身が読めなくても構いません**。構造（5セクション・箇条書き）だけ見てください。ご自身の言語に置き換えて考えましょう。

```markdown
# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM: SQLAlchemy 2.0

## コーディング規約
- インデント: 4スペース
- 型ヒント必須 (mypy strict)
- docstringはGoogle形式

## よく使うコマンド
- テスト: pytest tests/
- 起動: uvicorn app.main:app --reload
- lint: ruff check .

## 禁止事項
- 本番DBの直接操作禁止
- mainブランチへの直接push禁止

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = DBマイグレーション
```

**保存の手順 (3ステップ)**:
1. プロジェクトの一番上のフォルダ（ルート）を開く
2. `CLAUDE.md` という名前でファイルを作成し、上記を貼り付けて保存
3. Claude Codeを再起動 → 自動的に読み込まれる

> 💡 **確認方法**: Claude Codeを起動して「CLAUDE.mdの内容を教えて」と聞けば、正しく読み込まれているかすぐ確認できます

# セクション詳細 — 使用言語・FW と コーディング規約

**使用言語・フレームワーク**:
- 言語のバージョンまで書く → AIが使える機能を正しく判断
- 例: 「Python 3.12」と書けば、walrus演算子（`:=` で値を代入しつつ比較できる書き方）を使えると判断

**コーディング規約**:
- インデント幅、命名規則、リンター設定を書く
- AIが生成するコードの **スタイルが統一される**
- コードレビューの指摘事項が減る

> 「4スペース」と書くだけで、AIは常に4スペースでインデントする

<!--
最初の2セクションの詳細です。言語とフレームワークは、バージョンまで含めます。「Python 3.12」と書いておけば、3.12で追加された機能を使って良いとAIが判断します。逆に「Python 3.9」と書いておけば、それ以降の機能は使いません。コーディング規約は、インデント幅や命名規則を書いておくと、AIが生成するコードのスタイルが統一されます。地味ですが、レビューでの指摘を大幅に減らします。
-->

---

# セクション詳細 — よく使うコマンド と 禁止事項

**よく使うコマンド**:
- テスト、ビルド、Lint の実行コマンドを書く
- AIがコード修正後に **テストを実行して確認** する（実行前に人間へ確認することもあります）

| プロジェクト | テストコマンド |
|------------|--------------|
| Python | `pytest tests/` |
| JavaScript | `npm test` |
| Go | `go test ./...` |

**禁止事項**:
- 危険な操作 (`rm -rf` 等) の禁止
- 本番環境への直接アクセス禁止
- AIが **実行前に止まる** 安全装置

<!--
よく使うコマンドのセクションです。テストコマンドを書いておくと、AIはコードを修正した後に自動的にテストを実行してくれます。プロジェクトによってテストコマンドは違いますが、これを書いておけばAIが間違ったコマンドを使うことがなくなります。禁止事項は安全性の観点で非常に重要です。「本番DBの直接操作禁止」「mainブランチへの直接push禁止」と書いておけば、AIはこれらの操作を実行しようとした瞬間に立ち止まります。
-->

# よくある書き方の失敗 3選

**❌ 失敗1: 長すぎる**
- 100行以上のCLAUDE.md → AIが重要な部分を見落とす
- 解決: **30行以内**に収める。詳細は別ファイルに

**❌ 失敗2: 曖昧な表現**
- 「きれいなコードを書いて」→ AIの解釈がバラバラ
- 解決: **具体的な規約**を書く (例: 「4スペース」)

**❌ 失敗3: 矛盾する指示**
- 「TypeScriptを使って」+ 「JavaScriptで書いて」→ AIが混乱
- 解決: **一貫性**を確認してから保存する

<!--
CLAUDE.mdを書く時によくある失敗を3つ紹介します。1つ目は長すぎること。100行以上書くと、AIが重要な部分を見落とす可能性があります。最初は30行以内に収めましょう。2つ目は曖昧な表現。「きれいなコード」ではAIに伝わりません。「4スペース」「型ヒント必須」のように具体的に書きましょう。3つ目は矛盾する指示。TypeScriptとJavaScriptを同時に指定するとAIが混乱します。保存前に一貫性を確認することが大切です。
-->

# CLAUDE.mdを書く時のポイント

1. **短く始める** — 最初は5セクション30行で十分
2. **具体的に書く** — 「きれいに」ではなく「4スペース」
3. **禁止事項を必ず入れる** — 安全性の最低限
4. **随時アップデート** — プロジェクトの変更点があったら追記する（30行を超えそうなら、古いルールを見直して削る）
5. **AIの振る舞いを見て調整** — 期待と違ったら追記する

> **完璧を目指さない** — 30行のCLAUDE.mdが、0行の何倍も価値がある

<!--
CLAUDE.mdを書く時の5つのポイントです。一番大事なのは「短く始める」ことです。完璧なCLAUDE.mdを最初から書こうとすると、書けません。まずは5セクション30行で始めて、実際にAIを使いながら足りない部分を追加していくのが正しいアプローチです。追加する時は、古いルールを見直して削ることで30行を維持します。30行のCLAUDE.mdが、0行の何倍も価値があります。0行のまま悩むより、30行で始めることが何より重要です。
-->
