# 「書いた時の真実」は今も真実か

> 「このファイルパスを参照せよ」と書いたメモリ。
> 3ヶ月後にそのファイルは存在するだろうか？

**現実に起こること**:
- **リネームされた** → メモリのパスが 404
- **前提が変わった** → 「Python 3.11」と書いたが今は 3.13
- **担当が変わった** → メモリの所有者がもういない

> メモリに「賞味期限」がないのは設計の欠陥。
> これが **Context Rot** の正体

<!--
エモーショナルフックです。3ヶ月前に書いたメモリがあります。「このファイルパスを参照せよ」と。しかし3ヶ月後、そのファイルはリネームされているかもしれません。前提が変わっているかもしれません。担当者が異動しているかもしれません。メモリに賞味期限がないのは設計の欠陥であり、これがContext Rotの正体です。Context Rotは、書かれた時点では正確だった情報が、時間の経過とともに現実と乖離していく現象を指します。
-->

# Context Rot とは — 情報の腐敗

**Context Rot**: メモリの内容が現実と乖離している状態。

LangChainの研究コミュニティで報告されている統計的特徴:

| 指標 | 傾向 | 根拠 |
|------|------|------|
| **ファイルパスの陳腐化率** | 30日で約15%が無効 | リネーム・移動・削除 |
| **前提の陳腐化率** | 90日で約25%が古い | バージョンアップ・方針変更 |
| **影響度** | 古い1件が全体の判断を歪める | AIは古い前提を「真実」として扱う |

**Context Rotの連鎖**:
- 古いメモリが読み込まれる → AIが古い前提で判断 → 新しい出力が古い前提に基づく → さらに古い情報が蓄積

> LangChainのCompress戦略やIsolate戦略だけでは防げない。**検出と更新** が別途必要

<!--
Context Rotの統計的特徴を解説します。LangChainの研究コミュニティの報告では、ファイルパスは30日で約15%が無効になり、前提知識は90日で約25%が古くなります。問題は、古いメモリが1件でもAIの判断を歪めることです。AIは古い前提を真実として扱い、それに基づいた出力を生成します。この連鎖を断ち切るには、Compress戦略やIsolate戦略だけでは不十分で、検出と更新の仕組みが別途必要です。
-->

# Chroma — ベクトルDBによる意味的劣化検出

> 単なる「日付チェック」を超えて、内容の意味的変化を検出する

**Chromaとは**:
- LangChainエコシステムで広く使われる **オープンソースのベクトルデータベース**
- テキストをベクトル(数値の配列)に変換して保存
- 類似度検索で「意味的に近い情報」を高速に取り出す

**Context Rot検出への応用**:

1. メモリを **ベクトル化** してChromaに保存
2. 現在のコードベースも **ベクトル化**
3. メモリのベクトルと現在のベクトルの **距離を測定**
4. 距離が大きい = メモリが現実から乖離 = **Rot検出**

> 数学的な距離で「どれくらい古くなったか」を定量化できる

<!--
ChromaはLangChainエコシステムで広く使われるオープンソースのベクトルデータベースです。テキストを数値の配列に変換して保存し、類似度検索を高速に行えます。Context Rotの検出に応用する方法を解説します。まずメモリをベクトル化してChromaに保存します。現在のコードベースもベクトル化し、両者の距離を測定します。距離が大きいほど、メモリが現実から乖離していること、つまりRotが進行していることを意味します。単なる日付チェックを超えた、意味的な劣化検出が可能です。
-->

# 劣化検出パターン — 3ルール

> LangChain / Chroma を活用した3段階の劣化判定

**Rule 1: 構造的検出 (ファイル存在確認)**
- メモリに記載されたパスが実際に存在するか検証
- リネーム・移動・削除されたファイルを検出
- **確実度: 高** — パスがなければ確実にRot

**Rule 2: 統計的検出 (日付 + カテゴリ別しきい値)**
- タイムスタンプから経過日数を計算
- カテゴリ別しきい値: API仕様30日 / 構造60日 / 方針90日
- **確実度: 中** — 古いが内容は有効かもしれない

**Rule 3: 意味的検出 (Chroma類似度)**
- メモリのベクトルと現在のコンテキストの類似度を測定
- 類似度が低い = 内容が乖離している
- **確実度: 高** — 意味レベルで変化を検出

> 3ルールの組み合わせで、確実なRot検出が可能になる

<!--
劣化検出の3ルールを解説します。Rule 1は構造的検出です。ファイルパスが存在するかを確認し、存在しなければ確実にRotと判定します。Rule 2は統計的検出です。日付ベースでカテゴリ別のしきい値を設けます。API仕様は30日、構造は60日、方針は90日。古くても内容は有効かもしれないので確実度は中です。Rule 3はChromaを使った意味的検出です。メモリのベクトルと現在のコンテキストの類似度を測定し、低ければRotと判定します。LangChainエコシステムのChromaを使うことで、意味レベルでの変化検出が可能です。
-->

# Rule 1 実例 — 構造的検出

**メモリの記述**:
```

```

**スキャン結果**:
```
[EXPIRED] memory/feedback_login.md
  → パス "src/auth/legacy_login.py" が存在しません

```

**対処フロー**:
1. `git log -- src/auth/legacy_login.py` で変更履歴を確認
2. 新パスを特定 → メモリを更新
3. または機能自体が廃止 → メモリを削除

> Rule 1は最も確実性が高い。パスが存在しない = 100% Rot

**対処**: メモリの内容を現在の実装に合わせて更新

# 古いメモリの判断基準 — 3つのアクション

| 状態 | アクション | 条件 |
|------|-----------|------|
| **Rot確定** (EXPIRED) | **更新** または **削除** | パス不存在・意味的乖離確定 |
| **Rot疑い** (STALE) | **確認** → 更新/アーカイブ/削除 | 日付超過・内容は有効かも |
| **Rotなし** (FRESH) | そのまま | 現在も有効 |

**判断フロー**:
1. EXPIRED → まず「更新で済むか」を確認
2. 更新できない → 「アーカイブ価値があるか」を判断
3. どちらでもない → 削除

**自動化の境界線**:
- **自動でできる**: パス確認・日付計算・Chroma類似度測定・レポート生成
- **人間が判断すべき**: 「この前提はまだ有効か」「更新か削除か」

> 自動は「検出」まで。判断と実行は人間が行う — LangChainの基本方針

<!--
古いメモリの判断基準です。EXPIREDは更新か削除が確定です。STALEは確認してから判断します。FRESHはそのまま。判断フローは、まず更新で済むか確認し、無理ならアーカイブ価値を検討し、どちらでもなければ削除します。自動化の境界線も重要です。パス確認や日付計算、Chroma類似度の測定、レポート生成は自動でできます。しかし「この前提はまだ有効か」「更新するか削除するか」の判断は人間が行います。自動は検出まで。判断は人間が行う。これはLangChainの基本方針でもあります。
-->

# まとめ — Context Rot との付き合い方

**この章で学んだこと**:

1. **Context Rot** は避けられない — 情報は時間とともに必ず劣化する
2. **統計的特徴**: パスは30日で15%無効、前提は90日で25%が古くなる
3. **Chroma** による意味的検出で、パス・日付では検出できない劣化を捉える
4. **3ルール** で段階的に評価: 構造的 → 統計的 → 意味的
5. **自動は検出まで** — 判断と実行は人間が行う

```
Context Rot対策の3層:
  LangChain Write戦略 → 正しい情報を書き込む (予防)
  LangChain Compress戦略 → 重要情報だけを残す (最小化)
  Context Rot検出 → 古くなった情報を見つける (検出)
```

> 「メモリは書いた時点の真実であって、今の真実とは限らない」
> だからこそ、Chromaを使った定期検出が必要

<!--
まとめです。Context Rotは避けられない問題ですが、LangChainの戦略とChromaを組み合わせることで対処できます。統計的にはパスは30日で15%が無効になり、前提は90日で25%が古くなります。Chromaによる意味的検出は、パスや日付では捉えられない劣化を見つけます。3ルールで段階的に評価し、自動は検出まで、判断は人間が行います。LangChainのWrite戦略で予防し、Compress戦略で最小化し、Context Rot検出で見つける。この3層で、コンテキストを健全に保ちます。
-->

# 確認問題

**Q1**: Context Rotの統計的特徴を、パスと前提の2つの観点から述べよ。

**Q2**: Chromaによる意味的検出が、Rule 1(パス確認)やRule 2(日付チェック)とどう違うかを説明せよ。

**Q3**: なぜ自動更新ではなく「自動検出 + 手動判断」のバランスが推奨されるのか。LangChainの基本方針と関連づけて答えよ。

**解答**:

**A1**: パスの陳腐化率は30日で約15%が無効(リネーム・移動・削除による)。前提の陳腐化率は90日で約25%が古い(バージョンアップ・方針変更による)。古い情報が1件でもAIの判断を歪める連鎖が起きる。

**A2**: Rule 1はパスの物理的存在を確認するだけで内容の変化は検出できない。Rule 2は日付ベースで内容の有効性は分からない。Chromaはテキストのベクトル類似度を測定するため、「パスは存在するが内容が古い」パターンを意味レベルで検出できる。

**A3**: 前提の有効性や更新・削除の判断には文脈理解が必要であり、自動化のリスクが高いため。LangChainの基本方針では、確実な判定(パス不存在等)は自動で行い、文脈依存の判断は人間が行うことで安全性を担保する。
