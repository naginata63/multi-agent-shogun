# 「書いた時の真実」は今も真実か — Emotional Hook

> 「このファイルパスを参照せよ」と書いたメモリ。
> 3ヶ月後にそのファイルは存在するだろうか？

**現実に起こること**:
- **リネームされた** → メモリのパスが 404
- **前提が変わった** → 「Python 3.11」と書いたが今は 3.13
- **担当が変わった** → メモリの所有者がもういない

> メモリに「賞味期限」がないのは設計の欠陥

<!--
エモーショナルフックです。3ヶ月前に書いたメモリがあります。「このファイルパスを参照せよ」と。しかし3ヶ月後、そのファイルはリネームされているかもしれません。前提が変わっているかもしれません。担当者が異動しているかもしれません。メモリに賞味期限がないのは、実は設計の欠陥なんです。この章では、その欠陥をどう補うかを学びます。
-->

# staleness とは — メモリの「賞味期限」

**staleness（陳腐化）**: メモリの内容が現実と乖離している状態

| 状態 | 意味 | アクション |
|------|------|-----------|
| **fresh** | 現在も有効 | そのまま |
| **stale** | 古い可能性あり | 要確認 |
| **expired** | 確実に無効 | 更新または削除 |

**重要**: stale ≠ 削除。確認してから判断する。

<!--
stalenessとは、メモリの内容が現実と乖離している状態です。fresh、stale、expiredの3段階で評価します。staleは「古いかもしれない」であり、即座に削除するわけではありません。確認してから判断するのが重要です。ここが単純な期限切れ削除との違いです。
-->

---

# staleness 判定3ルール

**Rule 1: ファイル存在確認**
- メモリに記載されたパスが実際に存在するか検証
- リネーム・移動・削除されたファイルを検出

**Rule 2: 日付チェック**
- メモリのタイムスタンプから経過日数を計算
- しきい値（例: 90日）を超えたら stale フラグ

**Rule 3: 手動検証フラグ**
- 自動判定が困難な内容（前提知識の変化等）
- 人間によるレビューを要求するフラグ

> 3ルールの組み合わせで「確実に無効」から「要確認」まで段階的に評価

<!--
stalenessを判定する3つのルールを解説します。1つ目はファイル存在確認。メモリに書かれたパスが実際に存在するか検証します。2つ目は日付チェック。タイムスタンプから経過日数を計算し、しきい値を超えたらstaleフラグを立てます。3つ目は手動検証フラグ。自動では判定できない内容の変化を人間に確認してもらう仕組みです。この3ルールを組み合わせることで、確実に無効なものから要確認のものまで段階的に評価できます。
-->

---

# Rule 1 実例 — ファイル存在確認

**メモリの記述**:
```
参照: src/auth/legacy_login.py
```

**スキャン結果**:
```
[EXPIRED] memory/feedback_login.md
  → パス "src/auth/legacy_login.py" が存在しません
  → リネームまたは削除の可能性
```

**対処**:
1. `git log -- src/auth/legacy_login.py` で変更履歴を確認
2. 新パスを特定 → メモリを更新
3. または機能自体が廃止 → メモリを削除

<!--
Rule 1の実例です。メモリに「src/auth/legacy_login.pyを参照」と書いてあったとします。スキャンすると、そのパスが存在しません。git logで変更履歴を確認し、新パスを特定してメモリを更新するか、機能が廃止されていればメモリを削除します。このように、ファイル存在確認は最も確実な自動判定手法です。
-->

---

# Rule 2 実例 — 日付チェック

**しきい値の設計**:

| カテゴリ | しきい値 | 理由 |
|----------|---------|------|
| API仕様 | 30日 | 外部APIは頻繁に変わる |
| プロジェクト構造 | 60日 | リファクタリング周期 |
| 開発方針 | 90日 | 方針変更は比較的まれ |

**スキャン結果の例**:
```
[STALE] memory/project_api_design.md
  → 最終更新: 2026-01-15 (107日前)
  → カテゴリ: API仕様 (しきい値: 30日)
```

> しきい値は一律ではなく「変わる速度」に合わせて設計する

<!--
Rule 2の実例です。日付チェックのしきい値は一律ではなく、内容の「変わる速度」に合わせて設計します。API仕様なら30日、プロジェクト構造なら60日、開発方針なら90日。スキャン結果の例では、API仕様のメモリが107日前に更新されており、30日のしきい値を大幅に超えています。このメモリは要確認です。
-->

# 自動スキャンの仕組み

**スキャン対象**: `memory/*.md`

**アーキテクチャ**:

1. **スキャナースクリプト** が `memory/` 配下を走査
2. 各メモリの frontmatter から `type`・`date` を読取
3. Rule 1 (パス存在確認) + Rule 2 (日付チェック) を実行
4. 結果を **stalenessレポート** として出力

**実行タイミング**:
- **週次**: cron で毎週自動実行
- **手動**: 大きなリファクタリング後など必要に応じて

<!--
自動スキャンの仕組みです。スキャナーがmemoryディレクトリ配下を走査し、各メモリのfrontmatterからタイプと日付を読み取ります。Rule 1とRule 2を自動で実行し、結果をstalenessレポートとして出力します。実行タイミングは週次のcronと、大きなリファクタリング後の手動実行です。毎日実行するとノイズが増えるので、週次が適切なバランスです。
-->

# 古いメモリの判断基準

**3つのアクション**:

| 状態 | アクション | 条件 |
|------|-----------|------|
| **更新** | 内容を最新に書換 | 情報はまだ有効・パスや前提のみ古い |
| **削除** | ファイルを削除 | 完全に無効・他メモリと重複 |
| **アーカイブ** | `memory/archive/` へ移動 | 歴史的価値はあるが現役ではない |

**判断フロー**:

1. expired → まず「更新で済むか」を確認
2. 更新できない → 「アーカイブ価値があるか」を判断
3. どちらでもない → 削除

> 削除が最終手段。まずは更新・アーカイブを検討する。

<!--
古いメモリの判断基準です。3つのアクション、更新、削除、アーカイブから選びます。更新は情報がまだ有効でパスや前提だけ古い場合。削除は完全に無効で他メモリと重複している場合。アーカイブは歴史的価値があるが現役ではない場合です。判断フローは、まず更新で済むか確認し、無理ならアーカイブ価値を検討し、どちらでもなければ削除します。削除は最終手段です。
-->

# 自動更新と手動レビューのバランス設計

**自動でできること**:
- パスの存在確認 → **100%自動**
- 日付の経過計算 → **100%自動**
- expired レポートの生成 → **100%自動**

**人間が判断すべきこと**:
- 「この前提はまだ有効か」→ **手動レビュー**
- 「このメモリを更新するか削除するか」→ **手動判断**
- しきい値の調整 → **運用フィードバック**

> 自動は「検出」まで。判断と実行は人間が行う。

<!--
自動更新と手動レビューのバランス設計です。パスの存在確認、日付の経過計算、expiredレポートの生成は100%自動でできます。しかし、「この前提はまだ有効か」「更新か削除か」の判断は人間が行うべきです。しきい値の調整も運用フィードバックに基づいて人間が行います。自動は検出まで、判断と実行は人間が行う。この境界線を明確にすることが健全な運用の鍵です。
-->

# まとめ — コンテキスト劣化との付き合い方

1. **staleness は避けられない** — 時間が経てば必ず古くなる
2. **3ルールで段階的に評価** — 存在確認 / 日付 / 手動
3. **週次スキャンで早期発見** — 放置すると雪だるま式に悪化
4. **更新・アーカイブ・削除の判断基準** を持つ
5. **自動は検出まで** — 判断は人間が行う

> 「メモリは書いた時点の真実であって、今の真実とは限らない」
> だからこそ、定期的な点検が必要

<!--
この章のまとめです。stalenessは避けられない問題ですが、3ルールで段階的に評価し、週次スキャンで早期発見できます。更新、アーカイブ、削除の判断基準を持ち、自動は検出まで、判断は人間が行うバランス設計が重要です。メモリは書いた時点の真実であって今の真実とは限らない。だからこそ、定期的な点検が必要なんです。
-->

# 確認問題

**Q1**: staleness判定の3つのルールを挙げよ。

<details><summary>回答</summary>ファイル存在確認（パスが実際に存在するか）、日付チェック（しきい値を超えたか）、手動検証フラグ（自動判定困難な内容の人間レビュー）。</details>

**Q2**: expired と stale の違いは何か。

<details><summary>回答</summary>expiredは「確実に無効」（ファイルが存在しない等）。staleは「古い可能性あり」（日付がしきい値を超えているが内容は有効かもしれない）。stale ≠ 削除、確認してから判断する。</details>

**Q3**: なぜ自動更新ではなく「自動検出 + 手動判断」のバランスが推奨されるのか。

<details><summary>回答</summary>前提の有効性や更新・削除の判断には文脈理解が必要であり、自動化のリスクが高い。パス不存在等の確実な判定は自動で行い、文脈依存の判断は人間が行うことで安全性を担保するため。</details>
