# 伝わる ≠ 再利用できる — Emotional Hook

> 「AIに伝わればいい」と思っていませんか？
> 伝わるプロンプトと、**再利用できる**プロンプトは別物です。

**伝わるプロンプト**:
- その場では上手くいく
- 別の文脈では結果がブレる
- 毎回書き直す必要がある

**再利用できるプロンプト**:
- 入力を変えるだけで結果が変わる
- 別の文脈でも同じ品質
- 1度書けば何度でも使える

> この章では、1回書けて終わる指示を、何度でも使える「関数」に変えます。

<!--

# 関数プロンプトの概念 — 3パート分解

> プログラミングの関数: `output = f(input)`
> プロンプトも同じ構造で設計できる

| パート | 役割 | プロンプトでの例 |
|--------|------|-----------------|
| **入力 (input)** | 処理の対象 | ファイル・データ・条件 |
| **処理 (task)** | やってほしいこと | 分析・変換・生成 |
| **出力 (output)** | 結果の形式 | 箇条書き・表・コード |

**なぜ関数として設計するのか**:
- **入力だけを変えれば**、別のタスクに再利用できる
- **処理と出力は固定**されるので、結果が安定する
- **テスト可能**になる — 同じ入力なら同じ結果が返る

<!--
プロンプトを関数として設計するとは、入力・処理・出力の3パートに分解することです。プログラミングの関数と同じ考え方です。入力が処理の対象で、ファイルやデータや条件にあたります。処理がやってほしいことで、分析や変換や生成です。出力が結果の形式で、箇条書きや表やコードです。なぜ関数として設計するのか。入力だけを変えれば別のタスクに再利用できるからです。処理と出力は固定されるので結果が安定します。そして同じ入力なら同じ結果が返るので、テストも可能になります。
-->

# 構造化テンプレート — input / task / output

プロンプトを3パートに明示的に分ける構造が**構造化テンプレート**です:

```markdown
[入力]
対象: @src/auth/login.ts
前提: TypeScript 5.0 / Node.js 20

[処理]
このファイルに含まれる全関数について、
セキュリティ上の問題を特定する。

[出力]
各問題について以下の形式で出力:
- 関数名 / 深刻度(高/中/低) / 問題の概要 / 修正案
```

**ポイント**:
- **角カッコ `[入力]` `[処理]` `[出力]`** で3パートを明示的に区切る
- AIは「どこまでが素材で、何をしてほしいか」を**混同しなくなる**
- 入力部分だけを差し替えれば、別ファイルにも**即座に再利用**できる

<!--
構造化テンプレートの実例です。プロンプトを入力、処理、出力の3パートに角カッコで明示的に区切ります。入力には対象ファイルと前提条件を書き、処理にはやってほしいことを書き、出力には結果の形式を書きます。このように区切ることで、AIはどこまでが素材で何をしてほしいかを混同しません。そして入力部分だけを差し替えれば、別のファイルにも即座に再利用できます。API開発者向けのバージョンではXMLタグを使う手法もありますが、Claude Codeではこのように自然言語で十分です。
-->

# Before — 口頭指示の例

> 「このコード、問題ないか見ておいて」

**この指示の問題点**:

| 観点 | 状態 | 影響 |
|------|------|------|
| **入力** | 「このコード」が何を指すか不明 | AIが勝手に推測する |
| **処理** | 「問題ないか」の範囲が無限大 | セキュリティ? パフォーマンス? スタイル? |
| **出力** | どう返せばいいか未指定 | 段落? リスト? 表? |

**結果**: 毎回**違う結果**が返る。再利用不可能。

<!--
Beforeの例です。「このコード、問題ないか見ておいて」という指示。人間なら前後の文脈で何となく分かりますが、AIには伝わりにくい。入力の「このコード」が何を指すか不明です。処理の「問題ないか」の範囲が無限大で、セキュリティの話なのかパフォーマンスの話なのかスタイルの話なのか分かりません。出力も未指定で、段落で返すのかリストなのか表なのか分からない。結果として、毎回違う結果が返り、再利用不可能です。伝わることは伝わるかもしれない。でも再利用はできません。
-->

**結果**: 毎回**同じ構造**で安定出力。入力を `@src/api/users.ts` に変えれば、そのまま再利用可能。

<!--

# 再利用性の確認 — 同じテンプレート、別タスク

**1つのテンプレートを3つのタスクに適用**:

| タスク | 入力の変更箇所 | 処理・出力 |
|--------|---------------|-----------|
| セキュリティ監査 | 対象: `@src/auth/*.ts` | 同じ |
| パフォーマンス分析 | 対象: `@src/api/*.ts` + 処理を「ボトルネック特定」に変更 | 同じ |
| コード品質チェック | 対象: `@src/utils/*.ts` + 処理を「リファクタ候補特定」に変更 | 同じ |

> テンプレート = **処理と出力が固定された関数**
> タスクごとに変わるのは**入力**だけ — これが再利用性の正体

<!--
再利用性を確認します。1つのテンプレートを3つのタスクに適用する例です。セキュリティ監査なら対象ファイルをauth配下に変更。パフォーマンス分析なら対象をapi配下に変更し、処理をボトルネック特定に変更。コード品質チェックなら対象をutils配下に変更し、処理をリファクタリング候補特定に変更。出力形式はすべて同じです。つまりテンプレートとは、処理と出力が固定された関数です。タスクごとに変わるのは入力だけ。これが再利用性の正体です。プログラミングの関数と同じ発想です。
-->

# よくある失敗 ① — 入力不足

**症状**: 結果が一般的すぎる。的が外れている。

```
[処理]
バグを修正して

[出力]
修正内容をコードで示して
```

**原因**: `[入力]` が空 — どのファイル・どのバグか不明

**修正**: 入力を具体的に書く
```
[入力]
対象: @src/auth/login.ts
エラー: TypeError: Cannot read property 'email'

[処理]
バグの原因を特定し、修正コードを書く

[出力]
原因の説明 / 修正コード / テストケース
```

> 関数に引数を渡さないと、結果はランダムになる

<!--
1つ目の失敗パターンは入力不足です。症状は結果が一般的すぎる、的が外れていること。処理と出力は書いてあっても、入力が空です。どのファイルのどのバグか分からない状態で「バグを修正して」と言っても、AIは一般的な回答しか返せません。修正は入力を具体的に書くこと。対象ファイルを@file参照で示し、エラーメッセージを書き、前提条件を記載する。関数に例えると、引数を渡さないと結果はランダムになるのと同じです。入力は関数の引数です。必ず明示しましょう。
-->

# よくある失敗 ② — 処理の曖昧さ

**症状**: やってはくれるが、期待と違うことをする

```
[入力]
対象: @src/utils/helpers.ts

[処理]
いい感じに整理して

[出力]
コードで出力
```

**原因**: 「整理」の解釈が無限 — フォーマット? リファクタ? 削除?

**修正**: 処理を具体的な動詞で指定
```
[処理]
関数をアルファベット順に並べ替える。
import文は先頭にまとめる。
未使用の関数は削除しない。
```

> 「いい感じに」は関数の仕様として成立しない

<!--
2つ目の失敗パターンは処理の曖昧さです。症状は、AIは何かをやってくれるけれど、期待と違うことをする状態。入力も出力も書いてあるのに、処理が「いい感じに整理して」と曖昧。「整理」の解釈が無限にあります。コードフォーマットを整えるのか、リファクタリングするのか、不要な関数を削除するのか。AIは親切なので何かしらやりますが、あなたの期待とは違うかもしれません。修正は処理を具体的な動詞で指定すること。「関数をアルファベット順に並べ替える」「import文は先頭にまとめる」「未使用の関数は削除しない」。このように具体的に書けば、AIの解釈は1つに定まります。「いい感じに」は関数の仕様として成立しません。
-->

---

# よくある失敗 ③ — 出力形式の未指定

**症状**: 出力が長すぎる、短すぎる、または形式がバラバラ

```
[入力]
対象: @docs/api.md

[処理]
APIの使い方を説明して
```

**原因**: `[出力]` が空 — AIが形式を勝手に決める

**修正**: 出力形式を明示する
```
[出力]
各エンドポイントについて以下の3項目を箇条書きで:
- エンドポイント名とHTTPメソッド
- リクエスト例 (curl)
- レスポンス例 (JSON)
```

> 出力形式 = 関数の戻り値の型 — 指定しないと毎回違う型が返る

<!--
3つ目の失敗パターンは出力形式の未指定です。症状は出力が長すぎたり短すぎたり、形式がバラバラになること。入力も処理も書いてあるのに、出力形式がありません。AIは親切なので説明してくれますが、500字の段落になるか10項目のリストになるかは運次第です。修正は出力形式を明示すること。「各エンドポイントについて3項目を箇条書きで」と形式を指定すれば、常に同じ構造の結果が得られます。関数に例えると、出力形式は戻り値の型です。型を指定しないと毎回違う型が返る状態です。出力形式は必ず明示しましょう。
-->

# 確認問題

**Q1**: 関数プロンプトにおける「入力」の役割を、プログラミングの関数の何に例えられるか説明せよ。

**Q2**: 構造化テンプレートで3パートを区切ることで、AIにとってどんなメリットがあるか。

**Q3**: 「いい感じに整理して」という処理の問題点と、その修正方法を述べよ。

**Q4**: Claude Code でサブエージェントを定義する際、`.claude/agents/` と `CLAUDE.md` はそれぞれ何を定義するのに使うか。

**解答**:

**A1**: 関数の「引数 (argument)」に相当する。入力を変えることで同じテンプレートを別のタスクに再利用できる。

**A2**: AIが「どこまでが素材で、何をしてほしいか、どう返せばいいか」を混同しなくなる。3パートの境界が明確になることで、解釈の余地が減り、結果が安定する。

**A3**: 問題点は「整理」の解釈が複数あること（フォーマット、リファクタ、削除など）。修正は具体的な動詞で処理を指定すること（例: 「関数をアルファベット順に並べ替える」）。
