# 伝わる ≠ 再利用できる — Emotional Hook

> 「AIに伝わればいい」と思っていませんか？
> 伝わるプロンプトと、**再利用できる**プロンプトは別物です。

- その場では上手くいく

- 入力を変えるだけで結果が変わる
- 別の文脈でも同じ品質

# 関数プロンプトの概念 — 3パート分解

> Excelの関数: `=SUM(A1:A10)` と入れると、合計が返ってくる
> プロンプトも同じように「入力→処理→出力」で設計できる

| パート | 役割 | プロンプトでの例 |
|--------|------|-----------------|
| **入力 (input)** | 処理の対象 | ファイル・データ・条件 |
| **処理 (task)** | やってほしいこと | 分析・変換・生成 |
| **出力 (output)** | 結果の形式 | 箇条書き・表・コード |

**なぜ関数として設計するのか**:
- **入力だけを変えれば**、別のタスクに再利用できる
- **処理と出力は固定**されるので、結果が安定する
- **テスト可能**になる — 同じ入力なら同じ結果が返る

<!--
プロンプトを関数として設計するとは、入力・処理・出力の3パートに分解することです。プログラミングの関数と同じ考え方です。入力が処理の対象で、ファイルやデータや条件にあたります。処理がやってほしいことで、分析や変換や生成です。出力が結果の形式で、箇条書きや表やコードです。なぜ関数として設計するのか。入力だけを変えれば別のタスクに再利用できるからです。処理と出力は固定されるので結果が安定します。そして同じ入力なら同じ結果が返るので、テストも可能になります。
-->

# 構造化テンプレート — input / task / output

プロンプトを3パートに明示的に分ける構造が**構造化テンプレート**です:

```markdown
[入力]
対象: @議事録_2026-05-03.md
前提: 30名参加の全体会議

[処理]
この議事録から、決定事項と担当者を抽出する。

[出力]
各決定事項について以下の形式で出力:
- 決定内容 / 担当者 / 期限
```

**ポイント**:
- **角カッコ `[入力]` `[処理]` `[出力]`** で3パートを明示的に区切る
- AIは「どこまでが素材で、何をしてほしいか」を**混同しなくなる**
- 入力部分だけを差し替えれば、別の議事録にも**即座に再利用**できる

<!--
構造化テンプレートの実例です。プロンプトを入力、処理、出力の3パートに角カッコで明示的に区切ります。入力には対象ファイルと前提条件を書き、処理にはやってほしいことを書き、出力には結果の形式を書きます。このように区切ることで、AIはどこまでが素材で何をしてほしいかを混同しません。そして入力部分だけを差し替えれば、別の議事録にも即座に再利用できます。
-->

# Before — 口頭指示の例

> 「この議事録、まとめておいて」

**この指示の問題点**:

| 観点 | 状態 | 影響 |
|------|------|------|
| **入力** | 「この議事録」が何を指すか不明 | AIが勝手に推測する |
| **処理** | 「まとめて」の範囲が無限大 | 要約? 決定事項? アクション? |
| **出力** | どう返せばいいか未指定 | 段落? リスト? 表? |

**結果**: 毎回**違う結果**が返る。再利用不可能。

<!--
Beforeの例です。「この議事録、まとめておいて」という指示。人間なら前後の文脈で何となく分かりますが、AIには伝わりにくい。入力の「この議事録」が何を指すか不明です。処理の「まとめて」の範囲が無限大で、要約の話なのか決定事項の話なのかアクションの話なのか分かりません。出力も未指定で、段落で返すのかリストなのか表なのか分からない。結果として、毎回違う結果が返り、再利用不可能です。
-->

# After — 構造化プロンプトの例

```markdown
[入力]
対象: @議事録_2026-05-03.md
前提: 30名参加の全体会議

[処理]
この議事録から、決定事項と担当者を抽出する。

[出力]
各決定事項について以下の形式で出力:
- 決定内容 / 担当者 / 期限
```

**結果**: 毎回**同じ構造**で安定出力。入力を `@議事録_2026-05-10.md` に変えれば、そのまま再利用可能。

<!--
Afterの例です。先ほどの構造化テンプレートを使ったプロンプトです。入力に対象ファイルと前提条件を明示し、処理で決定事項と担当者を抽出することを指定し、出力で決定内容、担当者、期限という形式を指定しています。結果として毎回同じ構造で安定した出力が得られます。さらに、入力の対象ファイルを `@議事録_2026-05-10.md` に変えれば、そのまま別の議事録に再利用できます。これが「関数」として設計するメリットです。

# 再利用性の確認 — 同じテンプレート、別タスク

**1つのテンプレートを3つのタスクに適用**:

| タスク | 入力の変更箇所 | 処理・出力 |
|--------|---------------|-----------|
| 全体会議の議事録 | 対象: `@議事録_全体.md` | 同じ |
| 部署会議の議事録 | 対象: `@議事録_営業部.md` | 同じ |
| 顧客面談メモ | 対象: `@面談_A社.md` + 処理を「要望と課題を抽出」に変更 | 同じ |

> テンプレート = **処理と出力が固定された関数**
> タスクごとに変わるのは**入力**だけ — これが再利用性の正体

<!--
再利用性を確認します。1つのテンプレートを3つのタスクに適用する例です。全体会議なら対象を全体議事録に。部署会議なら対象を営業部議事録に。顧客面談なら対象を面談メモに変更し、処理を要望と課題の抽出に変更。出力形式はすべて同じです。つまりテンプレートとは、処理と出力が固定されたひな形です。タスクごとに変わるのは入力だけ。これが再利用性の正体です。

# よくある失敗 ① — 入力不足

**症状**: 結果が一般的すぎる。的が外れている。

```
[処理]
まとめておいて

[出力]
箇条書きで出力
```

**原因**: `[入力]` が空 — どの議事録・どうまとめるか不明

**修正**: 入力を具体的に書く
```
[入力]
対象: @議事録_全体_05-03.md
目的: 決定事項の確認

[処理]
決定事項と担当者を特定する

[出力]
決定内容 / 担当者 / 期限
```

> 関数（ひな形）に素材を渡さないと、結果はランダムになる

<!--
1つ目の失敗パターンは入力不足です。症状は結果が一般的すぎる、的が外れていること。処理と出力は書いてあっても、入力が空です。どの議事録か分からない状態で「まとめておいて」と言っても、AIは一般的な回答しか返せません。修正は入力を具体的に書くこと。対象ファイルを@file参照で示し、目的を書く。ひな形に素材を渡さないと、結果はランダムになるのと同じです。入力は必ず明示しましょう。
-->

# よくある失敗 ② — 処理の曖昧さ

**症状**: やってはくれるが、期待と違うことをする

```
[入力]
対象: @提案書_新規プロジェクト.md

[処理]
いい感じに整理して

[出力]
文章で出力
```

**原因**: 「整理」の解釈が無限 — 要約? 箇条書き? 添削?

**修正**: 処理を具体的な動詞で指定
```
[処理]
文章を論理的に並べ替える。
重複表現は1つにまとめる。
内容は削除しない。
```

> 「いい感じに」はひな形の仕様として成立しない

<!--
2つ目の失敗パターンは処理の曖昧さです。症状は、AIは何かをやってくれるけれど、期待と違うことをする状態。入力も出力も書いてあるのに、処理が「いい感じに整理して」と曖昧。「整理」の解釈が無限にあります。要約するのか、箇条書きにするのか、添削するのか。AIは親切なので何かしらやりますが、あなたの期待とは違うかもしれません。修正は処理を具体的な動詞で指定すること。「文章を論理的に並べ替える」「重複表現は1つにまとめる」「内容は削除しない」。このように具体的に書けば、AIの解釈は1つに定まります。「いい感じに」はひな形の仕様として成立しません。
-->

# よくある失敗 ③ — 出力形式の未指定

**症状**: 出力が長すぎる、短すぎる、または形式がバラバラ

```
[入力]
対象: @製品マニュアル.md

[処理]
使い方を説明して
```

**原因**: `[出力]` が空 — AIが形式を勝手に決める

**修正**: 出力形式を明示する
```
[出力]
各機能について以下の3項目を箇条書きで:
- 機能名
- 操作手順（3ステップ以内）
- 注意点
```

> 出力形式 = 結果の「枠」 — 指定しないと毎回違う形で返ってくる

<!--
3つ目の失敗パターンは出力形式の未指定です。症状は出力が長すぎたり短すぎたり、形式がバラバラになること。入力も処理も書いてあるのに、出力形式がありません。AIは親切なので説明してくれますが、500字の段落になるか10項目のリストになるかは運次第です。修正は出力形式を明示すること。「各機能について3項目を箇条書きで」と形式を指定すれば、常に同じ構造の結果が得られます。出力形式は結果の「枠」です。枠を指定しないと毎回違う形で返ってきます。出力形式は必ず明示しましょう。
-->

**Q3** (穴埋め): 「いい感じに整理して」という処理の問題点は、「整理」の □□□ が無限にあること。修正は具体的な □□ で処理を指定すること。

**Q4** (選択式): Claude Code でひな形に名前をつけて再利用する設定ファイルは？

- A: `.claude/agents/` の定義ファイル
- B: `.gitignore`
