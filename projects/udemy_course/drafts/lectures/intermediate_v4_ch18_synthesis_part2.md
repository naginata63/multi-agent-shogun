# 「これがあれば安心して任せられる」

> 「完了報告が嘘をついていないか、静かに失敗していないか、過去の解決策を見落としていないか——**3つのハーネスが同時に監視していれば、すべて自動検出できます。**」
>
> この仕組みを理解すれば、AIに任せることが「信頼」ではなく「仕組み」になります。

- 第11章: 完了Gate — 「完了」を機械的に検証する門番
- 第12章: silent fail検出 — エラーなし ≠ 成功 を見抜く監視員
- 第13章: cmd intake — 過去の知識を自動で引き出す検索係

<!--
このスライドがつかみです。3つのハーネスがそれぞれ単独で優れた機能を持つことは、すでに第11章から第13章で学びました。しかし、本当の力は3つが統合された時に発揮されます。完了Gateが未完了の報告をブロックし、silent fail検出がエラーなく進行する暗黙の失敗を見つけ出し、cmd intakeが過去の解決策を自動で参照する。この3つが連携すれば、AIに安心して任せられる環境が完成します。「信じて任せる」のではなく、「仕組みで保証する」という発想の転換が、この章の核心です。
-->

# 3つのハーネスを振り返る

| ハーネス | 検出対象 | hook種別 | 第何章 |
|----------|----------|----------|--------|
| 完了Gate | 未完了の「完了」報告 | PostToolUse | 第11章 |
| silent fail検出 | エラーなしの失敗 | inotifywait常駐 | 第12章 |
| cmd intake | 過去事例の見落とし | PreToolUse | 第13章 |

- それぞれ**独立して動作**するが、**連携することで相乗効果**が生まれる
- 1つの失敗を複数のハーネスが異なる角度から検出する

<!--
3つのハーネスのおさらいです。完了Gateは、AIが「完了しました」と報告した際に、本当に完了しているかを機械的に検証するPostToolUse hookです。silent fail検出は、エラーログには何も出ていないのに結果がおかしいという「静かな失敗」を見つける常駐監視です。cmd intakeは、新しいタスクを始める際に過去の類似事例を自動検索するPreToolUse hookです。これらは独立して動作しますが、連携することで1つの失敗を複数の角度から検出できるようになります。例えば、完了Gateがブロックした問題をsilent fail検出も同時に検知する、という二重の安全網ができます。
-->

# なぜ統合が必要か — 単体では見逃すパターン

- **パターンA**: 完了条件は満たしているが、**ログにsilent fail**が紛れている
  - 完了Gate → PASS（条件OK）
  - silent fail検出 → **WARN**（異常パターン検出）
- **パターンB**: 過去に解決済みの問題を**再発明**している
  - 完了Gate → PASS（条件OK）
  - cmd intake → **警告注入**（「3日前に解決済み」）
- **パターンC**: 完了報告自体が**不正確**
  - 完了Gate → **BLOCK**（条件不足）
  - silent fail検出 → WARN（ログ異常）

> 単体ハーネスは「検出できる範囲」が限られる。統合で**死角を補い合う**。

<!--
3つのハーネスがそれぞれ単独では見逃してしまうパターンを見てみましょう。パターンAは、完了Gateの5条件はすべて満たしているのに、ログの中にsilent failのサインが紛れているケースです。完了GateだけではPASSしてしまいますが、silent fail検出がWARNを出します。パターンBは、タスク自体は正しく完了しているのに、実は3日前に同じ問題をすでに解決していたというケースです。cmd intakeが「過去事例あり」と警告します。パターンCは、完了報告が不正確なケースで、完了GateがBLOCKし、silent fail検出もWARNを出します。このように、単体では検出できない死角を、ハーネス同士が補い合うのが統合の価値です。
-->

# 統合アーキテクチャの全体像

```
[タスク起票] -----> cmd intake (PreToolUse)
      |                  |
      |          過去事例検索結果注入
      |                  v
      v           +-----------------+
[実行開始] -----> |   AIエージェント   |
                  +--------+--------+
                           | 完了報告
                           v
                    完了Gate (PostToolUse)
                      | PASS / BLOCK
                      v
                ログ出力 --> silent fail検出 (常駐)
                                | WARN / OK
                                v
                          動作レポート
```

<!--
統合アーキテクチャの全体像を確認しましょう。タスクが起票されると、まずcmd intakeがPreToolUse hookとして発火し、過去の類似事例を検索して結果を注入します。AIエージェントはその情報を参考にしながら実行します。実行が終わって完了報告を出すと、完了GateがPostToolUse hookとして発火し、完了条件を機械的に検証します。PASSすれば次へ進み、BLOCKされれば修正が必要です。並行して、silent fail検出がログを常時監視しており、エラーなしの失敗パターンを検出します。最終的に、すべての結果が動作レポートに記録されます。この流れが自動的に回ることで、人間が逐一確認しなくても品質が保証されます。
-->

# failure注入テストとは

> **正常に動作していることを確認するには、わざと壊してみるのが一番確実**

- **failure注入** = ハーネスが正しく検出できるかを検証するための、意図的な故障テスト
- 消防設備の「定期的な防火訓練」と同じ発想
- ハーネスを導入した直後と、**定期的に実施**することで信頼性を維持

### 3つのパターン

| # | 注入する故障 | 確認するハーネス |
|---|-------------|-----------------|
| 1 | 完了条件を意図的に欠落させる | 完了Gate (BLOCK検出) |
| 2 | ログにsilent failパターンを紛れ込ませる | silent fail検出 (WARN検出) |
| 3 | 過去に解決済みの問題と同じタスクを起票 | cmd intake (過去事例ヒット) |

<!--
failure注入テストとは、ハーネスが正しく動作していることを確認するための手法です。わざと故障を起こし、ハーネスがそれを検出できるかをテストします。消防設備の定期的な防火訓練と同じ発想です。実際の火事が起きる前に、スプリンクラーが正常に動くか確認するのです。failure注入テストは3つのパターンがあります。1つ目は、完了条件を意図的に欠落させて完了GateがBLOCKするかを確認します。2つ目は、ログの中にsilent failのパターンを紛れ込ませて検出器がWARNを出すかを確認します。3つ目は、過去に解決済みの問題と同じ内容のタスクを起票して、cmd intakeが過去事例をヒットさせるかを確認します。
-->

---

# failure注入パターン1: 完了Gate検出テスト

```
# テスト: 完了条件を欠落させる
status: done
result:
  summary: "実装完了"
  files_modified: []
  # ← files_modified が空 → 完了Gate が BLOCK
```

- 完了Gateの検証項目: `files_modified` が空 → **BLOCK**
- 期待結果: `BLOCK: files_modified is empty`
- 修正して再実行 → **PASS** を確認

<!--
1つ目のfailure注入パターンを詳しく見ましょう。完了Gateが検証する5条件のうち、files_modifiedが空の状態で完了報告を提出します。これは「実装は完了したが、変更ファイルが記録されていない」という典型的な未完了パターンです。完了Gateはこれを検出してBLOCKを返します。テストでは、このBLOCKが正しく出力されることを確認します。その後、files_modifiedに正しい値を入れて再実行し、今度はPASSすることを確認します。これで完了Gateが正常に動作していることが検証できました。
-->

# failure注入パターン2: silent fail検出テスト

```
# テスト: ログにsilent failパターンを注入
[2026-05-03 14:00:01] Task completed successfully
[2026-05-03 14:00:01] exit_code: 0
[2026-05-03 14:00:01] result: None    ← 成功なのに結果がNone
```

- silent fail検出器が監視するパターン: `exit_code: 0` + `result: None`
- 期待結果: `WARN: success with empty result`
- この組み合わせは**エラーではないが、異常** — まさに静かな失敗

<!--
2つ目のfailure注入パターンです。終了コードが0で正常終了しているのに、結果がNoneになっているログを注入します。これは典型的なsilent failパターンです。エラーは出ていないので一見成功に見えますが、結果が空ということは実際には何もできていません。silent fail検出器はこの組み合わせを監視しており、WARNを出します。このテストで、検出器が正常にパターンを認識し、適切に警告を出せていることを確認します。silent failの怖さは、人間がログを眺めても気づきにくい点にあります。だからこそ機械的な検出が重要なのです。
-->

# failure注入パターン3: cmd intake検出テスト

```
# テスト: 過去に解決済みの問題と同じタスクを起票
task: "ffmpeg NVENC変換スクリプトの作成"
keywords: ["ffmpeg", "NVENC", "変換"]
# ← 過去3件の類似事例が mem-search に存在
```

- cmd intake (mem-search pre-hook) が過去事例を自動検索
- 期待結果: `HIT: 3 past cases found`
- **過去の解決策を再利用**できるかが検証ポイント
- hit=0なら「過去事例なし」の警告が注入される

<!--
3つ目のfailure注入パターンです。過去にすでに解決済みの問題と同じ内容のタスクを起票します。cmd intakeのmem-search pre-hookが、タスクのキーワードから過去の類似事例を自動検索します。テストでは、過去に3件の類似事例が存在する状態で、同じ内容のタスクを起票します。期待される結果は、過去3件がヒットすることです。これにより、過去の解決策を再利用できることが確認できます。逆に、本当に新しい問題の場合はhit=0となり、「過去事例なし」の警告が注入されます。このテストで、cmd intakeが正常に動作していることを検証します。
-->

# 全ハーネス連携フロー

```
  [failure注入開始]
       |
       +---> パターン1: 完了Gate --> BLOCK OK
       |
       +---> パターン2: silent fail --> WARN OK
       |
       +---> パターン3: cmd intake --> HIT OK
                    |
                    v
           3件すべて検出成功
                    |
                    v
           動作レポートに記録
```

- 3つのfailure注入が**すべて正しく検出**されたら、統合テストPASS
- 1つでも検出漏れがあれば、そのハーネスの**設定を見直す**

<!--
3つのfailure注入テストを順番に実行し、すべてのハーネスが正しく検出したかを確認する流れです。パターン1で完了GateがBLOCKを出すか。パターン2でsilent fail検出がWARNを出すか。パターン3でcmd intakeが過去事例をHITするか。3つすべてが正しく検出できたら、統合テストはPASSです。もし1つでも検出漏れがあれば、そのハーネスの設定やパターン定義を見直す必要があります。この一連の流れを動作レポートに記録します。
-->

# 動作レポートの書き方

```yaml
test_id: failure_injection_20260503
timestamp: "2026-05-03T14:30:00"
results:
  - harness: completion_gate
    pattern: "files_modified empty"
    expected: BLOCK
    actual: BLOCK
    status: PASS
  - harness: silent_fail_watcher
    pattern: "exit_code 0 + result None"
    expected: WARN
    actual: WARN
    status: PASS
  - harness: cmd_intake
    pattern: "duplicate task"
    expected: "HIT 3"
    actual: "HIT 3"
    status: PASS
overall: ALL_PASS
```

- **expected vs actual** を明記 — 期待と結果の比較が証拠
- 定期実行の結果を蓄積 → **品質のトレンド** が見える

<!--
動作レポートは、failure注入テストの結果を記録する文書です。ポイントは、expectedとactualを明記することです。期待した結果と実際の結果を比較することで、ハーネスが正しく動作したことが証明になります。レポートには、テストID、実行日時、各ハーネスごとの結果を記録します。statusがPASSであればそのハーネスは正常動作、FAILであれば設定を見直す必要があります。このレポートを定期的に蓄積することで、ハーネスの品質トレンドが見えるようになります。「先月は3回FAILがあったが、今月は0回」というように、改善の推移を定量的に把握できます。
-->
