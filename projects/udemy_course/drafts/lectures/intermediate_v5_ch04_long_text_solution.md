
# 主役①: chunk (長文分割)
## 長文を小片に分けて、必要なものだけ渡す

---

## chunk とは?

**chunk =「かたまり」**

長い文章を、AI が一度に処理できるサイズの小片に分割する技術です。

```
200 ページの仕様書
  ├── chunk 1: 第1章 概要 (7 ページ)
  ├── chunk 2: 第2章 要件定義 (12 ページ)
  ├── chunk 3: 第3章 画面設計 (8 ページ)
  ...
  └── chunk 30: 付録 (5 ページ)
```

各チャンクは 200-500 行 (A4 数枚分) 程度が目安。AI は LitM を起こさずに正確に読めます。

---

## なぜ chunk が効くのか?

| Before (全部添付) | After (chunk 分割) |
|---|---|
| 200 ページ = 約 50,000 トークン | 各チャンク = 約 1,500 トークン |

※ トークン = AI が文字を数える単位。日本語 1 文字 ≒ 1-2 トークン。50,000 トークン ≒ 日本語 3 万文字程度
| LitM で中央が読み飛ばされる | 各チャンクは短い → LitM 発生しない |
| 毎回全量を送る = コストも時間も無駄 | 必要なチャンクだけ送る = 効率的 |

**ch03 で学んだ LitM の根本対策そのもの** です。

---

## 実践: 仕様書を chunk に分ける

**ステップ**:

1. 200 ページ仕様書を章・節単位で分割
2. 各チャンクにタグ (章番号・トピック・キーワード) を付けて検索しやすくする
3. 質問が来たら「どのチャンクが関連するか」を判定
4. 関連チャンクだけ AI に渡す

**結果**: 「第5章のエラーハンドリングについて教えて」と聞けば、第5章チャンクだけが渡される。他の 29 チャンクは AI に読まれず、LitM も起きない。

---

## chunk の設計のコツ

**分割の単位は何がいいか?**

- 章単位 (大きな文書)
- 節単位 (中程度の文書)
- 段落単位 (短い文書)

**ポイント**: 「AI に一度に読ませたい最小単位」で分ける。チャンク同士が重複しないように境界を明確に。

# 主役②: RAG (検索拡張生成)
## 関連情報を自動で検索して AI に渡す仕組み

---

<div class="vocab-box">

### 用語: RAG (Retrieval-Augmented Generation)

**検索拡張生成** (Retrieval-Augmented Generation)

外部の知識ベースから関連する情報を検索し、その検索結果を AI に渡して回答を生成させる手法。

**ざっくりいうと**: 「AI に答えさせる前に、関連資料を検索して机の上に置いてから答えさせる」

</div>

---

## RAG のイメージ

```
質問: 「A社のQ3予算について教えて」
    │
    ▼
① 検索: 知識ベースから「A社」「予算」「Q3」に関連するチャンクを探す
    │
    ▼
② 取得: 関連チャンク 3 件を特定
    │
    ▼
③ 注入: AI に関連チャンクを渡す
    │
    ▼
④ 生成: AI がチャンクを根拠に回答
    → 「A社の予算は Q3 (7-9月) に集中する見込みです」
```

---

## RAG が解決する問題

| 問題 | RAG での解決 |
|------|-------------|
| 全部の資料を AI に渡すと LitM | 関連チャンクだけ渡す → 短いから LitM なし |
| 過去の決定事項を忘れる | 知識ベースに保存 → 検索で即呼び出し |
| 毎回ゼロから説明 | 検索結果が「過去の説明」の代わりに |

**要するに**: 「全部読む」から「必要な分だけ検索して読む」への転換

---

## RAG の構成要素

```
┌─────────────────────────────────────┐
│          知識ベース (Knowledge Base)  │
│  ┌──────┐ ┌──────┐ ┌──────┐        │
│  │chunk1│ │chunk2│ │chunk3│ ...    │
│  └──────┘ └──────┘ └──────┘        │
└──────────┬──────────────────────────┘
           │ 検索
           ▼
┌─────────────────────────────────────┐
│  AI (Claude)                         │
│  質問 + 関連チャンク → 回答を生成    │
└─────────────────────────────────────┘
```

**自分用 RAG**: 自分の業務メモ・決定事項・過去の AI 対話を知識ベースにすれば、「自分専用の検索エンジン + AI」が完成する。

# 主役③: claude-mem
## 過去の経験を意味で検索する「自分専用の検索」

---

## claude-mem とは?

**Claude Code に組み込まれた永続記憶データベース** (追加インストール不要)

- AI との対話で得られた発見・決定事項を自動・手動で保存
- 保存した内容を **意味で検索** できる (次スライドで詳しく)
- セッションが変わっても過去の記憶が残る

**= 自分専用の小さな RAG**

---

## 「意味で検索」とは?

**普通の検索 (キーワード検索)**:
- 「予算」という単語を含む文章を探す
- 「予算」が書いていないと見つからない

**意味で検索 (意味検索)**:
- 「お金が使える時期は?」と聞く
- → 「予算」「Q3 に集中」「7-9 月」など、**関連する内容** をまとめて見つける
- 同じ単語が書いていなくても、**内容が関連していれば** 命中する

**例え**: キーワード検索 = 索引で単語を引く / 意味検索 = 内容を理解している司書に聞く

---

## 3 つの検索コマンド

| コマンド | 役割 | 使いどころ |
|---------|------|-----------|
| `mem-search` | キーワードで記憶を検索 | 「あの件どうなったっけ?」 |
| `smart_search` | 文書やコードを構造検索 | 「あの提案書の承認フローの箇所はどこ?」 |
| `timeline` | 時系列で記憶を振り返る | 「先週何をやったっけ?」 |

---

## 実演: mem-search を使ってみる

**シナリオ**: 先月 A 社の商談で「Q3 予算集中」と合意したが、今月その情報が必要になった。

```
> mem-search "A社 予算時期"

検索結果:
━━━━━━━━━━━━━━━━━━━━━━━━━━━
📅 2026-04-15 | 商談メモ
「A社は Q3 (7-9月) に予算が集中する。
 6月までに提案書を提出することで、Q3 予算に間に合う。」
━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**結果**: 過去の商談記録が即座に呼び出され、正確な提案書が作成できる。

---

## 実演: timeline で振り返る

**シナリオ**: 今週やったことを振り返りたい。

```
> timeline "先週の作業"

2026-05-01: 顧客 B 社の提案書ドラフト作成
2026-05-02: 競合分析レポート提出
2026-05-03: バグ調査 (認証モジュール) → 原因特定済
```

**結果**: セッションが変わっても「何をやったか」が即座に思い出せる。

---

## claude-mem = 自分専用 RAG の具体例

| RAG の要素 | claude-mem での実装 |
|-----------|-------------------|
| 知識ベース | 永続記憶データベース (裏で自動管理) |
| チャンク | 各記憶エントリ (観測レコード) |
| 検索 | `mem-search` / `smart_search` |
| AI への注入 | 検索結果をプロンプトに自動挿入 |

**ch03 の根本対策「長文を作らない」が、ここで具体的な道具になりました。**

# 主役④: 手動でこまめなドキュメント化
## RAG の「入力」を自分で作る習慣

---

## なぜ「手動」が主役なのか?

RAG がいくら優れていても、**検索される中身がなければ意味がない**。

```
RAG の構造:
  知識ベース (保存する) → 検索 (探す) → AI に渡す (使う)
      ↑
  ここを自分で埋める = 手動ドキュメント化
```

**AI が勝手に覚えてくれるわけではない。** 人間が「これは重要」と判断して書き残す習慣が RAG の基盤。

---

## 何を書き残すべきか?

| 種類 | 例 | タイミング |
|------|----|-----------|
| 決定事項 | 「A社予算は Q3 に集中」 | 商談後すぐ |
| 教訓 | 「メールは週次フォーマットで統一」 | エラー解決後すぐ |
| 固有名詞 | 「B社の担当者は田中さん」 | 初回ヒアリング後 |
| 進捗 | 「認証バグは原因特定済」 | タスク完了時 |

**コツ**: 「後で書こう」ではなく「その場で書く」。1 行でいい。

---

## 実践: memory ファイルに書く

Claude Code では `memory/*.md` に決定事項を書き残します。

先頭の `---` で囲まれた部分は整理用のタグです。本編はその下の本文だけで OK です。

```markdown
---
name: a_client_budget
description: A社の予算時期についての商談メモ
type: project
---

A社は Q3 (7-9月) に予算が集中する。
6月までに提案書を提出することで Q3 予算に間に合う。
2026-04-15 商談にて確認。
```

**たったこれだけ**。でも、この 1 行が後で `mem-search` で呼び出され、大きなミスを防ぐ。

---

## 手動ドキュメント化 = 「未来の自分への手紙」

> **未来の自分**: 「A社の予算いつだっけ…?」
>
> **過去の自分**: (memory ファイルに書いておいた) 「Q3 だよ」

**この「書いておいたおかげで助かった」体験を繰り返すと、習慣になります。**

---

## 4 主役のまとめ

```
主役① chunk        : 長文を小片に分ける → 読み飛ばしを防ぐ
主役② RAG          : 関連小片を検索して渡す → 全部読ませない
主役③ claude-mem   : 過去の経験を意味検索 → 「忘れ」を防ぐ
主役④ 手動doc化    : 決定事項を都度書き残す → 検索される中身を作る
```

**共通の哲学**: 「全部一度に渡す」から「必要なときに必要な分だけ渡す」へ

# 副次: /handoff /rehydrate
## セッション切替時の引き継ぎ (LitM 対策ではありません)

---

## handoff / rehydrate とは?

| コマンド | 役割 | タイミング |
|---------|------|-----------|
| `/handoff` | 現在の作業状態をファイルに保存 | セッション終了前 |
| `/rehydrate` | 保存したファイルから状態を復元 | 新しいセッション開始時 |

**用途**: セッションが切れる前後の引き継ぎ。**会話の要約 (自動圧縮) が起きる前の安全策としても使えます。**

---

## ⚠️ handoff は LitM 対策ではありません

> **誤用**: 「handoff は長文読み飛ばし (LitM) 対策」
>
> **正しい用途**: 「**セッション切替時の引き継ぎ・圧縮の代替**」

**なぜ区別が重要か?**

- LitM = 長い参照情報の中央が読み飛ばされる問題 (ch03)
- handoff = セッションがリセットされる問題 (別系統)

混同すると「handoff すれば LitM は解決」という誤った設計に。

---

## handoff / rehydrate の使いどころ

**ケース**: 金曜に AI と仕様書レビューを終えた。月曜に続きをやりたい。

1. 金曜の終わり: `/handoff` → 引き継ぎファイル生成
2. 月曜の朝: `/rehydrate` → 文脈を復元
3. 月曜: 金曜の続きから作業再開

**自分の週次業務の引き継ぎに即適用できます。**

---

## 副次: PreCompact による情報退避

AI との対話が長くなると、AI が会話の要約 (自動圧縮) を行って古い内容を整理します。

この圧縮の直前に、重要情報を自動で退避する仕組みが `PreCompact` です。

- 圧縮直前: 重要な決定事項・進捗をファイルに自動保存
- 圧縮後: 保存したファイルから文脈を復旧

**handoff の自動版** と考えると分かりやすい。

---

## 4 主役 + 2 副次の全体図

```
■ 主役 (LitM 根本対策の4道具)
  ① chunk       : 長文を分割
  ② RAG         : 分割した中身を検索
  ③ claude-mem  : 過去の経験を意味検索 (= RAG の具体実装)
  ④ 手動doc化   : 検索される中身を自分で作る

■ 副次 (LitM とは別系統)
  /handoff /rehydrate : セッション切替時の引き継ぎ
  PreCompact          : 圧縮直前の情報退避
```

---

## 実例①: 営業の顧客対応

**現象**: A社との商談で「予算は Q3 に集中」と合意したのに、今月 AI がその情報を知らない

**解決**:
1. 商談後 → memory ファイルに「A社 Q3 予算集中」を書き残す (主役④)
2. 提案書作成時 → `mem-search "A社 予算"` で呼び出す (主役③)
3. 結果 → 正確な Q3 予算前提の提案書が完成

**置換**: 自社の顧客名・案件キーワードに変えて即応用

## 実例②: PM の仕様引き継ぎ

**現象**: 金曜に AI と仕様書レビューを終えた。月曜に続きをやろうとしたら AI が文脈を失っている

**解決**:
1. 金曜の終わり → `/handoff` で引き継ぎファイル生成 (副次)
2. 月曜の朝 → `/rehydrate` で文脈復元
3. 結果 → 金曜の続きから作業再開

**置換**: 自分の週次業務の引き継ぎに即適用

## 実例③: データ分析の必要時読込

**現象**: 分析プロジェクトに 20 個のドキュメントがある。全部渡すと LitM 発生

**解決**:
1. 20 個のドキュメントをチャンクに分割 (主役①)
2. 分析テーマに関連するチャンクだけを検索して渡す (主役②)
3. 結果 → 短いチャンクだけを渡す → LitM なし → 精度向上

**置換**: 自社のプロジェクトドキュメント構成に即適用

## 本章のまとめ

<!--
スピーカーノート:
4つの主役と2つの副次道具を振り返りましょう。chunk、RAG、claude-mem、手動ドキュメント化がLitMの根本対策。handoffとPreCompactはセッション切替用でLitMとは別系統です。
-->

- **chunk**: 長文を小片に分け、LitM を根本から防ぐ
- **RAG**: 関連情報を検索して渡すことで「全部添付」を回避
- **claude-mem**: 過去の経験を意味で検索する「自分専用検索」
- **手動ドキュメント化**: RAG の中身を自分で作る「未来への手紙」
- **handoff/rehydrate**: セッション切替用 (LitM 対策ではない)

**持ち帰り**:
「**長文を作らない仕組み** (chunk + RAG + 永続記憶 + 手動ドキュメント化) で、AI に忘れさせない設計ができる」

> **強化した層: コンテキスト層** — 長文を作らない仕組みで LitM の根本対策を実装した
