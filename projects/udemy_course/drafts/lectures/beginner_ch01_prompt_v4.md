# AIは「文脈」で判断する

> AIはあなたの **意図を推測** して動く。曖昧な指示 → 曖昧な結果。

**人間なら分かる**:
- 「これ、よろしく」→ 文脈と空気で何とか分かる

**AIには伝わらない**:
- 「これ」が何を指すか不明
- 「よろしく」の範囲が無限大

> 💡 プロンプトの品質 = **出力の品質**

<!--
AIは賢いですが、人間の「空気を読む」能力は持っていません。「これ、よろしく」と言われたら、人間なら前後の文脈で「この書類の提出をお願いします」と解釈できます。AIにはこれができません。「これ」が何か、「よろしく」の範囲がどこまでか、すべて言葉で明確にする必要があります。プロンプトの品質がそのまま出力の品質に直結する理由はここにあります。
-->

---

# 良いプロンプトの3要素

> 指示を **文脈・指示・形式** の3つに整理すると精度が劇的に上がる

| 要素 | 役割 | 例 |
|------|------|-----|
| **文脈** | AIに与える背景・素材 | 「この記事について…」「@src/app.ts を見て…」 |
| **指示** | AIにやってほしいこと | 「要約して」「バグを特定して」 |
| **形式** | 結果をどう返すか | 「箇条書き3つで」「表形式で」 |

**なぜ分けるのか**: AIは「どこまでが素材で、何をしてほしいのか」を混同しやすい。整理すると誤解が減る。

<!--
プロンプトの基本は3要素です。文脈には背景情報や素材を、指示にはやってほしいことを、形式には結果の返し方を書きます。人間なら「これを使って、ああして、こう返して」と文脈で分かりますが、AIは境界が曖昧だと混乱します。3要素を明確に分けることで、AIの誤解を防ぎます。特別な記法は不要。自然な文章で「このファイルについて〜」と始めれば、AIは文脈を理解してくれます。
-->

---

# テクニック1: @file でファイルを参照する

```
@src/app.ts のバグを特定して。
各バグについて深刻度（高/中/低）と
修正方法を箇条書きで教えて。
```

**ポイント**:
- `@ファイル名` でプロジェクト内のファイルを直接参照
- コードをコピペする必要がない
- AIが自動的にファイル内容を読み込む

> 💡 Claude Code なら、ファイルの中身を **貼り付け不要**

<!--
Claude Code最大の利点の一つが@file参照です。@src/app.tsと書くだけで、AIがそのファイルを自動的に読み込んでくれます。ブラウザ版のAIではコードをコピーして貼り付ける必要がありました。Claude Codeではその手間がゼロ。プロジェクト内の任意のファイルを@で参照できるので、長いコードもそのまま扱えます。これが3要素の「文脈」を簡単に与える方法です。
-->

---

# テクニック2: 段階分解 — 「まず○○、次に△△」

```
このプロジェクトで以下の手順で進めて：
1. まず @package.json の依存関係を確認して
2. 次に 使っていないパッケージを特定して
3. 最後に 削除しても安全か判断して
```

**ポイント**:
- 複雑な作業は **段階に分ける** と精度が上がる
- 「まず」「次に」「最後に」で順序を明示
- AIが途中で迷わない

<!--
2つ目のテクニックは段階分解です。複雑な作業を一度に頼むと、AIがどこから手をつけるべきか迷うことがあります。「まず○○して、次に△△して」と段階に分けて指示すると、AIが迷わずに進められます。特にデバッグやリファクタリングなど、複数ステップがある作業では効果的です。手順を番号付きで書くとさらに明確になります。
-->

---

# テクニック3: 具体的に書く — 曖昧→具体

**悪い例**:
```
いい感じにテストを書いて
```

**良い例**:
```
@src/utils.ts の各関数に対して
ユニットテストを書いて。
テストフレームワークはvitestを使い、
正常系・異常系・境界値の3パターンを
それぞれカバーすること。
```

> 「いい感じに」→ 毎回違う結果
> 「正常系・異常系・境界値」→ 毎回同じ品質

<!--
3つ目のテクニックは具体性です。「いい感じに」「適当に」は絶対にNG。AIは親切なので何かしら出力してくれますが、それが毎回同じ内容になる保証はありません。具体的な条件を書くことで、何度実行しても同じ品質の結果を得られます。テストなら「正常系・異常系・境界値」のように、期待する内容を明示しましょう。
-->

---

# テクニック4: 箇条書きで整理 + slash command

**箇条書きで複数条件を渡す**:
```
以下の観点で @README.md を改善して：
- セットアップ手順が分かりやすいか
- 必要な前提条件が書かれているか
- 使用例が含まれているか
```

**slash command で定型操作を一発実行**:
```
/init     → CLAUDE.md を自動生成
/review   → PRレビューを実行
/clear    → 会話をリセット
```

> 💡 定型作業は slash command、複数条件は箇条書きが確実

<!--
最後に2つのテクニックを紹介します。箇条書きは人間にもAIにも分かりやすい形式で、複数の条件や観点を伝える時に効果的です。もう一つはslash commandです。/から始まるコマンドでよく使う操作を一発実行できます。/initでCLAUDE.mdを自動生成、/reviewでPRレビューなど。毎回同じ指示をタイプする手間が省けます。さらに自分専用のコマンドも作れますが、詳しくは第5章hookの回で解説します。
-->

---

# Before / After — 出力の違い

**Before（曖昧な指示）**: 毎回違う形式で出力
```
1回目: 段落形式で要約
2回目: 箇条書きで要約
3回目: 表形式で要約
```

**After（3要素 + 具体性）**: 毎回同じ形式で安定出力
```
1回目: 箇条書き3項目 ✓
2回目: 箇条書き3項目 ✓
3回目: 箇条書き3項目 ✓
```

> **再現性** = 同じプロンプトで同じ品質の結果を得ること

<!--
BeforeとAfterの違いを可視化しました。曖昧なプロンプトは、実行するたびに出力形式がブレます。段落だったり箇条書きだったり表だったり。3要素を意識して具体的に書いたプロンプトは、毎回同じ形式。これが「再現性」です。実務ではプロンプトをチームで共有して使い回す場面が多くあります。その時に再現性がなければ意味がありません。
-->

---

# 再現性が重要な理由

**プロンプトは「1回限り」ではない**:

- **使い回し**: 同じ処理を別のファイルに適用
- **共有**: チームメンバーにプロンプトを渡す
- **自動化**: スクリプトの中で何度も実行

> 再現性のないプロンプト = 使い捨て
> 再現性のあるプロンプト = 資産

<!--
再現性がなぜ重要か。3つの理由があります。まず使い回し。別のファイルに同じ処理を繰り返す場面は多いです。次に共有。チームメンバーに「このプロンプトを使って」と渡す時に、相手が実行しても同じ結果が出なければ困ります。そして自動化。スクリプトの中でプロンプトを何百回と実行する時に、結果がブレると自動化の意味がありません。再現性のあるプロンプトは、あなたのチームの資産になります。
-->

# よくある失敗パターン 3選

| # | 失敗パターン | 例 | 修正方法 |
|---|-------------|-----|---------|
| 1 | **曖昧な指示** | 「いい感じに要約して」 | 具体的な条件を書く |
| 2 | **形式の指定なし** | 指示だけで「どう返すか」未指定 | 「箇条書きで」「表で」を必ず書く |
| 3 | **文脈不足** | 背景情報を何も伝えない | 前提条件・@file参照で文脈を与える |

> この3つを避けるだけで、プロンプトの品質が **劇的に向上** する

<!--
プロンプトで一番よくある失敗を3つ紹介します。1つ目は曖昧な指示。「いい感じに」「適当に」はAIには伝わりません。2つ目は形式の指定なし。これが一番多い失敗で、「どう返すか」を書かないと結果が不安定になります。3つ目は文脈不足。AIはあなたのプロジェクトの背景を知らないので、必要な情報は@file参照などで伝えてください。この3つを意識するだけで、プロンプトの品質が飛躍的に向上します。
-->

# 📚 まとめ — プロンプト3要素の復習

```
文脈  → 素材・背景（@file参照で簡単に）
指示  → やってほしいこと（段階分解で明確に）
形式  → 結果の形式（箇条書き・表など）
```

**この章で学んだこと**:
1. プロンプトを文脈・指示・形式の3要素で整理する
2. @file参照・段階分解・箇条書き・slash commandの実践テクニック
3. 具体的に書くことで再現性を担保する

<!--
まとめです。プロンプトは文脈、指示、形式の3要素で構成します。素材と指示と形式を整理することで、AIの誤解を防ぎ、再現性を確保します。特別な記法は不要。自然な日本語で、具体的に、整理して書く。それがClaude Codeでのプロンプトの基本です。この基礎は、この後の章でもずっと使います。CLAUDE.mdを書く時も、hookを設計する時も、基本は「AIに確実に伝える」こと。その基礎がこの3要素です。
-->
