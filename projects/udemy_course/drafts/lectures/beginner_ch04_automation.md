# 「質問」から「作業」へ — AIの役割が変わる

> 前章まで: AIに質問する → AIが回答する
> **この章から: AIに作業を頼む → AIが完了まで実行する**

**何が違うのか**:

- 質問 → AIは1回だけ答える
- 作業 → AIが **複数ステップを自ら実行** して完走する
- ファイルを読み → 原因を分析 → 修正 → テスト → 結果報告

> 💡 この「自律的な一連の動き」が Claude Code の本領

<!--
これまでの章では、AIにプロンプトを書いて質問し、回答を得る使い方を学びました。この章からは、AIに「作業」を頼みます。質問との違いは、AIが複数のステップを自ら判断して実行する点です。ファイルを読み、原因を分析し、修正を適用し、テストで確認する。この一連の流れをAIが自律的に完走します。これこそがClaude Codeの本来の強みです。
-->

# AIの自律作業フロー

> AIは **4つのステップ** を繰り返して作業を進める

| ステップ | AIの動き | 例 |
|---------|---------|-----|
| **Read** | ファイルを読み込む | `@src/utils.ts` の内容を把握 |
| **分析** | 原因・課題を特定 | 「インデックスが範囲外」と判断 |
| **Edit** | 修正を適用 | コードを書き換える |
| **Test** | 結果を確認 | テストを実行して合格を確認 |

> このサイクルを **何度も繰り返す** ことで、AIは自律的に作業を完走する

<!--
AIが自律的に作業する時の基本パターンを4つのステップで示しました。まずReadで対象ファイルを読み込み、次に内容を分析して問題の原因を特定します。そしてEditでコードを修正し、最後にTestで結果を確認する。この4ステップのサイクルを何度も繰り返すことで、AIは人間の介入なしに作業を完走します。人間が逐一指示を出す必要はありません。「このバグを修正して」という1つの指示だけで、AIがすべてのステップを実行します。
-->

# Step 1: Read — AIがコードを読み込む

```
@src/utils.ts のバグを特定して。
各バグについて深刻度と修正方法を
箇条書きで教えて。
```

**AIの動き**:
- `@src/utils.ts` を **自動的に読み込む**
- ファイル全体の構造を把握
- 関数・変数・ロジックの流れを理解

> 第1章で学んだ @file参照がここで活きる — AIが自らファイルを開く

<!--
AIの最初の動きはReadです。「このファイルのバグを特定して」と頼むと、AIはまずファイルを自動的に読み込みます。第1章で学んだ@file参照がここで活きてきます。AIはファイル全体の構造を把握し、関数がどう連携しているか、変数がどう使われているかを理解してから次のステップに進みます。人間がコピペする必要はありません。AIが自らファイルを開いて読み込みます。
-->

---

# Step 2: 分析 — 原因を特定する

**AIの分析プロセス**:

1. コードの **意図** を理解する（何をすべきか）
2. 実際の動きとの **ギャップ** を見つける（何がおかしいか）
3. 原因を **仮説→検証** で絞り込む（なぜ起きているか）

**具体例**:
- 関数の戻り値が `undefined` の可能性に気づく
- 配列アクセスで範囲外を検出
- 条件分岐の漏れを発見

> AIは「どこがおかしいか」を **段階的に推理** する

<!--
Readで読み込んだコードをもとに、AIは分析を行います。まずコードが「何を意図しているか」を理解し、「実際の動きとどこが違うか」のギャップを見つけます。そして「なぜそのズレが起きているか」の原因を絞り込みます。具体例で言うと、関数の戻り値がundefinedになる可能性に気づいたり、配列アクセスでインデックスが範囲外になることを検出したりします。AIは推理小説の探偵のように、段階的に原因を特定していきます。
-->

---

# Step 3: Edit — 修正を適用する

**AIの修正プロセス**:

1. 修正方針を **人間に提示**（「ここをこう変えます」）
2. コードの該当箇所を **書き換え**
3. 修正が他の箇所に **影響しないか確認**

> 💡 AIは修正前に方針を示す — 人間が **事前に確認できる**

<!--
分析で原因が特定できたら、AIは修正に取り掛かります。重要なのは、AIがいきなりコードを書き換えるのではなく、まず「ここをこう変えます」と方針を示す点です。人間はその方針を見て、問題なければそのまま進め、納得できなければストップできます。修正後は、その変更が他の箇所に影響していないかも確認します。AIは慎重に、段階的に修正を進めます。
-->

---

# Step 4: Test — 結果を確認する

**AIのテストプロセス**:

1. テストを **自動実行** — `npm test` や `pytest` 等
2. 結果を確認 — 全て **PASS** すれば修正完了
3. **FAIL** があれば → Readに戻って再分析

```
テスト結果:
  ✅ test_add_normal    — PASS
  ✅ test_add_negative  — PASS
  ✅ test_add_zero      — PASS

3/3 passed — 修正完了
```

> AIはテストが落ちれば **自動的に再トライ** する

<!--
修正後、AIはテストを実行して結果を確認します。テストが全てPASSすれば修正完了です。もしFAILがあれば、AIはReadのステップに戻って再度分析し、別の修正を試みます。この「テストして、失敗したらやり直す」サイクルを自動で繰り返すのがAIの自律的な動きの核心です。人間が「もう一度やって」と言う必要はありません。AIが自らテストして自ら修正を繰り返します。
-->

---

# バグ修正の実例 — 全体の流れ

> 「このファイルのバグを3つ修正して」と頼んだ時の実際の流れ

**AIの動き** (時系列):

1. `@src/calc.ts` を読み込む → コード全体を把握
2. 「型エラー・範囲外・論理エラー」の3つを特定
3. 1つ目の修正を適用 → テスト実行 → **PASS**
4. 2つ目の修正を適用 → テスト実行 → **PASS**
5. 3つ目の修正を適用 → テスト実行 → **PASS**
6. 「3つのバグを全て修正しました」と報告

> 人間がやったこと: **最初の1行の指示だけ**

<!--
バグ修正の実例を時系列で示します。「このファイルのバグを3つ修正して」と頼むと、AIはまずファイルを読み込み、3つのバグを特定します。そして1つずつ修正してはテストを繰り返し、全てPASSしたら完了報告を返します。人間がやったことは、最初の1行の指示だけです。あとはAIがRead→分析→Edit→Testのサイクルを自律的に繰り返して完走しました。これがAIに作業を丸投げする体験です。
-->

# AIの修正が常に正しいとは限らない

**AIが犯す可能性のあるミス**:

| パターン | 具体例 | 影響 |
|---------|--------|------|
| **過剰修正** | バグ以外の部分まで書き換える | 意図しない動作変更 |
| **見落とし** | 関連する別のファイルの影響を無視 | 修正が不完全 |
| **前提の誤り** | プロジェクト固有のルールを知らない | 規約違反のコード |

> AIは **賢いが完璧ではない** — 出力を鵜呑みにしない

<!--
AIは非常に賢いですが、決して完璧ではありません。3つの典型的なミスを紹介します。1つ目は過剰修正で、バグ以外の部分まで余計に書き換えてしまうケースです。2つ目は見落としで、別のファイルとの関連を考慮せずに修正するため不完全になるケースです。3つ目は前提の誤りで、プロジェクト固有のコーディング規約や暗黙のルールを知らないため、規約に違反する修正をしてしまうケースです。AIの出力を鵜呑みにせず、必ず人間が確認する必要があります。
-->

---

# 人間のレビューが必須な理由 3選

**1. 文脈の理解が違う**

AIはコードは読めるが **ビジネス要件** は知らない。「この機能は来月廃止予定」のような情報は人間しか知らない。

**2. 影響範囲の見えない部分がある**

AIが読めるのは指定したファイルだけ。**他システムとの連携** や **運用上の制約** は人間の領域。

**3. 「動けばOK」は危険**

テストが通っても、**意図通りに動くか** は別問題。仕様との整合性は人間が判断する。

> ✅ AIが作業 → **人間が確認** → このペアが正解

<!--
人間のレビューが必須な理由を3つ挙げます。1つ目、AIはコードは理解できても、プロジェクトのビジネス要件や背景事情までは知りません。「この機能は来月廃止予定だから触らないで」というような情報は、人間の頭の中にしかありません。2つ目、AIが読めるのは指示されたファイルだけで、他のシステムとの連携や運用上の制約は見えません。3つ目、テストが通ったからといって、仕様通りに動くとは限りません。AIが作業して人間が確認する、このペアこそが正しい使い方です。
-->

# CLAUDE.mdとの連携 — 設定が精度を左右する

> 前章で作った CLAUDE.md が、自動化の品質を大きく左右する

**CLAUDE.md あり/なしの違い**:

- **なし**: AIがプロジェクトのルールを知らない → 規約違反の修正をする
- **あり**: AIがコーディング規約・使用ライブラリを理解 → **プロジェクトに合った修正**

**効果的な記述例**:

```
このプロジェクトでは:
- TypeScript + vitest を使用
- テストは正常系・異常系・境界値の3パターン
- any型は禁止
```

> CLAUDE.md = AIに渡す **作業マニュアル**

<!--
前章で学んだCLAUDE.mdが、自動化の精度に直結します。CLAUDE.mdがない場合、AIはプロジェクト固有のルールを知らないため、規約に合わない修正をする可能性があります。CLAUDE.mdがあると、使用言語やテストフレームワーク、コーディング規約をAIが理解した上で修正を行います。「このプロジェクトではany型禁止」「テストは3パターン書く」といったルールをCLAUDE.mdに書いておけば、AIはそれに従います。CLAUDE.mdは、AIに渡す作業マニュアルです。
-->
