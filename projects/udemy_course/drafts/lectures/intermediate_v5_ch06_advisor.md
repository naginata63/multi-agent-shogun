# ch06. コンテキストを膨らませず別 AI の判断を取り込む — advisor

## この章で解決する「困りごと」

> **⑦ AI 出力を盲信して失敗した**

- AI が「完璧です。自信を持って提出できます」と言った
- そのまま提出したら、数字が間違っていて大恥をかいた
- **自分の AI に「これで合ってる?」と聞いても「はい合ってます」と返ってくるだけ**

**できる 1 動作:**
> 3 つの方法 (subagent / advisor / 別チャット) を context 流入量の観点で比較し、状況に合った方法を選べる

## 「盲信」で失敗した 3 つの実例

### 例① 数字の間違い
- AI に売上レポートを作成させた
- 「前年比 15% 増」という結論をそのまま報告
- 実は計算元のデータに欠損値が含まれていて、**本当は 3% 増** だった

### 例② 存在しない情報の引用
- AI が「〇〇省のガイドラインによれば…」と引用
- 提出先から「そのガイドライン、存在しませんよ」と指摘

### 例③ 一見正しそうなコード
- AI に書かせたコードがテストは通る
- でも本番で特定の条件下だけバグが発動 → 大騒ぎに

## なぜ「自分の AI に確認」は意味がないか

```
┌─────────────────┐     「これで合ってる?」     ┌─────────────────┐
│  あなたの AI     │  ──────────────────────→   │  同じ AI が      │
│  (A という偏り)  │                             │  「はい合ってます」│
└─────────────────┘                             └─────────────────┘
        ↑                                                │
        └──── 同じ偏りで再確認しているだけ ──────────────┘
```

- 同じ AI に確認 = **同じ偏りで再確認するだけ**
- 自己肯定しかしない → 客観的な品質チェックにならない
- **別の視点** からのレビューが必要

## ch03/ch04 の復習 — context window を節約する原則

- **ch03**: context window には上限がある → 大量テキストを詰め込むと重要情報が押し出される
- **ch04**: 長文を作らない → 必要な情報だけを必要なときに読み込む

> この「**長文を作らない**」原則は、
> 別の AI に見てもらう方法の選び方にも適用できる

---

<!--
スピーカーノート:
ここが本章の目玉。「別AIに見てもらう」には3つの実装方法があります。subagent・advisor・別チャット貼付の3つを並列提示します。まだ詳細は言わず「こういう選択肢がある」という紹介にとどめます。受講者に「違いは何？」と気になってもらうのが狙いです。
-->

## 別の AI に見てもらう — 3 つの実装

| 方法 | 概要 | ツール依存 |
|------|------|-----------|
| **① subagent** | 別の AI セッションを立ち上げて全文を渡す | Task tool 等 |
| **② advisor** | API に登録するだけで別 AI が自動レビュー | Claude API beta tool |
| **③ 別チャット貼付** | 手動で別チャットに成果物を貼り付ける | なし (ツール非依存) |

> どれも「別の AI に見てもらう」という目的は同じ
> → **何が違うのか？** 次のスライドで比較

---

<!--
スピーカーノート:
比較表の核心スライド。主軸は「親contextへの流入量」です。subagentは全文が親contextに戻ってくる(重い)、advisorは400〜700トークン程度の指摘だけが戻る(軽い)、別チャットは手動なので流入なし。ch04の「長文を作らない」原則と整合するのはどれか？→advisor。ここを強調してください。
-->

## 3 手法の比較 — 「context 流入量」で選ぶ

| 方法 | context 流入量 | 特徴 |
|------|---------------|------|
| **subagent** | 全文 (親 context が肥大) | 確実だが重い・ch04「長文を作らない」原則に違反しうる |
| **advisor** | 指摘のみ (400〜700 tok) | 軽量・ch04 の原則と整合 |
| **別チャット貼付** | なし (手動) | ツール非依存・貼り忘れやミスの余地あり |

**ch04 の原則「長文を作らない」と整合するのは advisor**

```
subagent:  全文が context に戻る → 肥大リスク
advisor:   指摘だけが context に戻る → 節約
別チャット: 手動で貼る → 確実性に課題
```

## advisor の正確な仕様 (公式ドキュメント準拠)

- **Claude API 公式 beta tool** (`type: advisor_20260301`)
- beta header: `advisor-tool-2026-03-01` が必要
- **server-side tool** — AI が `server_tool_use` block を emit して呼び出す
- 引数なし — 会話履歴は server が自動転送
- 手元で動かすには: **Claude API + beta header + tools 配列登録** が必要

```json
// tools 配列に追加するだけ
{
  "type": "advisor_20260301",
  "name": "advisor"
}
```

> **注意**: チャット欄に打ち込むものではありません
> API 経由で tools に登録することで有効化される仕組みです

## advisor の仕組み — 指摘だけが context に戻る

```
┌───────────────┐    会話全文    ┌───────────────┐
│  メイン AI     │ ──────────→ │  別の AI       │
│  (作業担当)    │              │  (server-side) │
└───────────────┘              └───────┬───────┘
        ↑                              │
        └── 指摘だけ (400〜700 tok) ←─┘
            (全文は戻らない = context 節約)
```

1. AI が作業を終えたタイミングで「advisor を呼ぶべきか」を自律判断
2. 呼ぶと決めたら、会話内容が別の AI に自動転送される
3. 別の AI が指摘事項を返す → **指摘だけ**が context に戻る

## 呼び出しタイミング 3 パターン

| タイミング | 目的 | 例 |
|-----------|------|-----|
| **着手前** | 計画の妥当性レビュー | 「この方針で進める前に見て」 |
| **着手後** | 成果物の品質レビュー | 「出来上がったものを確認して」 |
| **詰まったとき** | 方針の見直しレビュー | 「別のアプローチはないか」 |

一番使うのは **着手後** — できたものを第三者に見てもらう

---

<!--
スピーカーノート:
完了条件の重要性を説明。基準なしの「見てください」だと、レビューが表面的になりがち。基準ありだと具体的な観点で深く見てくれる。この差は大きいです。Before/Afterを見せて実感してもらってください。
-->

## 完了条件を明確にする (acceptance_criteria)

> advisor に「何を基準に見てほしいか」を伝える技術

**Before (基準なし):**
> 「このレポート、見てください」

**After (基準あり):**
> 「このレポートを以下の基準で見てください:
> - データの鮮度 (最新データを使っているか)
> - 数字の正確性 (計算に間違いがないか)
> - 結論と根拠の整合性 (飛躍がないか)」

基準を渡すと、レビューの精度が上がる

## 任意発動 vs 強制発動 — advisor と Hooks の使い分け

| | advisor | Hooks (ch07 で解説) |
|---|---------|---------------------|
| **発動** | AI が自律判断 (skip あり) | **強制発動** (skip なし) |
| **確実性** | 確実ではない | 確実 |
| **向いている場面** | 軽いレビュー・方針確認 | 絶対に通さないといけない検査 |

> **「advisor は AI が skip したら結局確認されないじゃないか」**
> → その通り。だからこそ ch07 で **Hooks (強制発動)** を学ぶ

**使い分けの指針**:
- 確実性が要る品質ゲート → **Hooks** (ch07)
- あればよい軽いレビュー → **advisor**

## 実例 — 営業の提案書レビュー

### 現象
- AI に提案書を作成させた

- 提出先から「このデータ、古くないですか」と指摘

### advisor で解決
```
> 提案書が完成しました。advisor で見てください。
> 基準: データの鮮度、数字の正確性
```

→ 別の AI が「競合スペック表の数字が、調べた範囲と一致しません」と指摘
→ 提出前に修正できた
