## この章で解決する「困りごと」

> **⑦ AI 出力を盲信して失敗した**

- AI が「完璧です。自信を持って提出できます」と言った
- そのまま提出したら、数字が間違っていて大恥をかいた
- **自分の AI に「これで合ってる?」と聞いても「はい合ってます」と返ってくるだけ**

---

## 章のゴール

**言える 1 文:**
> 自分の AI の出力を別の AI に見直してもらう習慣をつければ、盲信によるミスを防げる

**できる 1 動作:**
> advisor ツールを設定しておけば、AI が作業後に自律的に第三者レビューを呼べる (ただし skip もありうる)

## 「盲信」で失敗した 3 つの実例

### 例① 数字の間違い
- AI に売上レポートを作成させた
- 「前年比 15% 増」という結論をそのまま報告
- 実は計算元のデータに欠損値が含まれていて、**本当は 3% 増** だった

### 例② 存在しない情報の引用
- AI が「〇〇省のガイドラインによれば…」と引用
- 提出先から「そのガイドライン、存在しませんよ」と指摘

### 例③ 一見正しそうなコード
- AI に書かせたコードがテストは通る
- でも本番で特定の条件下だけバグが発動 → 大騒ぎに

---

## なぜ「自分の AI に確認」は意味がないか

```
┌─────────────────┐     「これで合ってる?」     ┌─────────────────┐
│  あなたの AI     │  ──────────────────────→   │  同じ AI が      │
│  (A という偏り)  │                             │  「はい合ってます」│
└─────────────────┘                             └─────────────────┘
        ↑                                                │
        └──── 同じ偏りで再確認しているだけ ──────────────┘
```

- 同じ AI に確認 = **同じ偏りで再確認するだけ**
- 自己肯定しかしない → 客観的な品質チェックにならない
- **別の視点** からのレビューが必要

## advisor — Claude API 公式 beta tool で別の AI に見てもらう

> 社内文書を提出する前に、上司や同僚に一回見てもらうのと同じ
> 自分が作ったものを **別の人** に見てもらう

```
┌───────────────┐    成果物    ┌───────────────┐
│  メイン AI     │ ──────────→ │  別の AI       │
│  (作業担当)    │             │  (レビュー担当) │
└───────────────┘             └───────┬───────┘
                                      │
                              問題点を指摘して返す
```

- メイン AI: 実際の作業をする
- **advisor**: Claude API の tools 配列に登録する公式 beta tool
- **別の独立した AI** が「問題ないか」を検査 — ただし AI が **skip することもある** (確実な品質ゲートではない)
- 別の AI = **別の判断基準** で見る → 同じ偏りによる見逃しを防ぐ

## advisor の仕組み — API tools 配列に登録するだけ

Claude API の **tools 配列** に 1 つ登録するだけで有効化:

```json
// tools 配列に追加
{
  "type": "advisor_20260301",
  "name": "advisor"
}
```

登録すると、AI が **自律判断で呼び出す**:

1. AI が作業を終えたタイミングで「advisor を呼ぶべきか」を自律判断
2. 呼ぶと決めたら、作業内容が別の AI に渡される
3. 別の AI が指摘事項を返す

> **重要**: AI が skip することもある — **確実な品質ゲートではない**
> 確実にチェックを通したい場面では、ch07 の **Hooks** を使う

---

## 呼び出しタイミング 3 パターン

| タイミング | 目的 | 例 |
|-----------|------|-----|
| **着手前** | 計画の妥当性レビュー | 「この方針で進める前に見て」 |
| **着手後** | 成果物の品質レビュー | 「出来上がったものを確認して」 |
| **詰まったとき** | 方針の見直しレビュー | 「別のアプローチはないか」 |

一番使うのは **着手後** — できたものを第三者に見てもらう

---

## 完了条件を明確にする (acceptance_criteria)

> advisor に「何を基準に見てほしいか」を伝える技術

**Before (基準なし):**
> 「このレポート、見てください」

**After (基準あり):**
> 「このレポートを以下の基準で見てください:
> - データの鮮度 (最新データを使っているか)
> - 数字の正確性 (計算に間違いがないか)
> - 結論と根拠の整合性 (飛躍がないか)」

基準を渡すと、レビューの **精度が劇的に上がる**

---

## 実例 — 営業の提案書レビュー

### 現象
- AI に提案書を作成させた
- 競合製品のスペック比較表の数字が **2 年前のもの** だった
- 提出先から「このデータ、古くないですか」と指摘

### advisor で解決
```
> 提案書が完成しました。advisor で見てください。
> 基準: データの鮮度、数字の正確性
```

→ 別の AI が「競合スペック表の数字が、調べた範囲と一致しません」と指摘
→ 提出前に修正できた

---

## 実例 — PM の仕様書レビュー

### 現象
- AI に仕様書ドラフトを書かせて、そのままエンジニアに共有
- 後で「**非機能要件が一切書かれていない**」と指摘 → 手戻り

### advisor で解決
```
> 仕様書ドラフトができました。advisor で見てください。
> 基準: 非機能要件の記載 (セキュリティ/性能/可用性)
```

→ 別の AI が「可用性要件 (SLA) の記載がありません」と指摘
→ エンジニアに渡す前に修正

---

## 実例 — データ分析の結果レビュー

### 現象
- AI に売上分析を頼んだ
- 「前年比 15% 増」という結論をそのまま報告
- 計算元のデータに **欠損値** が含まれていて、実質 3% 増だった

### advisor で解決
```
> 分析結果が出ました。advisor で見てください。
> 基準: データの完全性、欠損値の扱い
```

→ 別の AI が「欠損値の除外処理が不十分です」と指摘
→ 正確な分析に修正してから報告

## ch06 まとめ

**言える 1 文:**
> advisor ツールで AI に第三者レビューを任せれば、盲信によるミスを減らせる (ただし AI が skip することもある)

**advisor と Hooks の使い分け:**
> advisor = AI が **自律判断で呼ぶ** (skip あり)
> Hooks = **強制発動** (skip なし) — ch07 で解説
> 確実性が要る場面では Hooks が筋
