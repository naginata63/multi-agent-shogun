# プロンプトは「上手く書く」だけでは足りない

> 医者が病気を見分けるように、あなたはプロンプトの病気を見分けられるようになる

- 上手く書く技術 = **建設**の技術
- 壊れ方を見抜く技術 = **診断**の技術
- この2つが揃って初めて、**品質が安定**する

> 「上手くいった」は品質保証ではない — 「どこが壊れるか」を知ることこそが品質保証

<!--
ここで重要な考え方を提示します。プロンプトを上手く書く技術だけでは不十分です。医者に例えると、健康な人を作る技術と、病気を見つける技術は別物です。プロンプトも同じで、良いプロンプトを書く力と、悪いプロンプトのどこが悪いかを診断する力は別のスキルです。この章では後者、診断の力を鍛えます。前章で学んだ関数プロンプトの設計が、どこで壊れるのか。その壊れ方のパターンを知ることで、自分のプロンプトの品質を客観的に評価できるようになります。
-->

# 診断チェックリスト (1/2)

> プロンプトをこの10項目で診断する。**この章では特に 1, 2, 3, 7 に注目**してください。

| # | チェック項目 |確認すること |
|---|-------------|------------|
| 1 | **目的の明示** | 何をしてほしいかが1文で言えるか |
| 2 | **入力の明示** | 対象データ・ファイルが特定できているか |
| 3 | **出力形式の指定** | 箇条書き・表・コードなど形式が書いてあるか |
| 4 | **前提条件の記述** | 環境・バージョン・制約が書かれているか |
| 5 | **範囲の限定** | 「何を含めないか」が書いてあるか |

<!--
プロンプトの品質を診断するチェックリストを10項目紹介します。最初の5項目です。1つ目、目的の明示。何をしてほしいかが1文で言えるかを確認します。2つ目、入力の明示。対象となるデータやファイルが特定できているか。3つ目、出力形式の指定。結果を箇条書きで返すのか表で返すのか、形式が書いてあるか。4つ目、前提条件の記述。どの環境で、どのバージョンで、どんな制約の下で動くのか。5つ目、範囲の限定。これは意外と見落とされがちですが、「何を含めないか」を明示することで、AIが余計なことをしなくなります。
-->

# 診断チェックリスト (2/2)

| # | チェック項目 | 確認すること |
|---|-------------|------------|
| 6 | **一意性** | 解釈の余地が2つ以上ないか |
| 7 | **具体性** | 「いい感じに」等の曖昧語が含まれていないか |
| 8 | **段階の明示** | 複数ステップなら順序が書かれているか |
| 9 | **失敗時の指示** | エラー・不明時の対応方法が書かれているか |
| 10 | **再現性の確認** | 同じプロンプトで同じ結果が得られる構造か |

> 10項目全てYESなら、プロンプトは「健康」

<!--
残り5項目です。6つ目、一意性。プロンプトの解釈が1通りに定まるか。2通り以上の読み方ができるなら危険信号です。7つ目、具体性。「いい感じに」「適当に」「なんとなく」といった曖昧語が含まれていないか。8つ目、段階の明示。複数のステップがある場合、順序が書かれているか。9つ目、失敗時の指示。エラーが出た時や、AIが判断に迷った時の対応方法が書かれているか。10つ目、再現性の確認。同じプロンプトを何度使っても同じ品質の結果が得られる構造になっているか。この10項目全てにYESと答えられるなら、プロンプトは健康です。
-->

# 失敗解剖 ① 曖昧な指示

**症状**: 実行するたびに異なる結果が返る

```
このプロジェクトのコードを整理して
```

**原因**: 「整理」の範囲が無限大 — フォーマット? リファクタ? 不要ファイル削除?

**修正後**:
```
@src/utils.ts の関数をアルファベット順に
並べ替えて。import文は先頭にまとめること。
```
（`@src/utils.ts` は「このファイルを読んで」という指定方法。冒頭の用語集も参照）

**結果**: 毎回同じ並び順で安定出力

<!--
1つ目の失敗パターンは「曖昧な指示」です。症状は、実行するたびに異なる結果が返ること。「このプロジェクトのコードを整理して」というプロンプト。原因は「整理」の範囲が無限大なことです。コードフォーマットを整えるのか、リファクタリングするのか、不要なファイルを削除するのか、AIには判断できません。修正後は「@src/utils.ts の関数をアルファベット順に並べ替えて。import文は先頭にまとめること」と具体化しました。結果として、毎回同じ並び順で安定した出力が得られます。チェックリストで言うと、項目1の目的の明示と項目7の具体性に該当する失敗です。
-->

# 失敗解剖 ② 文脈不在

**症状**: 回答が的外れ、または一般的すぎる

```
バグを修正して
```

**原因**: どのファイル・どのバグ・どんなエラーメッセージか全く不明

**修正後**:
```
@src/auth/login.ts で
TypeError: Cannot read property 'email' of undefined
（＝ユーザー情報が空っぽの状態で email を取り出そうとしてエラーになっている）
が発生している。
user オブジェクトの null チェックを追加して。
```

**結果**: 的確な修正コードを1回で返す

<!--
2つ目の失敗パターンは「文脈不在」です。症状は回答が的外れ、または一般的すぎること。「バグを修正して」とだけ言われても、どのファイルの、どのバグで、どんなエラーメッセージが出ているのか全く不明です。AIは親切なので何かしらの回答を返しますが、的外れである可能性が高いです。修正後は「@src/auth/login.ts で TypeError が発生している。user オブジェクトの null チェックを追加して」と具体的に書きました。ファイル名、エラーメッセージ、期待する修正内容の3つが揃うことで、AIは的確な修正コードを1回で返せるようになります。チェックリストの項目2の入力の明示と項目4の前提条件に該当する失敗です。
-->

---

# 失敗解剖 ③ 形式指定なし

**症状**: 出力が長すぎる、短すぎる、または使えない形式

```
会議の議事録をまとめて
```

**原因**: 「まとめる」の形式が未定義 — 500字の文章? 箇条書き? 決定事項だけ?

**修正後**:
```
今日の全体会議の議事録を、
以下の3セクションでまとめて：
①決定事項（箇条書き3〜5項目）
②宿題（担当者名つき）
③次回日程
全体で500字以内。
```

**結果**: 常に同じ構造の議事録サマリが得られる

<!--
3つ目の失敗パターンは「形式指定なし」です。症状は出力が長すぎたり短すぎたり、使えない形式だったりすること。「会議の議事録をまとめて」というプロンプト。原因は「まとめる」の形式が未定義なことです。500字の文章でまとめるのか、箇条書きにするのか、決定事項だけを抜き出すのか。AIには分かりません。修正後は「決定事項・宿題・次回日程の3セクション、500字以内」と形式を明確に指定しました。これで常に同じ構造の議事録サマリが得られます。チェックリストの項目3の出力形式の指定に該当する失敗で、実務で最も頻度の高い失敗パターンです。
-->

# 修正前後の比較

| 観点 | 修正前 | 修正後 |
|------|--------|--------|
| **指示** | 「整理して」「修正して」「まとめて」 | 具体的な動作を明記 |
| **入力** | なし（対象が不明） | ファイル名や会議名を明示 |
| **形式** | なし（AIが勝手に決める） | 箇条書き・構造を指定 |
| **再現性** | 毎回異なる | 毎回同じ品質 |

> 共通パターン: **指示・入力・形式** の3要素が欠けている

<!--
3つの失敗パターンを横並びで比較しました。修正前は「整理して」「修正して」「まとめて」と動詞だけ。修正後は具体的な動作を明記しています。入力を見ると、修正前は何もなく、対象が分からない。修正後はファイル名や会議名で明示しています。形式は、修正前はAIが勝手に決める状態。修正後は箇条書きや文字数や構造を指定しています。結果として再現性が毎回異なるから毎回同じ品質に変わります。3つの失敗に共通するパターンは、前章で学んだ3要素、指示・入力・形式が欠けていることです。
-->

# 再発防止の設計原則 3点

**原則1: 具体化テスト**
> プロンプト内の各語句に「他の解釈は可能か?」と問う。YESなら書き直す

**原則2: 3要素セルフチェック**
> 投稿前に「入力・指示・形式」の3つが全て書かれているか確認する

**原則3: 再実行テスト**
> 同じプロンプトを2回実行し、結果の構造が一致するか確認する
> （AIは確率的に出力するため完全一致は稀ですが、構造化されていれば出力のブレは小さくなります）

> この3点を習慣化すれば、多くの失敗プロンプトを予防できる

<!--
3つの失敗から導き出した再発防止の設計原則です。原則1、具体化テスト。プロンプト内の各語句に「他の解釈は可能か」と問いかけます。YESなら書き直します。例えば「整理して」は他の解釈が可能なのでNG。「アルファベット順に並べ替えて」なら解釈は1通りなのでOK。原則2、3要素セルフチェック。プロンプトを送信する前に、入力・指示・形式の3つが全て書かれているか確認します。原則3、再実行テスト。同じプロンプトを2回実行し、結果の構造が一致するか確認します。構造がブレるなら、どこかに曖昧さが残っています。この3点を習慣化すれば、多くの失敗プロンプトを予防できます。
-->
