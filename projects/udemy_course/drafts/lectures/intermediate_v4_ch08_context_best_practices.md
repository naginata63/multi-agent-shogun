# 「コンテキストは設定して終わりではない」

> 「CLAUDE.mdを書いたから大丈夫」 — この思い込みが生産性を下げる

- コンテキストは **生き物** です。プロジェクトが変われば内容も変わる
- **放置されたコンテキスト** は、古い前提をAIに注入する毒になる
- 日々の運用で **読み・書き・使い回し** を続けることが鍵

> 「設定」は1回きり。「運用」は毎日。この差がAI開発の生産性を決める

<!--
ここで重要な考え方を提示します。「コンテキストは設定して終わりではない」ということです。CLAUDE.mdを書いたから大丈夫、という思い込みは危険です。コンテキストは生き物で、プロジェクトが進むにつれて内容を更新し続ける必要があります。放置されたコンテキストは、古い前提や存在しないファイルパスをAIに注入してしまい、かえって品質を下げます。設定は1回きりの作業ですが、運用は毎日の作業です。この差が、AI開発の生産性を決定づけます。
-->

# 読込効果の測定 — 何が効いているかを知る

**3層の読込タイミングと効果**:

| 層 | 読込タイミング | 測定方法 |
|----|---------------|---------|
| `CLAUDE.md` | セッション開始時 | 同名プロジェクトでの一貫性確認 |
| `instructions/` | セッション開始時 | ルール違反の発生率カウント |
| `memory/` | 毎セッション | 過去の教訓が活用された回数 |

**測定のポイント**:
- AIが **ルールを守っているか** = instructions効果の指標
- AIが **過去の失敗を回避しているか** = memory効果の指標
- AIが **プロジェクト固有の前提を理解しているか** = CLAUDE.md効果の指標

<!--
コンテキストの3層がそれぞれいつ読み込まれ、どう効果を測るかを解説します。CLAUDE.mdはセッション開始時に読み込まれます。効果の測り方は、同じプロジェクトで作業させた時に一貫性が保たれているかを確認することです。instructionsもセッション開始時に読み込まれます。効果はルール違反の発生率で測ります。memoryは毎セッション読み込まれ、過去の教訓が活用された回数で効果を測ります。重要なのは、AIがルールを守っているか、過去の失敗を回避しているか、プロジェクト固有の前提を理解しているかの3点を日常的に観察することです。

# 不要コンテキスト削減 5原則 (1/2)

> コンテキストを増やすより、減らす方が効果的

**原則1: 重複排除**
> 同じ情報がCLAUDE.mdとinstructionsの両方にある → 1箇所に集約する

**原則2: 期限切れ削除**
> 3ヶ月以上更新されていないmemory → 内容がまだ有効か確認し、無効なら削除

<!--
コンテキストを削減する5つの原則を紹介します。前半2つです。原則1、重複排除です。同じ情報がCLAUDE.mdとinstructionsの両方に書かれていると、更新時にどちらかを忘れる原因になります。1箇所に集約しましょう。原則2、期限切れ削除です。3ヶ月以上更新されていないmemoryファイルは、中身がまだ有効か確認が必要です。存在しないファイルパスや古いバージョンの前提が含まれている可能性が高いです。確認して無効なら削除します。
-->

# 不要コンテキスト削減 5原則 (2/2)

**原則3: 推論可能な情報は書かない**
> コードから読み取れることはCLAUDE.mdに書かない — ファイル構造・命名規則等

**原則4: 使わないプロジェクトのコンテキストを残さない**
> 終了したプロジェクト固有のルール → アーカイブまたは削除

**原則5: 1ファイル100行以内**
> 100行を超えるファイルは分割を検討 — 読込コストが増え、AIの理解も低下

> 5原則を1週間に1回見直すことで、コンテキストの健康状態を維持できる

<!--
残り3つの原則です。原則3、推論可能な情報は書かない。コードから読み取れること、例えばファイル構造や命名規則はCLAUDE.mdに書く必要はありません。AIがコードを読めば分かるからです。原則4、使わないプロジェクトのコンテキストを残さない。終了したプロジェクト固有のルールはアーカイブするか削除しましょう。残っていると無駄な読込コストになります。原則5、1ファイル100行以内。100行を超えると読込コストが増え、AIの理解も低下します。分割を検討しましょう。これら5原則を1週間に1回見直すことで、コンテキストの健康状態を維持できます。
-->

# セッション再開パターン 3選 — rehydrate活用法

**パターン1: 日次再開 (朝イチ)**
```
/rehydrate
```
- 前日の進捗をサマリから復元
- CLAUDE.md + instructions は自動読込 → すぐ作業再開

**パターン2: タスク切り替え**
```
/rehydrate → 新しいタスクの指示
```
- 現在のコンテキストをクリアして別タスクへ
- 不要な前タスク情報を引き継がない

**パターン3: コンテキスト圧縮後の復元**
- auto-compact後に `/clear` → `/rehydrate`
- 圧縮で失われた詳細をサマリから復元

<!--
rehydrateコマンドを使った3つのセッション再開パターンを解説します。パターン1は日次再開です。翌朝、rehydrateを実行するだけで前日の進捗をサマリから復元できます。CLAUDE.mdとinstructionsは自動的に読み込まれるので、すぐに作業を再開できます。パターン2はタスク切り替えです。rehydrateで現在のコンテキストをクリアし、新しいタスクの指示に切り替えます。前のタスクの不要な情報を引き継がないことがポイントです。パターン3はコンテキスト圧縮後の復元です。auto-compactが実行された後に/clearしてrehydrateすると、圧縮で失われた詳細をサマリから復元できます。
-->

# コンテキストROI計算 — 読む価値があるかを測る

**ROI = (生産性向上) / (読込コスト)**

| 指標 | 測り方 | 目安 |
|------|--------|------|
| 読込コスト | ファイルのトークン数 | 1ファイル 500〜2,000トークン |
| 生産性向上 | 指示の再説明が不要になった回数 | 1セッションで3回以上なら有効 |
| ROI判定 | 向上 / コスト | ROI > 1.0 なら継続、< 0.5 なら削除検討 |

**実践的な測定手順**:
1. 現在のmemoryファイルのトークン数を把握する
2. 1週間、AIがmemoryの内容を活用した場面をカウントする
3. カウント0のmemoryは削除候補

<!--
コンテキストのROI、投資対効果を計算する方法を解説します。ROIは生産性向上を読込コストで割ったものです。読込コストはファイルのトークン数で測ります。1ファイルあたり500から2,000トークンが目安です。生産性向上は、指示の再説明が不要になった回数で測ります。1セッションで3回以上AIが自発的にコンテキストを活用していれば有効です。ROIが1.0を超えていれば継続、0.5を下回っていれば削除を検討します。実践的な測定手順は、まず現在のmemoryファイルのトークン数を把握し、1週間AIがmemoryの内容を活用した場面をカウントし、カウント0のmemoryは削除候補とします。
-->

# チーム運用 — 共有 vs 個人の境界線

> 共有しすぎると「誰の責任か」が曖昧になる。分けすぎるとノウハウが分散する

| 種類 | 配置場所 | 更新頻度 | 例 |
|------|---------|---------|-----|
| **共有** | `CLAUDE.md` | プロジェクト変更時 | コーディング規約・禁止操作 |
| **チーム** | `instructions/` | フェーズ変更時 | レビュー手順・テスト方針 |
| **個人** | `memory/` | 毎日 | 個人の教訓・好み・学習メモ |

**境界線のルール**:
- **全員が守るべきルール** → 共有 (`CLAUDE.md`)
- **役割ごとの手順** → チーム (`instructions/`)
- **個人の経験則** → 個人 (`memory/`)

<!--
チームでのコンテキスト運用では、共有と個人の境界線を明確にすることが重要です。3つのレベルがあります。まず共有レベル。CLAUDE.mdに配置し、全員が守るべきルールを書きます。コーディング規約や禁止操作などです。プロジェクトが変わる時に更新します。次にチームレベル。instructionsディレクトリに配置し、役割ごとの手順を書きます。レビュー手順やテスト方針などです。フェーズが変わる時に更新します。最後に個人レベル。memoryディレクトリに配置し、個人の経験則を書きます。毎日更新します。境界線のルールは、全員が守るべきルールは共有、役割ごとの手順はチーム、個人の経験則は個人です。共有しすぎると責任が曖昧になり、分けすぎるとノウハウが分散します。

# 確認問題

**Q1**: コンテキスト削減5原則のうち、「コードから読み取れる情報は書かない」は何原則か。また、なぜそれが重要なのかを説明せよ。

**Q2**: rehydrateを使ったセッション再開パターンを3つ挙げ、それぞれの使い所を説明せよ。

**Q3**: チーム運用で「共有コンテキスト」と「個人コンテキスト」の境界線をどのように引くべきか、その理由とともに答えよ。

<!--
確認問題です。Q1は削減5原則に関する理解を問います。Q2はrehydrateの実践的な使い方を問います。Q3はチーム運用の設計思想を問います。各問題はこの章で学んだ内容の核となる概念です。じっくり考えてみてください。
-->

---

**解答**:

**A1**: 原則3「推論可能な情報は書かない」。AIがコードを読めば分かることをコンテキストに書くと、二重管理になり、コード変更時にコンテキストの更新漏れが発生するため。

**A2**: (1) 日次再開: 翌朝に前日の進捗を復元する場面 (2) タスク切り替え: 別のタスクに移る際に前タスクの情報を引き継がない場面 (3) 圧縮後復元: auto-compactで失われた詳細を復元する場面。

**A3**: 全員が守るべきルールは共有コンテキスト(CLAUDE.md)に、個人の経験則は個人コンテキスト(memory/)に配置する。理由は、共有範囲を広げすぎると更新責任が曖昧になり、逆に狭めすぎるとノウハウが分散してチーム全体の品質が安定しないため。

<!--
確認問題の解答です。Q1、原則3「推論可能な情報は書かない」です。AIがコードを読めば分かることをコンテキストに書くと、二重管理になります。コードが変更された時にコンテキストの更新が漏れ、かえって誤情報を注入する原因になります。Q2、3つのパターンは、日次再開、タスク切り替え、圧縮後復元です。Q3、境界線の引き方です。全員が守るべきルールは共有に、個人の経験則は個人に配置します。共有しすぎると更新責任が曖昧になり、分けすぎるとノウハウが分散します。
