# 4つの戦略 — Isolateの位置づけ

> 前章までで3つの戦略を学びました。この章は4番目です。

| 戦略 | やること | 対応章 |
|------|---------|--------|
| **Write** (書き込み) | AIに正しい情報を書き込む | 第5章 |
| **Select** (選択) | 必要な情報だけを取り出す | 第6章 |
| **Compress** (圧縮) | 情報を圧縮して容量を節約する | 第7章 |
| **Isolate** (分離) | 情報を分離して干渉を防ぐ | **本章 (第8章)** |

**Isolate戦略の本質**:
- 異なるタスクの情報が **混ざる** と、AIが混乱する
- タスクAの前提がタスクBに **漏れ出る** のを防ぐ
- 「部屋を分ける」ことで、それぞれの部屋の **整合性** を保つ

> Isolate = 「全部屋に荷物を広げる」のではなく「部屋ごとに整理する」

<!--
Isolate戦略は4つの戦略の4つ目です。Writeが書き込み、Selectが選択、Compressが圧縮なら、Isolateは分離による管理です。異なるタスクの情報が混ざると、AIがタスクAの前提をタスクBに適用してしまい、混乱が生じます。部屋の例えで言うと、全部屋に荷物を広げるのではなく、部屋ごとに整理するイメージです。
-->

# 「コンテキストの干渉」が起きる瞬間

> タスクA: 「Excelの売上マクロを修正中」
> タスクB: 「PowerPointの営業資料を作成中」
> **同じセッションで両方やるとどうなるか？**

**干渉の症状**:
- AIが **Excelの数式** をPowerPoint資料に書き込んでしまう
- タスクAの **未解決課題** がタスクBの判断に影響する
- AIの記憶容量が **両タスクの情報で埋まる**

**Isolate戦略の解決策**:
- タスクごとに **独立したコンテキスト** を維持する
- 物理的な分離 (別セッション) と 論理的な分離 (Context Window Budget)

<!--
コンテキストの干渉が起きる典型的な場面です。Excelのマクロ修正とPowerPointの資料作成を同じセッションで行うと、AIがExcelの数式をPowerPointに書き込んでしまったり、タスクAの未解決課題がBの判断に影響したりします。Isolate戦略の解決策は、タスクごとに独立したコンテキストを維持すること。物理的な分離として別セッションを使う方法と、論理的な分離としてContext Window Budgetを使う方法があります。
-->

# Context Window Budget — タスク別の「財布」設計

> AIの記憶容量を「財布」として管理する。タスクごとに別の財布を持つイメージ

**予算配分の考え方**:

> AIの記憶容量（コンテキストウィンドウ）は約 **10万〜20万トークン**（日本語で約7万〜15万字＝本1〜2冊分）。この限られた容量をどう配分するかが設計の鍵です。

| 項目 | 配分割合 | 内容 |
|------|---------|------|
| **システム指示** | 20-30% | 共通ルールファイル（前章で説明） |
| **タスク固有情報** | 40-50% | 対象ファイル + 参照コード |
| **会話履歴** | 20-30% | やり取り + ツール結果 |
| **余白** | 5-10% | 予期しない読込に備える |

**Isolate戦略のポイント**:
- **1タスク = 1つの財布**: 複数タスクの情報を同じ財布に詰め込まない
- **システム指示は固定費**: どのタスクでも一定額消費する
- **余白の確保**: 100%使い切ると容量不足でAIが前の記憶を忘れる

<!--
Context Window Budgetは、AIの記憶容量をタスク別に予算として管理する考え方です。AIの記憶容量は約10万から20万トークン、日本語で本1〜2冊分です。システム指示に20から30%、タスク固有の情報に40から50%、会話履歴に20から30%、余白に5から10%を配分します。家計簿ではなく「タスク別の財布」と考えるのがポイント。1つの財布に複数タスクのレシートを詰め込むと、どれがどのタスクの費用か分からなくなります。タスクごとに別の財布を持つことで、干渉を防ぎます。
-->

# 削減5原則 — 干渉源を取り除く

> Isolate戦略の前提: 不要な情報を減らしてから分離する

**原則1: 重複排除**
> 同じ情報がルールファイルの2箇所にある → 1箇所に集約

**原則2: 期限切れ削除**
> 3ヶ月以上更新されていない古い情報 → 確認して不要なら削除

**原則3: 推論可能な情報は書かない**
> AIがコードから自分で読み取れる内容はルールに書かない

**原則4: 終了プロジェクトの情報を残さない**
> 使わないルールはアーカイブまたは削除

**原則5: 1ファイル100行以内**
> （日本語約3,000字＝A4用紙1〜2枚程度）読込コストを抑えるための目安

<!--
Isolate戦略を実践する前に、まず不要な情報を減らします。5つの原則です。重複排除は同じ情報の二重管理を防ぎます。期限切れ削除は古い情報の混入を防ぎます。推論可能な情報の排除は、AIがコードから自分で読み取れる内容をコンテキストに書かないことです。終了プロジェクトの情報は不要です。そして1ファイル100行以内（日本語約3,000字、A4で1〜2枚程度）で読込コストを抑えます。
-->

# トレードオフ分析 — 分離コスト vs 干渉リスク

> 完全な分離は理想だが、常にコストがかかる

| アプローチ | 分離度 | コスト | 適用場面 |
|-----------|--------|--------|---------|
| **同一セッション** | 低 | なし | 関連する小タスク群 |
| **タスク切り替え** | 中 | 再開コスト | 独立した並行タスク |
| **別セッション** | 高 | 起動コスト | 完全に異なるプロジェクト |

**判断基準**:
- タスク間で **前提が共通** → 同一セッション可
- タスク間で **使うツールが異なる** → 分離推奨
- タスク間で **情報が競合** → 必須分離

> Isolateの度合いは「干渉リスク」と「分離コスト」のバランスで決める

<!--
Isolate戦略のトレードオフ分析です。完全な分離は理想ですが、常にコストがかかります。同一セッションは分離度が低いですがコストゼロ。タスク切り替えは中程度の分離。別セッションは完全分離ですが起動コストがかかります。判断基準は、タスク間で前提が共通なら同一セッションでもOK。使うツールが異なるなら分離推奨。情報が競合するなら必須分離です。
-->

# チーム運用 — 共有 vs 個人の分離線

> Isolate戦略の究極の形は「誰の情報か」を分離すること

| 種類 | 配置場所 | 更新頻度 | 役割 |
|------|---------|---------|------|
| **共有** | 共通ルールファイル | プロジェクト変更時 | 全員共通の固定費 |
| **チーム** | 役割別指示フォルダ | フェーズ変更時 | 役割ごとの分離 |
| **個人** | 個人記録フォルダ | 毎日 | 個人別の完全分離 |

**分離のルール**:
- **全員が守るべきルール** → 共有（共通ルールファイル）
- **役割ごとの手順** → チーム（役割別指示フォルダ）
- **個人の経験則** → 個人（個人記録フォルダ）

> 3レベル分離のメリット: 個人レベルになるほど「自分だけのルール」を自由に変えられる。共有ルールは変更にチーム合意が必要だが、個人ルールは自分の判断で即座に変更可能。この「変更の自由度」が分離レベルが高いほど増える理由です。

<!--
チームでのIsolate戦略では、「誰の情報か」を3レベルで分離します。共有レベルは全員が守るルール、チームレベルは役割別の手順、個人レベルは個人の経験則です。分離レベルが高いほど自由度が高くなる理由は、個人レベルなら自分の判断だけで変更できるからです。共有ルールはチーム合意が必要ですが、個人ルールは即座に変更可能。この自由度の違いが分離のメリットです。
-->

# まとめ — Isolate戦略の全体像

> Isolate戦略の核心は「部屋を分ける」こと。本章では6つの観点からこの戦略を解説しました。

# 確認問題

**Q1**: Isolate戦略の目的を、他の3つの戦略（Write/Select/Compress）と対比して説明せよ。

**Q2**: Context Window Budgetの配分割合と、「1タスク1つの財布」の考え方を説明せよ。

**Q3**: チーム運用で「共有情報」と「個人情報」を分離する理由と、その境界線の引き方を述べよ。

**解答**:

**A1**: Isolate戦略は「情報を分離して干渉を防ぐ」戦略。Write(書き込み)やSelect(選択)が情報の出入りを管理するのに対し、Isolateは情報間の境界を管理する点が異なる。Compress(圧縮)が情報量の削減なら、Isolateは情報の配置の最適化。

**A2**: システム指示20-30%（共通ルールの固定費）、タスク固有情報40-50%（対象ファイルと参照コード）、会話履歴20-30%（やり取りとツール結果）、余白5-10%（予期しない読込用）。「1タスク1つの財布」とは、1つの容量（財布）に複数タスクの情報を混ぜず、タスクごとに独立した容量を割り当てる考え方。

**A3**: 全員が守るルールは共有に、個人の経験則は個人記録に配置する。理由は、共有範囲を広げすぎると更新責任が曖昧になり、逆に狭めすぎるとノウハウが分散してチーム全体の品質が安定しないため。境界線は「変更頻度」と「適用範囲」で引く。
