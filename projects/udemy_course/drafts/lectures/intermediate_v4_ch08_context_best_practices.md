# LangChain 4戦略 — Isolateの位置づけ

| 戦略 | 目的 | 対応章 |
|------|------|--------|
| **Write** | AIに正しい情報を書き込む | 第5章 |
| **Select** | 必要な情報だけを選択して取り出す | 第6章 |
| **Compress** | 情報を圧縮して容量を節約する | 第7章 |
| **Isolate** | 情報を分離して干渉を防ぐ | **本章 (第8章)** |

**Isolate戦略の本質**:
- 異なるタスクのコンテキストが **混ざる** と、AIが混乱する
- タスクAの前提がタスクBに **漏れ出る** のを防ぐ
- 「部屋を分ける」ことで、それぞれの部屋の **整合性** を保つ

> Isolate = 「全部屋に荷物を広げる」のではなく「部屋ごとに整理する」

<!--
Isolate戦略はLangChainの4戦略の4つ目です。異なるタスクのコンテキストが混ざると、AIがタスクAの前提をタスクBに適用してしまい、混乱が生じます。Isolate戦略は、タスクごとにコンテキストを分離し、干渉を防ぐ手法です。部屋の例えで言うと、全部屋に荷物を広げるのではなく、部屋ごとに整理するイメージです。Writeが書き込み、Selectが選択、Compressが圧縮なら、Isolateは分離による管理です。
-->

# 「コンテキストの干渉」が起きる瞬間

> タスクA: 「Reactのコンポーネントをリファクタリング中」
> タスクB: 「PythonのAPIエンドポイントを追加中」
> **同じセッションで両方やるとどうなるか？**

**干渉の症状**:
- AIが **Reactの命名規則** をPythonコードに適用してしまう
- タスクAの **未解決課題** がタスクBの判断に影響する
- コンテキストウィンドウが **両タスクの情報で埋まる**

**Isolate戦略の解決策**:
- タスクごとに **独立したコンテキスト** を維持する
- 物理的な分離 (別セッション) と 論理的な分離 (Context Window Budget)

<!--
コンテキストの干渉が起きる典型的な場面です。ReactのリファクタリングとPythonのAPI追加を同じセッションで行うと、AIがReactの命名規則をPythonに適用してしまったり、タスクAの未解決課題がBの判断に影響したりします。コンテキストウィンドウが両タスクの情報で埋まるのも問題です。Isolate戦略の解決策は、タスクごとに独立したコンテキストを維持すること。物理的な分離として別セッションを使う方法と、論理的な分離としてContext Window Budgetを使う方法があります。
-->

# Context Window Budget — タスク別予算設計

> コンテキストウィンドウを「家計簿」として管理する

**予算配分の考え方**:

| 項目 | 配分割合 | 内容 |
|------|---------|------|
| **システム指示** | 20-30% | `CLAUDE.md` + `instructions/*.md` |
| **タスク固有情報** | 40-50% | ターゲットファイル + 参照コード |
| **会話履歴** | 20-30% | やり取り + ツール結果 |
| **余白** | 5-10% | 予期しない読込に備える |

**Isolate戦略のポイント**:
- **1タスク = 1予算**: 複数タスクの情報を同じ予算に押し込めない
- **システム指示は固定費**: どのタスクでも一定額消費する
- **余白の確保**: 100%使い切るとCompactionが頻発する

<!--
Context Window Budgetは、コンテキストウィンドウの容量をタスク別に予算として管理する考え方です。システム指示に20から30%、タスク固有の情報に40から50%、会話履歴に20から30%、余白に5から10%を配分します。Isolate戦略のポイントは、1タスクに1つの予算を割り当てること。複数タスクの情報を同じ予算に押し込めると、干渉と容量不足の両方の問題が起きます。システム指示は固定費として必ず消費されるため、残りの予算でタスク固有の情報をどれだけ載せられるかが設計の鍵です。
-->

# 削減5原則 — 干渉源を取り除く

> Isolate戦略の前提: 不要な情報を減らしてから分離する

**原則1: 重複排除**
> 同じ情報が `CLAUDE.md` と `instructions/` の両方にある → 1箇所に集約

**原則2: 期限切れ削除**
> 3ヶ月以上更新されていないmemory → 確認して無効なら削除

**原則3: 推論可能な情報は書かない**
> コードから読み取れることは `CLAUDE.md` に書かない

**原則4: 終了プロジェクトのコンテキストを残さない**
> 使わないルールはアーカイブまたは削除

**原則5: 1ファイル100行以内**
> 100行超のファイルは分割を検討

<!--
Isolate戦略を実践する前に、まず不要な情報を減らします。5つの原則です。重複排除は同じ情報の二重管理を防ぎます。期限切れ削除は古い情報の混入を防ぎます。推論可能な情報の排除は、AIがコードから自分で読み取れる内容をコンテキストに書かないことです。終了プロジェクトのコンテキストは不要です。そして1ファイル100行以内で読込コストを抑えます。これらを1週間に1回見直すことで、Isolate戦略の前提となるクリーンな状態を維持できます。
-->

# トレードオフ分析 — 分離コスト vs 干渉リスク

> 完全な分離は理想だが、常にコストがかかる

| アプローチ | 分離度 | コスト | 適用場面 |
|-----------|--------|--------|---------|
| **同一セッション** | 低 | なし | 関連する小タスク群 |
| **タスク切り替え** | 中 | 再開コスト | 独立した並行タスク |
| **別セッション** | 高 | 起動コスト | 完全に異なるプロジェクト |

**判断基準**:
- タスク間で **前提が共通** → 同一セッション可
- タスク間で **技術スタックが異なる** → 分離推奨
- タスク間で **コンテキストが競合** → 必須分離

> Isolateの度合いは「干渉リスク」と「分離コスト」のバランスで決める

<!--
Isolate戦略のトレードオフ分析です。完全な分離は理想ですが、常にコストがかかります。同一セッションは分離度が低いですがコストゼロ。タスク切り替えは中程度の分離。別セッションは完全分離ですが起動コストがかかります。判断基準は、タスク間で前提が共通なら同一セッションでもOK。技術スタックが異なるなら分離推奨。コンテキストが競合するなら必須分離です。LangChainのIsolate戦略では、このトレードオフを定量的に評価することが重要です。
-->

> 本講座の制作環境でのIsolate戦略の実装例

**実例1: タスク別YAMLによる論理的分離** 『本講座サンプル実装』

- 各タスクを独立したYAMLファイル (`queue/tasks/task_001.yaml`) で管理
- 1セッション = 1タスクYAML → タスク間のコンテキスト干渉を物理的に防止
- システム指示 (`CLAUDE.md`) は共有、タスク固有情報はYAMLに分離

**実例2: 役割別instructionsによる分離** 『本講座サンプル実装』

- `instructions/` 配下に役割別のファイル (`reviewer.md`, `implementer.md`)
- 同一プロジェクト内でも、役割が違えば読み込む指示が違う
- Context Window Budget: システム指示30% + タスク情報50% + 余白20%

> プロジェクト独自の運用マニュアルで、分離ルールを明文化する

**5〜7セクション構成**:

| # | セクション | Isolate戦略での役割 |

| 3 | よく使うコマンド | ビルド・テスト・デプロイ手順 |

| 5 | 品質基準 | テスト方針・レビュー観点 |
| 6 | デプロイ手順 | 環境別の反映方法 |
| 7 | 連絡・参照 | ドキュメントURL・チャットチャンネル |

> このフォーマットを `CLAUDE.md` のベースにすることで、Isolate戦略の境界線を明確にする

# チーム運用 — 共有 vs 個人の分離線

> Isolate戦略の究極の形は「誰のコンテキストか」を分離すること

| 種類 | 配置場所 | 更新頻度 | Isolate観点 |
|------|---------|---------|------------|
| **共有** | `CLAUDE.md` | プロジェクト変更時 | 全員共通の固定費 |
| **チーム** | `instructions/` | フェーズ変更時 | 役割別の分離 |
| **個人** | `memory/` | 毎日 | 個人別の完全分離 |

**分離のルール**:
- **全員が守るべきルール** → 共有 (`CLAUDE.md`)
- **役割ごとの手順** → チーム (`instructions/`)
- **個人の経験則** → 個人 (`memory/`)

> Isolate戦略の3レベル: 共有 → チーム → 個人。分離レベルが高いほど自由度が高い

<!--
チームでのIsolate戦略では、「誰のコンテキストか」を3レベルで分離します。共有レベルはCLAUDE.mdに配置し、全員が守るルールを書きます。Context Window Budgetの固定費として全員に読み込まれます。チームレベルはinstructionsに配置し、役割別の手順を分離します。reviewerとimplementerで読み込む指示が違うのはIsolate戦略の実践例です。個人レベルはmemoryに配置し、個人の経験則を完全に分離します。分離レベルが高いほど自由度が高くなります。
-->

# まとめ — Isolate戦略の全体像

**この章で学んだこと**:

1. **Isolate戦略** = タスク間のコンテキスト干渉を防ぐ LangChain の体系的手法
2. **Context Window Budget** でタスク別に予算を設計・配分する
3. **削減5原則** で不要コンテキストを特定し、干渉源を取り除く

```
  物理的分離 → 別セッション・別YAML

  組織的分離 → 共有/チーム/個人の3レベル
```

> Isolate = 「部屋を分ける」ことで、それぞれの部屋の品質を最大化する

<!--

# 確認問題

**Q1**: LangChainのIsolate戦略の目的を、他の3戦略と対比して説明せよ。

**Q2**: Context Window Budgetの配分割合と、各項目の役割を説明せよ。

**Q3**: チーム運用で「共有コンテキスト」と「個人コンテキスト」を分離する理由と、その境界線の引き方を述べよ。

**解答**:

**A1**: Isolate戦略は「情報を分離して干渉を防ぐ」戦略。Write(書き込み)やSelect(選択)が情報の出入りを管理するのに対し、Isolateは情報間の境界を管理する点が異なる。Compress(圧縮)が情報量の削減なら、Isolateは情報の配置の最適化。

**A2**: システム指示20-30% (CLAUDE.md等の固定費)、タスク固有情報40-50% (対象ファイルと参照コード)、会話履歴20-30% (やり取りとツール結果)、余白5-10% (予期しない読込用)。1タスクに1予算を割り当て、複数タスクの混載を防ぐ。

**A3**: 全員が守るルールは共有(CLAUDE.md)に、個人の経験則は個人(memory/)に配置する。理由は、共有範囲を広げすぎると更新責任が曖昧になり、逆に狭めすぎるとノウハウが分散してチーム全体の品質が安定しないため。境界線は「変更頻度」と「適用範囲」で引く。

<!--
確認問題です。Q1はIsolate戦略の位置づけを4戦略の中で説明させる問題です。Q2はContext Window Budgetの実践的な設計を問います。Q3はチーム運用でのIsolate戦略の応用を問います。3問とも正解できれば、Isolate戦略を実践的に理解できています。
