# 「覚えてないの？」— 毎回起こる会話

> AI「了解しました。そのファイルを修正します」
> （翌日）
> あなた「昨日の修正、続きを頼みたいんだけど」
> AI「はい、まず前提を教えてください」— **すべて忘れている**

- コンテキストウィンドウには限界がある (前章)
- `/clear` や新セッションで **会話はリセット** される
- compaction で詳細が **要約に圧縮** される
- **重要な情報はファイルに保存しなければ永続しない**

<!--
これがつかみです。AIを使っていると、このような状況に頻繁に出会います。昨日の作業を翌日引き継ごうとしたら、AIがすべて忘れている。前章で学んだ通り、コンテキストウィンドウには限界があり、/clearや新規セッションで会話はリセットされます。compactionでも詳細が要約に圧縮されます。つまり、会話の中に残した情報は永続しません。重要な情報を永続させるには、ファイルに保存する仕組みが必要です。それが永続コンテキスト層です。
-->

# 3つの永続化バックエンド

| バックエンド | 仕組み | 強み | 向いている用途 |
|------------|--------|------|---------------|
| **Claude Code memory** | `memory/MEMORY.md` + 自動読込 | 標準機能・設定不要 | ユーザー設定・基本ルール |
| **MCP memory** | MCP サーバー経由の key-value | 構造化検索・共有可能 | プロジェクト固有データ |
| **claude-mem** | SQLite + semantic search | 高速検索・履歴管理 | 大量の過去事例検索 |

> どれか1つではなく、**用途に応じて使い分ける**のが正解

<!--
AIの記憶を永続化するバックエンドは大きく3つあります。1つ目はClaude Code標準のmemory機能です。memory/MEMORY.mdというファイルに情報を書き込み、新セッション開始時に自動的に読み込まれます。設定不要ですぐに使えます。2つ目はMCP memoryです。MCPサーバー経由でkey-value形式でデータを保存し、構造化された検索が可能です。3つ目はclaude-memです。SQLiteデータベースに情報を保存し、セマンティック検索で過去の事例を高速に検索できます。どれか1つを選ぶのではなく、用途に応じて使い分けるのが実践的なアプローチです。
-->

# Claude Code memory — 最もシンプルな永続化

**仕組み**: `memory/MEMORY.md` に書くだけで、次回セッションで自動読込

- **自動読込**: 新セッション開始時に MEMORY.md が自動的にコンテキストに含まれる
- **手動書込**: AIに「これを覚えて」と指示するだけで更新される
- **階層構造**: `memory/*.md` に個別ファイルを作成し、MEMORY.md に索引を書く

```
memory/
  MEMORY.md           ← 索引 (自動読込)
  feedback_xxx.md     ← フィードバック記録
  project_notes.md    ← プロジェクト固有情報
```

> **メリット**: 設定不要・標準機能・直感的
> **デメリット**: 200行制限・検索機能なし

<!--
Claude Code memoryの詳細です。最もシンプルで、設定不要ですぐに使える永続化方法です。memoryディレクトリにMEMORY.mdというファイルを置くだけで、新セッション開始時に自動的に読み込まれます。AIに「これを覚えておいて」と指示するだけで、AIがファイルを更新してくれます。個別のトピックごとに別ファイルを作成し、MEMORY.mdに索引としてリストを書くのが推奨される構成です。シンプルですが、200行という制限があり、検索機能もありません。大量の情報を管理するには次のバックエンドが適しています。
-->

# 4カテゴリ分類 — 何をどう保存するか

永続コンテキストは **4つのカテゴリ** に分類して管理する

| カテゴリ | 内容 | 例 |
|---------|------|-----|
| **user** | ユーザーの役割・好み・知識レベル | 「React経験3年・Python初心者」 |
| **feedback** | AIへの指摘・訂正・改善指示 | 「テストは実ファイルで確認せよ」 |
| **project** | プロジェクト固有の状況・判断 | 「認証方式はJWTに決定」 |
| **reference** | 外部システムへのポインタ | 「バグ管理はLinearプロジェクトX」 |

> **分類の意図**: カテゴリごとに **更新頻度** と **寿命** が違う
> - user: ほぼ不変 / feedback: 中頻度 / project: 高頻度 / reference: 低頻度

<!--
メモリの内容を4つのカテゴリに分類します。1つ目のuserは、ユーザーの役割や好み、知識レベルです。React経験が3年でPythonが初心者というような情報は、ほぼ変わらないので一度書けば長く使えます。2つ目のfeedbackは、AIへの指摘や訂正です。「テストは実ファイルで確認せよ」というような指摘は、AIが同じミスを繰り返さないために重要です。3つ目のprojectは、プロジェクト固有の状況や決定事項です。更新頻度が高く、常に最新に保つ必要があります。4つ目のreferenceは、外部システムへのポインタです。更新頻度は低いですが、探す手間を省くために記録しておきます。この分類により、何をいつ更新すべきかが明確になります。
-->

# MEMORY.md 索引 — なぜ「目次」が必要か

**MEMORY.md は「本棚の目次」** — 全文ではなく索引を置く

```
# Memory Index
- [Inbox送信ルール](feedback_inbox_spam.md) — 1回送って待て
- [Dashboard完了行削除](feedback_dashboard_cleanup.md) — 自動削除あり
```

### 200行制限の理由

- MEMORY.mdは **セッション開始時に毎回読み込まれる**
- 200行を超えると、他のコンテキストを圧迫する
- **詳細は個別ファイル** に書き、MEMORY.mdには **1行のリンク** だけ

> **アンチパターン**: MEMORY.mdに長文を書く → 読込コスト増 → コンテキスト枯渇

<!--
MEMORY.mdは本棚の目次のようなものです。全文を書くのではなく、各メモリファイルへのリンクと1行の説明だけを書きます。なぜ200行という制限があるのか。MEMORY.mdはセッション開始時に毎回自動的に読み込まれるからです。200行を超えると、CLAUDE.mdやソースコードなど、他のコンテキストを圧迫してしまいます。詳細は個別のmdファイルに書き、MEMORY.mdには1行のリンクだけを残すのが正しい使い方です。長文をMEMORY.mdに書くのはアンチパターンです。読込コストが増え、結果的にコンテキストが枯渇します。
-->

# メモリのライフサイクル

```
作成 → 参照 → 検証 → 更新 → 削除
```

| フェーズ | 内容 | 注意点 |
|---------|------|--------|
| **作成** | 新しい情報を `memory/*.md` に書く | カテゴリを選んで分類 |
| **参照** | 新セッションで自動読込される | MEMORY.md索引が正しいか確認 |
| **検証** | メモリの内容がまだ正しいか確認 | ファイルパスの存在チェック |
| **更新** | 古い情報を新しい情報に書き換え | **追記**ではなく**置換**が原則 |
| **削除** | 不要になったメモリを削除 | MEMORY.mdからもリンクを削除 |

> **重要**: メモリは「書いて終わり」ではない — **定期的な検証** が必要

<!--
メモリにはライフサイクルがあります。作成、参照、検証、更新、削除の5つのフェーズです。作成時は適切なカテゴリを選んで分類します。参照は新セッションで自動的に行われます。検証では、メモリの内容がまだ正しいかを確認します。例えば、ファイルパスがまだ存在するか、前提条件が変わっていないかなどです。更新では、古い情報を新しい情報に書き換えます。ここで重要なのは、追記ではなく置換が原則であることです。古い情報と新しい情報が両方残ると矛盾が生じます。最後に、不要になったメモリは削除し、MEMORY.mdからもリンクを削除します。メモリは書いて終わりではなく、定期的な検証が必要です。
-->

# 自動読込の仕組み — セッション開始時の動き

新セッション開始時、Claude Code は以下を **自動的に** コンテキストに含める:

1. **CLAUDE.md** — プロジェクト全体ルール (常時)
2. **memory/MEMORY.md** — 索引 (常時)
3. **instructions/*.md** — 役割別ルール (指示があれば)

**読込の流れ**:
```
セッション開始
  → CLAUDE.md を読込
  → memory/MEMORY.md を読込
  → 必要に応じて instructions/*.md を読込
  → メモリ内のリンク先を必要に応じて読込
```

> **設計上の注意**: 自動読込されるファイルは **簡潔** に — 肥大化すると前章の「コスト爆発」を引き起こす

<!--
新セッションが開始されると、Claude Codeはいくつかのファイルを自動的に読み込みます。まずCLAUDE.mdが読み込まれ、プロジェクト全体のルールが認識されます。次にmemory/MEMORY.mdが読み込まれ、過去の記憶の索引が認識されます。そして必要に応じてinstructions/*.mdが読み込まれます。メモリ内のリンク先のファイルは、必要に応じてAIが個別に読み込みます。ここで注意すべきは、自動読込されるファイルは簡潔に保つことです。肥大化すると前章で学んだコスト爆発を引き起こします。MEMORY.mdを200行以内に保つ理由がここにあります。
-->

# 肥大化防止 — 何を書かないか

**メモリに書くべきでないもの**:

- **コードパターン・規約** → `CLAUDE.md` や `instructions/*.md` に書くべき
- **Git履歴・変更内容** → `git log` で確認できる
- **デバッグの解決策** → コードに反映済みなら不要
- **プロジェクト構造・ファイルパス** → 現在のファイルシステムから確認できる
- **CLAUDE.md に既にある情報** → 重複は矛盾の元

> **判断基準**: 「コードやGitから確認できるか？」→ はい → **書かない**
> 「人間の判断・教訓・好み」か？ → はい → **書く**

<!--
メモリの肥大化を防ぐには、何を書くべきでないかを知ることが重要です。コードパターンや規約はCLAUDE.mdやinstructionsに書くべきで、メモリには書きません。Git履歴はgit logで確認できるので不要です。デバッグの解決策も、コードに反映済みならメモリに残す必要はありません。プロジェクトの構造やファイルパスも、現在のファイルシステムから確認できるので書きません。CLAUDE.mdに既にある情報を重複して書くのも避けましょう。判断基準はシンプルです。「コードやGitから確認できるか？」がYesなら書かない。「人間の判断、教訓、好みか？」がYesなら書く。これだけです。
-->
