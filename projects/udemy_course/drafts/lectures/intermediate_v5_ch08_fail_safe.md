# 「完了しました」— 本当に完了していますか?

> AIがコードを書く際、エラーが起きても**無視する書き方**（Pythonでは`try/except`と呼ばれます）があります。エラーを捕捉して握り潰し、「完了」と報告 — 実際にはデータが壊れていた、という経験はありませんか?

- エラー時に**代替値**を返して誤魔化す → 何が起きたか**痕跡が残らない**
- 後日、同じエラーが起きても**原因調査が不可能**
- 「そのコマンド使うな」と指示したのに、AIが普通に実行して本番を壊した

<!--

## 「黙って失敗」の3パターン

| パターン | 具体例 | 何が問題か |
|----------|--------|-----------|
| **① 例外の握り潰し** | エラーが起きても無視する構文を書く | エラーが完全に消える |
| **② 代替値による隠蔽** | エラー時にダミーデータを返す | 失敗が成功に見せかけられる |
| **③ 条件分岐の漏れ** | 想定外の入力で無言の不正動作 | 誰も気づかない |

> どのパターンも**「失敗が見えない」**という共通点があります。見えないものは直しようがありません。

<!--
黙って失敗には大きく3つのパターンがあります。例外を握り潰す、代替値で誤魔化す、条件の漏れで意図しない動作をする — どれも「失敗が見えない」という点で同じです。見えない敵は倒せません。まずはこの3パターンを正しく認識することが第一歩です。
-->

## 解決策① — 段階確認で関所を置く

> 各工程の終わりに**チェックポイント**を置き、異常を検知したら即座に停止・報告する

```
データ取得 → [検証: 件数 > 0 ?]
                    ↓ NG → 即STOP + 報告
                    ↓ OK
データ前処理 → [検証: フォーマット正常 ?]
                    ↓ NG → 即STOP + 報告
                    ↓ OK
レポート出力 → 完了
```

- 各工程 = **1つの責任** に分割
- 関所 = 「この条件を満たさなければ次に進めない」という**強制停止点**
- 関所で止まれば、**原因は直前の工程に絞り込める**

<!--
1つ目の解決策は「段階確認」です。処理を複数の工程に分け、それぞれの終わりにチェックポイントを置きます。関所で異常を検知したら、即座に処理を停止して報告します。こうすれば「どこで壊れたか」が一目で分かります。
-->

---

## 段階確認の具体例 — 週次レポート生成

> **困りごと**: AIに週次レポートを生成させたら、過去データの取得に失敗したのに代替値を入れて「完了」と報告。気づかずに上司に提出してしまった

**Before (関所なし)**:
```
データ取得 → レポート生成 → 「完了しました」
                ↑ 失敗を代替値で隠蔽
```

**After (関所あり)**:
```
データ取得 → [Gate: 取得件数 > 0] → レポート生成 → [Gate: 前週差分が妥当] → 出力
```

- 関所1: データが0件なら即STOP — 「取得失敗」と報告
- 関所2: 前週比が異常なら確認待ち — 「データ異常の可能性」と報告

<!--
具体例を見てみましょう。週次レポートの自動生成で、データ取得に失敗したのに代替値を入れて完了報告するケース。関所を2つ置けば、データ取得後とレポート生成後のそれぞれで異常を検知できます。自分の現場のレポートやKPI名前に置き換えて、すぐに応用できます。
-->

## 解決策② — 検知の仕組み (watcher)

> 段階確認は「その場で検知」する仕組み。**watcher** は「後から異常を見つける」仕組み

- cron (スケジュール実行) やhookで**定期的に監視** — 段階確認が届かないケース (全体クラッシュ・外部サービスダウン等) も捕捉
  - Claude Codeではhookと組み合わせて、バッチ処理後に自動で結果をチェックさせることができます
- 異常パターンを自動検出:
  - 終了コードが0以外（プログラムが正常終了すると0、異常終了すると別の番号を返す仕組みです）
  - 出力ファイルが存在しない
  - 処理件数が前回と大きく乖離 (例: 前日比50%以上の変動)
  - ログにエラーメッセージが含まれる

```
毎朝6:00 バッチ実行
  → watcher が「前日比-70%」を検知
  → 即座に通知: "⚠ 件数異常: 昨日1000件 → 今日300件"
```

<!--
2つ目の解決策は「watcher」です。段階確認はその場での検知ですが、watcherは後から異常を見つける仕組みです。毎日自動で動くバッチ処理などで、件数の急減やファイルの不在を自動検知して通知します。自分の現場の監視条件に置き換えて応用できます。
-->

---

## 解決策③ — `--dry-run` で安全に試す

> **本番実行の前に、結果だけを確認する** — 2段階で確実性を担保する技術

| フラグ | 動作 | 用途 |
|--------|------|------|
| `--dry-run` | 実行の**プレビューのみ**表示 (実際の変更なし) | 「何が起きるか」を事前確認 |
| `--no-confirm` | 確認なしで即座実行 | 信頼済みの定型処理向け |

**運用フロー**:

1. まず `--dry-run` で結果を確認
2. 想定通りなら本番実行
3. 想定と違えば原因を調査

```
$ python import_data.py --dry-run
→ "取り込む予定: 1,247件 / フォーマット: CSV / 先頭: 2026-01-15"
# ← AIに「--dry-run モードを作って」と頼めば、この機能を実装してくれます
# ← これを見てOKなら本番実行
```

<!--
3つ目の解決策はdry-runです。本番実行の前に「何が起きるか」だけを確認する仕組みです。まずdry-runで結果を確認し、想定通りなら本番実行する — この2段階で確実性を担保します。自分のバッチ処理やデータ投入に応用できます。
-->

## 解決策④ — 禁止操作リストで「止める」

> 指示で「使うな」と書いても、AIは**実行時に忘れる**ことがあります。「仕組み」で止める

**禁止操作リスト** = 「絶対に実行してはいけない操作」をリスト化して明文化したもの

| ID | 禁止操作 | 理由 |
|----|---------|------|
| D001 | `rm -rf /` | OS全体を破壊 |
| D002 | 本番データベースへの直接変更 | データ消失の危険 |
| D003 | `--force` 付きのリモートへの反映 | 他の人の作業を上書き |
| D004 | 確認なしの全ファイル削除 | 取り返しがつかない |
| D005 | 管理者権限での実行 | 権限の不当な昇格 |

> 前章の`PreToolUse` hookと連動させると、禁止操作をAIが実行しようとした**瞬間**に阻止できます

<!--
4つ目の解決策は禁止操作リストです。「使うな」と指示するだけでなく、「仕組み」として強制的に阻止します。D001からD005は代表例ですが、自分の現場で「絶対にやってはいけないこと」を列挙して設定します。前章のPreToolUse hookと連動させると、実行の瞬間に自動阻止できます。
-->

---

## 禁止操作リストの具体例 — 本番デプロイ

> **困りごと**: 「本番DBのDROP TABLEは禁止」と指示したのに、AIが開発環境だと思って本番で実行

**Before (指示のみ)**:
```
CLAUDE.md に「本番DBを触るな」と書いた
→ AIが文脈を勘違い → 本番でDROP TABLEを実行
```

**After (仕組みで阻止)**:
```
PreToolUse hook が SQL文を検査
→ "DROP" / "TRUNCATE" / 本番ホスト名 を含む SQL を即ブロック
→ 「禁止操作が検出されました: D002 本番DBへの直接変更」
```

- **指示**はAIが忘れることがある
- **仕組み**は忘れない — 実行の瞬間に強制阻止

<!--
具体例を見ましょう。本番DBのDROP TABLE禁止を指示だけに頼ると、AIが文脈を勘違いして実行してしまいます。PreToolUse hookで禁止パターンを定義しておけば、実行の瞬間に確実にブロックできます。「指示」ではなく「仕組み」で止める — これが本章の核心です。
-->

## ch07との連動 — hookが禁止操作の「実行時強制力」を担保する

> 前章で学んだ`PreToolUse` hook（AIがツールを実行する直前に発動する仕組み）が、禁止操作リストの**実行時の強制力**を担う

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ AIがコマンド構築  │ ──→ │ PreToolUse hook  │ ──→ │ 禁止操作リストと  │
│ "DROP TABLE ..."  │     │ がコマンドを検査  │     │ 照合 → 一致!     │
└──────────────────┘     └──────────────────┘     └────────┬─────────┘
                                                           │
                                                           ▼
                                                  ┌──────────────────┐
                                                  │ 実行を BLOCK     │
                                                  │ 「禁止操作 D002」 │
                                                  └──────────────────┘
```

- ch07: hookで「いつ検査するか」を制御
- ch08: 禁止操作リストで「何を禁止するか」を定義
- **この2つの組み合わせ**で、失敗を見逃さない仕組みが完成

<!--
前章で学んだPreToolUse hookと本章の禁止操作リストが連動します。hookが「いつ検査するか」を制御し、禁止リストが「何を禁止するか」を定義する。この2つの組み合わせで、失敗を見逃さない体制が完成します。自動化と安全装置が揃って初めて、安心してAIに任せられるのです。
-->

## 4つの解決策 — まとめ

| 解決策 | 何を防ぐか | 仕組み |
|--------|-----------|--------|
| **段階確認** | 工程途中の異常を見逃す | 各工程の終わりに関所を置く |
| **watcher** | 事後に気づく遅延 | 定期的な異常パターンの自動検知 |
| **`--dry-run`** | 確認なしの本番実行 | 事前プレビュー → OKなら本番 |
| **禁止操作リスト** | AIが禁止操作を実行 | PreToolUse hook + 禁止パターン定義 |

> この4つを組み合わせることで、**「黙って失敗」も「止まらない暴走」も防げる**

## 確認問題

**Q1**: AIがエラーを無視する書き方をして「完了しました」と報告しました。実際には処理が失敗しています。これは「黙って失敗」のどのパターンですか?

**Q2**: 段階確認 (チェックポイント) を設計するとき、各関所では何を確認すべきですか?

**Q3**: 禁止操作リストを「指示」ではなく「仕組み」として機能させるには、前章のどの技術と組み合わせる必要がありますか?

---

**A1**: パターン①「例外の握り潰し」です。エラーが起きても無視する構文で処理され、何も記録されず、結果として失敗が完全に隠蔽されます。

**A2**: 「その工程の出力が、次の工程の入力として妥当か」を確認します。例: データ取得後なら「件数が0より多いか」、前処理後なら「フォーマットが正常か」など、工程ごとの妥当性条件を定義します。

**A3**: 前章で学んだ`PreToolUse` hookと組み合わせます。hookがAIの実行直前にコマンドを検査し、禁止操作リストと照合して、一致すれば実行をブロックします。指示は忘れることがありますが、仕組みは忘れません。

<!--
確認問題の解答です。Q1は例外の握り潰しパターン。Q2は工程ごとに妥当性条件を定義すること。Q3はPreToolUse hookとの組み合わせです。指示ではなく仕組みで止める — この考え方が本章の核心です。
