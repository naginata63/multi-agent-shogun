# 「同じ問題を2度解いていませんか？」

> AIエージェントはセッションをまたぐと、過去に解決した問題を忘れます。

- すべての過去履歴を常時ロードすることは不可能

# hookとは何か？ — 自動発動の仕組み

> **hook** = 「特定のタイミングで勝手に動く小さなプログラム」（一般的なソフトウェアの概念です）

例え話:
- 防犯カメラの「動体検知センサー」に似ています
- センサーが反応すると自動で録画が始まるように、
  **AIが特定の操作をする直前** に自動でプログラムが走ります

- **どこに書く？** → Claude Code の設定ファイル（プロジェクト内の `.claude/settings.json` または `~/.claude/settings.json`）に書くと、この一般概念が Claude Code で実装できます
- **何がいい？** → 人間が毎回指示しなくても、自動で安全確認が走る

<!--
hookの基本概念を説明します。hookは「特定のタイミングで自動実行される小さなプログラム」です。防犯カメラの動体検知センサーに例えると分かりやすいでしょう。センサーが反応したら自動で録画が始まるように、AIが特定の操作（ファイルへの書き込みなど）をする直前に、自動でチェック処理が走ります。設定はsettings.jsonというファイルに書きます。人間が毎回指示しなくても自動で安全確認が走るのが最大の利点です。
-->

# hookの登録方法

設定ファイル（`settings.json`）に書くだけでOK（`.claude/settings.json` をテキストエディタで開いて追記します）:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "bash scripts/search_and_validate.sh"
      }
    ]
  }
}
```

- **`"matcher"`**: どの操作で発動するかの条件
  - `Write|Edit` = 「ファイル書き込み**または**編集」の意味（`|` は「または」）
- **`"command"`**: 発動時に実行するプログラム
- **`"PreToolUse"`**: 「ツールを使う前」というタイミング
- **スクリプトは？** → チームの共通スクリプトを使うか、自分で `scripts/` フォルダに用意します

<!--
hookの登録方法です。settings.jsonという設定ファイルに、このように書くだけでhookが有効になります。matcherは発動条件で、WriteかEditが呼ばれたときに反応します。commandは発動時に実行するスクリプトです。PreToolUseは「ツールを使う前」というタイミングを指定しています。
-->

# 工程検問（Phase-Gating）— 品質の検問所

> **工程検問** = 工事現場の「安全確認チェックポイント」のようなもの
> 各工程が完了・安全確認できるまで、次の工程に進ませない仕組み

この講座では **3つのタイミング** でチェックポイントを置きます:

| タイミング | 何を確認するか | いつ発動 |
|-----------|--------------|---------|
| **ツール実行前** | 実行していいか条件確認 | AIが操作する直前 |
| **ツール実行後** | 結果が正しいか検証 | 操作が終わった直後 |
| **セッション開始時** | 前回の続きが正しく復旧 | 新しい作業開始時 |

> この章では、特に **ツール実行前** の工程検問に焦点を当てます。

<!--
工程検問の概念です。工事現場で作業ごとに安全確認のチェックポイントがあるように、AIの処理の流れにも確認ポイントを置きます。この講座では3つのタイミングでチェックポイントを設けます。この章では特にツール実行前の工程検問に焦点を当てます。

# AIの行動を予測可能にする設計

> AIに「何でも自由にやっていい」と任せると、何が起きるか分からなくなる。
> そこで **安全な範囲を決めておく** 設計が重要になります。

考え方（3つのポイント）:

1. **範囲を決める**: AIが実行できる操作をあらかじめ限定する
2. **同じ入力には同じ結果**: 毎回同じ条件なら同じ結果を返す
3. **順序を守る**: 処理を決まった順番で実行する

| 無制約（危険） | 範囲を決めた設計（安全） |
|---------------|----------------------|
| AIが任意の操作を実行可能 | 許可された操作のみ実行 |
| 結果が予測不能 | 同じ条件で同じ結果 |
| 原因追跡が困難 | ログで原因が追える |

<!--
AIの行動を予測可能にする設計パターンの説明です。AIに自由にやらせると何が起きるか分からなくなるため、安全な範囲を決めておくことが重要です。3つのポイントは、行動範囲の制限、同じ入力への同じ結果、処理順序の定義です。無制約の場合と比較すると、範囲を決めた設計の利点が分かりやすいでしょう。
-->

# セマンティック検索パイプライン

**3段階の流れ**で過去事例を自動検索:

| 段階 | 処理 | 役割 |
|------|------|------|
| **1. 抽出** | タスク情報からキーワードを取り出す | 検索クエリの作成 |
| **2. 検索** | 記憶領域（＝過去の事例が保存されているデータベースやファイル）を検索 | 過去事例の発見 |
| **3. 注入** | 検索結果をAIの入力に追加 | AIが過去事例を認識 |

> **「注入」とは？** → AIが読む入力テキスト（プロンプト）の末尾に、検索結果を自動的に追記することです。人間がコピペする必要はありません。

- この3段階を **ツール実行前のhook** として自動化
- タスク開始前に必ず過去事例を検索 — 人間の介入なし

> **具体例 — プロンプトの末尾に自動追記されるイメージ**:
> ```
> AIへの指示: 「データベース移行のスクリプトを作成して」
> ↓ （自動注入）
> [自動検索結果] 過去の類似事例:
>   1. "DB移行" → 手動移行はミスが起きやすいとの記録あり
>   2. "DB変更" → バックアップ必須の手順が過去に確立済み
> ```

<!--
セマンティック検索パイプラインの全体像です。第1段階はキーワード抽出、第2段階は記憶領域の検索、第3段階は結果をAIの入力に追加します。この3段階をツール実行前のhookとして登録しておけば、新しいタスクを開始するたびに自動的に過去事例が検索されます。
-->

# Embedding（埋め込み）とは

> 前のスライドの「検索」段階で「意味」で検索する仕組みが Embedding です。

> **Embedding** = テキストの「意味」を数字の列に変換する技術
> **何が嬉しい？** → 言葉が違っても「意味が似ている」過去事例を自動で見つけてくれる

- 上位N件（通常3〜5件）を結果として返す
- 注入量は **1件あたり約200トークン**（日本語で約130文字程度）に制限

> **安心してください**: Embeddingの変換計算は **裏側で自動的に行われます**。あなたが数式を書く必要はありません。検索APIを呼ぶだけで、似ている文書が自動で見つかります。

**仕組み（興味がある人向け）**:
- 「データベースの移行」→ `[0.23, -0.15, 0.87, ...]`
- 「DBの引越し」→ `[0.21, -0.14, 0.85, ...]`
- 2つの数字の列は **とても似ている** → 「似ている話題」と判定できる
- **Embeddingの似ている度合い** を数値で計測する

**自分で書くとこうなる**（Pythonの例）:
```python
from claude_mem import search
results = search(query="データベース移行", limit=3)
# → 過去事例が自動でEmbedding検索され、上位3件が返ってくる
```

<!--
Embeddingの説明です。テキストの意味を数字の列に変換する技術です。「データベースの移行」と「DBの引越し」は言葉は違いますが、意味が似ているため、変換後の数字の列も似たものになります。この似ている度合いを数値で計測することで、意味ベースの検索が可能になります。1件あたり約200トークン（日本語で約130文字程度）に制限して注入します。
-->

---

# 検索結果が0件 — 「過去事例なし」の警告

> 検索結果が0件だった場合、それは単なる結果ではなく **注意信号** です。

- **結果ゼロの3パターン**:
  - 初回の問題（新しい問題）→ 正常、心配ない
  - 過去に解決済みだが記録漏れ → 問題、記録が必要
  - キーワード抽出が不適切 → 調整が必要

- **警告の仕組み**:
  ```
  [search] 注意: 過去事例なし — 新規問題の可能性あり
  → 完了後に必ず記憶に記録すること
  ```

- **設計意図**: 記録漏れを自動検知し、**知識を蓄積するサイクルを完成させる**

<!--
検索結果が0件だった場合の対応です。結果ゼロは単なる「なし」ではなく、文脈によって意味が変わります。初回の新規問題なら正常ですが、過去に解決済みなのに記録されていなければ問題です。結果ゼロのときは注意信号を出し、タスク完了後の記録を促すことで、知識蓄積のサイクルを完成させます。
-->

# タスク定義の検索設定

検索動作をタスク定義ファイルで制御できます（このYAMLはタスク管理ファイルに書きます）:

```yaml
task_id: task_1001            # ← タスクごとの固有ID（任意の名前でOK）
description: "工程検問の仕組みの設計"
search_config:
  enabled: true
  keywords: ["工程検問", "検証"]
  exclude_keywords: ["初心者", "入門"]
  max_results: 5
```

| 設定項目 | 役割 | 初期値 |
|---------|------|--------|
| `enabled` | 検索のON/OFF | `true` |
| `keywords` | 手動で検索語を指定 | 自動抽出 |
| `exclude_keywords` | 除外したい語 | なし |
| `max_results` | 最大結果件数 | 3 |

> **設計思想**: 自動抽出を基本としつつ、**手動で微調整** も可能にする

<!--
タスク定義に検索設定を追加する方法です。search_configフィールドで検索動作を制御できます。enabledで有効無効、keywordsで手動指定、exclude_keywordsで除外、max_resultsで件数制限を設定します。自動抽出を基本としつつ手動で微調整できる柔軟性を持たせています。
-->

# 検索精度のチューニング — 5つの調整ポイント

> **最初は #2 の除外キーワードだけ設定すればOK** です。それ以外は初期値のままで十分機能します。必要に応じて少しずつ調整してください。

| # | 調整ポイント | 方法 | 効果 |
|---|------------|------|------|
| **★2** | **除外キーワード** ← まずこれ | 「初心者」「入門」などの汎用語を除外指定 | ノイズ削減 |
| 1 | キーワード重み | タスクの説明文（description）を一番重視し、メモ（notes）、タスクID（task_id）の順に重み付け | 重要な項目を優先 |
| 3 | 結果件数 | `max_results` を調整 | 入力サイズの制御 |
| 4 | Embeddingモデル | より高精度なモデルに変更 | 意味検索の精度向上 |
| 5 | 類似度の閾値 | 最低閾値を設定 | 関連の薄い事例を除外 |

> **チューニングのコツ**: 初期設定で1週間運用 → ログファイル（検索結果ゼロの回数・除外された事例など）を確認 → 1点ずつ調整。一度に全部変えると効果判定ができなくなります。

<!--
検索精度のチューニング方法です。5つの調整ポイントがあります。キーワード重み付け、除外キーワード、結果件数、Embeddingモデル、類似度の閾値の順に調整できます。コツは初期設定で運用してログ分析してから1点ずつ調整することです。一度に全部変えると、どの変更が効果的だったか分からなくなります。
-->
