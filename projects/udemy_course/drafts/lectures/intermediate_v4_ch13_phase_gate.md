# 「また同じ問題を解いていた」 — 過去の忘却

> 「1ヶ月前に同じバグを直したはずなのに、また最初から調査し直している」
> — これは人間だけでなく、**AI開発でも** 毎日起こっています。

- AIはセッションをまたぐと **過去の解決履歴を忘れる**
- 同じエラーの調査、同じ設計の議論を何度も繰り返す
- この **「過去の忘却コスト」** は、プロジェクトが長くなるほど増大する

<!--
このスライドがつかみです。AIを使って開発していると、セッションをまたぐたびに過去の解決履歴がリセットされます。1ヶ月前に直したバグと同じ問題に再び遭遇し、また最初から調査し直す。この無駄をなくす仕組みが、この章のテーマです。コンテキストウィンドウには限界があるため、すべての過去履歴を常時ロードすることは不可能です。だからこそ、必要なときに必要な過去事例だけを自動検索する仕組みが必要なのです。
-->

# mem-search pre-hook — キーワード抽出から結果注入まで

**3段階のパイプライン**:

| 段階 | 処理 | 役割 |
|------|------|------|
| **1. 抽出** | cmd YAMLからキーワードを抽出 | 検索クエリの生成 |
| **2. 検索** | メモリバックエンドを検索 | 過去事例の発見 |
| **3. 注入** | 検索結果をコンテキストに注入 | AIが過去事例を認識 |

- この3段階を **PreToolUse hook** として自動化する
- タスク開始前に必ず過去事例を検索 — 人間の介入なし

<!--
mem-search pre-hookの全体像です。3段階のパイプラインで構成されます。第1段階はキーワード抽出で、新しいタスクのYAMLから重要なキーワードを抽出します。第2段階は検索で、メモリバックエンドにある過去の事例を検索します。第3段階は注入で、検索結果をAIのコンテキストに自動的に注入します。この3段階をPreToolUse hookとして登録しておくことで、新しいタスクを開始するたびに自動的に過去事例が検索され、AIが過去の解決策を参照できるようになります。
-->

# 第1段階: キーワード抽出の仕組み

**cmd YAML からのキーワード抽出**:

```
# 入力: cmd YAML
description: "SQLite移行: dual-path書込実装"
parent_cmd: cmd_1488

# 抽出されるキーワード:
→ "SQLite", "移行", "dual-path", "書込", "実装"
```

- **抽出方法**: `awk` + 形態素解析ライブラリで名詞を抽出
- **重み付け**: `description` の語句を最重視、`parent_cmd` のIDは軽視
- **除外フィルタ**: ストップワード（"する""いる""こと"等）を除去

<!--
第1段階のキーワード抽出の詳細です。cmd YAMLのdescriptionフィールドやparent_cmdフィールドから、重要なキーワードを抽出します。抽出にはawkコマンドや形態素解析ライブラリを使い、名詞を中心に取り出します。descriptionに含まれる語句を最も重視し、cmd ID自体は軽視します。また「する」「いる」「こと」などの一般的なストップワードは除外することで、検索精度を高めます。
-->

# 第2段階: メモリ検索の仕組み

**検索の流れ**:

1. 抽出されたキーワードでメモリバックエンドを検索
2. **セマンティック検索** (Embedding類似度) と **キーワード検索** のハイブリッド
3. 上位 N件（通常3〜5件）を結果として返す

| 検索方式 | 特徴 | 用途 |
|---------|------|------|
| キーワード検索 | 高速・精密一致 | 固有名詞・コマンド名 |
| セマンティック検索 | 意味ベースの類似度 | 概念的な関連事例 |

> **設計ポイント**: 2方式を組み合わせることで「表現ゆれ」に強い検索を実現

<!--
第2段階のメモリ検索です。抽出されたキーワードを使ってメモリバックエンドを検索します。キーワード検索とセマンティック検索の2方式をハイブリッドで使うのがポイントです。キーワード検索は高速で精密な一致に強く、コマンド名や固有名詞の検索に向いています。セマンティック検索は意味ベースの類似度で、「SQLiteの移行」と「データベースの引越し」のような表現ゆれにも対応できます。2方式を組み合わせることで、精度の高い検索が可能になります。
-->

# 第3段階: 結果注入 — AIが過去を認識する瞬間

**検索結果の注入イメージ**:

```
[mem-search] 過去3件の類似事例を発見:
1. cmd_1481 "SQLite設計書" — dual-path設計の詳細
2. cmd_1491 "SQLite p1実装" — 初期実装の知見
3. cmd_1476 "inbox_write改修" — SQLite書込の実例
```

- 検索結果は **プレーンテキスト** でAIのコンテキストに注入
- AIは自動的に過去事例を参照し、**同じ間違いを避ける**
- 注入量は **1件あたり200トークン程度** に制限（コンテキスト節約）

<!--
第3段階の結果注入です。検索結果はプレーンテキストでAIのコンテキストに注入されます。過去3件の類似事例が発見された場合、cmd IDと要約がAIに提示されます。AIはこの情報を参照して、過去の解決策を活用したり、同じ間違いを避けたりできます。注入量は1件あたり200トークン程度に制限し、コンテキストウィンドウを節約する設計になっています。5章で学んだコンテキスト経済学の原則がここでも活きています。
-->

# hit=0 — 「過去事例なし」の警告

> 検索結果が0件だった場合、**それは情報ではなく「警告」です**。

- **hit=0 の意味**:
  - 初回の問題（新規性が高い）✅ 正常
  - 過去に解決したはずが記録されていない ❌ 問題
  - キーワード抽出が不適切 ❌ チューニング必要

- **警告注入の仕組み**:
  ```
  [mem-search] ⚠ 過去事例なし — 新規問題の可能性
  → 完了後に必ずメモリに記録すること
  ```

- **設計意図**: 「記録漏れ」を自動検知し、**知識の蓄積サイクルを閉じる**

<!--
検索結果が0件だった場合の対応です。hit=0は単なる「結果なし」ではなく、文脈によって意味が変わります。初回の新規問題なら正常ですが、過去に解決したはずなのに記録されていないなら問題です。またキーワード抽出が不適切で検索に失敗している可能性もあります。このため、hit=0のときは警告を注入し、タスク完了後に必ずメモリに記録するよう促します。これにより、知識の蓄積サイクルが閉じ、次回は同じ問題がhit=0にならなくなります。
-->

# PreToolUse hook としての登録方法

**settings.json への登録**:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "bash scripts/mem_search_prehook.sh"
      }
    ]
  }
}
```

- **matcher**: `Write` または `Edit` ツールの呼出時に発火
- **command**: キーワード抽出 → 検索 → 結果注入のスクリプト
- **発火タイミング**: AIがファイルを書き込む直前

> **なぜ Write/Edit か**: ファイル書込 = 実作業の開始点。ここで過去事例を注入するのが最も効果的

<!--
PreToolUse hookとしての登録方法です。settings.jsonのhooksセクションにPreToolUse hookを登録します。matcherにはWriteまたはEditを指定し、AIがファイルに書き込む直前にフックを発火させます。commandにはキーワード抽出、検索、結果注入を行うスクリプトを指定します。なぜWriteかEditのタイミングかというと、ファイルへの書込は実作業の開始点であり、この直前に過去事例を注入するのが最も効果的だからです。プロンプト入力時ではなく、実作業の直前に注入することで、AIが過去事例を確実に参照できます。
-->

# cmd YAML schema 拡張 — メモリ検索の制御

**mem_search フィールドの追加**:

```yaml
# cmd YAML (拡張版)
task_id: cmd_1595
description: "Phase間ゲートハーネスの設計"
mem_search:
  enabled: true
  keywords: ["Phase gate", "ハーネス", "cmd intake"]
  exclude_keywords: ["初心者", "入門"]
  max_results: 5
```

| フィールド | 役割 | デフォルト |
|-----------|------|-----------|
| `enabled` | 検索の有効/無効 | `true` |
| `keywords` | 手動指定キーワード | 自動抽出 |
| `exclude_keywords` | 除外キーワード | なし |
| `max_results` | 最大結果件数 | 3 |

> **設計思想**: 自動抽出を基本としつつ、**手動での微調整** を可能にする

<!--
cmd YAMLのスキーマ拡張です。mem_searchフィールドを追加することで、メモリ検索の動作を細かく制御できます。enabledで検索の有効無効を切り替え、keywordsで手動でキーワードを指定することもできます。exclude_keywordsで特定のキーワードを除外し、max_resultsで最大結果件数を制限します。設計思想は、自動抽出を基本としつつ手動での微調整を可能にすることです。通常は自動抽出で十分ですが、検索精度に課題がある場合に手動で調整できる柔軟性を持たせています。
-->

# 検索精度のチューニング — 5つの調整ポイント

| # | 調整ポイント | 方法 | 効果 |
|---|------------|------|------|
| 1 | **キーワード重み** | description > notes > parent_cmd | 重要フィールドを優先 |
| 2 | **除外キーワード** | `exclude_keywords` で汎用語を除外 | ノイズ削減 |
| 3 | **結果件数** | `max_results` を調整 | コンテキスト消費の制御 |
| 4 | **Embedding モデル** | より高精度なモデルに変更 | 意味検索の精度向上 |
| 5 | **閾値調整** | 類似度の最低閾値を設定 | 低関連事例の除外 |

> **チューニングのコツ**: まずデフォルト設定で1週間運用 → ログ分析 → 1点ずつ調整

<!--
検索精度のチューニング方法です。5つの調整ポイントがあります。1つ目はキーワードの重み付けで、descriptionフィールドの語句を最も重視します。2つ目は除外キーワードで、汎用的すぎる語句を除外します。3つ目は結果件数の調整で、コンテキスト消費を制御します。4つ目はEmbeddingモデルの変更で、より高精度なモデルを使うことで意味検索の精度が向上します。5つ目は類似度の閾値調整で、低関連の事例を除外します。チューニングのコツは、まずデフォルト設定で1週間運用し、ログを分析してから1点ずつ調整することです。一度に全部変えると、どの変更が効果的だったか分からなくなります。
-->
