# CLAUDE.mdとは — AIに読ませるREADME

> 🪞 「新しいプロジェクトに入った時、最初に何を読みますか？」

- 新しいプロジェクト → まず **README.md** を読む
- AIも同じ → **CLAUDE.md** を読む
- **CLAUDE.md = AIに読ませるプロジェクトの説明書**

> Claude Codeは起動時に自動的にCLAUDE.mdを読み込み、プロジェクトのルールを理解する

<!--
人間が新しいプロジェクトに入った時、まずREADME.mdを読んで「このプロジェクトは何で、どう動いているのか」を理解しますよね。Claude Codeも同じことをします。起動すると自動的にCLAUDE.mdというファイルを探して読み込みます。このファイルにプロジェクトのルールを書いておけば、AIは毎回そのルールを守って作業してくれます。
-->

---

# なぜCLAUDE.mdが必要なのか

**CLAUDE.mdなしの会話**:
- 「このプロジェクトはPythonです」→ 次の会話で忘れる
- 「テストはpytestで実行して」→ 毎回言わないといけない
- 「TypeScriptのstrictモードです」→ また忘れる

**CLAUDE.mdありの会話**:
- 一度書けば **毎セッション自動で読み込まれる**
- プロジェクトのルールを **AIが自動で守る**
- 同じ前提説明の **繰り返しがゼロに**

<!--
CLAUDE.mdがないと、毎回AIに「このプロジェクトはPythonです」「テストはpytestで」「TypeScriptのstrictモードです」と説明し直す必要があります。会話が長くなると、AIが前の方を忘れてしまうこともあります。CLAUDE.mdがあれば、プロジェクトのルールを一度だけ書けば、AIは起動のたびに自動で読み込み、ルールを守り続けます。
-->

---

# Before / After — 効果の実感

**Before (CLAUDE.md なし)**:
- AI「テストを書きました」 → **Jestで書いてる** (プロジェクトはpytest)
- AI「コードを修正しました」 → **2スペースインデント** (プロジェクトは4スペース)
- 毎回指示を繰り返す必要がある

**After (CLAUDE.md あり)**:
- AI「pytestでテストを書きました」 → ✅ 自動で適切なツールを選択
- AI「4スペースインデントで修正しました」 → ✅ コーディング規約を遵守
- **指示なしで正しい振る舞い**

<!--
この違いを見てください。CLAUDE.mdがないと、AIはプロジェクトの事情を知らないので、デフォルトの振る舞いをします。PythonプロジェクトなのにJestでテストを書いたり、4スペースのプロジェクトなのに2スペースでコードを書いたり。CLAUDE.mdがあれば、AIは最初からプロジェクトのルールを理解しているので、指示しなくても正しいツールと規約を使います。この違いが、CLAUDE.mdの価値です。
-->

# 5セクション構成 — 30行で書ける

CLAUDE.mdは以下の5セクションで構成するのが基本:

| # | セクション | 役割 | 例 |
|---|----------|------|-----|
| 1 | 使用言語・FW | 何を使っているか | Python 3.12 / FastAPI |
| 2 | コーディング規約 | スタイルの統一 | 4スペース / 型ヒント必須 |
| 3 | よく使うコマンド | ビルド・テスト・実行 | `pytest` / `uvicorn` |
| 4 | 禁止事項 | やってはいけないこと | `rm -rf` 禁止 / `main`にpush禁止 |
| 5 | プロジェクト構造 | ディレクトリの意味 | `src/` = 本体 / `tests/` = テスト |

> 💡 これだけで **約30行**。最初はこれだけで十分

<!--
CLAUDE.mdは5つのセクションで構成します。使用している言語とフレームワーク、コーディング規約、よく使うコマンド、禁止事項、そしてプロジェクトのディレクトリ構造です。全部書いても約30行。最初はこのミニマム構成で十分です。後から必要に応じて追加していけばOKです。
-->

# 30行ミニマム版の例

```markdown
# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM: SQLAlchemy 2.0

## コーディング規約
- インデント: 4スペース
- 型ヒント必須 (mypy strict)
- docstringはGoogle形式

- テスト: pytest tests/

- lint: ruff check .

## 禁止事項

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = DBマイグレーション
```

# セクション詳細 — 使用言語・フレームワーク

**何を書くか**:
- プログラミング言語とバージョン
- フレームワークとライブラリ
- パッケージマネージャー

**書くべき理由**:
- AIが **適切な構文** でコードを生成する
- AIが **正しいライブラリ** を選択する
- バージョン違いのエラーを **未然に防ぐ**

> 例: 「Python 3.12」と書けば、AIはwalrus演算子を使えると判断する

<!--
最初のセクション「使用言語・フレームワーク」の詳細です。ここには言語のバージョンも含めます。例えば「Python 3.12」と書いておけば、AIは3.12で追加された機能を使って良いと判断します。逆に「Python 3.9」と書いておけば、3.10以降の機能は使わなくなります。バージョン情報は意外と重要です。
-->

---

# セクション詳細 — コーディング規約

**何を書くか**:
- インデント幅 (2スペース / 4スペース / タブ)
- 命名規則 (camelCase / snake_case)
- リンター・フォーマッターの設定

**書くべき理由**:
- AIが生成するコードの **スタイルが統一される**
- プロジェクトの一貫性が **自動で保たれる**
- コードレビューの **指摘事項が減る**

> 例: 「4スペース」と書けば、AIは2スペースでコードを書かなくなる

<!--
コーディング規約のセクションです。インデント幅や命名規則を書いておくと、AIが生成するコードのスタイルが統一されます。「4スペース」と書いておけば、AIは常に4スペースでインデントします。これは地味ですが、コードレビューでの指摘を大幅に減らします。
-->

---

# セクション詳細 — よく使うコマンド

**何を書くか**:
- テストの実行コマンド
- ビルド・起動コマンド
- Lint・フォーマットコマンド

**書くべき理由**:
- AIが **正しいコマンド** を使ってテストを実行する
- 「どうやってテスト動かすんだっけ」が **なくなる**
- 自動修正後に **自動でテストが走る**

| プロジェクト | テストコマンド |
|------------|--------------|
| Python | `pytest tests/` |
| JavaScript | `npm test` |
| Go | `go test ./...` |

<!--
よく使うコマンドのセクションです。テストの実行コマンドを書いておくと、AIはコードを修正した後に自動的にテストを実行してくれます。プロジェクトによってテストコマンドは違います。Pythonならpytest、JavaScriptならnpm test、Goならgo test。これを書いておけば、AIが間違ったコマンドを使うことがなくなります。
-->

---

# セクション詳細 — 禁止事項

**何を書くか**:
- 危険な操作の禁止 (`rm -rf` 等)
- 本番環境への直接アクセス禁止
- 特定ブランチへのpush禁止

**書くべき理由**:
- AIが **危険な操作を実行前に止まる**
- プロジェクトの安全性が **自動で担保される**
- 人間が「やめて」と言う **手間がなくなる**

> 例: 「本番DBの直接操作禁止」と書けば、AIは本番DBを触るコマンドを実行しない

<!--
禁止事項のセクションは、安全性の観点で非常に重要です。AIに「この操作は絶対にしないで」と明示的に伝えることができます。「本番DBの直接操作禁止」「mainブランチへの直接push禁止」と書いておけば、AIはこれらの操作を実行しようとした瞬間に立ち止まります。後の章で学ぶhookと組み合わせると、さらに強力な安全装置になります。
-->

# よくある書き方の失敗 3選

**❌ 失敗1: 長すぎる**
- 100行以上のCLAUDE.md → AIが重要な部分を見落とす
- 解決: **30行以内**に収める。詳細は別ファイルに

**❌ 失敗2: 曖昧な表現**
- 「きれいなコードを書いて」 → AIの解釈がバラバラ
- 解決: **具体的な規約**を書く (例: 「4スペース」)

**❌ 失敗3: 矛盾する指示**
- 「TypeScriptを使って」+ 「JavaScriptで書いて」 → AIが混乱
- 解決: **一貫性**を確認してから保存する

<!--
CLAUDE.mdを書く時によくある失敗を3つ紹介します。1つ目は長すぎること。100行以上書くと、AIが重要な部分を見落とす可能性があります。最初は30行以内に収めましょう。2つ目は曖昧な表現。「きれいなコード」ではAIに伝わりません。「4スペース」「型ヒント必須」のように具体的に書きましょう。3つ目は矛盾する指示。TypeScriptとJavaScriptを同時に指定するとAIが混乱します。保存前に一貫性を確認してください。
-->

---

# CLAUDE.mdを書く時のポイント

1. **短く始める** — 最初は5セクション30行で十分
2. **具体的に書く** — 「きれいに」ではなく「4スペース」
3. **禁止事項を必ず入れる** — 安全性の最低限
4. **随時アップデート** — プロジェクトが変わったら更新する
5. **AIの振る舞いを見て調整** — 期待と違ったら追記する

> 💡 **完璧を目指さない** — 30行のCLAUDE.mdが、0行の何倍も価値がある

<!--
CLAUDE.mdを書く時の5つのポイントです。一番大事なのは「短く始める」ことです。完璧なCLAUDE.mdを最初から書こうとすると、書けません。まずは5セクション30行で始めて、実際にAIを使いながら足りない部分を追加していくのが正しいアプローチです。30行のCLAUDE.mdが、0行の何倍も価値があります。
-->
