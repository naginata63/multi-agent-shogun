# 「完了しました」— 本当に完了していますか?

> AIエージェントが「タスク完了」と報告したのに、実際にはファイルが存在しなかったり、テストが通っていなかったりする。これは**毎日起きている**ことです。

- エージェントは「主観的には完了」を報告しがち
- 人間の目視チェックは属人的で抜けが生じる
- **機械的な検証** だけが確実な完了保証を与える

> **Supervisor Pattern (監督パターン)**: 上司が部下の完了報告を**必ず再確認**する考え方 — 信頼するが、検証する

> **被害の実例**: AIに「テスト通過」と報告させてマージした結果、本番環境が1時間ダウン。原因はAIが「通過したつもり」で書いたテスト結果でした。

<!--
つかみのスライドです。エージェントにタスクを任せて完了報告を受け取ったあと、実際に確認したら違っていた、こういう経験は多くの開発者が持っているはずです。Supervisor Patternは、上司が部下の完了報告を必ず再確認するパターンです。信頼はするが、必ず検証するという姿勢が、エージェント開発での品質保証の基本です。
-->

# Supervisor Pattern と Phase-Gating

**Supervisor Pattern (監督パターン)**: AIの作業に対して、外部から検証・承認を挟む考え方

> 日常の例え: レストランの厨房で、シェフが完成した料理を必ず味見してから配膳に出す仕組みと同じです。料理人を信用していないのではなく、ミスを防ぐための最後の関所です。

**Phase-Gating (工程関所)**: 作業の各段階の終わりに**Gate (関所)** を置き、合格しないと次の段階に進めない仕組み

| 役割 | 何をするか | 日常の例え |
|------|-----------|-----------|
| **Supervisor** | 外部からの完了検証 | シェフの味見チェック |
| **Phase-Gating** | 段階ごとの通過条件 | 調理→盛付→配膳の各チェック |

> どちらも「人間のあいまいな判断」を「機械的な合格/不合格」に置き換えるのがポイントです

<!--
Supervisor PatternとPhase-Gatingは、エージェント開発における品質保証の2つの柱です。Supervisorは外部からの完了検証、Phase-Gatingは各Phaseの通過条件を強制する仕組みです。どちらも、人間の目視確認に頼らない自動化された品質保証が核心です。
-->

# 完了条件5項目 — 何を検証すべきか

AIのタスク完了を宣言する前に、5つの条件を満たしているか:

| # | 検証項目 | 理由 | 料理の例え |
|---|---------|------|-----------|
| 1 | **成果物が存在する** | 存在しないものは完了ではない | 料理が皿に盛られている |
| 2 | **成果物が正しい内容** | 存在するだけでは不十分 | 注文通りの料理である |
| 3 | **品質チェック済み** | 第三者による確認 | シェフが味見済み |
| 4 | **報告が提出された** | 完了の記録 | 配膳伝票にサイン |
| 5 | **事後手続き完了** | 保存・共有等の必須ステップ | 料理を冷蔵庫に保存 |

> これらを毎回人間が確認するのは非現実的 → **自動化が必須**

<!--
完了を宣言する前に検証すべき5つの項目です。料理の例えで考えてみるとイメージしやすいです。皿に盛られているか、注文通りか、味見済みか、伝票にサインされたか、保存されたか。これらを毎回人間が確認するのは現実的ではありません。だから自動化が必要です。
-->

# 宣言型検証 Gate — 設計思想

**宣言型検証** = AIが事前に完了条件を**YAML** (設定ファイル) に書いておき、Gateがそれを機械的にチェックする

> **YAML (ヤムル)**: AIへの指示書を書くためのファイル形式。インデント (字下げ) で階層を表します。メモ書きのようなシンプルな見た目が特徴です。

```yaml
# task YAML の宣言例 (AIへの指示書)
post_steps:
  - path/to/output_file.csv    # このファイルが存在するか検証
  - reports/report.yaml        # 報告書が提出されたか検証
verify:
  command: "npm test"          # このコマンドでテストを実行
  timeout_seconds: 60          # 60秒以内に終わらなければ失敗
```

**3つの設計原則**:

1. **宣言型**: YAMLに条件を書く — Gateは推測しない
2. **opt-in**: verify欄を書かなければ素通り — いきなり全部導入しなくてOK
3. **自動連鎖**: hook → 検証 → Gate が自動で回る

<!--
宣言型検証の設計思想です。AIが事前に完了条件をYAMLに宣言しておき、Gateはそれを機械的に照合します。YAMLは「AIへの指示書」というシンプルな設定ファイルです。3つの設計原則があります。宣言型はYAMLに条件を書くことでGateの推測を排除します。opt-inはverify欄を書かなければ素通りする設計で、段階的な導入が可能です。自動連鎖はhookが自動的に検証を起動する仕組みです。
-->

# hook による自動検証

> **hook (フック)**: 「〇〇が起きたら、自動で△△を実行する」という仕掛け。ここでは「ファイル編集が終わったら、自動でテストを実行」します。

**実行フロー** (4ステップ):

1. AIがYAMLファイルを編集する
2. **hook** が「編集された!」を検知して自動発火
3. `verify_result: pending` を見つけたら → 検証コマンドを実行
4. 結果で `verify_result` を `pass` (合格) / `fail` (不合格) に更新

> hookの設定は**設定ファイル (settings.json)** に書きます。JSONは `{ "キー": "値" }` という形式の設定ファイルです。

> ※本講座では設定ファイルの書き方の概念を解説します。実際のプロジェクトではスクリプトのパスやタイムアウト値を環境に合わせて調整してください。

<!--
hookによる自動検証の仕組みです。AIがYAMLファイルを編集するとhookが発火し、検証が自動で走ります。verify_resultがpendingの場合、宣言されたコマンドを自動実行し、結果でpassまたはfailに上書きします。JSONの詳細な書き方はここでは触れませんが、「設定ファイルにhookを登録しておくと自動で動く」というイメージを持ってください。
-->

# Gate が BLOCK する3つのパターン

| # | BLOCK条件 | 対応 |
|---|----------|------|
| 1 | **成果物が存在しない** — 指定されたファイルがない | 成果物を生成する |
| 2 | **検証が未実行または失敗** — テストが通っていない | テストを実行して合格を確認 |
| 3 | **品質チェック不足** — レビューが未実施 | レビューを実施する |

- **BLOCK メッセージ** が「何が足りないか」「どう直せばいいか」を具体的に指示
- AIはメッセージを読んで**自力で修正**可能

> つまり「怒られる」のではなく「次に何をすべきか」がわかる仕組みです

<!--
GateがBLOCKする3つのパターンです。成果物なし、検証不合格、品質チェック不足の3つです。BLOCKされた場合、具体的な対応方法がメッセージで指示されるため、AIは自力で問題を修正できます。これがこの仕組みの実用的な強みです。
-->

# 「未完了の完了報告」の再現例

AIが `status: done` (完了) を宣言したが、実際には:

```yaml
# YAML (AIが書いた状態)
- task_id: subtask_build_widget
  status: done            # ← AIは「完了」と宣言
  verify:
    command: "npm test"
  verify_result: pending  # ← まだテストが実行されていない
  post_steps:
    - dist/widget.js      # ← このファイルがまだ存在しない
```

> **なぜ矛盾が起きる?** AIは「自分のタスクは終わった」と主観的に判断します。しかし「テストを走らせる」「ファイルを保存する」といった事後手続きを忘れているケースがよくあります。人間でも「作業は終わったけど報告書まだ」がありますよね。

**Gate の反応**:
```
BLOCK: subtask_build_widget
  - post_steps 未完了 (1件欠落): dist/widget.js
  - verify_result: pending (テスト未実行)
対応: 成果物を生成し、テストを実行してから status:done を宣言せよ
```

<!--
未完了の完了報告の再現例です。AIはstatus:doneを宣言しましたが、verify_resultはpendingのままで、post_stepsに指定されたファイルも存在していません。AIは「作業自体は終わった」と思っているが、事後手続きを忘れているのです。Gateはこれを検出してBLOCKし、具体的な対応方法を指示します。
-->

# 実運用での調整ポイント

**条件の追加**:
- 新しい検査項目はGateスクリプトにブロックを追加するだけ
- 例: コードの書き方チェック、セキュリティスキャン

**条件の除外**:
- `verify:` 欄を書かなければ自動的にSKIP (opt-in設計)
- 最初は軽いチェックだけ → 慣れてきたら厳しく、が可能

**閾値 (しきいち) の変更**:
- 品質チェックの最低回数を調整
- 検証のタイムアウト秒数を調整

**ログの活用**:
- 検証コマンドの実行結果ログ — どこで失敗したか追跡可能
- 品質チェックの全履歴ログ — 改善のデータに

<!--
実運用での調整ポイントです。条件の追加はGateスクリプトに新しい検査ブロックを追加するだけで済みます。条件の除外はverify欄を書かないだけで自動的にSKIPされます。閾値の変更やログの活用も柔軟に対応できます。opt-in設計のおかげで、プロジェクトの成熟度に合わせて段階的に導入できるのが大きな利点です。

# この章のまとめ

- **Supervisor Pattern (監督パターン)**: AIの完了報告を外部から機械的に検証する考え方
- **Phase-Gating (工程関所)**: 各工程の完了をGateで強制検証する仕組み
- **宣言型検証**: YAML (設定ファイル) に完了条件を宣言 → Gateが機械的に照合
- **hook** が検証を自動起動 → `pending` (未実行) を `pass`/`fail` に更新
- **opt-in 設計** で段階的導入が可能 — 最初は軽く、慣れてから厳しく
- **BLOCK メッセージ** が具体的な対応方法を指示 → AIは自力で修正可能

<!--
この章のまとめです。Supervisor PatternはAIの完了を外部から検証するパターンで、Phase-Gatingは各工程の通過を強制する仕組みです。宣言型検証、hookによる自動化、opt-in設計、具体的なBLOCKメッセージの4つが実装の柱です。

# 確認問題

**Q1**: Supervisor Patternの「Supervisor」とは何を指すか？
- A: AI自身の自己評価機能
- B: AIの完了報告を外部から検証する仕組み
- C: AIを監視するカメラ

> 実務ヒント: チーム開発のコードレビューと同じ考え方です。「完了」の判断を作業者本人ではなく第三者に任せる仕組みです。

**Q2**: YAMLにverify欄を書かなかった場合、どうなるか？
- A: 常にBLOCKされる
- B: 検査をスキップして素通りする
- C: デフォルトのコマンドが自動実行される

> 実務ヒント: 最初は何も書かなくてOK。慣れてきたら少しずつ検証を追加できるのがopt-in設計の良さです。

**Q3**: hookがverify_result: pendingを検出したとき、何をするか？
- A: メールで通知を送る
- B: 検証コマンドを実行し、結果でverify_resultを上書きする
- C: タスクを自動的に削除する

> 実務ヒント: 「テスト未実行」を見つけたら自動でテストを走らせるイメージです。
