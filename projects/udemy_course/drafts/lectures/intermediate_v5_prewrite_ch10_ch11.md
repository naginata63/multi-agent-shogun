## ch10. 困った時の逆引き辞典 — 症状 → 現象 → 対策 → 該当章

### [1] 対象問題

実務で AI が想定外の動きをしたとき、**「講座のどこを見直せばいいか」が分からない**という問題を解決する。

具体的には、本講座で扱った 11 件の困りごと全てを、2 つの視点から即座に引けるようにする:

- **症状からの逆引き**: 現場で起きた現象 → その原因 → 対策道具 → 該当章
- **3 層からの逆引き**: 「プロンプト層 / コンテキスト層 / ハーネス層のどれが欠けているか」→ 該当する困りごと → 解決章

11 件の困りごと一覧 (ch00 で提示済み、本章で再構成):

| # | 困りごと | 欠如している層 | 解決章 |
|---|---------|---------------|-------|
| ① | AI が推測で動く・確認せず突進 | プロンプト層 | ch02 |
| ② | AI 出力を盲信して失敗 | ハーネス層 | ch06 |
| ③ | 既にあるものを確認せず新規作成 | プロンプト層 | ch01 |
| ④ | 失敗を fallback で隠す | ハーネス層 | ch08 |
| ⑤ | 「止めろ」が AI に伝わらない | ハーネス層 | ch08 |
| ⑥ | 「言ったのにやってくれない」 | コンテキスト層 | ch05 |
| ⑦ | 完了したことを次に伝え忘れる | コンテキスト層 | ch04 |
| ⑧ | API 変更後にテストせず壊れる | ハーネス層 | ch07 |
| ⑨ | 大事なドキュメントを確認しない | コンテキスト層 + ハーネス層 | ch07 / ch04 |
| ⑩ | 同じ作業を毎回手動で繰り返している | プロンプト層 | ch01 |
| ⑪ | 長文の中央の参照情報が読み飛ばされる | コンテキスト層 | ch03 |

3 層別の集計:
- プロンプト層欠如由来: 3 件 (① ③ ⑩)
- コンテキスト層欠如由来: 3 件 (⑥ ⑦ ⑪)
- ハーネス層欠如由来: 4 件 (② ④ ⑤ ⑧)
- 複合層欠如由来: 1 件 (⑨)

### [2] 持ち帰り

受講者が章末で言える 1 文:
> **「詰まったら、まず『どの層が欠けているか』を特定し、3 段逆引き表で該当章を即座に見つけられる」**

できるようになる 1 動作:
- 症状が起きたとき → 3 段早見表で「症状 → 現象 → 対策 → 該当章」を 30 秒で引く
- 3 層インデックスで「どの層が足りないか」から解決章にたどり着く

### [3] ストーリー (現象 → 対策)

**現象**:
受講者が実際の業務で AI を使っていて「何か変だ」と感じた瞬間、どうするか。講座で学んだ内容は多い。ch01 から ch09 まで、プロンプト層・コンテキスト層・ハーネス層と多数の道具を学んだ。しかし「AI が勝手に進めた」「前回の修正を忘れている」「エラーを隠した」という症状が起きたとき、「あれ、どの章で対策を学んだっけ?」と迷子になる。

**対策**:
症状から原因を逆引きする 2 つの辞書を渡す。
1. **3 段早見表**: 症状 (現場で起きること) → 現象 (AI の中で何が起きているか) → 対策 (使うべき道具) → 該当章、の 4 列で全 11 件を整理
2. **3 層インデックス**: 「プロンプト層が足りない」「コンテキスト層が足りない」「ハーネス層が足りない」の 3 区分から、該当する困りごとを一覧表示

この 2 つの辞書を持てば、実務で詰まった瞬間に「自分は今、どの層を強化すべきか」が即座に分かる。11 件の困りごとは、すべて 3 層のいずれかが欠けているから起こる。だから「どの層が欠けているか」が分かれば、解決策も自動的に決まる。

### [4] 実例

**3 段早見表 (11 件全件・症状→現象→対策→該当章)**:

| 症状 (現場で起きること) | 現象 (AI の何が原因か) | 対策 (持ち帰った道具) | 該当章 |
|------------------------|---------------------|--------------------|-------|
| AI が確認なしに突進した | プロンプトに確認指示が弱い | 診断 10 項目 + 確認問いかけテンプレ | ch02 |
| AI 出力を信じて失敗した | 第三者レビューが入っていない | `/advisor` 呼び出し | ch06 |

| エラー握り潰しで本番ダウン | 失敗検知の仕組みなし | 段階確認 + watcher | ch08 |

| 長文 doc の中央が読まれない | Lost in the Middle | TL;DR + 末尾サマリ + 必要時取得 | ch03 / ch04 |

**3 層インデックス (層別分類)**:

| 不足している層 | 該当する困りごと | 対策章 |
|--------------|----------------|-------|
| **プロンプト層不足** | ① ③ ⑩ | ch01 / ch02 |
| **コンテキスト層不足** | ⑥ ⑦ ⑪ (+ ⑨ 補足) | ch03 / ch04 / ch05 |
| **ハーネス層不足** | ② ④ ⑤ ⑧ (+ ⑨ 補足) | ch06 / ch07 / ch08 |
| **複合層不足 (要組合せ)** | ⑨ | ch04 + ch07 |

**受講者の現場に置き換えた具体例**:

- **営業担当のケース**: AI に「先週の商談メモを要約して」と頼んだら、AI が勝手に「案件はクローズ」と出力した (症状: AI が確認なしに突進 = ①)。→ プロンプト層不足 → ch02 の診断 10 項目で「確認問いかけ」を追加
- **PM サポートのケース**: 仕様変更メールを AI に読ませたら、古い仕様の箇所を無視して回答した (症状: 長文 doc の中央が読まれない = ⑪)。→ コンテキスト層不足 → ch03 の TL;DR 配置 + ch04 の必要時取得

**似た症状の判別基準**:

  - ① = AI が**出力前に**確認をスキップ → プロンプト層 (ch02)
  - ② = AI の**出力を**人が鵜呑みにした → ハーネス層 (ch06)

## ch11. 自分専用の Skill を作る — 3 層統合の到達点

### [1] 対象問題

ここまで 3 つの層 (プロンプト層 / コンテキスト層 / ハーネス層) を個別に学んできたが、**自分の業務に常駐する AI 道具がまだ 1 つも完成していない**という問題を解決する。

`/command` はプロンプトの資産化 (ch01) として学んだ。しかし `/command` は「人が意図的に呼び出す」前提。**AI が文脈を読み取って自律的に呼び出す仕組み**が欲しい。それが Skill。

Skills の縦糸 (講座全体での繋がり):
- **ch01**: `/command` で「プロンプトの資産化」を学ぶ + Skill の布石 (「`/command` の先にあるもの」として言及)
- **ch07**: hook と Skill の連携を学ぶ (hook が発動タイミングを制御し、Skill が処理内容を提供)
- **ch11**: 自分専用の Skill を**完成させる** (3 層統合の到達点)

本章は、講座で学んだ 3 層を統合して 1 つの「自分専用の常駐道具」を作る、講座の最終到達点。

### [2] 持ち帰り

受講者が章末で言える 1 文:
> **「自分の業務知識を Skill に書き込み、AI が文脈で自律的に呼び出す常駐道具を完成させた」**

できるようになる 1 動作:
- `.claude/skills/` に自分専用の Skill ファイル (frontmatter + 本文) を作成できる
- Skill 内でプロンプト層・コンテキスト層・ハーネス層を統合していることを説明できる
- Skill と hook を連携させて、特定のタイミングで自動発動する構想を立てられる

### [3] ストーリー (現象 → 対策)

**現象**:
受講者は ch01 で `/command` を使い、プロンプトの資産化を体験した。ch07 で hook を学び、AI の動作を自動化する仕組みを知った。しかし、`/command` は「毎回自分で呼び出す」必要があり、hook は「タイミングの制御」であって「処理の中身」ではない。**プロンプトの資産化**と**自動化の仕組み**の間に、もう一段上の統合が必要だと気づく。

例えば、「毎週月曜に議事録を要約して日報を作る」という業務がある。`/command` で「日報作成プロンプト」を資産化しても、毎回「議事録データを探して、memory から前週の記録を呼び出して、フォーマットに当てはめる」という手順は自分で指示しなければならない。

**対策**:
**Skill** を作る。Skill = `.claude/skills/{name}.md` に業務知識を書き込み、AI が文脈 (「日報の季節だ」「議事録が更新された」等) を読み取って自律的に呼び出す仕組み。

Skill は 3 層を統合する:
- **プロンプト層**: Skill 内に書き込む「日報のフォーマット」「要約の方針」という指示
- **コンテキスト層**: Skill 実行時に memory から過去の日報を検索し、関連情報を自動取得
- **ハーネス層**: hook 連携で「議事録ファイルが更新されたら自動発動」という仕組み

これにより、`/command` の「毎回呼び出す」から、Skill の「AI が自律的に動く」へ進化する。講座の到達点は、**自分専用の常駐 AI 道具**を作ること。

### [4] 実例

**題材: 「週次業務日報の半自動化 Skill」の実装演習**

受講者が実際に手を動かして作る Skill の構造:

```
.claude/skills/weekly_report.md
```

**frontmatter (Skill のメタ情報)**:
```yaml
---
name: weekly-report
description: 週次業務日報を半自動生成する。議事録要約 + 過去記憶検索 + ドラフト出力。
---
```

**Skill 本文 (業務知識の書き込み)**:
```markdown
## この Skill がやること
1. 今週の議事録ファイル (.md) を読み込む
2. 議事録から「決定事項」「宿題」「次週の予定」を抽出
3. 過去の週報 (memory) を検索し、継続中のタスクを参照
4. 指定フォーマットで週報ドラフトを出力

## 出力フォーマット
### 今週のハイライト (3 行以内)
### 決定事項
### 宿題 (担当・期限付き)
### 来週の予定

## 注意事項
- 推測で補完せず、議事録にない情報は「未記載」と明記
- 前週からの継続タスクは必ず参照
```

**3 層統合の解説 (受講者が理解すべきポイント)**:

| 層 | この Skill での役割 | 対応箇所 |
|----|-------------------|---------|
| プロンプト層 | 日報フォーマット・抽出ルールの指示 | Skill 本文の「やること」「フォーマット」「注意事項」 |
| コンテキスト層 | 過去の週報を memory から自動検索 | 「過去の週報を検索」ステップ |
| ハーネス層 | hook 連携で議事録更新時に自動発動 | ch07 で学んだ hook と連携 (構想レベル) |

**受講者の現場に置き換えたバリエーション**:
- **営業担当**: 「商談メモから提案書のドラフトを作る Skill」— 顧客要件を memory から検索、提案テンプレに当てはめ
- **PM サポート**: 「ミーティング議事録からタスク管理表を更新する Skill」— 担当者・期限を抽出、既存タスクと統合
- **データ分析担当**: 「分析レポートの定型フォーマット生成 Skill」— クエリ結果をテンプレに流し込み、前回との差分を memory から参照

**演習の流れ**:
1. 自分の業務で「毎週繰り返している手作業」を 1 つ選ぶ (5 分)
2. その手作業を「何を読み込み → 何を出力するか」に分解する (10 分)
3. `.claude/skills/` に frontmatter + 本文を書く (10 分)
4. 実際に呼び出して動作確認 (5 分)

**章末まとめ**:
- 講座修了: 3 層 (プロンプト / コンテキスト / ハーネス) を学び、それらを統合する Skill を完成させた
- 次の一手: 自分の現場で「週に 1 回使う Skill」を 1 つ作り、1 ヶ月運用してみる
- 発展: ch07 の hook と連携して「ファイル更新で自動発動」を実現する
