# 「上手くいったと思う」は品質保証ではない

> プロンプトを書いて、AIの出力を見て、「よし、これでいいだろう」 — この判断を誰が検証していますか？

- **自己評価の罠**: 書いた本人は「意図通り」に見えても、第三者には別の解釈ができる
- **沈黙の失敗**: エラーが出ない ≠ 正しい結果
- **再現性の欠如**: 今日うまくいったプロンプトが明日も同じ結果を出す保証はない

<!--
ここが最初のつかみです。プロンプトを書いてAIの出力を確認した時、「上手くいったと思う」と満足していませんか？ その判断を検証する人は誰でしょう？ 答えは「誰も検証していない」です。これが品質保証の最大の罠です。書いた本人は意図通りに見えても、別の視点では別の解釈ができます。エラーが出ないからといって結果が正しいとは限りません。今日うまくいったプロンプトが、明日も同じ結果を出す保証もありません。ここに品質Gateが必要な理由があります。
-->

# なぜ「別のAI」に審査させるのか

**裁判所のメタファー**:

- **被告** = プロンプトを書いた本人（AI自身）
- **検察** = 品質基準（QC template）
- **裁判官** = advisor（別のAI）

**別のAIに審査させる3つの利点**:

1. **盲点の検出** — 書いた本人には見えない問題を発見
2. **バイアスの排除** — 「自分の書いたものは良く見える」を防止
3. **客観的な基準** — 感覚ではなく判定基準で PASS/FAIL を決定

<!--
裁判所を想像してください。被告はプロンプトを書いた本人、つまりAI自身です。検察は品質基準、QC templateです。そして裁判官がadvisor、別のAIです。なぜ裁判官を別のAIにする必要があるのか。3つの理由があります。1つ目、書いた本人には盲点があります。自分では気づかない問題を、別の視点が見つけます。2つ目、人間もAIも自分の書いたものを良く見る傾向があります。これがバイアスです。別のAIならこのバイアスを排除できます。3つ目、感覚ではなく明確な判定基準で合格・不合格を決められます。
-->

---

# QC template の構造 [L1: Evaluate]

QC template は **品質の採点表** です:

```
QC checklist:
- 期待する出力が得られたか → PASS/FAIL
- 指示の曖昧さがないか   → PASS/FAIL
- 境界条件をカバーしているか → PASS/FAIL
  ...（項目数はプロジェクトに応じて可変）
```

**判定ルール**:
- **全項目 PASS** → 品質 Gate 通過
- **1項目でも FAIL** → 修正が必要（advisor が修正案を提示）

<!--
QC templateは品質の採点表です。プロンプトの出力を複数の項目でチェックし、それぞれPASSかFAILかを判定します。判定ルールはシンプルです。全項目がPASSなら品質Gateを通過できます。1項目でもFAILがあれば、advisorが修正案を提示してくれます。重要なのは、この判定を人間の感覚ではなく、機械的な基準で行うことです。これにより、品質のばらつきを防ぎます。
-->

---

# QC template の判定基準例

| # | チェック項目 | 判定 |
|---|-------------|------|
| 1 | 出力フォーマットが仕様通りか | PASS / FAIL |
| 2 | 指示に曖昧な表現がないか | PASS / FAIL |
| 3 | 境界条件（空入力・最大長等）を処理するか | PASS / FAIL |
| 4 | エラーハンドリングが定義されているか | PASS / FAIL |
| 5 | セキュリティ上の懸念がないか | PASS / FAIL |

> プロジェクトの要件に合わせて項目を増減させる — **5〜15項目** が実用的

<!--
具体的な判定基準の例を示します。出力フォーマットが仕様通りか、曖昧な表現がないか、境界条件を処理するか、エラーハンドリングが定義されているか、セキュリティ上の懸念がないか。これらは一例ですが、プロジェクトの要件に合わせて項目を増減させます。実務的には5〜15項目程度が適正です。多すぎるとチェックが形骸化し、少なすぎると品質の網羅性が下がります。
-->

# advisor + QC 連携フロー [L1: Evaluate]

**品質 Gate の全体像**:

1. **プロンプト実行** → AIがタスクを完了
2. **advisor 呼出** → 別のAIが会話全体を審査
3. **QC template 適用** → チェック項目ごとに PASS/FAIL 判定
4. **結果に応じた分岐**:
   - 全 PASS → 品質 Gate 通過 → 次へ
   - FAIL あり → 修正案を受け取り → 再実行 → 再度 Gate

> このサイクルを **自動化** することで、品質確認の手間をゼロに近づける

<!--
advisorとQC templateがどう連携するか、全体像を説明します。まず、プロンプトが実行されてAIがタスクを完了します。次に、advisorが呼び出され、別のAIが会話全体を審査します。審査の際にQC templateが適用され、各チェック項目がPASSかFAILか判定されます。結果に応じて分岐します。全PASSなら品質Gateを通過して次に進みます。FAILがあれば、advisorの修正案を受け取って再実行し、再度Gateにかけます。このサイクルを自動化することで、品質確認の手間を大幅に減らせます。
-->

# 品質 Gate が検出する盲点5選

| # | 盲点 | 具体例 | 検出方法 |
|---|------|--------|---------|
| 1 | **指示の暗黙前提** | 「適切に」と書いたが基準が不明 | QC: 曖昧語句チェック |
| 2 | **エラー隠蔽** | 失敗時に空文字を返す | advisor: 異常系レビュー |
| 3 | **セキュリティ穴** | ユーザー入力を直接コマンドに埋め込み | QC: インジェクション検査 |
| 4 | **文脈の忘却** | 長い会話で最初の指示を無視 | advisor: 全文走査 |
| 5 | **過剰な一般化** | 「すべての場合に対応」は不可能 | QC: 境界条件テスト |

<!--
品質Gateが実際にどんな盲点を検出するか、5つの典型的な例を紹介します。1つ目、指示の暗黙前提です。「適切に処理して」と書いた場合、「適切」の基準が不明確です。QC templateの曖昧語句チェックで検出します。2つ目、エラー隠蔽です。失敗した時に空文字を返すコードは、表面上はエラーがなくても実は壊れています。3つ目、セキュリティ穴です。ユーザー入力をそのままコマンドに埋め込むとインジェクション攻撃の危険があります。4つ目、文脈の忘却です。長い会話の中で、AIが最初の指示を忘れてしまう問題です。5つ目、過剰な一般化です。「すべての場合に対応する」という指示は実現不可能で、境界条件テストで検出します。
-->

---

# 盲点の具体例: Before / After

**Before**（品質Gateなし）:
- AIの出力を見て「まあいいだろう」で進める
- 本番環境で **予期しない挙動** が発覚

**After**（品質Gateあり）:
- advisorが **暗黙前提を3件** 検出
- QC templateが **セキュリティ懸念を1件** 検出
- 修正後に再実行 → **全PASS** で安心してリリース

> Before: 品質は「運」に左右される / After: 品質は「基準」で保証される

<!--
BeforeとAfterを比較してみましょう。品質Gateがない場合、AIの出力を見て「まあいいだろう」という主観的な判断で進めます。そして本番環境で予期しない挙動が発覚する、という悪いパターンに陥ります。一方、品質Gateがある場合、advisorが暗黙前提を検出し、QC templateがセキュリティ懸念を検出します。問題を修正して再実行すれば、全PASSで安心してリリースできます。品質が運に左右される状態から、基準で保証される状態へ。これが品質Gateの価値です。
-->
