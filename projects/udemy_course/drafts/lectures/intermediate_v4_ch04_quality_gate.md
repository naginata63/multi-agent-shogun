# 「上手くいったと思う」は品質保証ではない

> プロンプトを書いて、AIの出力を見て、「よし、これでいいだろう」 — この判断を誰が検証していますか？

- **自己評価の罠**: 書いた本人は「意図通り」に見えても、第三者には別の解釈ができる
- **沈黙の失敗**: エラーが出ない ≠ 正しい結果
- **再現性の欠如**: 今日うまくいったプロンプトが明日も同じ結果を出す保証はない

<!--
ここが最初のつかみです。プロンプトを書いてAIの出力を確認した時、「上手くいったと思う」と満足していませんか？ その判断を検証する人は誰でしょう？ 答えは「誰も検証していない」です。書いた本人は意図通りに見えても、別の視点では別の解釈ができます。エラーが出ないからといって結果が正しいとは限りません。今日うまくいったプロンプトが、明日も同じ結果を出す保証もありません。ここに品質Gateが必要な理由があります。
-->

# advisor とは — 公式レビュー機能 [L1: Evaluate]

**Claude Code に標準搭載された品質レビュー機能**:

- **呼出方法**: CLI上で `/advisor` と入力するだけ — 設定不要
- **仕組み**: 会話の全履歴を、より強力なAIモデル（Opus等）に転送して客観レビューを即座に取得
- **モデル選択**: 自動的に上位モデルが選ばれる — 利用者がモデルを指定する必要なし

> advisor は **「自分の作業全体を、より強いAIが一から見直す」** 仕組み — 追加費用は呼出1回分のみ

<!--
advisorはClaude Codeに標準搭載されている公式機能です。CLI上でスラッシュコマンド「/advisor」と入力するだけで、会話の全履歴がより強力なAIモデルに転送され、客観的なレビュー結果が即座に返ってきます。設定ファイルの編集やAPIキーの追加取得は一切不要です。重要なのは、advisorが「別の、より強いAI」に審査させるという点です。利用者自身がどのモデルを使うか指定する必要はなく、自動的に上位モデルが選ばれます。自分では気づかない問題を見つけることができる強力な機能です。
-->

# model pairing 戦略 — なぜOpusを最初から使わないのか

**強いモデル = 高品質 ではない**:

| 観点 | 最初からOpus | model pairing（推奨） |
|------|-------------|---------------------|
| **コスト** | 全工程で高額（3〜5倍） | レビュー時のみ追加費用 |
| **速度** | 全工程で遅い（2〜3倍） | 作業は速く、レビューは丁寧 |
| **品質** | 同じ視点での自己評価 | 別モデルによる客観レビュー |

**model pairing の考え方**:
- 作業モデル（Sonnet等）で **素早く実装** — 日常的な作業に十分な精度
- レビューモデル（Opus等）で **客観的に審査** — より深い推論で見落としを検出
- Opusを「最初から」使うと、**自分で書いて自分で見直す** のと同じ罠に陥る
- 別モデルに見せるからこそ **新しい視点** が得られる

<!--
ここが重要なポイントです。一番強いAIモデル、たとえばOpusを最初から使えば品質が最高になると思うかもしれませんが、そうではありません。理由は3つあります。1つ目、コストです。OpusはSonnetの3〜5倍のコストがかかり、全工程で使うと非常に高額になります。2つ目、速度です。Opusは処理が遅く、日常的な反復作業に使うと効率が大きく下がります。3つ目、そして最も重要なのが、同じモデルで書いて見直すと自己評価の罠に陥ることです。model pairingは、作業用の軽いモデルで素早く実装し、レビュー用の強いモデルで客観的に審査するという戦略です。強いモデルは「最初から」ではなく「レビューの時だけ」使うのが正解です。別のモデルに見せるからこそ、新しい視点による指摘が得られます。
-->

# 呼ぶタイミング — 3つの場面 [L1: Evaluate]

| タイミング | 状況 | 目的 |
|-----------|------|------|
| **Before substantive work** | 実装を始める前 | 方針の確認・見落としの検出 |
| **When stuck** | エラーが続く・方針が不明 | 原因の特定・代替案の提案 |
| **When complete** | タスク完了後 | 成果物の最終レビュー |

> advisor は **「早めに呼ぶほど効果的」** — 実装後に呼ぶと修正コストが大きい

<!--
advisorをいつ呼ぶべきか、3つの場面を解説します。1つ目、実装を始める前です。コードを書き始める前に呼ぶことで、方針の見落としを早期に検出できます。2つ目、行き詰まった時です。エラーが続いたり方針が不明になったりした時は、すぐにadvisorを呼びましょう。原因の特定や代替案の提案をしてくれます。3つ目、タスク完了後です。成果物の最終レビューとして呼びます。重要なのは、早めに呼ぶほど効果的だという点です。実装後に問題が見つかると修正コストが大きくなります。実装前の確認が最も効率的です。
-->

# 応答ルール — 簡潔で実用的なフィードバック

advisor のレビューは **100語以内・列挙型ステップ** で返ってくる:

1. **簡潔さ**: 100語以内 — 長文解説ではなく、次に取るべき行動がすぐ分かる
2. **列挙型**: 番号付きステップ — 「1. Xを確認 2. Yを修正 3. Zを実行」の形式
3. **実用性**: 理由の説明ではなく、具体的なアクションが中心

> **「次に何をすべきか」** が分かれば十分 — advisor は長文解説ではなく行動指針を返す

<!--
advisorの応答には明確なルールがあります。100語以内で、番号付きステップで返ってきます。長文の解説ではなく、「1. 〇〇を確認、2. 〇〇を修正、3. 〇〇を実行」という具体的なアクションが中心です。理由はシンプルです。あなたが次に何をすべきかをすぐ知りたいからです。長文の解説を読む時間があれば、その分早く修正に取り掛かれます。advisorは「なぜそうすべきか」の解説ではなく「次に何をすべきか」の行動指針を返すのが役割です。
-->

# サブエージェント (Agent) との違い

| 観点 | advisor | Agent（サブエージェント） |
|------|---------|------------------------|
| **役割** | レビュー・指摘 | タスクの実行・コード作成 |
| **アクション** | 読むだけ・変更しない | ファイル編集・コマンド実行可能 |
| **コンテキスト** | 会話全体を見る | 新しい独立セッションで動く |
| **コスト** | 低い（レビューのみ） | 高い（実行タスク分） |

**使い分けの基準**:
- 「この実装に問題ないか確認したい」 → **advisor**
- 「この処理を別のプロセスに委任したい」 → **Agent**

<!--
advisorとAgentツール、両方ともClaude Codeの機能ですが、役割が全く違います。advisorはレビュー専門です。会話全体を読んで問題を指摘しますが、ファイルを編集したりコマンドを実行したりはしません。一方、Agentはタスクの実行担当です。独立したセッションで動き、ファイルの編集やコマンドの実行ができます。コンテキストの見え方も違います。advisorは会話全体を見渡せますが、Agentは独立セッションなので親セッションの一部しか見えません。使い分けの基準はシンプルです。確認したい時はadvisor、委任したい時はAgentです。
-->

# QC template と連携フロー

**品質 Gate の全体像**:

1. **プロンプト実行** → AIがタスクを完了
2. **advisor 呼出** → より強いAIが会話全体を審査
3. **QC template 適用** → チェック項目ごとに PASS/FAIL 判定
4. **結果に応じた分岐**:
   - 全 PASS → 品質 Gate 通過 → 次へ
   - FAIL あり → 修正 → 再実行 → 再度 Gate

> QC template は品質の採点表 — プロジェクトごとに 5〜15項目 を定義

<!--
advisorとQC templateがどう連携するか、全体像を説明します。まず、プロンプトが実行されてAIがタスクを完了します。次に、advisorが呼び出され、より強いAIが会話全体を審査します。審査の際にQC templateが適用され、各チェック項目がPASSかFAILか判定されます。QC templateは品質の採点表で、プロジェクトごとに5〜15項目程度を定義します。全PASSなら品質Gateを通過して次に進みます。FAILがあれば、修正して再実行し、再度Gateにかけます。このサイクルを回すことで、品質確認の手間を大幅に減らせます。
-->

# 品質 Gate が検出する盲点5選

| # | 盲点 | 具体例 | 検出方法 |
|---|------|--------|---------|
| 1 | **指示の暗黙前提** | 「適切に」と書いたが基準が不明 | QC: 曖昧語句チェック |
| 2 | **エラー隠蔽** | 失敗時に空文字を返す | advisor: 異常系レビュー |
| 3 | **セキュリティ穴** | ユーザー入力を直接コマンドに埋め込み | QC: インジェクション検査 |
| 4 | **文脈の忘却** | 長い会話で最初の指示を無視 | advisor: 全文走査 |
| 5 | **過剰な一般化** | 「すべての場合に対応」は不可能 | QC: 境界条件テスト |

<!--
品質Gateが実際にどんな盲点を検出するか、5つの典型的な例を紹介します。1つ目、指示の暗黙前提です。「適切に処理して」と書いた場合、「適切」の基準が不明確です。QC templateの曖昧語句チェックで検出します。2つ目、エラー隠蔽です。失敗した時に空文字を返すコードは、表面上はエラーがなくても実は壊れています。3つ目、セキュリティ穴です。ユーザー入力をそのままコマンドに埋め込むとインジェクション攻撃の危険があります。4つ目、文脈の忘却です。長い会話の中で、AIが最初の指示を忘れてしまう問題です。5つ目、過剰な一般化です。「すべての場合に対応する」という指示は実現不可能で、境界条件テストで検出します。これらはすべて、advisorとQC templateを組み合わせることで機械的に検出できます。
-->
