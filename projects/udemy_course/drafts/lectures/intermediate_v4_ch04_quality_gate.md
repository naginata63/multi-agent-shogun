# 「上手くいったと思う」は品質保証ではない

> プロンプトを書いて、AIの出力を見て、「よし、これでいいだろう」 — この判断を誰が検証していますか？

- **自己評価の罠**: 書いた本人は「意図通り」に見えても、第三者には別の解釈ができる
- **沈黙の失敗**: エラーが出ない ≠ 正しい結果
- **再現性の欠如**: 今日うまくいったプロンプトが明日も同じ結果を出す保証はない

<!--
ここが最初のつかみです。プロンプトを書いてAIの出力を確認した時、「上手くいったと思う」と満足していませんか？ その判断を検証する人は誰でしょう？ 答えは「誰も検証していない」です。書いた本人は意図通りに見えても、別の視点では別の解釈ができます。エラーが出ないからといって結果が正しいとは限りません。今日うまくいったプロンプトが、明日も同じ結果を出す保証もありません。ここに品質Gateが必要な理由があります。
-->

# advisor とは — 公式レビュー機能

**Claude Code に標準搭載された品質レビュー機能**:

- **呼出方法**: Claude Code を起動している画面の入力欄に `/advisor` と打つだけ — 設定ファイルの編集は一切不要
- **仕組み**: あなたがこれまでAIとやり取りした内容（チャット画面の全ログ）を、別のより強力なAIモデルに転送して客観レビューを即座に取得
- **モデル選択**: advisor 内部で自動的に上位モデルが選ばれる — 利用者がモデルを選ぶ必要はない

> **まず知っておくべきこと**: `/advisor` と打てば、別のAIがあなたの作業を見直してアドバイスを返してくれます。中身の仕組みを知らなくても、これだけ覚えていれば使えます
>
> **費用の目安**: 1回の呼出で数十円程度（月額課金のプラン内であれば実質無料の範囲で収まる場合が多い）

<!--
advisorはClaude Codeに標準搭載されている公式機能です。Claude Codeを起動している画面の入力欄に「/advisor」と打つだけで、それまでの会話の全ログがより強力なAIモデルに転送され、客観的なレビュー結果が即座に返ってきます。設定ファイルの編集やAPIキーの追加取得は一切不要です。まずは「/advisorと打てば別のAIが見直してくれる」という事実だけ知っていれば使えます。その上で、次のスライドで「なぜ別のAIに見せるのか」を説明します。
-->

# model pairing 戦略 — なぜOpusを最初から使わないのか

> **ここは「知らなくても使える」補足知識です**。/advisor と打てば自動的に最適なAIが選ばれるので、以下の内容を全部理解していなくても問題ありません

> **Sonnet / Opus とは**: Anthropic社が提供するAIモデルの名前です。
> Sonnet = 速度と精度のバランス型（日常作業向け） / Opus = 最高性能だが低速・高コスト（深い推論向け）

**強いモデル = 高品質 ではない**:

| 観点 | 最初からOpus | model pairing（推奨） |
|------|-------------|---------------------|
| **コスト** | 全工程で高額（3〜5倍） | レビュー時のみ追加費用 |
| **速度** | 全工程で遅い（2〜3倍） | 作業は速く、レビューは丁寧 |
| **品質** | 同じAIが書いて同じAIが見直す → 見落としに気づきにくい | 別のAIが見直す → 新しい視点で指摘してくれる |

**model pairing の考え方**:
- 作業モデル（Sonnet等）で **素早く実装** — 日常的な作業に十分な精度
- レビューモデル（Opus等）で **客観的に審査** — より深い推論で見落としを検出
- Opusを「最初から」使うと、**自分で書いて自分で見直す** のと同じ罠に陥る
- 別のモデルに見せるからこそ **新しい視点** が得られる

> 「Opusだけ使えばいい」は「自分で書いて自分で見直す」のと同じ — 別の目で見るから価値がある

<!--
ここは補足知識です。/advisorと打てば自動的に最適なAIが選ばれるので、利用者がSonnetやOpusを意識する必要はありません。ただし「なぜ別のAIに見せるのか」を理解しておくと、advisorをより効果的に使えるようになります。一番強いAIを最初から使えば品質が最高になると思うかもしれませんが、そうではありません。同じAIが書いて同じAIが見直すと、自分では気づかない見落としがそのまま残ってしまいます。別のAIに見せるからこそ、新しい視点による指摘が得られます。
-->

# 呼ぶタイミング — 3つの場面

| タイミング | 状況 | 目的 |
|-----------|------|------|
| **① 実装を始める前** | コードを書き始める前 | 方針の確認・見落としの検出 |
| **② 行き詰まった時** | エラーが続く・方針が不明 | 原因の特定・代替案の提案 |
| **③ 完了後** | タスク完了後 | 成果物の最終レビュー |

> advisor は **「早めに呼ぶほど効果的」** — 実装後に呼ぶと修正コストが大きい

<!--
advisorをいつ呼ぶべきか、3つの場面を解説します。1つ目、実装を始める前です。コードを書き始める前に呼ぶことで、方針の見落としを早期に検出できます。2つ目、行き詰まった時です。エラーが続いたり方針が不明になったりした時は、すぐにadvisorを呼びましょう。原因の特定や代替案の提案をしてくれます。3つ目、タスク完了後です。成果物の最終レビューとして呼びます。重要なのは、早めに呼ぶほど効果的だという点です。実装後に問題が見つかると修正コストが大きくなります。実装前の確認が最も効率的です。
-->

# 応答ルール — 簡潔で実用的なフィードバック

advisor のレビューは **英語換算で100語以内・列挙型ステップ** で返ってくる:

1. **簡潔さ**: 100語以内（日本語なら約200〜300文字程度） — 長文解説ではなく、次に取るべき行動がすぐ分かる
2. **列挙型**: 番号付きステップ — 「1. Xを確認 2. Yを修正 3. Zを実行」の形式
3. **実用性**: 理由の説明ではなく、具体的なアクションが中心

> **「次に何をすべきか」** が分かれば十分 — advisor は長文解説ではなく行動指針を返す

<!--
advisorの応答には明確なルールがあります。100語以内で、番号付きステップで返ってきます。長文の解説ではなく、「1. 〇〇を確認、2. 〇〇を修正、3. 〇〇を実行」という具体的なアクションが中心です。理由はシンプルです。あなたが次に何をすべきかをすぐ知りたいからです。長文の解説を読む時間があれば、その分早く修正に取り掛かれます。advisorは「なぜそうすべきか」の解説ではなく「次に何をすべきか」の行動指針を返すのが役割です。
-->

# サブエージェント (Agent) との違い

| 観点 | advisor | Agent（サブエージェント） |
|------|---------|------------------------|
| **役割** | レビュー・指摘 | タスクの実行・コード作成 |
| **アクション** | 読むだけ・変更しない | ファイル編集・コマンド実行可能 |
| **コンテキスト** | 会話全体を見る | 新しい独立した作業部屋で動く（親の会話は一部しか見えない） |
| **コスト** | 低い（レビューのみ） | 高い（実行タスク分） |

**使い分けの基準**:
- 「この実装に問題ないか確認したい」 → **advisor**
- 「この処理を別のプロセスに委任したい」 → **Agent**

<!--
advisorとAgentツール、両方ともClaude Codeの機能ですが、役割が全く違います。advisorはレビュー専門です。会話全体を読んで問題を指摘しますが、ファイルを編集したりコマンドを実行したりはしません。一方、Agentはタスクの実行担当です。独立したセッションで動き、ファイルの編集やコマンドの実行ができます。コンテキストの見え方も違います。advisorは会話全体を見渡せますが、Agentは独立セッションなので親セッションの一部しか見えません。使い分けの基準はシンプルです。確認したい時はadvisor、委任したい時はAgentです。
-->

# QC template と連携フロー

**QC template（品質チェック表）**: プロジェクトごとに「確認すべき項目」を定義した採点表のひな形。テキストファイル等で項目をリスト化しておく。

> **具体例**: 「禁止用語が含まれていないか」「ファイル名の命名規則に従っているか」「テストが全て通るか」など、5〜15項目程度をリスト化しておく

**品質 Gate の全体像**:

1. **プロンプト実行** → AIがタスクを完了
2. **advisor 呼出** → より強いAIが会話全体を審査
3. **QC template 適用** → チェック項目ごとに PASS/FAIL 判定
4. **結果に応じた分岐**:
   - 全 PASS → 品質 Gate 通過 → 次へ
   - FAIL あり → 修正 → 再実行 → 手順2に戻る

> **重要**: advisor は「審査する人」、QC template は「採点表」。QC template は **あなた（人間）が事前に定義** し、advisor がそれに沿って審査結果を返します。自動採点ではありません — あなたが基準を決め、advisor がその基準で確認します

<!--
QC templateとは品質チェック表のひな形です。プロジェクトごとに「この項目を確認する」という5〜15項目のチェックリストをYAMLファイル等で定義しておきます。例えば「禁止用語が含まれていないか」「ファイル名の規則に従っているか」「テストが全て通るか」などです。重要なのは、QC templateは人間が事前に定義するもので、advisorがそれに沿って審査します。自動採点ではなく、あなたが基準を決めてadvisorが確認する仕組みです。まずプロンプトを実行してAIがタスクを完了し、次にadvisorを呼び出して審査し、QC templateに照らしてPASSかFAILかを判定します。全PASSなら次に進み、FAILがあれば修正して再実行します。
-->

# 品質 Gate が検出する盲点5選

| # | 盲点 | プロンプトで起こる具体例 | どう見つけるか |
|---|------|------------------------|---------------|
| 1 | **指示の暗黙前提** | 「適切に」と書いたが「適切」の基準が不明 | QC template に「曖昧語句」のチェック項目を入れておき、引っかかるか確認 |
| 2 | **エラー隠蔽** | AIが失敗したのに「完了しました」と答える | advisor に「エラーになりそうなケースも審査して」と依頼しておく |
| 3 | **セキュリティの弱点** | AIに「このデータを使って」と指示した際、データの中に悪意のある指示文が含まれているとAIが騙される | QC template に「入力内容の安全性確認」を入れておき、危険な指示が混入していないか確認 |
| 4 | **文脈の忘却** | 長い会話の中でAIが最初の指示を忘れてしまう | advisor は会話全体を見るので、冒頭の指示と最後の出力が矛盾していないか検出 |
| 5 | **過剰な一般化** | 「すべてのパターンに対応して」と指示するが、現実には例外がある | QC template に「例外ケースの確認」を入れておき、極端な例でも壊れないか確認 |

> 検出のカギは **advisor（審査する人）× QC template（採点表）の組み合わせ** — 単独では見逃す問題も、2つ合わせれば高率で検出できる
