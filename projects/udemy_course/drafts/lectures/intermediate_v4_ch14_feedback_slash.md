# ステークホルダーの怒りは、最高の品質改善データ

> 「なんでこんな簡単なこともできないんだ！」
> — この一見ネガティブな発言の中に、**3つの改善ヒント**が埋まっています。

- **何が期待されていたか** = 品質基準の再定義
- **何が期待を裏切ったか** = 改善ポイントの特定
- **どうすべきだったか** = 具体的な修正方向

> ほとんどの場合、この情報は口頭で消え、再発防止に活かされません

<!--
このスライドがつかみです。ステークホルダーの怒りや不満の裏には、品質改善のヒントが隠れています。「なんでこんな簡単なこともできないんだ」という発言から、何が期待されていたか、何が期待を裏切ったか、どうすべきだったかの3つの情報が読み取れます。しかし残念なことに、ほとんどの場合この情報は口頭で消えてしまい、二度と活用されません。この章では、この消えてしまう情報を自動的に永続化する仕組みを解説します。
-->

# feedback-capture slash — 設計思想

**核心**: 人間の感情発言を、AIが構造化フィードバックに変換するパイプライン

| 段階 | 入力 | 出力 |
|------|------|------|
| **入力** | ステークホルダーの生の発言 | テキスト |
| **分析** | 感情・文脈・改善ヒントの抽出 | 構造化データ |
| **草案生成** | feedback文書の自動作成 | .mdファイル |
| **レビュー** | 人間による承認・修正・破棄 | 最終版 |

- **なぜ自動化するか**: 手動では感情が先立ち、客観的な記録が困難
- **なぜAIにやらせるか**: 感情を除去し、事実ベースの構造化が得意

<!--
feedback-capture slashの設計思想を解説します。核心は、人間の感情のこもった発言を、AIが客観的な構造化フィードバックに変換するパイプラインです。4つの段階があります。入力段階ではステークホルダーの生の発言を受け取ります。分析段階では感情、文脈、改善ヒントを抽出します。草案生成段階ではfeedback文書を自動作成します。最後のレビュー段階で人間が承認、修正、または破棄します。なぜ自動化が必要かというと、手動では感情が先立ち客観的な記録が困難だからです。AIは感情を除去し事実ベースの構造化が得意なので、この変換に適しています。
-->

---

# テンプレートの構造 — 4段階パイプライン

### Stage 1: 入力受け取り
```
/f feedback-capture
「またテスト落ちてるじゃないか！マージ前に確認しろって
何度も言ったのに！」
```

### Stage 2: AIによる分析
- **感情ラベル**: 怒り (強)
- **本質**: マージ前のテスト実行が漏れている
- **改善ヒント**: PreToolUse hook でテスト実行を強制

### Stage 3: 草案生成 → Stage 4: レビュー (次スライド)

<!--
テンプレートの構造を具体的に見ていきましょう。Stage 1では、ユーザーがslash commandを呼び出し、ステークホルダーの発言をそのまま入力します。カギカッコや感嘆符が含まれていてもそのまま渡します。Stage 2でAIが分析します。感情のラベリング、本質の抽出、改善ヒントの提案を自動で行います。この例では、怒りの背後にある本質は「マージ前のテスト実行漏れ」であり、改善ヒントとしてPreToolUse hookの導入が提案されています。Stage 3で草案を生成し、Stage 4でレビューします。
-->

# レビュー方式 — y / e / n の3パターン

AIが生成した草案に対し、人間が3つのアクションから選択:

| アクション | 意味 | 次のステップ |
|-----------|------|-------------|
| **y** (承認) | この内容で確定 | `memory/` に保存 |
| **e** (修正) | 内容を調整したい | AIに修正指示 → 再生成 |
| **n** (破棄) | このフィードバックは不要 | 破棄して終了 |

> **重要**: すべてのフィードバックを保存する必要はない。
> **取捨選択を人間が行う**ことが品質の鍵。

<!--
AIが生成した草案に対するレビュー方式です。3つのアクションから選択します。yで承認すればmemoryディレクトリに保存されます。eで修正したい場合はAIに修正指示を出して再生成させます。nで破棄すれば、そのフィードバックは保存されません。重要なのは、すべてのフィードバックを保存する必要はないということです。取捨選択を人間が行うことが、フィードバック品質の鍵になります。AIは構造化を担当し、人間は判断を担当するという役割分担です。
-->

# カスタムslash commandの作成方法

Claude Codeのslash command = `.claude/commands/` に置いた `.md` ファイル

```
.claude/
  commands/
    feedback-capture.md    ← /feedback-capture で呼び出せる
```

**ファイルの中身** = プロンプトテンプレート:
```markdown
# Feedback Capture

以下のステークホルダー発言を分析し、
構造化フィードバックの草案を作成せよ。

発言内容: $ARGUMENTS
```

- `$ARGUMENTS` = slash commandの後に続くテキストが自動挿入される
- ファイル名がコマンド名になる
- 複数階層も可能: `commands/team/feedback.md` → `/team feedback`

<!--
カスタムslash commandの作成方法です。Claude Codeでは、.claude/commands/ディレクトリに.mdファイルを置くだけで、slash commandとして認識されます。ファイルの中身はプロンプトテンプレートで、$ARGUMENTSという変数にslash commandの後に続くテキストが自動的に挿入されます。ファイル名がそのままコマンド名になります。例えばfeedback-capture.mdというファイルを置けば、/feedback-captureで呼び出せます。複数階層のディレクトリ構造もサポートされており、commands/team/feedback.mdなら/team feedbackで呼び出せます。
-->

# 実例 1: コード品質への不満

**入力** (ステークホルダー発言):
> 「この関数、引数の型すら書いてない。プロとしてどうなの？」

**AI分析結果**:
- **感情**: 呆れ (中)
- **本質**: 型アノテーションの欠如
- **改善**: 型ヒント追加のルール化

**生成されたフィードバック草案**:
> `feedback_type_annotations.md`
> - **What**: 型アノテーションが欠如していた
> - **Impact**: コードの可読性と保守性が低下
> - **Fix**: 全関数に型ヒントを追加するルールを設定
> - **Prevention**: `ruff` や `mypy` による自動チェック

<!--
1つ目の実例です。コード品質に対する不満の変換例です。「この関数、引数の型すら書いてない。プロとしてどうなの？」という発言を入力すると、AIは呆れの感情を検出し、本質が型アノテーションの欠如であることを抽出します。そして改善として型ヒント追加のルール化を提案します。生成されたフィードバック草案は、何が問題だったか、どういう影響があったか、どう修正すべきか、どう再発を防止するかの4つの構造で記述されます。これがyで承認されれば、memoryディレクトリに保存され、今後の開発で参照されるようになります。
-->

# 実例 2: スケジュール遅延への不満

**入力**:

**AI分析結果**:
- **感情**: 焦りと怒り (強)
- **本質**: 要件伝達から実装までの追跡が欠如
- **改善**: 要件→実装のトレーサビリティ導入

**生成されたフィードバック草案**:
> `feedback_requirement_tracking.md`
> - **What**: 要件伝達後に実装ステータスが追跡されていなかった
> - **Impact**: リリース直前の未実装発覚
> - **Fix**: 要件→タスク→実装の追跡フローを導入
> - **Prevention**: 週次で要件の実装状況をレビュー

<!--
2つ目の実例です。スケジュール遅延に対する不満の変換例です。「来週リリースなのにAPIのエラーハンドリングまだ？何週間前に要件伝えたと思ってるの？」という発言から、AIは焦りと怒りの感情を検出します。本質は要件伝達から実装までの追跡が欠如していることです。生成されるフィードバックは、要件のトレーサビリティ導入を提案します。要点件を伝えたのに実装されなかった原因を、個人の責任ではなくプロセスの問題として構造化するのがポイントです。

# 実例 3: 仕様誤解への不満

**入力**:
> 「要件書いてあるでしょ？ なんで勝手にUI変えてるの？」

**AI分析結果**:
- **感情**: 不信感 (中)
- **本質**: AIの自律的な変更と要件の乖離
- **改善**: 変更前の確認プロセス (advisor呼出)

**生成されたフィードバック草案**:
> `feedback_scope_control.md`
> - **What**: 要件外の変更が無断で行われていた
> - **Impact**: ステークホルダーの意図との乖離
> - **Fix**: スコープ外変更の禁止ルールを明文化
> - **Prevention**: 実装前に `advisor()` で要件適合性を確認

<!--
3つ目の実例です。仕様の誤解に対する不満の変換例です。「要件書いてあるでしょ？なんで勝手にUI変えてるの？」という発言から、AIは不信感を検出します。本質はAIが自律的に要件外の変更を行ったことです。改善として、変更前の確認プロセス、つまり第4章で学んだadvisorの呼出を提案します。フィードバック草案では、スコープ外変更の禁止ルールの明文化と、実装前の要件適合性確認を推奨しています。

# チームでの運用方法

### ロール分担

| ロール | 担当 | 頻度 |
|-------|------|------|
| **入力者** | 全メンバー | 不満発生時いつでも |
| **レビュー者** | テックリード / PM | 週次 |
| **蓄積先** | `memory/` (Claude Code) | 自動 |

### 運用サイクル

1. **Capture**: メンバーが `/feedback-capture` で入力
2. **Review**: テックリードが y/e/n で判定
3. **Store**: 承認されたものを `memory/` に保存
4. **Act**: 週次で蓄積されたフィードバックを確認し、改善策を実行

> **ポイント**: フィードバックは「誰が言ったか」ではなく「何が課題か」に焦点を当てる

<!--
チームでの運用方法を解説します。ロール分担は3つあります。入力者は全メンバーで、不満が発生したらいつでも入力できます。レビュー者はテックリードやプロジェクトマネージャーが担当し、週次で承認判定を行います。蓄積先はClaude Codeのmemoryディレクトリで、承認されたものが自動的に保存されます。運用サイクルは、キャプチャ、レビュー、保存、実行の4ステップです。重要なポイントは、フィードバックを「誰が言ったか」ではなく「何が課題か」に焦点を当てることです。個人の攻撃ではなく、プロセスや仕組みの改善に繋げるのが目的です。
-->
