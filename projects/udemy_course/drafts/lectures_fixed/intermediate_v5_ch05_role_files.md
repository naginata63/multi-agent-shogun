## 現象 — こんな経験ありませんか?

**ケース A**: 営業の AI に「メール返信」「提案書作成」「競合分析」を全部 `CLAUDE.md` に書いている。指示が長すぎてメールのトーン指定が無視される

**ケース B**: PM の AI に「タスク分解して」と頼んだのに、勝手に進捗確認まで始めてしまった

**ケース C**: 100 ページの企画書を 1 つの AI にチェックさせたら、中盤の重要な指摘を見落とした (ch03 LitM)

---

## 対策の方向性 — 「全部入り」からの脱却

```
Before:  1つのAIに全部頼む → 指示が膨大 → 抜け漏れ + LitM発生

After:   役割別にファイルを分ける + 長文は別AIに分担
         → 指示が短く明確 → LitMも起きない
```

**核心**: 1 つの AI に「何でも屋」をやらせない。役割を分けて、それぞれに最小限の指示を渡す = **方向性② 物理分割**

---

## 本章の解決策 — 2 本の柱

| # | 解決策 | 効果 | 該当層 |
|---|--------|------|--------|
| ① | **3 段重ねの役割別ファイル** | 指示が短く明確になり、抜け漏れが激減 | コンテキスト層 (L2) |
| ② | **サブエージェントで長文を物理分割** | 主文脈に長文を持ち込まず LitM を回避 | ハーネス層 (L3) |

> ※ 役割別ファイルは「AI に渡す指示の整理」=コンテキスト層性格。サブエージェントは「Claude の動作を拡張する仕組み」=ハーネス層。**本章は L2 と L3 にまたがります**

## 「全部入り」の罠

<!--
スピーカーノート:
CLAUDE.md に全部書く罠を実例で示します。メール・提案書・データ分析のルールが混在 → ch03 の LitM で後半が読み飛ばされる。受講者に「自分の CLAUDE.md は何行あるか?」と問いかけてください。

よくある `CLAUDE.md`:

```markdown
# 私のAIルール

## 基本方針
- 丁寧な言葉遣い
- 日本語で回答

## メール返信のルール
- 宛先は CC に営業部長を入れる
- 署名は自社テンプレートを使う
- 返信は24時間以内

## 提案書作成のルール
- 競合比較表を必ず含める
- 価格は標準プランから提示
- 図表は3枚以上

## データ分析のルール
- 欠損値は除外してから分析
- 可視化は棒グラフと折れ線を併用
- 統計的有意差も確認する

... (まだ 60 行続く)
```

**問題**: ch03 で学んだ通り、100 行の指示は後半が読み飛ばされる

---

## なぜ「全部入り」が失敗するのか?

| 問題 | 原因 |
|------|------|
| 指示の最後のルールが無視される | ch03 Lost in the Middle — 長い指示の中央〜末尾が読み飛ばされる |
| 無関係なルールが混在 | メール返信時にデータ分析ルールまで読む無駄 |
| 役割が曖昧 | 「この AI は何担当?」が不明確 |
| 更新が怖い | 100 行の中に重要なルールが埋もれていて、編集すると壊れそう |

**ch04 の「長文を作らない」設計思想と同じ問題** — 指示ファイルが長文になっている

---

## 解決策: 3 段重ねに分ける

<!--
スピーカーノート:
3 層アーキテクチャの図を説明します。第1層 CLAUDE.md は常駐・全役割共通・30 行以内。第2層役割別ファイルは必要な役割だけ。第3層個別指示は使い捨て。各層の役割分担が明確になることがポイントです。
-->

```
┌────────────────────────────────────────┐
│  第1層: CLAUDE.md (常駐ルール)          │
│  全役割で共通する最低限のルール          │
│  例: 「日本語で回答」「丁寧な言葉遣い」  │
│  → 会話開始時に読み込まれる (短く保つ)  │
├────────────────────────────────────────┤
│  第2層: 役割別ファイル                   │
│  各役割に固有のルール                    │
│  例: instructions/email.md (メール担当)  │
│       instructions/proposal.md (提案担当)│
│  → 必要な役割のファイルだけ読み込む      │
├────────────────────────────────────────┤
│  第3層: 個別指示 (タスク)               │
│  その都度の具体的な指示                  │
│  例: 「このメールに返信して」            │
│  → そのタスク限定の指示                  │
└────────────────────────────────────────┘
```

## 第1層: CLAUDE.md の読込仕様 (Claude Code 固有)

> `CLAUDE.md` は **Claude Code 固有の自動読込ファイル**です。他ツールには別の規約があります:

| ツール | 自動読込ファイル |
|-------|----------------|
| **Claude Code** | `CLAUDE.md` |
| Cursor | `.cursorrules` |
| Windsurf | `.windsurfrules` |
| VSCode + GitHub Copilot | `.github/copilot-instructions.md` / `.github/instructions/*.instructions.md` |

**`CLAUDE.md` の読込挙動 (公式)**:
- **タイミング**: 新しい会話/セッションの開始時に読込 (プロセス再起動は不要)。`/clear` や `/compact` の後もディスクから再読込
- **階層 (4 階層)**: ① 管理ポリシー / ② ユーザー `~/.claude/CLAUDE.md` / ③ プロジェクト `./CLAUDE.md` または `./.claude/CLAUDE.md` / ④ ローカル `./CLAUDE.local.md`
- **マージ挙動**: 発見されたすべての `CLAUDE.md` は**連結**される (上書きではない)。矛盾時は後に読まれるプロジェクト指示が優先されやすい

<!-- Speaker note: CLAUDE.md は Claude Code 固有のファイルであり、Cursor や Windsurf は別の規約 (.cursorrules / .windsurfrules) を使っていることを明示します。受講者が他ツールに移行した時に混乱しないよう、ツール別の対応表を示してください。読込タイミングは「プロセス再起動」ではなく「新しい会話/セッション開始時」が正しい表現です -->

## 分類の基準 — 変更頻度 × 適用範囲

| | 全役割で共通 | 特定の役割のみ |
|---|---|---|
| **変更頻度: 低** | CLAUDE.md (第1層) | 役割別ファイル (第2層) |
| **変更頻度: 高** | (CLAUDE.md に入れない) | 個別指示 (第3層) |

**コツ**:
- CLAUDE.md は **30 行以内** に保つ (= ch03 配置技術の実践)
- 役割別ファイルは **その役割の AI だけが読む** (= ch04 必要時読込の実践)
- 個別指示は **使い捨て** で書く (常駐ルールに混ぜない)

## Before / After 実例

**Before** (全部入り 100 行):
```
CLAUDE.md (100行)
├── 基本方針 (5行)
├── メールルール (20行)
├── 提案書ルール (25行)
├── データ分析ルール (30行)

**After** (3 段重ね):

instructions/analysis.md (30行) — 分析担当のルール

個別指示: 「このメールに返信して」(その都度)
```

## ⚠️ `instructions/` ディレクトリの出所 (重要)

> `instructions/<role>.md` は **本リポジトリ独自の運用パターン**です。Claude Code 標準ではありません。

**由来**: VSCode + GitHub Copilot 公式の **`.github/instructions/NAME.instructions.md`** 規約に着想を得て、Claude Code 用にアレンジしたものです。

| 項目 | VSCode + GitHub Copilot (公式) | 本リポジトリ (派生・独自) |
|------|------------------------------|-----------------------|
| 配置場所 | `.github/instructions/` | top-level `instructions/` |
| ファイル形式 | `<NAME>.instructions.md` | `<role>.md` |
| 読み込み機構 | VSCode 設定 `chat.instructionsFilesLocations` で自動 | CLAUDE.md からの明示的 Read 指示で実現 |
| frontmatter | YAML `applyTo` フィールド | (本リポジトリでは未使用) |

**Claude Code 単体で類似機能を実現する公式の選択肢**: ①`.claude/skills/` (オンデマンド読込) / ②`@path` imports (CLAUDE.md 内) / ③ `.claude/agents/` (サブエージェント定義)

## 役割別ファイルはどう読み込まれるのか?

**Claude Code 標準のしくみ** (CLAUDE.md 経由):

```
CLAUDE.md (会話開始時に自動読込)
  ↓
  「メール案件の時は instructions/email.md を Read せよ」
  「提案書案件の時は instructions/proposal.md を Read せよ」
  ↓
ユーザーが「このメールに返信して」と指示
  ↓
AI が CLAUDE.md の指示に従って instructions/email.md を Read tool で読込
  ↓
個別指示はその都度プロンプトに直接書く
```

**ポイント**: `instructions/` ディレクトリのファイルは Claude Code が自動では読まない (`CLAUDE.md` ではない)。CLAUDE.md に「いつ・どれを Read するか」の指示を書くことで、AI が必要時に Read する仕組み。

> ※ より自動化したい場合は ch07 の hook (UserPromptSubmit hook で関連ファイルを注入) と組み合わせる選択肢もあります

---

## 3 段重ねが機能する理由

```
メール返信を頼む:
  1. CLAUDE.md → 「日本語で回答」「丁寧な言葉遣い」(会話開始時に読む)
  2. instructions/email.md → 「CC に営業部長を入れる」「署名テンプレ使用」(メール案件で Read)
  3. 個別指示 → 「この顧客メールに返信して」(今回のみ)

提案書作成を頼む:
  1. CLAUDE.md → 「日本語で回答」「丁寧な言葉遣い」(会話開始時に読む)
  2. instructions/proposal.md → 「競合比較表を含める」「図表3枚以上」(提案案件で Read)
  3. 個別指示 → 「A社向けの提案書を作成して」(今回のみ)
```

**無関係なルールが混入しない** → 指示が短く明確 → 抜け漏れ激減

# 柱②: サブエージェントで長文を物理分割
## ch03 LitM のもう一つの根本対策

## 長文レビューの問題

**シナリオ**: 100 ページの企画書を AI にチェックしてほしい。

```
あなた: 「この 100 ページの企画書をレビューして」

AI:  (100 ページを読み込む)
     → ch03 LitM が発生
     → 中盤の重要なリスク指摘を見落とす
     → 「問題ありません」と回答 ← 嘘
```

**ch04 の chunk + RAG も一つの解決策**ですが、別のアプローチがあります。

---

## サブエージェント = 「別の AI に丸投げ」

<!--
スピーカーノート:
サブエージェントの図を説明します。メイン AI から別 AI に仕事を振る → 別 AI が独立して処理 → 結果だけ報告。メインの文脈が短いまま保たれる = LitM 発生なし、という流れを強調してください。
-->

<div class="vocab-box">

### 用語: サブエージェント (Subagent)

**メインの AI** から **別の AI** に作業を依頼する Claude Code 公式の仕組み。

ざっくりいうと: 「AI が別の AI に『この部分お願い』と仕事を振る」

</div>

```
メインAI (あなたが話しているAI)
  │
  ├──「企画書のレビュー、100ページあるから別AIにお願いして」
  │
  └──→ サブエージェント (別のAI)
         │
         ├── 100ページを独立して読む
         ├── レビュー結果をまとめる
         │
         └──→ メインAIに報告: 「リスク指摘が3件見つかりました」
```

---

## サブエージェントの定義方法 (公式仕様)

**配置場所**:
- **プロジェクト用**: `.claude/agents/<name>.md`
- **ユーザー用** (全プロジェクト共通): `~/.claude/agents/<name>.md`

**ファイル形式**: YAML frontmatter 付き Markdown

```markdown
---
name: reviewer
description: 長文ドキュメントをレビューしてリスク指摘を抽出する。Use when ユーザーが企画書・提案書・契約書のレビューを依頼するとき
tools: Read, Grep, Glob
---

# サブエージェント: reviewer の指示書

あなたは長文レビュー専門のサブエージェントです。
渡されたドキュメントを通読し、以下の観点でリスクを抽出してください:
- セキュリティ上の問題
- 法務リスク
- 数値の不整合
...
```

> ※ これは初級 ch03 で予告した `.claude/agents/` の **公式定義方法**です。本章「柱①」の `instructions/` (本リポジトリ独自) とは別物。

## サブエージェントの 5 つの呼出方式 (公式)

| # | 方式 | 例 | 用途 |
|---|------|----|----|
| ① | **自動委譲** (description ベース) | 自動 | description のキーワードマッチで自動起動 |
| ② | **自然言語でのサブエージェント名指定** | 「reviewer エージェントで企画書を見て」 | 明示的に指定したいとき |
| ③ | **@-mention** | `@"reviewer (agent)"` でレビューを依頼 | チャット内での明示的呼出 |
| ④ | **CLI フラグ `--agent`** | `claude --agent reviewer` | CLI 起動時に直接指定 |
| ⑤ | **Agent ツール経由** (旧 Task) | コード/スクリプト内から API 経由で起動 | 自動化スクリプトから呼出 |

> ⚠️ `/agents` は **管理用** のタブ付きインターフェース (呼出用ではない)。`/agent <name>` という個別呼出コマンドは公式に**存在しません**。

## なぜ LitM が防げるのか?

| | メイン AI が全部やる | サブエージェントに分担 |
|---|---|---|
| **メインの文脈** | 100 ページの企画書 + 会話履歴 | サマリだけ (3-5 行) |
| **LitM リスク** | 高い (長文が混入) | なし (メインは短いまま) |
| **レビュー品質** | 中盤が読み飛ばされる | サブエージェントは 100 ページだけに集中 |

**核心**: 長文をメインの文脈に**持ち込まない**。別の AI に読んでもらって、結果だけを受け取る。

---

## サブエージェントが効く場面

| 場面 | 何を分担するか | 効果 |
|------|--------------|------|
| 企画書レビュー | 100 ページの企画書全体 | メインは結果だけ受け取る |
| 複数資料の調査 | 10 個の PDF レポート | 各資料の要約だけ受け取る |
| 大規模な文書作成 | 対象ファイル 20 個 | 変更提案だけ受け取る |
| テスト一括確認 | 100 個のテストケース | 結果サマリだけ受け取る |

**共通パターン**: 「長文を読む作業」を別 AI に丸投げ → メインは短い結果だけ処理

## ch04 との使い分け

| アプローチ | 手法 | 適している場面 |
|-----------|------|--------------|
| **ch04: chunk + RAG** | 長文を小片に分けて検索 | 「特定の箇所だけ知りたい」場合 |
| **ch05: サブエージェント** | 長文を別 AI に丸投げ | 「全体をちゃんと見てほしい」場合 |
| **組み合わせ** | chunk → サブエージェント | まず chunk で関連部分を特定 → その部分の深掘りを別 AI に依頼 |

**どちらも ch03 LitM の根本対策** — 「長文をメインの文脈に持ち込まない」という共通の哲学

## 実践: サブエージェントの使い方

Claude Code では、ユーザーが「長文のレビューをして」と頼むと、AI が自動的にサブエージェント (自動委譲) を使って分担します。

```
あなた: 「この 100 ページの企画書をレビューして」

AI:
1. description マッチで reviewer サブエージェントを起動 (自動委譲)
2. サブエージェントに企画書全体を渡す
3. サブエージェントが独立してレビュー
4. レビュー結果を受け取る
5. あなたに報告: 「リスク指摘が 3 件見つかりました」
```

**メイン AI の文脈には、レビュー結果のサマリだけが残る** → LitM 発生なし

> 明示的に指定したい場合は `@"reviewer (agent)"` や `--agent reviewer` を使う

---

## サブエージェントの注意点

> **⚠️ サブエージェントは万能ではありません**

- **指示は明確に**: 「レビューして」より「セキュリティ上の問題がないか確認して」
- **結果は検証する**: サブエージェントも AI。ch06 で学ぶ第三者レビュー (advisor) で検証を
- **使いどころを見極める**: 短い文章ならメイン AI で十分。長文 (数十ページ以上) でこそ本領発揮
- **description が呼出の鍵**: 自動委譲で起動させたいなら、description に「何をするか + Use when」を明記

## 本章のまとめ

<!--

- **3 段重ね** (CLAUDE.md / `instructions/<role>.md` / 個別指示) で指示を短く明確にする
