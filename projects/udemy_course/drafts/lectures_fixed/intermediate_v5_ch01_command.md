# ch01. プロンプトを資産にする — カスタムスラッシュコマンドと Skill 布石

---

## この章で解決する「困りごと」

前章で挙げた **11 の困りごと** のうち、本章で解決するのは:

> **① 既にあるものを確認せず新規作成してしまう**
> **② 同じ作業を毎回手動で繰り返している**

- 先週も似たプロンプトを書いた気がする…
- 議事録要約のプロンプト、どこに保存したっけ？
- 毎回ゼロからプロンプトを書いている。書くたびに品質がブレる

> プロンプトが「使い捨て」だから、同じ労力を何度も払っている

<!-- Speaker note: 前章 ch00 で提示した 11 件の困りごとリストのうち、①「確認せず新規作成」と②「同じ作業を毎回手動」が本章のターゲットです。どちらもプロンプト層 (L1) の欠如が原因。「先週も似たプロンプト書いた気がする」という感覚は多くの人が持っているはず。ここで「自分のことだ」と思ってもらいます。 -->

---

## 章のゴール

**言える 1 文:**
> プロンプトをカスタムスラッシュコマンド (`.claude/commands/<名前>.md` または `.claude/skills/<名前>/SKILL.md`) として保存すれば、次から 1 コマンドで同じ品質の結果を得られる

**できる 1 動作:**
> カスタムスラッシュコマンドを 1 つ作って動かせる。自分の業務でカスタムスラッシュコマンド化候補を 3 件挙げられる

## 「使い捨て」と「資産」の違い

| | 使い捨てプロンプト | 資産化プロンプト |
|---|---|---|
| 書く回数 | 毎回ゼロから | **1 回だけ** |
| 品質 | 毎回ブレる | **安定** |
| 探す手間 | 「どこに書いたっけ?」 | **コマンド 1 つ** |
| 改善 | 毎回一から直す | **蓄積される** |

> 1 回書いて何度でも使える = **資産**

<!-- Speaker note: 使い捨てと資産の対比表を解説します。ここで伝えたいのは「書く回数・品質・探す手間・改善」の4軸すべてで資産化が有利ということ。特に「改善が蓄積される」は重要で、使い捨てでは毎回ゼロリセットですが、資産化すれば一度の改善が永続します。 -->

---

## カスタムスラッシュコマンド — プロンプトを資産にする仕組み

```
.claude/commands/
  └── summarize-meeting.md   ← このファイルがプロンプト本体
```

- `.claude/commands/` フォルダに Markdown ファイルを置くだけ
- ファイル名がそのままコマンド名になる
- 例: `summarize-meeting.md` → `/summarize-meeting` で呼び出し

> ※ Claude Code 公式に `/command` という単一コマンドは存在しません。
> 「カスタムスラッシュコマンド」とは、自分で `/<任意の名前>` を作る仕組みのことです。

<!-- Speaker note: カスタムスラッシュコマンドの仕組みを初めて紹介するスライドです。仕組みはシンプル：Markdown ファイルを置くだけ。フォルダ名は `.claude/commands/` 固定。ファイル名 = コマンド名という直感的な対応関係を強調してください。「プログラミング不要」であることも安心材料として伝えます。注意点として「/command」という単一コマンドは Claude Code 公式に存在しないこと、私たちは「自作のスラッシュコマンドの仕組み」を学んでいることを明示します。 -->

---

## 実演: カスタムスラッシュコマンドを作ってみよう

### Step 1 — フォルダを作る

プロジェクトのルートに `.claude/commands` フォルダを作ります:

```bash
mkdir -p .claude/commands
```

> ※ VS Code のエクスプローラーから「新しいフォルダー」で作っても OK です

### Step 2 — プロンプトファイルを書く

```markdown
<!-- .claude/commands/summarize-meeting.md -->
以下の議事録を読み、要点を 3 つにまとめてください。

## 出力形式
- 決定事項
- 次回のアクション
- 未解決の課題

## 対象ファイル
$ARGUMENTS
```

> `$ARGUMENTS` には、コマンド呼び出し時に渡した内容が入ります。
> 後ほど実演します。

<!-- Speaker note: 実演の Step1 と Step2 です。ここでは実際に VS Code かターミナルを開いてデモすることを推奨。mkdir -p .claude/commands を実行し、summarize-meeting.md をその場で作ることで、受講者に「これだけ？」という驚きを与えます。$ARGUMENTS の概念は次のスライドで詳しく説明します。 -->

---

## 実演: カスタムスラッシュコマンドを呼び出す

### Step 3 — 使う

プロンプト欄にこう入力するだけ:

```
/summarize-meeting @meeting-2026-05-06.md
```

- `/summarize-meeting` = コマンド名
- `@meeting-2026-05-06.md` = 対象ファイル (引数)
  - `@ファイル名` で AI にそのファイルを読み込ませる Claude Code の参照記法

> **毎回プロンプトを書く必要がなくなりました**

<!-- Speaker note: Step3 の実演です。実際に /summarize-meeting @meeting-2026-05-06.md と入力して AI がどう動くかを見せます。ポイントは「プロンプトの中身は一切書いていない」こと。コマンド名とファイル参照だけで、あらかじめ用意したフォーマット通りの出力が得られる驚きを伝えてください。@file 記法は Claude Code の汎用機能で、commands/skills 内に限らず使えます。 -->

---

## 引数付き呼出の仕組み

```markdown
<!-- .claude/commands/weekly-report.md -->
以下の進捗データから、週次レポートを作成してください。

## フォーマット
1. 今週の成果 (3 行以内)
2. 課題と対応状況
3. 来週の予定

## 対象
$ARGUMENTS
```

```
/weekly-report @progress.md
```

- `$ARGUMENTS` の位置に `@progress.md` の中身が差し込まれるイメージ
- **プロンプトの中身は変えずに、対象だけ変えられる**

<!-- Speaker note: $ARGUMENTS の仕組みを図解します。「テンプレートの穴埋め」というイメージで伝えると分かりやすい。プロンプトの中身 (出力形式や指示) は固定で、対象ファイルだけ差し替える。これが「資産化」の核心です。プログラミングの関数に慣れている人は「引数のようなもの」と説明してもよいです。なお、個別位置の引数は `$N` 記法 (`$0` が第 1 引数、`$1` が第 2 引数 — 0 ベース番号付け) が公式仕様にありますが、本講座は `$ARGUMENTS` (全引数まとめて) のシンプル形のみ扱います。 -->

## Script / Slash Command / Skill — 3 つの道具の使い分け

| 観点 | Script (Bash 等) | カスタムスラッシュコマンド (`/name`) | Skill |
|------|------------------|------------------------------|-------|
| 起動 | コマンドライン手動 | ユーザーが `/<名前>` と入力 | **description のキーワードマッチで AI が自律呼出** |
| 配置 | 任意のフォルダ | `.claude/commands/<名前>.md` (legacy) | `.claude/skills/<名前>/SKILL.md` (公式標準・ディレクトリ形式) |
| 引数 | CLI 引数 | `$ARGUMENTS` / `@file` 参照 | プロンプト本文から AI が抽出 |
| 学習負荷 | 低 | 中 | やや高 (但し再利用最大) |
| 用途 | 手動繰り返し作業 | 定型プロンプトの資産化 | **業務領域知識のオンデマンド読込** |
| 本講座での位置 | 補助 | **← 今ここ (ch01)** | 到達点 (ch11) |

> 公式は「カスタムコマンドはスキルにマージされました」と明記。本章では分かりやすさのため legacy の `.claude/commands/` から入り、ch11 で Skill (ディレクトリ形式) に進みます。

<!-- Speaker note: 3 つの道具を表で対比します。Slash Command の配置パスは legacy 形式の単一 .md と、推奨される Skill のディレクトリ形式 (`.claude/skills/<名前>/SKILL.md`) があることを示します。Skill の自律発動条件は「description のキーワードマッチ」で、これが ch11 で扱う最も重要なポイントです。ここで「Skill = 業務領域知識をオンデマンドで呼ばれる小さなプレイブック」というイメージを植え付けてください。 -->

---

## 3 つの道具 — ざっくり言うと

- **Script**: シェルスクリプトと同じ。手動で叩く
- **カスタムスラッシュコマンド**: `/<任意名>` で呼び出す定型プロンプト。**本章の主役**
- **Skill**: description のキーワードに合致すると AI が文脈を判断して**自律的に呼び出す**。ch11 で作る

> まずはカスタムスラッシュコマンドをマスターして、最後に Skill に到達する。

<!-- Speaker note: 3 つの道具のざっくりとした説明です。ここで重要なのは「段階的に学ぶ」というメッセージ。まず Script (手動) → カスタムスラッシュコマンド (半自動) → Skill (自律) と進む流れを強調。Skill の自律呼出は「description のキーワードマッチ」が発火条件、という公式仕様も一言添えます。受講者が「Skill まで行かないと意味ない？」と不安にならないよう、「カスタムスラッシュコマンドだけでも十分に実用的」であることを伝えてください。 -->

## 業務 3 例 — カスタムスラッシュコマンド化候補あぶり出し

### 営業パーソンの例
- 毎回やっている: 顧客への提案メールをゼロからプロンプトで書く
- カスタムスラッシュコマンド化: `.claude/commands/proposal-email.md`
  - 「顧客名・商談内容・条件を入力して提案メールを作成」
  - 次から `/proposal-email` で 1 発

### PM サポートの例
- 毎回やっている: 進捗レポートのフォーマットから毎回作り直す
- カスタムスラッシュコマンド化: `.claude/commands/weekly-report.md`
  - `/weekly-report @進捗.md` で即座に生成

### データ分析者の例
- 毎回やっている: 「このデータの傾向を教えて」と毎回手動で書く
- カスタムスラッシュコマンド化: `.claude/commands/analyze-trend.md`
  - `/analyze-trend @sales.csv` で 1 発

## Skill 布石 — 次の段階への予告

> カスタムスラッシュコマンドは **ユーザーが手動で呼ぶ** 道具 (`/<名前>` と入力)
> **Skill** は AI が **description のキーワードマッチで自律的に呼ぶ** 道具

**例:**
- カスタムスラッシュコマンド版: `/proposal-email` と手動で呼ぶ
- **Skill** 版: 商談メモを渡すと、description に「提案メール作成」相当のキーワードがマッチして AI が**勝手に**過去の提案パターンを参照してメールを作る

> Skill の詳細は **ch11** で。まずはカスタムスラッシュコマンドで土台を固めましょう。

<!-- Speaker note: Skill は「description のキーワードマッチで AI が自律判断で呼ぶ道具」という布石のスライドです。ch11 で詳しくやる予告ですが、ここでは「カスタムスラッシュコマンドをマスターすれば自然と Skill への道が開ける」という安心感を与えます。「今はカスタムスラッシュコマンドに集中すれば OK」というメッセージを忘れずに。Skill の発火条件 = description のキーワード一致、という公式仕様も予告しておくと ch11 でスムーズに入れます。 -->
