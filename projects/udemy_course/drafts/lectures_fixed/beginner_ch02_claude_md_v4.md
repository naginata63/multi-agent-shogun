# CLAUDE.mdとは — AIに読ませるREADME

> 🪞 「新しいプロジェクトに入った時、最初に何を読みますか?」

- 新しいプロジェクト → まず **README.md** を読む
- AI も同じ → **CLAUDE.md** を読む
- **CLAUDE.md = AI に読ませるプロジェクトの説明書**

**読込挙動 (Claude Code 公式)**:
- **新しい会話/セッション開始時に自動読込** (プロセス再起動は不要)
- `/clear` や `/compact` の **後もディスクから再読込される**
- 一度書けば **毎セッション同じルールが効く**

**Before (CLAUDE.md なし)** → AI が Jest でテスト生成 (本当は pytest) / 2 スペース (本当は 4 スペース) / 毎回前提説明を繰り返す
**After (CLAUDE.md あり)** → AI が pytest を選ぶ / 4 スペースで書く / **指示なしで正しい振る舞い**

<!--
人間が新しいプロジェクトに入ったら README.md を読むように、AI は CLAUDE.md を読みます。新しい会話やセッションを開始すると自動で読み込まれ、`/clear` や `/compact` の後もディスクから再読み込みされるので、プロセスを再起動する必要はありません。CLAUDE.md がないと、AI は毎回デフォルトの判断をするので、Python プロジェクトなのに Jest でテストを書いたり、4 スペースのプロジェクトを 2 スペースで書いたりします。CLAUDE.md があれば最初から正しい選択をしてくれる。これが最大の価値です。
-->

# 📁 CLAUDE.md は 4 階層 — 「連結」して読込

| 階層 | 配置パス | 何のため |
|------|----------|---------|
| ① 管理ポリシー | OS 配下 (例: `/Library/Application Support/ClaudeCode/CLAUDE.md`) | 組織レベル |
| ② **ユーザー指示** | `~/.claude/CLAUDE.md` | 自分の全プロジェクト共通 |
| ③ **プロジェクト指示** | `./CLAUDE.md` または `./.claude/CLAUDE.md` | このプロジェクト固有 |
| ④ ローカル | `./CLAUDE.local.md` | 個人用 (`.gitignore` 推奨) |

**重要な挙動**:
- ディレクトリツリーを **上方向に歩いて発見された全 CLAUDE.md を「連結」** (上書きではない)
- 矛盾時は **後に読まれるプロジェクト指示が従われやすい** (上書きではなく「後勝ち」)
- サブディレクトリの CLAUDE.md は **オンデマンド読込**

> 最初は **③ プロジェクト指示 (`./CLAUDE.md`) 1 つだけ**で十分。②③④ の使い分けは第 3 章で扱います

<!--
CLAUDE.md は 4 階層構造で、組織レベルからユーザー全体、プロジェクト、個人用までに分かれています。④ のローカルファイルは正式名が `CLAUDE.local.md` で、個人メモや作業中のメモを書いて `.gitignore` でリポジトリ管理から外すのが定石です。Claude Code はこれらを上書きではなく「連結」で読み込み、矛盾時は後に読まれるプロジェクト指示の方が従われやすい後勝ち挙動になります。最初は `./CLAUDE.md` 1 つで十分、複雑な使い分けは第 3 章で見ます。
-->

# 5 セクション構成 — 30 行で書ける

CLAUDE.md は以下の 5 セクションで構成するのが基本:

| # | セクション | 役割 | 例 |
|---|----------|------|-----|
| 1 | 使用言語・FW (フレームワーク) | 何を使っているか | Python 3.12 / FastAPI |
| 2 | コーディング規約 | スタイルの統一 | 4 スペース / 型ヒント必須 |
| 3 | よく使うコマンド | ビルド・テスト・実行 | `pytest` / `uvicorn` |
| 4 | 禁止事項 | やってはいけないこと | `rm -rf` 禁止 / `main` に push 禁止 |
| 5 | プロジェクト構造 | ディレクトリの意味 | `src/` = 本体 / `tests/` = テスト |

> これだけで **約 30 行**。**完璧を目指さない** — 30 行の CLAUDE.md は 0 行の何倍も価値がある

# 30 行ミニマム版の例

> Python の例ですが **中身が読めなくても構いません**。JS でも Go でも構造は同じ。**5 セクション** という構造だけ見てください

```markdown
# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM (データベースを扱いやすくする仕組み): SQLAlchemy 2.0

## コーディング規約
- インデント: 4 スペース
- 型ヒント必須 (変数の種類を明示する書き方)
- 関数の説明書き (docstring) は Google 形式

## よく使うコマンド
- テスト: pytest tests/
- 起動: uvicorn app.main:app --reload
- コード検査 (lint): ruff check .

## 禁止事項
- 本番 DB の直接操作禁止
- main ブランチへの直接 push 禁止

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = データベースの変更履歴 (マイグレーション)
```

**保存 3 ステップ**: ① プロジェクトのルートを開く ② `CLAUDE.md` という名前で保存 ③ Claude Code で**新しい会話/セッションを開始** → 自動的に読み込まれる (プロセス再起動不要)

# セクション解説 — 何をどう書くか

> 💡 ここから先は Python 固有の用語も出てきます。**「そういう設定項目がある」程度で OK**・ご自身の言語に置き換えて読んでください

**① 使用言語・FW**: 言語の**バージョンまで書く** → AI が使える新機能を正しく判断
- 例: 「Python 3.12」と書けば **PEP 701 で追加された f-string クォート再利用** — `f"playlist: {", ".join(items)}"` のように外側と内側で同じクォートを再利用できる書き方 (3.11 以前ではクォート衝突でエラーだった) — が使えると判断する

**② コーディング規約**: インデント幅・命名規則・リンター設定 → AI 生成コードの**スタイルが揃いレビュー指摘が減る**

**③ よく使うコマンド**: テスト・ビルド・Lint のコマンド → AI が**修正後に確認実行**できる (Python は `pytest tests/`, JS は `npm test`, Go は `go test ./...`)

**④ 禁止事項**: `rm -rf` 禁止 / 本番 DB 直接操作禁止 / `main` 直 push 禁止 → AI が**実行前に止まる安全装置**

**⑤ プロジェクト構造**: `src/` `tests/` 等のディレクトリの役割 → AI が**ファイルを置く場所を間違えない**

<!--
5 セクションを 1 枚にまとめました。言語・FW はバージョンまで書くのが大事で、Python 3.12 と書けば PEP 701 の f-string クォート再利用が使えると AI が判断します。3.11 以前ではクォート衝突でエラーだった書き方が 3.12 でそのまま通るようになった、と。逆に 3.9 と書けばそれ以降の新機能は使わない。コーディング規約・コマンド・禁止事項・構造もそれぞれ役割が明確です。禁止事項は特に重要で、本番 DB の直接操作や main への直 push といった危険な操作を AI が実行前に止まる安全装置になります。
-->

# よくある失敗 3 選 + 書く時のポイント 5 つ

**❌ 失敗 3 選**:
1. **長すぎる** (100 行超) → AI が重要な部分を見落とす → **30 行以内**に収める
2. **曖昧な表現** (「きれいなコード」) → AI の解釈がバラバラ → **「4 スペース」**のように具体的に
3. **矛盾する指示** (TypeScript と JavaScript 両方指定) → AI が混乱 → **一貫性を確認**してから保存

**✍️ 書く時のポイント 5 つ**:
1. **短く始める** — 5 セクション 30 行で十分
2. **具体的に書く** — 「きれいに」ではなく「4 スペース」
3. **禁止事項を必ず入れる** — 安全性の最低限
4. **随時アップデート** — 変更があったら追記 (30 行超えそうなら古いルール削る)
5. **AI の振る舞いを見て調整** — 期待と違ったら追記

> **完璧を目指さない** — 30 行の CLAUDE.md は 0 行の何倍も価値がある

<!--
失敗パターン 3 つとポイント 5 つを 1 枚にまとめました。失敗の共通点は「長い・曖昧・矛盾」。ポイントの共通点は「短く始めて具体的に書いて随時調整」。0 行のまま悩むよりも、まず 30 行書いてみる、それが最大のコツです。
