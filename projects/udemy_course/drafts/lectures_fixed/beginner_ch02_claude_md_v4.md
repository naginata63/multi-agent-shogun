# CLAUDE.mdとは — AIに読ませるREADME

> 🪞 「新しいプロジェクトに入った時、最初に何を読みますか？」

- 新しいプロジェクト → まず **README.md** を読む
- AIも同じ → **CLAUDE.md** を読む
- **CLAUDE.md = AIに読ませるプロジェクトの説明書**

> Claude Codeは **新しい会話/セッションを開始すると自動的に CLAUDE.md を読み込み**、プロジェクトのルールを理解する。`/clear` や `/compact` の後も**ディスクから再読込される** (プロセスの再起動は不要)

<!--
人間が新しいプロジェクトに入った時、まずREADME.mdを読んで「このプロジェクトは何で、どう動いているのか」を理解しますよね。Claude Codeも同じことをします。新しい会話やセッションを開始すると、自動的にCLAUDE.mdというファイルを探して読み込みます。`/clear` で会話をリセットしたり `/compact` で要約した後も、ディスクから再読み込みされるので、ルールが消えてしまうことはありません。プロセスを再起動する必要はなく、新しい会話を始めるだけで毎回反映されます。これがCLAUDE.mdの基本的な役割です。
-->

# 📁 CLAUDE.md は階層構造 — 4階層を「連結」して読込

| 階層 | 配置パス | 何のため |
|------|----------|---------|
| ① 管理ポリシー | OS 配下 (例: `/Library/Application Support/ClaudeCode/CLAUDE.md`) | 組織レベル |
| ② **ユーザー指示** | `~/.claude/CLAUDE.md` | 自分の全プロジェクト共通 |
| ③ **プロジェクト指示** | `./CLAUDE.md` または `./.claude/CLAUDE.md` | このプロジェクト固有 |
| ④ ローカル | `./CLAUDE.local.md` | 個人用 (`.gitignore` 推奨) |

**重要な挙動 (Claude Code 公式)**:
- ディレクトリツリーを **上方向に歩いて発見された全 CLAUDE.md を「連結」** (上書きではない)
- 矛盾時は **後に読まれるプロジェクト指示が従われやすい** (上書きではなく「後勝ち」)
- サブディレクトリの CLAUDE.md は **オンデマンド読込**

> 最初は ③ プロジェクト指示 (`./CLAUDE.md`) **1つだけ**で十分。②③ の使い分けは第3章で詳しく扱います

<!--
CLAUDE.mdは実は4階層の構造を持っています。組織レベルの管理ポリシー、自分の全プロジェクト共通のユーザー指示、プロジェクト固有のプロジェクト指示、そして個人用のローカル。Claude Codeはディレクトリツリーを上方向に歩いて、見つけた全てのCLAUDE.mdを「連結」してコンテキストに読み込みます。これがポイントで、後に読まれるファイルが先に読まれたファイルを「上書き」するわけではなく、すべてが連結されてAIに渡されます。矛盾する指示があった場合は、後に読まれるプロジェクト指示の方が従われやすい、という「後勝ち」の挙動になります。最初は難しく考えず、プロジェクトのルートに `./CLAUDE.md` を1つ置くだけで十分です。複数階層の使い分けは第3章で詳しく見ていきます。
-->

# なぜCLAUDE.mdが必要なのか

**CLAUDE.mdなしの会話**:
- 「このプロジェクトはPythonです」→ 次の会話で忘れる
- 「テストはpytestで実行して」→ 毎回言わないといけない
- 「TypeScriptのstrictモードです」→ また忘れる

**CLAUDE.mdありの会話**:
- 一度書けば **毎セッション自動で読み込まれる**
- プロジェクトのルールを **AIが自動で守る**
- 同じ前提説明の **繰り返しがゼロに**

<!--
CLAUDE.mdがないと、毎回AIに「このプロジェクトはPythonです」「テストはpytestで」「TypeScriptのstrictモードです」と説明し直す必要があります。会話が長くなると、AIが前の方を忘れてしまうこともあります。CLAUDE.mdがあれば、プロジェクトのルールを一度だけ書けば、AIは新しいセッションのたびに自動で読み込み、ルールを守り続けます。この「繰り返しのゼロ」が最大のメリットです。
-->

# 5セクション構成 — 30行で書ける

CLAUDE.mdは以下の5セクションで構成するのが基本:

| # | セクション | 役割 | 例 |
|---|----------|------|-----|
| 1 | 使用言語・FW（フレームワーク） | 何を使っているか | Python 3.12 / FastAPI |
| 2 | コーディング規約 | スタイルの統一 | 4スペース / 型ヒント必須 |
| 3 | よく使うコマンド | ビルド・テスト・実行 | `pytest` / `uvicorn` |
| 4 | 禁止事項 | やってはいけないこと | `rm -rf` 禁止 / `main`にpush禁止 |
| 5 | プロジェクト構造 | ディレクトリの意味 | `src/` = 本体 / `tests/` = テスト |

> これだけで **約30行**。最初はこれだけで十分

<!--
CLAUDE.mdは5つのセクションで構成します。使用している言語とフレームワーク、コーディング規約、よく使うコマンド、禁止事項、そしてプロジェクトのディレクトリ構造です。全部書いても約30行。最初はこのミニマム構成で十分です。後から必要に応じて追加していけばOKです。完璧を目指さず、まずは30行で始めるのがコツです。

# 30行ミニマム版の例

> 以下はPythonの例ですが、**中身が読めなくても構いません**。この例はPythonですが、JavaScriptでもGoでも構造は同じです。セクションの構成（5つ）が重要で、中身の技術用語はご自身の使う言語に置き換えてください。構造（5セクション・箇条書き）だけ見てください。

```markdown
# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM（データベースを扱いやすくする仕組み）: SQLAlchemy 2.0

## コーディング規約
- インデント: 4スペース
- 型ヒント必須（変数の種類を明示する書き方）
- 関数の説明書き（docstring）はGoogle形式

## よく使うコマンド
- テスト: pytest tests/
- 起動: uvicorn app.main:app --reload
- コード検査（lint）: ruff check .

## 禁止事項
- 本番DBの直接操作禁止
- mainブランチへの直接push禁止

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = データベースの変更履歴（マイグレーション）
```

> 💡 **確認方法**: 新しい会話を始めて「CLAUDE.mdの内容を教えて」と聞けば、正しく読み込まれているかすぐ確認できます

# セクション詳細 — 使用言語・FW と コーディング規約

> 💡 **ここから先はPython固有の用語が出てきます。「そういう設定項目があるんだな」程度に読み流してOK** です。ご自身の言語に置き換えて考えましょう。

**使用言語・フレームワーク**:
- 言語のバージョンまで書く → AIが使える機能を正しく判断
- 例: 「Python 3.12」と書けば、最新の文法機能（例: 値を代入しつつ比較できる `:=` 演算子）を使えると判断

**コーディング規約**:
- インデント幅、命名規則、リンター設定を書く
- AIが生成するコードの **スタイルが統一される**
- コードレビューの指摘事項が減る

> 「4スペース」と書くだけで、AIは常に4スペースでインデントする

<!--
最初の2セクションの詳細です。言語とフレームワークは、バージョンまで含めます。「Python 3.12」と書いておけば、3.12で追加された機能を使って良いとAIが判断します。逆に「Python 3.9」と書いておけば、それ以降の機能は使いません。コーディング規約は、インデント幅や命名規則を書いておくと、AIが生成するコードのスタイルが統一されます。地味ですが、レビューでの指摘を大幅に減らします。
-->

# セクション詳細 — よく使うコマンド と 禁止事項

**よく使うコマンド**:
- テスト、ビルド（プログラムを動かせる形に変換すること）、Lint（コードの問題を自動検査すること）の実行コマンドを書く
- AIがコード修正後に **テストを実行して確認** する（実行前に人間へ確認することもあります）

| プロジェクト | テストコマンド |
|------------|--------------|
| Python | `pytest tests/` |
| JavaScript | `npm test` |
| Go | `go test ./...` |

**禁止事項**:
- 危険な操作 (`rm -rf` 等) の禁止
- 本番環境への直接アクセス禁止
- AIが **実行前に止まる** 安全装置

<!--
よく使うコマンドのセクションです。テストコマンドを書いておくと、AIはコードを修正した後に自動的にテストを実行してくれます。プロジェクトによってテストコマンドは違いますが、これを書いておけばAIが間違ったコマンドを使うことがなくなります。禁止事項は安全性の観点で非常に重要です。「本番DBの直接操作禁止」「mainブランチへの直接push禁止」と書いておけば、AIはこれらの操作を実行しようとした瞬間に立ち止まります。
-->

# よくある書き方の失敗 3選

**❌ 失敗1: 長すぎる**
- 100行以上のCLAUDE.md → AIが重要な部分を見落とす
- 解決: **30行以内**に収める。詳細は別ファイルに

**❌ 失敗2: 曖昧な表現**
- 「きれいなコードを書いて」→ AIの解釈がバラバラ
- 解決: **具体的な規約**を書く (例: 「4スペース」)

**❌ 失敗3: 矛盾する指示**
- 「TypeScriptを使って」+ 「JavaScriptで書いて」→ AIが混乱
- 解決: **一貫性**を確認してから保存する

<!--
CLAUDE.mdを書く時によくある失敗を3つ紹介します。1つ目は長すぎること。100行以上書くと、AIが重要な部分を見落とす可能性があります。最初は30行以内に収めましょう。2つ目は曖昧な表現。「きれいなコード」ではAIに伝わりません。「4スペース」「型ヒント必須」のように具体的に書きましょう。3つ目は矛盾する指示。TypeScriptとJavaScriptを同時に指定するとAIが混乱します。保存前に一貫性を確認することが大切です。
-->

# CLAUDE.mdを書く時のポイント

1. **短く始める** — 最初は5セクション30行で十分
2. **具体的に書く** — 「きれいに」ではなく「4スペース」
3. **禁止事項を必ず入れる** — 安全性の最低限
4. **随時アップデート** — プロジェクトの変更点があったら追記する（30行を超えそうなら、古いルールを見直して削る）
5. **AIの振る舞いを見て調整** — 期待と違ったら追記する

> **完璧を目指さない** — 30行のCLAUDE.mdが、0行の何倍も価値がある

<!--
CLAUDE.mdを書く時の5つのポイントです。一番大事なのは「短く始める」ことです。完璧なCLAUDE.mdを最初から書こうとすると、書けません。まずは5セクション30行で始めて、実際にAIを使いながら足りない部分を追加していくのが正しいアプローチです。追加する時は、古いルールを見直して削ることで30行を維持します。30行のCLAUDE.mdが、0行の何倍も価値があります。0行のまま悩むより、30行で始めることが何より重要です。

## **第3章: CLAUDE.mdの仲間たち — commands / skills / agents** (約 30 min)

> CLAUDE.md以外にもAIに渡せる .md ファイル群があります。CLAUDE.md (プロジェクト全体) から特定用途のファイルへ広げる流れを学びます。

- `.claude/skills/<name>/SKILL.md` — 業務領域のワークフロー定義 (必要時にオンデマンド読込)
- `.claude/agents/<name>.md` — サブエージェント (役割別の専門担当)
- `@path` imports — CLAUDE.md 内から他ファイルを取り込む構文

> CLAUDE.md がプロジェクト全体の説明書なら、これらは **特定用途の補強パッケージ**
