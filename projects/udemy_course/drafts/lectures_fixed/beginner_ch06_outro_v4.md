> コンテキスト = AIに「プロジェクトの文脈」を永続的に与える仕組み

<!--
第 2 章から第 4 章で学んだコンテキスト層。CLAUDE.md は「AI 向けの README」として、プロジェクトのルールを AI に教える仕組みです。第 3 章では .md ファイル群を整理しましたが、ここで一つ重要な訂正があります。以前は「CLAUDE.md・commands・skills の 3 大カテゴリ」と言われていましたが、Anthropic 公式が「カスタムコマンドはスキルにマージされました」と明記した通り、現在は **CLAUDE.md と Skills の二系統** が正しい整理で、`.claude/commands/<name>.md` という単一ファイル形式は Skills の **旧形式 (legacy)** として継続サポートされている位置付けです。新しく作るなら原則 Skills (`.claude/skills/<skill-name>/SKILL.md` 形式) を選びます。imports は別立てのファイル種別ではなく、CLAUDE.md の中で他のファイルを取り込む `@path` 構文として位置付くものなので、ここでは並列ではなく「CLAUDE.md 内の取り込み構文」として整理しています。そして第 4 章で、DevOps から SRE、Platform Engineering を経て AI Engineering に至る歴史の流れの中で、このコンテキスト層がなぜ必要なのかを理解しました。人間が毎回説明しなくても、AI がプロジェクトの文脈を理解して動く。これが L2 の価値です。

# 到達点3: ハーネス [L3] — AIを制御・拡張する周辺基盤

**何を学んだか**:
- hookの仕組み — AIの **ライフサイクル(起動 → 指示受付 → ツール実行 → 終了までの一連の動作)** 上のイベント (公式 29 種類) に任意の処理を**自動で起動させる**仕組み
- PreToolUse Hook でコマンド実行前に検査・ブロック
  - **前章 (第 5 章) のシェルスクリプトのあの部分の復習**:
    - `INPUT=$(cat)` で **stdin (標準入力)** から JSON を受け取る
    - 危険検知時は **`exit 2`** で終了する (`2` は「AI にブロック理由を伝えるための特別な終了コード」)
    - 同時に **stderr (標準エラー出力)** にブロック理由を出力すれば、AI が理由を読んで別案を出す

**何が解決できたか**:
- `rm -rf` や `sudo` 等の **危険操作を自動検知**
- AIが勝手にファイルを消すリスクを **排除**

> ハーネス = AI モデルを「実際に仕事を完遂する work engine」に変える周辺基盤
> 安全装置はその中の代表的な一用途

<!--
第 5 章で学んだ hook。3 階層の最上層、L3 にあたります。ここでこの章で初めて「ライフサイクル」という言葉を本格的に使うので、改めて括弧で「起動 → 指示受付 → ツール実行 → 終了までの一連の動作」と注釈を添えました。Hook はその一連の動作中の特定のイベント、公式 29 種類のタイミングに任意の処理を自動で起動させる仕組みで、本章ではその中の最も基本となる PreToolUse を入口に体験しました。前章の用語を覚えていない方のためにあえてここで再掲しています。データの受け渡しは標準入力 (stdin) 経由の JSON、ブロックは exit 2 で終了する形を取りました。この「2」は普通の正常終了 (0) でもエラー終了 (1) でもなく、「AI にブロック理由を伝えるための特別な終了コード」として Claude Code 側で特別扱いされる値です。同時に標準エラー出力 (stderr) に理由を書けば、AI がその理由を読んで別の案を出してくれる。これがコピペで動く公式仕様の最小形でした。L3 ハーネスは Anthropic 公式の業界標準概念で、hook だけでなく skill やサブエージェントなど、AI を「実際に仕事を完遂する work engine」に変える周辺基盤全体を指します。安全装置は、その中で最も初心者が体験しやすい代表用途のひとつです。プロンプトで指示し、コンテキストで文脈を与え、ハーネスで制御・拡張する。この 3 層が揃って初めて、AI は安全で頼れる開発パートナーになります。
-->

# 中級編のプレビュー

> 中級編では、今日学んだ3階層を **深掘り** し、新しい技術と運用を加えていきます

| 項目 | 初級編 (今) | 中級編 (次) |
|------|------------|---------------|
| 所要時間 | 約 3 時間 / 7 章 | 約 13 時間 / 12 章（1日30分×4週間程度） |
| L1 プロンプト | 基礎の構造化 | 高度なテンプレート化 |
| L2 コンテキスト | 30行ミニマム | 永続化・マルチプロジェクト |
| L3 ハーネス | 1本の入門 PreToolUse Hook | 主要カテゴリ別に **計 18 本** の実装例 (ファイル保護・コマンド検査・通知など、各カテゴリ 3〜4 本ずつ) を学ぶ |
| **MCP** | **今回未対応** | **自前構築 + 永続化バックエンド** |

> **MCP**（Model Context Protocol）= LLM と外部ツール・データソース・API を繋ぐ **オープンソース標準プロトコル**。Claude Code を数百の外部ツールに接続するための公式仕様。claude-mem のような「記憶を拡張する MCP サーバー」はその一用途例にすぎず、課題追跡・監視データ分析・DB クエリ・デザイン統合など幅広い領域で使われている

<!--
初級編が各階層の「入り口」だったのに対し、中級編は「実戦」に踏み込みます。L3 ハーネスは入門の 1 本から、主要カテゴリ別に計 18 本の実装例に広がります。「網羅する」とは言わず、ファイル保護・コマンド検査・通知といった主要カテゴリそれぞれに 3〜4 本ずつ、合計で 18 本を体験する位置付けです。なぜこの数になるかというと、各カテゴリで「最小・典型・応用」程度の段階を踏むと、自然にこの本数に落ち着くからです。特に注目してほしいのが MCP です。MCP は「Model Context Protocol」の略で、LLM と外部ツール・データソース・API を繋ぐオープンソース標準プロトコル、つまり業界標準の規格そのものです。「AI の記憶を拡張する仕組み」と狭く説明されることがありますが、それは claude-mem などの一用途例で、本来はもっと広い接続規格です。中級編では自前の MCP サーバーを構築して、その仕組み自体を体験します。プロンプト・コンテキスト・ハーネスの深化に加えて、この MCP が中級編の大きな柱です。
