# 道具は覚えた。でも自分の現場ではどう使う?

## 3層の復習 — 何を強化したいかで選ぶ

| 層 | 強化する内容 | 代表的な道具 | こんな困りごとに |
|----|------------|------------|----------------|
| **プロンプト層** | 1回の指示の質 | Skills (旧カスタムコマンド) / 診断チェックリスト | 同じプロンプトを毎回書いている |
| **コンテキスト層** | AIに覚えさせる知識 | chunk / RAG / memory / 検索 | 過去の知識をAIが知らない |
| **ハーネス層** | 自動化と安全装置 | hook / advisor / 禁止操作リスト | AIに任せきりで品質が見えない |

> 自分の困りごとが「どの列」に当てはまるか — それが「どの層から始めるか」のヒント
> ※ **ハーネス層**は Anthropic 公式概念 (Engineering Blog「Effective harnesses for long-running agents」2025-11)。Agent = Model + Harness の Harness 層に hook / 安全装置 / advisor を実装します。

<!--
3層の復習です。横に「こんな困りごとに」の列を追加しました。自分が抱えている困りごとがどの列に当てはまるかを考えると、自然と「どの層から始めるべきか」が見えてきます。プロンプト層の代表道具「Skills」は公式に「カスタムコマンドはスキルにマージされました」と明記された機能で、旧 `.claude/commands/<name>.md` 単一ファイル形式と新 `.claude/skills/<name>/SKILL.md` ディレクトリ形式の 2 つの書き方を持つ同一機能です (ch01 で詳述)。ハーネスは Anthropic 公式概念で、Agent = Model + Harness という業界標準定義です。本章ではこの 3 層をどう組み合わせるかを 3 ケースで具体化します。
-->

## ケース① — 営業の3層組み合わせ

> **現場の声**: 「顧客への提案メールを毎回ゼロから書いている。前回の要望も調べ直している」

**プロンプト層** — Skills (旧カスタムコマンド) で定型化 (ch01 で作成)
```
/propose-mail @顧客A議事録.md
  ※「@ファイル名」はAIにそのファイルを読み込ませる指定です
  ※ 公式仕様: カスタムコマンドは Skills 機能にマージ済み
     (公式 docs/ja/skills「カスタムコマンドはスキルにマージされました」)
  ※ 同じ機能の 2 つの書き方:
     ・新形式 `.claude/skills/<name>/SKILL.md` (ディレクトリ・推奨)
     ・旧形式 `.claude/commands/<name>.md` (単一ファイル・併存)
→ 件名・トーン・締めの一言が毎回ゼロからではなく、
   テンプレートベースで生成される
```

**コンテキスト層** — memory検索で過去を呼び戻す
- 前回の顧客要望・クレーム履歴を意味検索 (ch04 で紹介)
- 例: 外部 MCP サーバー `claude-mem` で「この顧客について検索」と自然文入力するイメージ (MCP の自然文呼出は公式仕様・claude-mem 固有 UI 詳細は別途公式 README で確認)
- 「この顧客は前回〇〇に不満だった」を自動参照

**ハーネス層** — advisor で念押し
- 提案内容に「前回の要望との矛盾」がないか第三者レビュー (Anthropic 公式 `advisor_20260301` ツール・ch06 で詳述・beta 機能)

<!--
1つ目のケースは営業です。顧客提案メールを毎回ゼロから書いている場合、プロンプト層の Skills 化が最初の一手です。重要な前提として、旧称「カスタムスラッシュコマンド」は公式仕様で Skills 機能にマージ済みで、公式 docs/ja/skills に「カスタムコマンドはスキルにマージされました」と明記されています。つまり別物 2 機能ではなく、同じ機能の旧 (`commands/`・単一ファイル) ・新 (`skills/`・ディレクトリ) 2 つの書き方として併存している関係です。新形式が推奨される理由は (1) サポートファイル同梱可能 (2) YAML フロントマターで宣言可能 (3) description マッチでオンデマンド自動読込、の 3 点で、いずれも常駐コストを抑える設計です (ch01 で詳述)。次にコンテキスト層で過去の顧客対応履歴を意味検索できるようにします。MCP は LLM と外部ツールを繋ぐオープン標準で、自然文での MCP サーバー呼出は公式仕様。ただし claude-mem 固有の UI 表記や検索体験は claude-mem 公式 README/GitHub で別途確認してください。最後にハーネス層で advisor (Anthropic 公式 advisor_20260301 ツール・beta 機能) による念押しレビューを入れます。
-->

## ケース② — PMサポートの3層組み合わせ

> **現場の声**: 「タスクの粒度がバラバラで、AIに任せると抜けが出る。進捗も見えない」

**プロンプト層** — タスクを構造化して書く
- ch02の診断10項目を適用: タスクの前提・出力形式・完了条件を明記

**コンテキスト層** — memory で過去の決定事項を記憶
- 先週の進捗・「なぜこの方針にしたか」を ch04 の意味検索で呼び戻し
- 同じ前提説明を毎セッション繰り返す必要がなくなる
- プロジェクト固有の規約は `CLAUDE.md` (公式) に書いて全セッションで自動ロード

**ハーネス層** — advisor + 自動チェック
- WBS(プロジェクトのタスク分解表)の妥当性を advisor (Anthropic 公式 tool) にレビュー
- ドキュメントやスプレッドシート更新後に自動で内容チェック (ch07 で学んだ **PostToolUse hook** — 公式 hook 29 種類の 1 つで「ツール実行が成功した直後」に発火)

<!--
2つ目のケースはPMサポートです。タスク管理の粒度がバラバラなら、まずプロンプト層でタスクの書き方を構造化します。次にコンテキスト層で過去の決定事項を memory に蓄積し、ch04 で学んだ意味検索で呼び戻せるようにします。プロジェクト固有の規約 (用語集・命名規則・ドキュメント置き場等) は公式機能の CLAUDE.md に書いておけば全セッションで自動ロードされます。最後にハーネス層で advisor (Anthropic 公式 advisor_20260301 tool) による自動レビューと、PostToolUse hook (公式 hook 29 種類の 1 つ・「ツール実行成功後」に発火) によるテスト実行を入れます。
-->

## ケース③ — データ分析の3層組み合わせ

> **現場の声**: 「分析のたびに前提条件を説明している。前処理でミスに気づかない」

**プロンプト層** — 分析要件をタグで構造化(ch02で学んだ「確認指示」の応用です。`<目的>`のように山括弧で囲むと、AIが「ここは目的・ここはデータ」と構造を理解しやすくなります)
```
<目的>売上の前年比較</目的>
<データ>sales_2025.csv</データ>
<出力>グラフ付きレポート</出力>
```
- 目的・データソース・出力形式をタグで明記 → AIが「何を分析すべきか」を勘違いしない
  (このように`< >`で囲む書き方を「XMLタグ」と呼びます。HTMLを知らなくても、AIに渡す指示を構造化するだけならこの書き方だけで十分です)

**コンテキスト層** — memoryで過去の分析結果を記憶
- 「前回は〇〇のデータ品質に問題があった」等の注意事項を記憶
- 同じミスを繰り返さない

**ハーネス層** — AIに役割を分けて分業 + 段階確認(ch05で学んだ役割別ファイル + ch08の段階確認)
- AIに「データ取得担当」「分析担当」「レポート担当」の役割を割り当てて分業
- 前処理後のデータ件数を段階確認 (チェックポイント) で検証 (ch08 解決策①)

<!--
3つ目のケースはデータ分析です。分析前提を毎回説明しているなら、プロンプト層で構造化します。過去の分析で得た知見はmemoryに蓄積し、ハーネス層で役割別の分業と段階確認を入れます。自社の分析対象データやレポート形式に置き換えて応用できます。役割別ファイルは ch05、段階確認は ch08 解決策①で扱った道具です。役割別ファイルの構成パターンは VSCode Copilot 由来の `.github/instructions/` 規約を Claude Code 用にアレンジした本リポジトリ独自運用ですが、その背景は ch05 で詳述しています。
-->

## 「どの層から始めるか」の判断基準

> 自分の困りごとに合う層を**最初の一手**にする — 全部一度に導入しなくていい

| 困りごとの特徴 | 最初の一手 | 具体的にやること |
|---------------|-----------|----------------|
| 「毎回同じプロンプトを書いている」 | **プロンプト層** | Skills (旧 Commands とマージ済・同機能) を 1 つ作る |
| 「過去の知識をAIが知らない」 | **コンテキスト層** | memoryに1件書く / 意味検索を試す |
| 「AIに任せきりで品質が見えない」 | **ハーネス層** | advisor を 1 回呼ぶ / hook を 1 つ入れる |

> 最初は**1層・1つ**でいい。効果を確認してから次の層に進む

<!--
重要なのは「全部一度にやらない」ことです。自分の困りごとに合う層を1つ選び、そこから始めます。効果を確認してから次の層に進む。この段階的なアプローチが確実です。プロンプト層の最初の一手は Skills (`.claude/skills/<name>/SKILL.md`) を 1 つ作るところから始めるのが推奨。公式仕様で旧カスタムコマンド (`.claude/commands/<name>.md`) は Skills にマージ済みなので、これから新しく作るなら新形式 Skills を選んでおくと、サポートファイル同梱・description 自動読込といった新形式特有の利点をそのまま享受できます。

## 確認問題

**Q1**: 「毎回同じプロンプトを書いている」という困りごとがある場合、どの層から始めるのが適切ですか?

**Q2**: 営業ケースで「前回の顧客クレーム」をAIに自動参照させるには、どの層の技術を使いますか?

**Q3**: 3ケースに共通する「定型化 → 検索 → 担保」のうち、「担保」にあたる代表的な道具を2つ挙げてください。

---

**A1**: プロンプト層です。まず Skills (旧カスタムコマンドが公式仕様で Skills にマージ済・同じ機能の旧 `commands/`・新 `skills/` 2 形式が併存) で、繰り返し使うプロンプトを 1 つ資産化します。これから新規作成するなら推奨の新形式 `.claude/skills/<name>/SKILL.md` を選んでおくと、サポートファイル同梱・description 自動読込の利点をそのまま得られます。「毎回同じものを書いている」= プロンプト層の課題と判断できます。

**A2**: コンテキスト層の技術 (memory検索 / RAG / MCP サーバー) を使います。過去の顧客対応履歴をmemoryに蓄積しておき、プロンプト送信時に意味検索で関連履歴を自動参照させます。外部 MCP サーバー (例: `claude-mem`) を組み合わせる構成が代表例ですが、claude-mem 固有の UI/コマンド体系は claude-mem 公式 README で別途確認してください。

**A3**: **advisor** (Anthropic 公式 `advisor_20260301` ツール・第三者AIレビュー) と **段階確認** (チェックポイント・ch08 解決策①) の2つです。いずれもハーネス層 (Anthropic 公式概念「Agent = Model + Harness」の Harness 層) に属する「品質を担保する」仕組みです。
