# 🚨 AIに「掃除して」と頼んだら、プロジェクトが消えた

> 「不要なファイルを整理して」と頼んだ瞬間、AI が `rm -rf .` を実行 — **プロジェクト全消失**。
> ChatGPT 普及以降、業界で実際に報告されているエージェント暴走事故の典型形です。

- AI は強力だが、**自律性が高まるほど誤操作リスクも増大** (前章 L3 ハーネス誕生の動機そのもの)
- `rm -rf` (一括削除) や `sudo` (管理者権限実行) を毎回人間が「実行していい?」と確認するのは **非現実的**
- 必要なのは「AI が危険な操作をする前に **自動で立ち止まらせる仕組み**」 = 本章で作る **Hook**

**この 30 分で得られること**:
1. **Hookとは何か** — AI のライフサイクルイベントに任意の処理を発火させる仕組みを説明できる
2. **PreToolUse Hook** の仕組みと役割を理解し、何が起きているかを読める
3. **Claude に頼んで Hook を作ってもらう** プロンプトが書ける — シェルスクリプトの暗記は不要

# Hookとは — AIのライフサイクルに割り込む仕組み

**Hook = AI のライフサイクルイベント (セッション開始 / プロンプト送信 / ツール実行前後 / 圧縮前 / 終了 など) に、任意の処理を発火させる仕組み**

- Claude Code が **ツールを使う前** にチェック処理を挟める (本章のテーマ)
- セッション開始時に資料を自動で渡す、終了時にログを残す、なども同じ仕組み
- **「安全装置」はそのうちの代表的な一用途**にすぎない

| イベント | タイミング | 代表的な用途 |
|----------|-----------|-------------|
| **PreToolUse** ★ | ツール実行の **前** | 危険なコマンドをブロック (本章で扱う) |

- Hook は公式で **全 29 種類**。本章で覚えるのは **PreToolUse 1 つだけ** で十分
- 残り 28 種類 (PostToolUse / SubagentStop / SessionStart / PreCompact / Notification / Stop など) は応用編で扱う

> 💡 銀行 ATM で「大金を引き出す前に暗証番号を確認する」のと同じ — 実行前に一旦止めて確認する用途は Hook の使い道のひとつ

<!--
Hook の本質を正しく定義します。「AI の動きに制限をかける仕組み」と狭く捉えるのは誤りで、本質は「AI のライフサイクルイベントに任意処理を発火させる広義の機構」です。安全装置はそのうちの代表的な一用途。公式で全 29 種類のイベントがあることを明示し、本章では PreToolUse 1 つに集中することを宣言します。残り 28 種類のうちよく使うイベントを名前だけ紹介して安心情報にしておきます。SubagentStop は a が小文字、これは公式表記なので大文字 A で書くと動きません — 応用編で扱うときに改めて触れますが、ここで一度名前を見せておきます。ATM の例えで「実行前にちょっと止めて確認する」が Hook の根本の発想だと感覚で掴んでもらいます。

# PreToolUse Hook の動き — Before / After

**PreToolUse = ツールが実行される「直前」に検査を挟む仕組み**

```
AIがコマンドを生成
       ↓
  【PreToolUse Hook】← ここで検査！
       ↓
  安全 → 実行
  危険 → ブロック (STOP)
```

**Hookなし**: AI が `rm -rf /project` を生成 → そのまま実行 → **プロジェクト消滅**
**Hookあり**: AI が `rm -rf /project` を生成 → Hook が検知 → `STOP: 危険な操作を検知しました` → **未然に防止**

> 🛡️ たった 1 本の Hook で、致命的な事故を未然に防げる — これが L3 ハーネスの威力

<!--
PreToolUse の動きを図と Before/After で 1 枚にまとめました。AI が生成したコマンドはまっすぐ実行されるわけではなく、PreToolUse Hook が間に入って検査します。安全なら通過、危険ならブロック。人間がいちいち判断しなくても、瞬時に止められるのが価値です。冒頭の痛点フックで提示した `rm -rf` 事故も、この仕組み 1 本で未然に防げる。L3 ハーネスの効き目を最初に体感してもらいます。次のスライドからは「では実際にどう作るか」 — その第一歩として、Claude が裏側で知っている公式仕様を 4 行だけ概念で押さえます。
-->

# Hook 公式仕様の核心 — Claude が知っている 4 つのお約束

Hook を「Claude に作ってもらう」前に、**Claude が裏側で何を満たしているか** を 4 行で押さえる:

| 項目 | 内容 | なぜ重要か |
|------|------|-----------|
| 設定ファイル | `.claude/settings.json` の `hooks` セクションでイベントと実行スクリプトを宣言 | Claude Code はここを読んで Hook を有効化する |
| データ受渡し | スクリプトは **stdin (標準入力) 経由で JSON ペイロードを受け取る** | コマンド内容は `tool_input.command` フィールドに入っている |
| ブロック合図 | スクリプトを **`exit 2` で終了** | Claude Code はこの値を「危険検知 → 実行中断」と解釈する |
| 理由の伝達 | **stderr (標準エラー出力) にメッセージ** を流す | AI はこの内容を「ブロック理由」として読み、別案を考える |

> 💡 受講者がこの 4 行を **暗記する必要はない** — Claude にプロンプトを書く際、この 4 項目を「仕様」として渡せばよい。**Claude 自身がこの 4 行を知っている** ので、適切に頼めば公式仕様に沿った実装が出てくる。
>
> ⚠️ 古い記事には `$1` (引数) や独自環境変数で受け取る書き方が散見されるが、**公式仕様に存在せず動かない**。Claude に頼む際も「stdin から JSON で受け取って」と明示すると安全。

<!--
ここがこの章の概念的な要です。Hook の公式仕様は 4 つの要素 — settings.json での宣言、stdin JSON 受取、exit 2 でブロック、stderr で理由伝達。受講者にこれを暗記させるのではなく、「Claude が知っているお約束」として概念で掴んでもらいます。なぜなら、次のスライドで Claude にプロンプトを書く際、この 4 項目を仕様として含めて頼めば Claude は正確な実装を返してくれるから。「stdin から JSON で受け取って・exit 2 + stderr でブロック」の一文を覚えるだけで、Claude が出す実装が公式仕様通りになります。逆に古い記事に流通している誤った受け取り方 — $1 引数や独自環境変数 — は公式に存在せず動きません。Claude に頼む際は「stdin から JSON を受け取って」と明示することで、Claude が古い情報に引きずられるのを防げます。「概念を掴んで Claude に正しく頼む」 — これが本章の核心アプローチです。

# 実装は Claude に頼む — プロンプト例

**手書きで `.claude/settings.json` や hook スクリプトを書く必要はない**。Claude にこう頼むだけで、両方とも生成してくれる:

```
Claude Code 用の PreToolUse hook を作って:
- 対象ツール: Bash
- 動作: 実行されようとしているコマンドに `rm -rf` が含まれていたらブロック
- データ受け取り: stdin から JSON ペイロード・コマンド本体は tool_input.command
- ブロック方法: exit 2 + stderr にブロック理由メッセージ
- 出力ファイル:
  1. `.claude/settings.json` (既存があればマージ)
  2. `.claude/hooks/safety.sh` (実行権限 chmod +x も付けて)
- 完了したら、新しいセッションを開始して動作確認するための手順も教えて
```

**Claude が返してくるもの**:
- `.claude/settings.json` — Hook を有効化する設定 (PreToolUse の Bash 用 hook 登録)
- `.claude/hooks/safety.sh` — stdin から JSON を受け取り、`rm -rf` を含む場合に exit 2 でブロックするスクリプト
- chmod +x の実行コマンドと、新セッションでの動作確認手順

> 💡 受講者は **「動く Hook が手元に出てきたか」** だけを確認すればよい。シェル / jq / JSON の文法を覚える必要はない。
> 💡 上のプロンプトはコピペして使ってよい。要件を書き換えれば自分の現場用にカスタマイズできる。

<!--
新方針の核心スライドです。前のスライドで概念を掴んだ Hook 公式仕様 (stdin JSON / exit 2 / stderr) を、Claude へのプロンプトに「仕様」として書き込みます。すると Claude は公式仕様通りの実装を一発で返してくれます。受講者が学ぶのは「シェルスクリプトの書き方」ではなく「AI に作業を正確に依頼するプロンプトの書き方」 — これが本講座の本筋です。プロンプトの中で「対象ツール」「動作」「データ受け取り」「ブロック方法」「出力ファイル」と要素を並べているのに注目してください。これがプロンプトエンジニアリングの基本パターン (要件を分解して箇条書きで渡す) で、次のスライドで「良いプロンプトの作り方」として一般化します。Claude が返してくる中身を「読まなくてもよい」と明言するのは、初学者の心理的ハードルを下げるためです。「動いた / 動かない」だけ確認できれば十分、という割り切りが本講座の方針です。

# 良いプロンプトの作り方 — 仕様を 5 要素で渡す

Claude に Hook (や任意の実装) を頼むとき、**5 つの要素** を箇条書きで渡すと精度が上がる:

| 要素 | 例 |
|------|-----|
| **① 何を作るか** | 「PreToolUse hook を作って」 |
| **② どこで動くか** | 「対象ツール: Bash」「macOS / Linux 想定」 |
| **③ どう振る舞うか** | 「rm -rf を含むコマンドをブロック」 |
| **④ 仕様の制約** | 「stdin から JSON 受取・exit 2 + stderr でブロック」 |
| **⑤ 出力先と後処理** | 「`.claude/settings.json` と `.claude/hooks/safety.sh` に分けて保存・chmod +x も付けて」 |

> 💡 ④ の「仕様の制約」を入れるのが要 — 古い情報や勘違いを排除し、Claude を **正しい公式仕様に向けて誘導** できる。
> 💡 不確かな部分は「公式仕様に従って実装して」と添えれば、Claude は最新の正解に寄せてくる。
> 💡 同じパターンが Skill やカスタムスラッシュコマンドなど、より発展的な依頼にも応用可能 — 「何を / どこで / どう / 制約 / 出力」の 5 要素は普遍。

<!--
プロンプトエンジニアリング (L1) の基礎を、Hook という具体例で実演しています。要素を 5 つ並べて渡すと AI は迷わず正解に到達できる、これは Hook 以外の依頼にも応用できる普遍パターンです。特に④の「仕様制約」を入れるのが重要 — Hook の場合は「stdin JSON / exit 2 / stderr」を明示することで、Claude が古い記事の $1 引数方式に流れるのを防げます。「公式仕様に従って実装して」の一言を添えるのも有効。受講者がこの先 Skill やカスタムスラッシュコマンドなど発展的な機能を頼むときも、同じ 5 要素パターンが使えます。プロンプトエンジニアリングは「魔法のフレーズ」ではなく「要件を構造化して渡す技術」 — これを Hook 作成という現場で身につけてもらう構成です。受講者がこのスライドを「コピペして自分用に書き換える原型」として持ち帰れるようにしました。

# 拡張も同じ — 「もっと止めて」と追加で頼む

**よくある危険パターン**:

| 危険パターン | 具体例 | 被害 |
|-------------|--------|------|
| 再帰削除 | `rm -rf /` | OS / プロジェクト全体を消去 |
| 特権実行 | `sudo`, `su` | システム設定を変更 |
| データ破壊 | `DROP TABLE` | データベースのテーブルを削除 |
| 強制 push | `git push --force` | リモートの履歴を破壊 |

**追加で守りたくなったら、Claude に追加プロンプトを送るだけ**:

```
さっきの safety.sh に、`sudo` と `git push --force` の検知も追加して。
他のパターンには触らないで。
```

> 💡 「最初は最小から始めて、被害が見えたら追加していく」 — これが現場での Hook 育て方。コードを自分で書き直す必要はない。
> 💡 「他のパターンには触らないで」を添えると、AI が良かれと思って関係ない部分まで書き換えるのを防げる (差分指示の基本)。

<!--
Hook が防ぐべき危険なパターンを 4 つ示します。再帰削除・特権実行・データ破壊・強制 push は実務でも頻出の事故源です。重要なのは、これらを後から追加するときも「Claude に追加プロンプトを送るだけ」で済むということ。コードに書き慣れていない受講者でも、「sudo も検知して」と一言頼めば Claude が判定ロジックを更新してくれます。「他のパターンには触らないで」と書くのは、AI が良かれと思って関係ないところまで書き換えるのを防ぐため — これも実践的なプロンプト技術 (差分指示) の一例です。最初から完璧を狙わず、最小から始めて使いながら育てる、これが現場での Hook 運用の鉄則です。「拡張も Claude に任せる」が習慣化すれば、Hook はどんどん厚くなっていきます。
-->

# Hook の実行フローまとめ

```
1. ユーザーがプロンプトを入力
2. AI がツールの使用を判断（例: Bash で rm -rf を実行）
3. PreToolUse Hook が発動
4. Claude Code がチェックスクリプトに JSON を stdin で渡す
5. スクリプトが JSON から .tool_input.command を取り出して検査
6. 安全（exit 0）→ 実行継続
   危険（exit 2 + stderr メッセージ）→ ブロック + 警告表示
7. AI がブロック理由 (stderr の内容) を受け取り、別の方法を検討
```

> Hook は **「検閲」ではなく「安全ネット」** — AI も理由を理解して別案を出す
> このフロー全体を **Claude が裏側で実装してくれる** ので、受講者は「何が起きているか」を理解していれば十分

# 🖐️ ミニ演習 — プロンプトで Hook を作ってもらおう

**手順 (5 分・観察用の安全な Hook)**:

**Step 1**: Claude Code を起動して、以下のプロンプトをそのまま送る:

```
PreToolUse hook を作って:
- 対象ツール: Bash
- 動作: 実行されようとしているコマンドを stderr に「[CHECK] <コマンド>」の形で表示するだけ・ブロックはしない (exit 0)
- データ受け取り: stdin から JSON ペイロード・コマンド本体は tool_input.command
- 出力ファイル:
  1. `.claude/settings.json` (既存があればマージ)
  2. `.claude/hooks/hook_demo.sh` (chmod +x も付けて)
- 完了したら新セッションでの動作確認手順を教えて
```

**Step 2**: Claude が `.claude/settings.json` と `.claude/hooks/hook_demo.sh` の **2 ファイルを作るのを確認** (中身は読まなくて OK)

**Step 3**: **新しいセッションを開始** (`/clear` または ターミナル終了→`claude` 再起動) — Claude Code は新セッション開始時に `.claude/settings.json` をディスクから読み直す

**Step 4**: 「現在のディレクトリのファイル一覧を表示して」と入力 → `[CHECK] ls -la` のような出力が出れば **Hook が動いている証拠**

> 💡 これは exit 0 (ブロックしない) の安全な観察用。実際の `rm -rf` ブロック版が欲しければ、本章で先に紹介した exit 2 のプロンプト例を使う。
> 💡 動作しないときは Claude にエラーメッセージをそのまま貼って「動かないので直して」と頼めばよい — 受講者が文法を直す必要はない。

# 📚 まとめ + 次のステップ

**この章で学んだ 3 つのポイント**:

1. **Hook = L3 ハーネスの公式実装機能** — AI のライフサイクルイベント (公式 29 種類) に任意の処理を発火させ、動作を制御・拡張する仕組み。「安全装置」はその代表的な一用途
2. **PreToolUse Hook** — コマンド実行前に検査し、危険な操作を自動ブロック (**`exit 2` + stderr メッセージ** で AI にブロック理由を渡す・データは **stdin JSON 経由**)
3. **実装は Claude に頼めば書いてくれる** — 受講者が学ぶのは「**仕様を 5 要素で渡すプロンプト**」 (① 何を ② どこで ③ どう ④ 制約 ⑤ 出力先)。シェルスクリプトの暗記は不要

> **3 階層モデルの全体像**:
> **L1 プロンプト** + **L2 コンテキスト** + **L3 ハーネス (Agent = Model + Harness)** = 安心して AI に任せる基盤

**次の章 (第6章 終章)**: これまでの 3 つの武器を振り返り、より進んだ自動化システムへの道筋を確認する。
