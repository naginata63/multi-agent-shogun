# こんな指示を出していませんか? — 3要素で解決

> 「いい感じにテストを書いて」「これ、よろしく」 — **同じ指示なのに毎回違う結果** が返ってくる

**人間なら分かる、AIには伝わらない**:
- 「これ」が何を指すか不明・「よろしく」の範囲が無限大
- AIは **意図を推測** して動く → 曖昧な指示 = 曖昧な結果

**解決策: プロンプトを 3 要素に分けて書く**

| 要素 | 役割 | 例 |
|------|------|-----|
| **文脈** | AIに与える背景・素材 | 「この記事について…」「@src/app.ts を見て…」 |
| **指示** | AIにやってほしいこと | 「要約して」「バグを特定して」 |
| **形式** | 結果をどう返すか | 「箇条書き3つで」「表形式で」 |

> 💡 3要素を整理すると、AIの誤解が減って **出力品質が安定** します

<!--
スピーカーノート:
ここで一度、皆さんに質問です。AIに「いい感じにテストを書いて」とか「これ、よろしく」と頼んだことはありませんか? そして同じ指示なのに毎回違う結果が返ってきて、困った経験はないでしょうか。これがまさに序章で見た「壁① 意図が伝わらない」の正体です。人間なら「これ、よろしく」と言われたら前後の文脈で「この書類の提出をお願いします」と解釈できますが、AIにはこれができません。「これ」が何か、「よろしく」の範囲がどこまでか、すべて言葉で明確にする必要があります。解決策がプロンプトの3要素です。文脈には背景や素材、指示にはやってほしいこと、形式には結果の返し方を書きます。AIは「どこまでが素材で、何をしてほしいのか」を混同しやすいので、3要素を明確に分けることで誤解を防ぎます。特別な記法は不要。自然な文章で大丈夫です。
-->

# テクニック1: @file でファイルを参照する

```
@src/app.ts のバグを特定して。
各バグについて深刻度（高/中/低）と
修正方法を箇条書きで教えて。
```

**3要素に分解してみよう**:
| 要素 | 対応部分 |
|------|---------|
| 文脈 | `@src/app.ts`（Webアプリのコードファイルを読み込む） |
| 指示 | 「バグを特定して」 |
| 形式 | 「深刻度と修正方法を箇条書きで」 |

**ポイント**:
- `@ファイル名` でプロジェクト内のファイルを直接参照（コードをコピペ不要）
- AIが自動的にファイル内容を読み込む

> 💡 Claude Code なら、ファイルの中身を **貼り付け不要**

<!--
スピーカーノート:
@file参照の例を見てみましょう。`@src/app.ts`はWebアプリのコードファイルの例です。この1行で文脈を与え、「バグを特定して」が指示、「箇条書きで」が形式。たった3行で3要素が揃います。ブラウザ版のAIではコードをコピーして貼り付ける必要がありましたが、Claude Codeでは@一つで完了です。これが3要素の「文脈」を簡単に与える方法です。
-->

# テクニック2: 段階分解 — 「まず○○、次に△△」

```
このプロジェクトで以下の手順で進めて：
1. まず @package.json（パッケージ管理ファイル）の依存関係（どのライブラリを使っているか）を確認して
2. 次に 使っていないパッケージを特定して
3. 最後に 削除しても安全か判断して
```

**ポイント**:
- 複雑な作業は **段階に分ける** と精度が上がる
- 「まず」「次に」「最後に」で順序を明示
- AIが途中で迷わない

<!--
スピーカーノート:
2つ目のテクニックは段階分解です。複雑な作業を一度に頼むと、AIがどこから手をつけるべきか迷うことがあります。「まず○○して、次に△△して」と段階に分けて指示すると、AIが迷わずに進められます。特にデバッグやリファクタリングなど、複数ステップがある作業では効果的です。手順を番号付きで書くとさらに明確になります。
-->

# テクニック3: 具体的に書く — 曖昧→具体

**悪い例**:
```
いい感じにテストを書いて
```

**良い例**:
```
@src/utils.ts の各関数に対して
ユニットテストを書いて。
テストフレームワークはvitest（JavaScriptのテストツール）を使い、
正常系（普通の入力）・異常系（おかしい入力）・境界値（ギリギリの値）
の3パターンを
それぞれカバーすること。
```

> 「いい感じに」→ 毎回違う結果
> 「正常系・異常系・境界値」→ ブレが大幅に減る

<!--
スピーカーノート:
3つ目のテクニックは具体性です。「いい感じに」「適当に」は絶対にNG。AIは親切なので何かしら出力してくれますが、それが毎回同じ内容になる保証はありません。具体的な条件を書くことで、出力のブレが大幅に減ります。テストなら「正常系＝普通の入力、異常系＝おかしい入力、境界値＝ギリギリの値」のように、期待する内容を明示しましょう。
-->

# おまけ: slash command — 定型操作を一発実行

**Claude Code の中で** `/` から始まるコマンドが打てます:

```
/init     → CLAUDE.md（設定ファイル）を自動生成
/review   → PRレビュー（コード変更の確認）を実行
/clear    → 会話をリセット
```

> 💡 ターミナルではなく **Claude Code のチャット画面の中** で打つコマンドです。`/init`・`/review`・`/clear` などは Claude Code 公式の組込コマンドで、ほかに公式同梱の `/security-review` `/simplify` などもあります。自分専用のコマンドも作れます (詳しくは第3章「.md 仲間たち」で解説)

<!--
スピーカーノート:
slash command は Claude Code の中で使える便利なコマンドです。ターミナルではなく、Claude Code のチャット画面で `/init` や `/review` と打つだけ。よく使う操作を一発で実行できます。`/init` `/review` `/clear` などは Claude Code 公式の組込コマンド、`/security-review` `/simplify` などは Anthropic が公式に同梱しているバンドルスキルです。自分専用のコマンドも作れますが、作り方や種類の詳細は第3章「.md 仲間たち」で扱います。本章では「便利な操作の入口がここにある」とだけ押さえてください。

# Before / After で見る再現性

**Before（曖昧な指示）**: 毎回違う形式で出力
```
1回目: 段落形式で要約
2回目: 箇条書きで要約
3回目: 表形式で要約
```

**After（3要素 + 具体性）**: 形式が揃い、ブレが大幅に減る
```
1回目: 箇条書き3項目 ✓
2回目: 箇条書き3項目 ✓（細部は少し変わることも）
3回目: 箇条書き3項目 ✓
```

**再現性が重要な3つの場面**:
- **使い回し**: 同じ処理を別のファイルに適用
- **共有**: チームメンバーにプロンプトを渡す
- **自動化**: スクリプトの中で何度も実行

> 再現性のないプロンプト = 使い捨て / 再現性のあるプロンプト = **資産**

<!--
スピーカーノート:
BeforeとAfterの違いを可視化しました。曖昧なプロンプトは、実行するたびに出力形式がブレます。段落だったり箇条書きだったり表だったり。3要素を意識して具体的に書いたプロンプトは、形式が揃います。細部が完全一致するとは限りませんが、ブレは大幅に減ります。これが「再現性」です。なぜ再現性が重要か。3つの理由があります。まず使い回し、別のファイルに同じ処理を繰り返す場面は多いです。次に共有、チームメンバーに「このプロンプトを使って」と渡す時に同じ結果が出なければ困ります。そして自動化、スクリプトの中で何百回と実行する時に結果がブレると意味がありません。再現性のあるプロンプトは、あなたのチームの資産になります。
-->

# よくある失敗パターン 3選

| # | 失敗パターン | 例 | 修正方法 |
|---|-------------|-----|---------|
| 1 | **曖昧な指示** | 「いい感じに要約して」 | 具体的な条件を書く |
| 2 | **形式の指定なし** | 指示だけで「どう返すか」未指定 | 「箇条書きで」「表で」を必ず書く |
| 3 | **文脈不足** | 背景情報を何も伝えない | 前提条件・@file参照で文脈を与える |

> この3つを避けるだけで、プロンプトの品質が **劇的に向上** する

<!--
スピーカーノート:
プロンプトで一番よくある失敗を3つ紹介します。1つ目は曖昧な指示。「いい感じに」「適当に」はAIには伝わりません。2つ目は形式の指定なし。これが一番多い失敗で、「どう返すか」を書かないと結果が不安定になります。3つ目は文脈不足。AIはあなたのプロジェクトの背景を知らないので、必要な情報は@file参照などで伝えてください。この3つを意識するだけで、プロンプトの品質が飛躍的に向上します。
-->

# 📚 まとめ — プロンプト3要素の復習

```
文脈  → 素材・背景（例: @src/app.ts でファイルを読み込む）
指示  → やってほしいこと（例: 「バグを特定して」）
```

**この章で学んだこと**:
1. プロンプトを文脈・指示・形式の3要素で整理する
2. @file参照・段階分解・箇条書き・slash command の実践テクニック
3. 具体的に書くことで再現性を担保する
