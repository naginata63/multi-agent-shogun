## 旧 Commands 形式から Skills 新形式へ — なぜもう一段上が必要か

**具体例**: 「毎週月曜に議事録を要約して日報を作る」業務

| やり方 | 手順 | 問題点 / 利点 |
|--------|------|--------|
| 毎回ゼロから書く | プロンプトを毎週新規作成 | 非効率・品質バラつき |
| **旧 Commands 形式** で資産化 | `.claude/commands/weekly-report.md` 単一ファイルにプロンプトを保存 | 手動 `/weekly-report` 呼出のみ・サポートファイル不可 |
| **Skills 新形式** で資産化 ★ | `.claude/skills/weekly-report/SKILL.md` + 周辺ファイルで業務知識パッケージ化 | **AI が description で自律発動 + サポートファイル同梱可能** |

> **手動呼出**: 人が `/<コマンド>` で意図的に呼び出す (旧 Commands 形式・新 Skills 形式ともに可)
> **自律発動**: **description のキーワードマッチ**で AI が自律的に Skill 本体を読み込む (Skills 新形式のみ)

<!--
スピーカーノート: この表の一番下「Skills 新形式」が重要です。旧 Commands 形式までは人が「日報作って」と指示する必要があります。しかし Skills 新形式で description を整え、発動条件を Skill 側に書き込むと、AI が「これは日報の文脈だ」と判断して Skill 本体をオンデマンド読込します。同じ「カスタムスラッシュコマンド」という機能の中で、旧 Commands 形式と新 Skills 形式の使い分け、特に新形式が持つ「自律発動」の能力が鍵です。
-->

## Skill の構造 — ディレクトリ + エントリポイント + 追加機能

```
.claude/skills/
└── weekly-report/        ← ディレクトリ名がコマンド名 /weekly-report になる
    ├── SKILL.md          ← エントリポイント (frontmatter + 本文)
    ├── templates/        ← サポートファイル (テンプレート同梱可)
    │   └── weekly.md
    └── refs/             ← サポートファイル (参考データ同梱可)
        └── format.md
```

> **Skills 新形式 (推奨・2026年1月公式統合)**: `.claude/skills/<skill-name>/SKILL.md`
> **personal scope**: `~/.claude/skills/<skill-name>/SKILL.md`
> **旧 Commands 形式 (legacy・併存サポート)**: `.claude/commands/<name>.md` (単一 .md)

**Skills 新形式が旧 Commands 形式に持つ追加機能**:

| 機能 | 内容 |
|------|------|
| サポートファイル同梱 | テンプレート・スクリプト・参考データを同ディレクトリ配置 |
| frontmatter | `name` / `description` で自律発動条件を定義 |
| 自動読込制御 | description 常駐 + 本体オンデマンド読込 (コンテキスト節約) |
| サブエージェント呼出 | Task tool 経由で別エージェントから起動可能 |
| パス制限 | `.claude/skills/<name>/` 配下スコープに閉じる |

<!--
スピーカーノート: Skill の配置は単一の .md ファイル (旧 Commands 形式) ではなく、「ディレクトリ + SKILL.md エントリポイント」(Skills 新形式) が 2026年1月の公式統合以降の推奨です。ディレクトリ形式にすることで、Skill 本体以外にサポートファイル (テンプレート、スクリプト、参考データ等) を同梱できる点が最大の違いです。さらに frontmatter による自律発動条件の定義、description のみ常駐の自動読込制御、Task tool 経由のサブエージェント呼出、パス制限によるアクセス管理など、旧 Commands 形式単一ファイルでは実現できなかった機能が一通り揃います。新規作成は Skills 新形式のディレクトリ形式で進めてください。
-->

## Skill ファイルの中身 — frontmatter + 本文

**`.claude/skills/weekly-report/SKILL.md`**:

```yaml
---
name: weekly-report
description: 週次業務日報を半自動生成する。議事録要約 + 過去記憶検索 + 指定フォーマットのドラフト出力まで一貫して担当。Use when ユーザーが週報・日報・週次サマリの作成を依頼したとき、または議事録ファイル更新後に週次集約が必要なとき。
---
```

```markdown
## この Skill がやること
1. 今週の議事録ファイル (.md) を読み込む
2. 「決定事項」「宿題」「次週の予定」を抽出
3. 過去の週報 (MEMORY.md / トピックファイル) を必要時に参照し、継続中タスクを補完
4. 指定フォーマットで週報ドラフトを出力

## 出力フォーマット
### 今週のハイライト (3 行以内)
### 決定事項
### 宿題 (担当・期限付き)
### 来週の予定

## 注意事項
- 推測で補完せず、議事録にない情報は「未記載」と明記
- 前週からの継続タスクは必ず参照
```

<!--
スピーカーノート: description は公式推奨形式「〇〇する。Use when ××のとき。」に揃えています。これが Skills 新形式の「自律発動の引き金」になります。description は常駐し、本文 (## 以下) は使用時にオンデマンド読込される二段構造を意識してください。旧 Commands 形式の単一 .md ファイルには frontmatter がないため、この自律発動の仕組みは Skills 新形式の特権です。
-->

## Skill と hook の連携 — さらに先へ

ch07 で学んだ hook (公式 29 種類イベント) と Skill を組み合わせると、**完全な自動化**が実現する:

```
┌───────────────────┐
│  議事録が更新された │ ← タイミング (FileChanged 等の公式 hook が検知)
└────────┬──────────┘
         │
         ▼
┌───────────────────┐
│  Skill が発動する   │ ← 業務知識 (Skill 本体をオンデマンド読込)
│  「週報作成」       │
│  1. 議事録を読む    │
│  2. MEMORY.md を参照│
│  3. ドラフトを出力  │
└───────────────────┘
```

> **hook = 「いつ動くか」の引き金** (公式 29 種類のライフサイクルイベント)
> **Skill = 「何を呼ぶか」の業務知識セット** (description で自律発動条件を定義)
