## 3. 3 層の世界観 (本講座の地図)

> **本講座の核心**: AI 開発で起きる困りごとは、必ず以下の 3 層のどこかが欠けている。

### 3.1 3 層モデル

```
┌────────────────────────────────────────┐
│  プロンプト層 (Prompt)                  │
│  「AI への 1 回の指示の質」              │
│  ── /command・診断・XML タグ・関数化     │
├────────────────────────────────────────┤
│  コンテキスト層 (Context)               │
│  「AI に何を覚えさせ、何を見せるか」      │
│  ── memory・handoff・必要時取得・配置技術 │
├────────────────────────────────────────┤
│  ハーネス層 (Harness)                   │
│  「AI を動かす仕組み・自動化の枠組み」    │
│  ── 役割別ファイル・/advisor・hook・Skill │
└────────────────────────────────────────┘
```

### 3.2 各層の役割と本講座の章配置

| 層 | 役割 | 担当章 | 持ち帰る道具 |
|----|------|-------|------------|
| **プロンプト層** | 1 回の指示を資産化・診断する | ch01 / ch02 | `/command` / 診断チェックリスト 10 項目 |
| **コンテキスト層** | 必要なときに必要な情報を AI に届ける | ch03 / ch04 | TL;DR 配置 / claude-mem / handoff / 必要時取得 |
| **ハーネス層** | AI を分業・第三者レビュー・自動化させる | ch05-08 | 役割別ファイル / `/advisor` / 4 種の hook / Skill |

#### Script / Slash Command / Skill 比較表 (必須スライド)

| 観点 | Script (Bash 等) | Slash Command (`/name`) | Skill |
|------|------------------|------------------------|-------|
| 起動 | コマンドライン手動 | プロンプト中で AI が呼ぶ | **AI が自律判断で呼ぶ** |

| 用途 | 手動繰り返し作業 | 定型プロンプトの資産化 | **業務領域知識の常駐化** |
| 本講座での位置づけ | 補助 | 中心 (ch01) | 到達点 (ch11) |

- 1 つの `/command` を作って動かせる
- 自分の業務で `/command` 化候補を 3 件挙げられる
- Skill が ch11 の到達点であると展望できる

### ch02. よくある失敗 3 パターンの診断 [プロンプト層]

**所要**: 30 min / 担当する困りごと: **① AI が推測で動く・確認せず突進する**

#### Emotional Hook (現象 → 対策)

- **対策**: プロンプトを **10 項目で診断** して、推測で動く原因を切り分ける。医者が問診で病気を絞るように、症状を体系的に診る

#### 学習目標
- 失敗 3 パターン (推測突進 / 確認不足 / 範囲過大) を診断できる
- 自分のプロンプトを 10 項目で自己診断できる
- 「症状 → 原因 → 修正」の 3 セクション構成で自分のプロンプトを直せる

#### 解説アウトライン
1. 失敗 3 パターンの典型例 — 受講者が「あるある」と頷く実プロンプト
2. 診断チェックリスト 10 項目の内訳 (前提・出力形式・検証手段 等)
3. 失敗プロンプト 3 本の解剖 — 症状/原因/修正/結果の Before-After
4. 「推測 (esper) で動く」を防ぐ確認問いかけテンプレ 3 種
5. 再発防止のためのプロンプト設計原則 3 点
6. 章末ミニ演習: 自分のプロンプト 1 本を 10 項目で診断

#### 持ち帰り
- 自分のプロンプトを 10 項目で診断できる
- AI が推測で動かないようにする問いかけパターンを 3 つ持ち帰る

#### 強化した層
> **プロンプト層** を強化 — プロンプトの**診断と修正**ができるようになる。

#### Emotional Hook (現象 → 対策)
- **現象**: 「ちゃんと書いたのに AI が読み飛ばす」「会話が長くなると応答品質が落ちる」「2 万トークンの仕様書を渡しても核心を捉えてくれない」
- **対策**: AI の context window には **3 つの制約** がある。それらを理解し、配置技術 (TL;DR / 末尾サマリ / 必要時読込) で対処する

#### 学習目標
- 制約① 容量上限 (context window のサイズ限界) を説明できる
- 制約② Lost in the Middle の発生条件 (中央の参照情報が読み飛ばされる現象) を説明できる
- 制約③ Context Rot (会話が長く続くと AI の応答品質が劣化する現象) を説明できる
- 重要情報を冒頭/末尾に配置する書き換え技術を使える

#### 解説アウトライン
1. **3 制約スライド①: 容量上限** — context window のサイズと、超えると何が起きるか
2. **3 制約スライド②: Lost in the Middle** — 「コンテキスト/参照情報の **中央**」が読み飛ばされる (Liu et al. 2023 を引用)。**「指示が中央にある」ではなく「参照情報が長く、その中央にある」のが正確な定義**
3. **3 制約スライド③: Context Rot** — 会話履歴が長く積み上がると応答の質が劣化する。古い対話を圧縮・破棄する判断が必要
4. 配置技術① 冒頭 TL;DR — 重要結論を最初の 3 行に
5. 配置技術② 末尾サマリ — 章末・指示末尾の再要約
6. 章末ミニ演習: 自分の長文プロンプトに TL;DR を追加して比較

#### Lost in the Middle の正確な解釈 (殿指摘の反映)

> ❌ 誤: 「指示が長すぎると、指示の中央が読み飛ばされる」
> ✅ 正: 「**参照情報** (添付ドキュメント・過去のログ等) が長く、その**中央に重要な事実**があると AI が読み飛ばしやすい現象」
>
> → 対策: 重要な参照情報は **冒頭または末尾** に置く・添付の長さ自体を**必要時取得**で削る (ch04 へ繋がる)

#### 持ち帰り
- 3 つの制約を区別して説明できる
- 長文プロンプトを TL;DR + 末尾サマリで再構成できる

#### 強化した層
> **コンテキスト層** を強化 — context window の制約を理解し、配置で対処できる。
