## 3. 3 層の世界観 (本講座の地図)

> **本講座の核心**: AI 開発で起きる困りごとは、必ず以下の 3 層のどこかが欠けている。

### 3.1 3 層モデル

```
┌────────────────────────────────────────┐
│  プロンプト層 (Prompt)                  │
│  「AI への 1 回の指示の質」              │
│  ── /command・診断・XML タグ・関数化     │
├────────────────────────────────────────┤
│  コンテキスト層 (Context)               │
│  「AI に何を覚えさせ、何を見せるか」      │
│  ── chunk・RAG・claude-mem・手動ドキュメント化   │
├────────────────────────────────────────┤
│  ハーネス層 (Harness)                   │
│  「AI を動かす仕組み・自動化の枠組み」    │
│  ── 役割別ファイル・/advisor・hook・Skill │
└────────────────────────────────────────┘
```

### 3.2 各層の役割と本講座の章配置

| 層 | 役割 | 担当章 | 持ち帰る道具 |
|----|------|-------|------------|
| **プロンプト層** | 1 回の指示を資産化・診断する | ch01 / ch02 | `/command` / 診断チェックリスト 10 項目 |
| **コンテキスト層** | 長文を作らない仕組み + 必要なときに呼び戻す | ch03 / ch04 | LitM 認識 / chunk / RAG / claude-mem / 手動ドキュメント化 (handoff は session 切替用・別系統) |
| **ハーネス層** | AI を分業・第三者レビュー・自動化させる | ch05-08 | 役割別ファイル / `/advisor` / 4 種の hook / Skill |

### 3.3 章構成図 (3 層対応)


- `/command` の概念 (`.claude/commands/{name}.md`) を説明できる

#### 解説アウトライン

2. `/command` の作り方 — `.claude/commands/{name}.md` への配置
3. 引数付き呼出 — `/summarize-meeting @file.md` の動作実演
4. **Script / Slash Command / Skill 比較表スライド** (改訂版で必須追加・下記)

#### Script / Slash Command / Skill 比較表 (必須スライド)

| 観点 | Script (Bash 等) | Slash Command (`/name`) | Skill |
|------|------------------|------------------------|-------|
| 起動 | コマンドライン手動 | プロンプト中で AI が呼ぶ | **AI が自律判断で呼ぶ** |
| 配置 | 任意のフォルダ | `.claude/commands/*.md` | `.claude/skills/*.md` (or 公開 Skill) |
| 引数 | CLI 引数 | プロンプトの `@file` 等の参照 | AI がプロンプトから抽出 |
| 学習負荷 | 低 | 中 | やや高 (但し再利用最大) |
| 用途 | 手動繰り返し作業 | 定型プロンプトの資産化 | **業務領域知識の常駐化** |
| 本講座での位置づけ | 補助 | 中心 (ch01) | 到達点 (ch11) |

#### 持ち帰り
- 1 つの `/command` を作って動かせる
- 自分の業務で `/command` 化候補を 3 件挙げられる
- Skill が ch11 の到達点であると展望できる

### ch02. よくある失敗 3 パターンの診断 [プロンプト層]

**所要**: 30 min / 担当する困りごと: **① AI が推測で動く・確認せず突進する**

#### Emotional Hook (現象 → 対策)
- **現象**: 「AI が確認なしに勝手に進めて壊した」「曖昧な指示なのに『理解しました』と言われた」「症状はわかるが原因がわからない」

### ch03. Lost in the Middle と長文 context の根本対策 [コンテキスト層]

**所要**: 30 min / 担当する困りごと: **④ 長文の中央の参照情報が読み飛ばされる**

> ⚠️ **重要前提**: 本章の対策は **通常の命令 (200-500 token 程度) では起きない問題** を扱う。**長文 context (5,000 token+) のコードレビュー・仕様書・議事録・多文献参照** で発生する LitM (Lost in the Middle) 現象が対象。短いプロンプトでは無関係。

#### Emotional Hook (現象 → 対策)
- **現象**: 「5,000 行のコードレビューを依頼したら核心バグを見落とした」「30 ページの議事録を要約させたら中盤の決定事項が抜けた」「200 ページの仕様書から重要要件を引き出すよう頼んだら見落とされた」「複数文献を渡しても中盤の引用が無視される」
- **対策**: LitM は **配置技術 (TL;DR / 末尾サマリ) で『緩和』はできるが根本対策にはならない**。本章は **長文 context そのものを作らない・分割する仕組み化** の方向性を示し、ch04/ch05/ch07 で具体的な道具を扱う

#### 学習目標
- LitM が **長文 context (5,000 token+) でのみ発生** する現象であると説明できる (短いプロンプトでは無関係)
- 制約① 容量上限・制約② Lost in the Middle・制約③ Context Rot の 3 つを区別できる
- 「**LitM の真の対策は長文を作らない仕組み化**」という視点を持てる
- ch04 (chunk + RAG + 手動ドキュメント化) / ch05 (subagent 物理分割) / ch07 (hook 自動 grep) への橋渡しを理解する

#### 解説アウトライン
1. **長文 context が起きるユースケース 4 例**: コードレビュー (5,000 token+) / 仕様書 (200 ページ+) / 議事録 (30 ページ+) / 多文献参照 — **通常の業務命令では発生しない** ことを強調
2. **3 制約スライド①: 容量上限** — context window のサイズと、超えると何が起きるか
3. **3 制約スライド②: Lost in the Middle** — 「**参照情報** (添付ドキュメント・過去のログ等) が長く、その**中央に重要な事実**があると AI が読み飛ばしやすい現象」(Liu et al. 2023)。**「指示が中央にある」ではなく「参照情報が長く、その中央にある」のが正確な定義**
4. **3 制約スライド③: Context Rot** — 会話履歴が長く積み上がると応答の質が劣化する現象
5. **配置技術 (緩和策)**: TL;DR・末尾サマリ・重要結論を冒頭/末尾に置く — **しかし長文化が進むほど効果は薄れる・根本対策ではない**
6. **根本対策: 長文を作らない仕組み化 (本章末・橋渡し)**:
   - **ch04 へ**: 長文 → 物理 chunk + RAG (検索拡張生成) + claude-mem 永続記憶 + **手動でこまめなドキュメント化** で必要時のみ短く呼び戻す
   - **ch05 へ**: 長文 → **subagent (Task tool) で別文脈に物理分割** (主文脈に長文を持ち込まない)
   - **ch07 へ**: 長文 → **PreToolUse hook で関連 doc を自動 grep 注入** (人手で長文添付しない)

#### Lost in the Middle の正確な解釈 (殿指摘の反映)

> ❌ 誤: 「指示が長すぎると、指示の中央が読み飛ばされる」
> ✅ 正: 「**参照情報** (添付ドキュメント・過去のログ等) が長く、その**中央に重要な事実**があると AI が読み飛ばしやすい現象」
>
> ❌ 誤対策: 「TL;DR と末尾サマリで全部解決する」
> ✅ 正対策: 「配置技術は緩和策・**長文を作らない仕組み化** が根本対策。ch04/05/07 で扱う」

#### 持ち帰り
- LitM が長文 context (5,000 token+) でのみ発生することを説明できる
- 配置技術は緩和策に過ぎず、根本対策は ch04/05/07 の仕組み化であると理解する
- 自分の長文ユースケースを列挙し、どの章で対処するかを判別できる

#### 強化した層
> **コンテキスト層** を強化 — LitM 現象の正確な定義と、根本対策の仕組み化方向性 (ch04/05/07) を把握する。
