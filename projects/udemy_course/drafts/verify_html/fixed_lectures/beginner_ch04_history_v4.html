2020年 — 大規模言語モデルの幕開け

GPT-3 (1750億パラメータ)

  • OpenAIがGPT-3をリリース
  • 従来のAIとは次元が違う 文章生成能力
  • 「Few-shot learning（例示学習）」— 例を数個見せるだけで新しいタスクを実行
  • しかしAPI経由のみで、一般ユーザーには遠い存在

GPT-3は「AIが人間らしい文章を書ける」という事実を世界に示した

5年で「AIと話す」から「AIに開発させる」へ — そして「AIを制御・拡張する」へ

L1誕生 — プロンプトエンジニアリング

2022年: ChatGPTが新しい仕事の分野を生んだ

  • 2022年11月: ChatGPTが一般公開、2ヶ月で1億ユーザーに
  • AIとの対話が「誰でもできる」ことに — プロンプトが結果を左右する
  • 「どう指示を出すか」がスキルとして認識され、プロンプトエンジニアリング という仕事の分野が確立
  • Few-shot（例示学習）、Chain-of-Thought（思考の連鎖 — 推論ステップを段階的に出力させる手法）、システムプロンプト等の技術が体系化

プロンプトは新しいプログラミング言語 — 自然言語でAIを動かす時代の始まり

L2誕生① — プロンプトが足りない瞬間

プロンプトだけでは実務で破綻する3つの壁

  1. 同じ指示を毎回繰り返す疲弊 — プロジェクトのルールを毎チャットで説明し直す非効率
  2. プロジェクト規約をAIが守れない — 「うちはTabインデント」と伝えてもSpaceで出力される
  3. 複数ファイル横断の文脈が消える — 前回の会話内容をAIが忘れ、一貫性が崩れる

プロンプトは「その場の指示」には強いが、「継続的な指示」には弱い

L2誕生② — コンテキストファイルの革命

2023-2024年: AIにプロジェクトを教える指示書が生まれた

  • RAG (2023年) — Retrieval-Augmented Generation（検索拡張生成）。外部ドキュメントを検索してプロンプトに埋め込む。外部知識活用の先駆け
  • CLAUDE.md (2024年) — プロジェクトのルール・方針・制約を1ファイルに永続化。AIが新しい会話/セッション開始時に自動読み込み (4 階層を連結)
  • .cursorrules / AGENTS.md — ドット始まりは「隠しファイル」（設定用ファイル）の慣習。同じ概念がエディタ横断で共通化。「プロジェクトにAIを教える指示書」として定着
  • コンテキストエンジニアリング がL2として確立

「毎回説明する」から「ファイルに書いておく」へ — プロンプトの外に知識を置く転換

L3誕生① — AIが暴走する瞬間

エージェントの自律化が生んだ恐怖の事例

  • 危険コマンドの実行 — AIが rm -rf（フォルダ丸ごと削除）や git push --force（履歴を上書き破壊）を意図せず実行
  • 無限ループとコスト爆発 — エージェントが同じエラーを繰り返し返し、API利用料が膨張
  • 意図しないファイル削除 — 自律的なファイル操作で重要なファイルが消える事故が多発
  • 2023-2024年: AIが「自律する」ほど リスクが増大 するジレンマ

AIが「自律する」なら、人間は「制御」しなければならない — しかし制御の仕組みがない

L3誕生② — ハーネスという設計分野が現れた

2024-2025年: ハーネスエンジニアリングの誕生

  • Claude Code Hook — AI のライフサイクルイベントに任意の処理を差し込む仕組み。公式では 全 29 種類のイベント (SessionStart / UserPromptSubmit / PreToolUse / PostToolUse / Stop / PreCompact 等) が用意され、本書では代表的な数個を扱う
  • Permission モデル — 2 軸構造で AI の操作を制御。①権限モード (default / acceptEdits / plan / auto / dontAsk / bypassPermissions の 6 種) で全体動作・②ツール個別ルール (Allow / Ask / Deny の 3 種) で個別操作
  • サブエージェント・スキル・サンドボックス — モデルを「実際に仕事を完遂する work engine」に変える周辺コード・設定・実行ロジック一式
  • 業界では 「Agent = Model + Harness」 が公式定義として定着 (Anthropic Engineering Blog 2025-11「Effective harnesses for long-running agents」)

ハーネス = 安全装置だけではない。hooks/skills/sub-agents/コンテキストポリシー/回復経路 等を束ねた AI を実運用で信頼できる形にする設計分野

3階層統合 — 揃って初めて成り立つ

階層 役割 なぜ必要か
L1 プロンプト AIに意図を伝える ChatGPTで証明された基本インターフェース
L2 コンテキスト プロジェクト知識を永続化 プロンプトだけでは文脈が消える
L3 ハーネス hooks/skills/sub-agents 等で AI を制御・拡張する周辺基盤 (Anthropic 公式: Agent = Model + Harness) エージェントが自律するからこそ境界と回復経路が必要

L1だけ — 毎回ゼロから説明する非効率
L2だけ — 危険な操作を止め・回復させる仕組みがない
L3だけ — 「何をするべきか」の文脈がない

車に例えれば: L1 = ハンドル（方向を指示）、L2 = ナビゲーション（目的地までの知識）、L3 = シャシー・ブレーキ・サスペンション一式（車として走らせる土台と安全装置）。3つ揃って初めて安全に走れる。

3つ揃って「文脈を理解し、安全に、再現性よく」自律動作できる

この章のまとめ

  1. 2020年のGPT-3 から始まり、ChatGPT がL1プロンプトエンジニアリングを誕生させた
  2. プロンプトの限界から RAG → CLAUDE.md へ、L2コンテキストエンジニアリングが確立した
  3. エージェントの暴走事故と自律化の必要性から Hook・Permissions・サブエージェント が整備され、L3ハーネスエンジニアリングが誕生した (Anthropic 公式概念「Agent = Model + Harness」)
  4. 3階層 (L1/L2/L3) は生成AIの5年間の進化が導いた必然的な設計

プロンプト・CLAUDE.md・hook — 生成AIの5年間の歴史が結実した3つの柱

確認問題

Q1: 2022年のChatGPT登場により、どのような新しい職能が確立されたか？

Q2: プロンプトだけでは実務で破綻する「3つの壁」とは何か？

Q3: AIエージェントの自律化により多発した事故例を2つ以上挙げ、L3ハーネスがどのように防ぐかを説明せよ。

回答は次のスライドにあります

確認問題 — 回答

A1: プロンプトエンジニアリング。AIへの指示の出し方で結果が劇的に変わることがわかり、「AIにどう伝えるか」が職能として認識・体系化された。

A2:
1. 同じ指示を毎回繰り返す疲弊 — プロジェクトのルールを毎チャットで説明し直す
2. プロジェクト規約をAIが守れない — 伝えたルールが反映されない
3. 複数ファイル横断の文脈が消える — 前回の会話内容をAIが忘れる

A3: 例：AIが git push --force（履歴を上書き破壊）を意図せず実行してリモート履歴を破壊。エージェントがエラーで無限ループに陥りAPI利用料が膨張。L3ハーネスは PreToolUse hook（ツール使用前の割り込みチェック・公式 29 種類のイベントの 1 つ）で危険な操作を事前にブロックし、Permission の 2 軸構造（権限モード × Allow/Ask/Deny ルール）で「Deny」または「Ask」に設定することで事故を防止できる。
