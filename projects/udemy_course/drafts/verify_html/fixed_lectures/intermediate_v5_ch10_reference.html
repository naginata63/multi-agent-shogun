スラッシュコマンドの 4 階層 — 区別の前提

本章で頻出する /command 系の用語は 4 階層に分かれる。「公式」と「本リポジトリ独自」を混同しないこと

階層 例 出所
機構 (built-in mechanism) スラッシュコマンドという仕組み自体 Claude Code 本体機能
組込コマンド /clear /compact /init /memory /agents /model 等 公式コマンド表に掲載
バンドルスキル /review /security-review /simplify 等 Anthropic 公式同梱
独自カスタム /handoff /advisor /senkyou 等 本リポジトリ独自実装 (各自で作成)

作成方式: Skills (.claude/skills/<name>/SKILL.md) が推奨・legacy .claude/commands/<name>.md 単一ファイルも併存。公式は「カスタムコマンドはスキルにマージ」と明記

3段早見表 (1/3) — プロンプト層の困りごと

症状 (現場で起きること) 現象 (AIの中で何が起きているか) 対策 (使う道具) 章

3段早見表 (2/3) — コンテキスト層の困りごと

症状 現象 対策 章
長文の中央にある重要情報が無視された 長文の中央が読み飛ばされる現象 (Lost in the Middle: 長い文章の真ん中にある情報が抜け落ちる現象) TL;DR + 重要情報を冒頭/末尾に配置 ch03
先週の修正内容をAIが知らない 過去の記憶が引き継がれていない 自動メモリ (公式 MEMORY.md + トピックファイル・~/.claude/projects/<project>/memory/ に格納) + 引き継ぎファイル ch04
関連資料を確認せず回答した 必要な資料が自動的に検索されていない 必要時取得 (必要な時だけ読み込む) + 自動取得 hook ch07 / ch04

共通パターン: AIに「知らせるべき情報」が届いていない、または忘れている

症状 現象 対策 章

共通パターン: AI の動きを監視・制御する「仕組み」(= ハーネス層) が足りない

3層インデックス — 「どの層が足りないか」から引く

症状から引くだけでも便利ですが、「どの層が欠けているか」 を特定できれば、もっと早く解決できます。

11の困りごとはすべて、3つの層のいずれかが足りない から起きています。

3層別分類 — 全11件を層で整理

不足している層 該当する困りごと 対策章
プロンプト層不足 ①既存を無視 ②手動繰返 ③推測で突進 ch01 / ch02
コンテキスト層不足 ④中央が読まれない ⑤完了を忘れる ch03 / ch04
ハーネス層不足 (Anthropic 公式概念) ⑥言ったのに違う ⑦盲信 ⑧仕様変更で壊れる ⑩失敗を隠す ⑪止まらない ch05 / ch06 / ch07 / ch08
複合 (要組み合わせ) ⑨資料未確認 (コンテキスト + ハーネスの両方) ch04 + ch07

使い方 — 現場で詰まったら

ステップ1: 症状を確認する
- 「AIが何かおかしい」と感じたら、具体的に何が起きているかを言葉にする

ステップ2: 3段早見表の「症状」列から該当行を探す
- 似た症状が複数あれば、「現象」列で見分ける

ステップ3: 「対策」列の道具を使う
- 該当章を見直せば、具体的な手順が載っている

所要時間: 慣れれば 30秒 で該当箇所にたどり着ける

現場の具体例で練習

3つのパターンを実際の業務で追ってみましょう。

パターン1: 営業担当のケース

症状: AIに「先週の商談メモを要約して」と頼んだら、AIが勝手に「案件はクローズ」と出力した

分析:
1. AIが確認せず推測で出力した = 困りごと③
2. どの層? → プロンプト層不足
3. 対策 → ch02 の診断10項目で「確認問いかけ」を追加

改善後のプロンプト例:

先週の商談メモを要約して。
ただし、不明な点は推測せず「不明」と明記すること。

パターン2: PMサポートのケース

症状: 仕様変更メールをAIに読ませたら、古い仕様の箇所を無視して回答した

分析:
1. 長文メールの中央にある変更箇所が読み飛ばされた = 困りごと④
2. どの層? → コンテキスト層不足
3. 対策 → ch03 の TL;DR 配置 + ch04 の必要時取得

改善策:
- メールの先頭に「変更点サマリ」を3行で書く (ch03)
- 過去の仕様変更履歴を自動メモリから検索できるようにする (ch04)

パターン3: データ分析担当のケース

症状: 毎週同じ前処理プロンプトをコピペしている

分析:
1. 同じ作業を毎回手動で繰り返している = 困りごと②
2. どの層? → プロンプト層不足
3. 対策 → ch01 の カスタムスラッシュコマンド (Skills) → ch11 の Skill 化深掘りへ

改善ステップ:
1. まずカスタムスラッシュコマンド (Skills 形式 .claude/skills/<name>/SKILL.md 推奨・legacy .claude/commands/<name>.md も可) でプロンプトを資産化 (ch01)
2. 慣れたらより高度な自動化へ (詳しくはch11で解説)

似た症状の見分け方

困りごとによっては症状が似ています。ここで間違えると「違う章を見直す」ことになります。
2つの見分け方を押さえましょう。

見分け方①: 「推測で突進」と「盲信して失敗」

③ 推測で突進 (AIが確認せず勝手に進める) ⑦ 盲信して失敗 (AI出力を信じ込む)
いつ起きる? AIが出力する前 AIの出力を見た後
誰が悪い? AI (指示不足) 人 (確認不足)
どの層? プロンプト層 ハーネス層
対策章 ch02 ch06
対策 プロンプトに確認指示を追加 advisor ツール (Anthropic 公式・beta) で第三者レビュー

判断のコツ: AIが出力する前に起きるなら③、出力を見た後に気づくなら⑦

見分け方②: 「忘れている」と「読み飛ばしている」

⑤ 完了を忘れる (過去の作業を覚えていない) ④ 中央が読まれない (長文の真ん中が抜け落ちる)
症状 過去にやったことをAIが知らない 長文の中ほどが無視される
原因 記憶が引き継がれていない 配置の問題 (Lost in the Middle)
どの層? コンテキスト層 コンテキスト層
対策章 ch04 (自動メモリ・引き継ぎ) ch03 (配置技術)

判断のコツ: 「前回やったこと」を覚えていない → ch04 / 長文の特定箇所が無視される → ch03

見分け方③: 「言ったのに違う」と「止まらない」

⑥ 言ったのに違う (指示を部分的に無視) ⑪ 止まらない (止めても実行が続く)
症状 AIが指示を部分的に無視する 「止めて」と言っても実行が続く
原因 役割・指示が1人に詰め込まれている 実行を止める仕組みがない
どの層? ハーネス層 (役割分業) ハーネス層 (ガード機構)
対策章 ch05 (役割別ファイル) ch08 (禁止操作リスト = PreToolUse hook で実装)

判断のコツ: AIが部分的に間違える → ch05 / AIが止まらない → ch08

最初に試すべき道具 5選

困りごとが起きたとき、どれか1つでも試すと効果的な道具を5つ厳選しました。
まずはこの5つから試してみてください。

順位 道具 公式/独自 なぜ効くのか 詳細は
1 カスタムスラッシュコマンド (Skills) 機構は公式・コマンド本体は自分で作る 同じ作業の繰り返しを即座に終わらせる ch01
2 診断チェックリスト 本講座独自の運用ガイド プロンプトの「推測で突進」を防ぐ ch02

4 advisor ツール (advisor_20260301) Anthropic 公式 (beta) 別のAIに見直してもらうことで盲信を防ぐ ※ /advisor 名スラッシュコマンドは本リポジトリ独自 ch06
