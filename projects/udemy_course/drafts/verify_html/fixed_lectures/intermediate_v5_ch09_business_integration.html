ch01〜ch08 は 道具の章、本章は 使い方の章。3 層を統合して業務に組み込む型を提示します。

この章で学ぶこと

  3. 「どの層から始めるか」の優先順位判断ができる

本章の略語: RAG = Retrieval-Augmented Generation (検索拡張生成) / MCP = Model Context Protocol (LLM と外部ツール・データソース・API を繋ぐオープンソース標準プロトコル・記憶拡張はその一用途例)

道具は覚えた。でも自分の現場ではどう使う?

ch01〜ch08で3層の道具を学んだ。しかし、自分の現場に戻ったら何から始めればいいか分からない — このギャップを埋めるのが本章です

  • 道具は9種類以上 (カスタムコマンド / 診断 / chunk / RAG (Retrieval-Augmented Generation) / memory / 役割別ファイル / advisor / hook / 禁止操作...)
  • でも「全部一度に導入」は現実的ではない
  • 自分の困りごとに合う層から始めるのが正解

3層の復習 — 何を強化したいかで選ぶ

層 強化する内容 代表的な道具 こんな困りごとに
プロンプト層 1回の指示の質 カスタムスラッシュコマンド / 診断チェックリスト 同じプロンプトを毎回書いている
コンテキスト層 AIに覚えさせる知識 chunk / RAG / memory / 検索 過去の知識をAIが知らない
ハーネス層 自動化と安全装置 hook / advisor / 禁止操作リスト AIに任せきりで品質が見えない

自分の困りごとが「どの列」に当てはまるか — それが「どの層から始めるか」のヒント
※ ハーネス層は Anthropic 公式概念 (Engineering Blog「Effective harnesses for long-running agents」2025-11)。Agent = Model + Harness の Harness 層に hook / 安全装置 / advisor を実装します。

ケース① — 営業の3層組み合わせ

現場の声: 「顧客への提案メールを毎回ゼロから書いている。前回の要望も調べ直している」

プロンプト層 — カスタムスラッシュコマンドで定型化 (ch01 で作成)

/propose-mail @顧客A議事録.md
  ※「@ファイル名」はAIにそのファイルを読み込ませる指定です
  ※ カスタムスラッシュコマンドは Skills (`.claude/skills/<name>/SKILL.md`) で定義
   (legacy `.claude/commands/<name>.md` 単一ファイル形式も併存)
→ 件名・トーン・締めの一言が毎回ゼロからではなく、
   テンプレートベースで生成される

コンテキスト層 — memory検索で過去を呼び戻す
- 前回の顧客要望・クレーム履歴を意味検索 (ch04 で紹介)
- 例: 外部 MCP サーバー claude-mem で「この顧客について検索」と自然文入力するイメージ (MCP の自然文呼出は公式仕様・claude-mem 固有 UI 詳細は別途公式 README で確認)
- 「この顧客は前回〇〇に不満だった」を自動参照

ハーネス層 — advisor で念押し
- 提案内容に「前回の要望との矛盾」がないか第三者レビュー (Anthropic 公式 advisor_20260301 ツール・ch06 で詳述・beta 機能)

ケース② — PMサポートの3層組み合わせ

現場の声: 「タスクの粒度がバラバラで、AIに任せると抜けが出る。進捗も見えない」

プロンプト層 — タスクを構造化して書く
- ch02の診断10項目を適用: タスクの前提・出力形式・完了条件を明記

コンテキスト層 — /handoff でセッション跨ぎの引き継ぎ
- 前回の進捗・決定事項を次セッションに引き継ぎ
- 「先週どこまで進んだか」をゼロから説明しなおす必要がなくなる
- ⚠️ /handoff は 本リポジトリ独自のカスタムスラッシュコマンド (本講座の Skills として定義) であり、Claude Code 標準のスラッシュコマンドではない。自分の現場で類似機能を使うには自前で Skill (.claude/skills/handoff/SKILL.md) を作成する

ハーネス層 — advisor + 自動チェック
- WBS（プロジェクトのタスク分解表）の妥当性を advisor (Anthropic 公式 tool) にレビュー
- ドキュメントやスプレッドシート更新後に自動で内容チェック (ch07 で学んだ PostToolUse hook — 公式 hook 29 種類の 1 つで「ツール実行が成功した直後」に発火)

ケース③ — データ分析の3層組み合わせ

現場の声: 「分析のたびに前提条件を説明している。前処理でミスに気づかない」

プロンプト層 — 分析要件をタグで構造化（ch02で学んだ「確認指示」の応用です。<目的>のように山括弧で囲むと、AIが「ここは目的・ここはデータ」と構造を理解しやすくなります）

<目的>売上の前年比較</目的>
<データ>sales_2025.csv</データ>
<出力>グラフ付きレポート</出力>
  • 目的・データソース・出力形式をタグで明記 → AIが「何を分析すべきか」を勘違いしない
    （このように< >で囲む書き方を「XMLタグ」と呼びます。HTMLを知らなくても、AIに渡す指示を構造化するだけならこの書き方だけで十分です）

コンテキスト層 — memoryで過去の分析結果を記憶
- 「前回は〇〇のデータ品質に問題があった」等の注意事項を記憶
- 同じミスを繰り返さない

ハーネス層 — AIに役割を分けて分業 + 段階確認（ch05で学んだ役割別ファイル + ch08の段階確認）
- AIに「データ取得担当」「分析担当」「レポート担当」の役割を割り当てて分業
- 前処理後のデータ件数を段階確認 (チェックポイント) で検証 (ch08 解決策①)

「どの層から始めるか」の判断基準

自分の困りごとに合う層を最初の一手にする — 全部一度に導入しなくていい

困りごとの特徴 最初の一手 具体的にやること
「毎回同じプロンプトを書いている」 プロンプト層 カスタムスラッシュコマンド (Skills) を 1 つ作る
「過去の知識をAIが知らない」 コンテキスト層 memoryに1件書く / 意味検索を試す
「AIに任せきりで品質が見えない」 ハーネス層 advisor を 1 回呼ぶ / hook を 1 つ入れる

最初は1層・1つでいい。効果を確認してから次の層に進む

3ケースの比較 — 共通パターンを見抜く

観点 営業 PMサポート データ分析
最初の一手 カスタムコマンドでメール定型化 タスク構造化 分析要件の構造化
検索の活用 顧客対応履歴の検索 進捗の引き継ぎ 過去分析結果の記憶
安全装置 advisor 念押し 自動チェック (PostToolUse hook) 段階確認
共通点 プロンプト定型化 + 知識の検索 + 品質の担保

3ケースに共通する「定型化 → 検索 → 担保」の3ステップが、業務への適用パターン

この章のまとめ

  • 3ケース (営業・PM・データ分析) で3層の組み合わせ方を具体的に見た
  • 「どの層から始めるか」は自分の困りごとに合う層を最初の一手にする
  • 共通パターン: 定型化 → 検索 → 担保 の3ステップ
  • 最初は1層・1つでいい — 効果を確認してから次に進む
  • 自分の業務に当てはめた構想図を1枚書き出すことが、次章への準備
  • ⚠️ 本章で扱った /handoff は 本リポジトリ独自のカスタムスラッシュコマンド であり、Claude Code 標準には含まれない (自前 Skill 化が必要)。一方 advisor ツール / PostToolUse hook / MCP は 公式機能

強化した層: 3層を統合 — 業務に組み込む型を持てる
※ ハーネス層 = Anthropic 公式概念 (Agent = Model + Harness)

確認問題

Q1: 「毎回同じプロンプトを書いている」という困りごとがある場合、どの層から始めるのが適切ですか?

Q2: 営業ケースで「前回の顧客クレーム」をAIに自動参照させるには、どの層の技術を使いますか?

Q3: 3ケースに共通する「定型化 → 検索 → 担保」のうち、「担保」にあたる代表的な道具を2つ挙げてください。

A1: プロンプト層です。まずカスタムスラッシュコマンド (Skills 形式 .claude/skills/<name>/SKILL.md 推奨) で繰り返し使うプロンプトを1つ資産化します。「毎回同じものを書いている」= プロンプト層の課題と判断できます。

A2: コンテキスト層の技術 (memory検索 / RAG / MCP サーバー) を使います。過去の顧客対応履歴をmemoryに蓄積しておき、プロンプト送信時に意味検索で関連履歴を自動参照させます。外部 MCP サーバー (例: claude-mem) を組み合わせる構成が代表例ですが、claude-mem 固有の UI/コマンド体系は claude-mem 公式 README で別途確認してください。

A3: advisor (Anthropic 公式 advisor_20260301 ツール・第三者AIレビュー) と 段階確認 (チェックポイント・ch08 解決策①) の2つです。いずれもハーネス層 (Anthropic 公式概念「Agent = Model + Harness」の Harness 層) に属する「品質を担保する」仕組みです。
