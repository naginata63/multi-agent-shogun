今日のテーマ: 別 AI の判断を context window を膨らませずに取り込む = ハーネス層 (L3) で品質を支援する

この章で解決する「困りごと」

⑦ AI 出力を盲信して失敗した

  • AI が「完璧です。自信を持って提出できます」と言った
  • そのまま提出したら、数字が間違っていて大恥をかいた
  • 自分の AI に「これで合ってる?」と聞いても「はい合ってます」と返ってくるだけ

章のゴール

言える 1 文:

別の AI に見直してもらうには、context window を節約する方法を選ぶことが大切

できる 1 動作:

3 つの方法 (サブエージェント / advisor / 別チャット) を context 流入量の観点で比較し、状況に合った方法を選べる

「盲信」で失敗した 3 つの実例

例① 数字の間違い

  • AI に売上レポートを作成させた
  • 「前年比 15% 増」という結論をそのまま報告
  • 実は計算元のデータに欠損値が含まれていて、本当は 3% 増 だった

例② 存在しない情報の引用

  • AI が「〇〇省のガイドラインによれば…」と引用
  • 提出先から「そのガイドライン、存在しませんよ」と指摘

例③ 一見正しそうなコード

  • AI に書かせたコードがテストは通る
  • でも本番で特定の条件下だけバグが発動 → 大騒ぎに

なぜ「自分の AI に確認」は意味がないか

┌─────────────────┐     「これで合ってる?」     ┌─────────────────┐
│  あなたの AI     │  ──────────────────────→   │  同じ AI が      │
│  (A という偏り)  │                             │  「はい合ってます」│
└─────────────────┘                             └─────────────────┘
        ↑                                                │
        └──── 同じ偏りで再確認しているだけ ──────────────┘
  • 同じ AI に確認 = 同じ偏りで再確認するだけ
  • 自己肯定しかしない → 客観的な品質チェックにならない
  • 別の視点 からのレビューが必要

ch03 / ch04 / ch05 の復習 — context window を節約する原則

  • ch03: context window には上限がある → 大量テキストを詰め込むと重要情報が押し出される (LitM)
  • ch04: 長文を作らない → 必要な情報だけを必要なときに読み込む (方向性① 分割+検索)
  • ch05: 長文は別 AI に分担 → メイン context に持ち込まない (方向性② 物理分割)

この「context window を節約する」原則は、
別の AI に見てもらう方法の選び方にも適用できる

別の AI に見てもらう — 3 つの実装

方法 概要 ツール依存
① サブエージェント (ch05 既習) 別の AI セッションを立ち上げて全文を渡す Claude Code 公式 (.claude/agents/)
② advisor API に登録すると別 AI が自動レビュー Anthropic Claude API beta tool
③ 別チャット貼付 手動で別チャットに成果物を貼り付ける なし (ツール非依存)

どれも「別の AI に見てもらう」という目的は同じ
→ 何が違うのか? 次のスライドで比較

3 手法の比較 — 「context 流入量」で選ぶ

方法 context 流入量 特徴
サブエージェント 全文 (親 context が肥大) 確実だが重い・ch04「長文を作らない」原則に違反しうる
advisor 指摘のみ (400〜700 tok) 軽量・ch04 の原則と整合
別チャット貼付 なし (手動) ツール非依存・貼り忘れやミスの余地あり

ch04 の原則「長文を作らない」と整合するのは advisor

サブエージェント:  全文が context に戻る → 肥大リスク
advisor:           指摘だけが context に戻る → 節約
別チャット貼付:    手動で貼る → 確実性に課題

advisor の正確な仕様 (Anthropic 公式ドキュメント準拠)

  • Anthropic Claude API 公式 beta tool (type: advisor_20260301)
  • beta header: advisor-tool-2026-03-01 が必要
  • server-side tool — AI が server_tool_use block を emit して呼び出す
  • 引数なし — 会話履歴は server が自動転送
  • 手元で動かすには: Claude API + beta header + tools 配列登録 が必要
// tools 配列に追加するだけ
{
  "type": "advisor_20260301",
  "name": "advisor"
}

⚠️ beta 機能のため identifier が将来変わる可能性あり — 公式ドキュメントの最新版を都度確認してください
注意: チャット欄に打ち込むものではありません — API 経由で tools に登録することで有効化される仕組みです

advisor の仕組み — 指摘だけが context に戻る (ハイブリッド構造)

┌───────────────┐    会話全文    ┌───────────────┐
│  メイン AI     │ ──────────→ │  別の AI       │
│  (executor:    │              │  (server-side: │
│   client側)    │              │   Anthropic側) │
└───────────────┘              └───────┬───────┘
        ↑                              │
        └── 指摘だけ (400〜700 tok) ←─┘
            (全文は戻らない = context 節約)
  1. AI が作業を終えたタイミングで「advisor を呼ぶべきか」を自律判断
  2. 呼ぶと決めたら、会話内容が別の AI に自動転送される
  3. 別の AI が指摘事項を返す → 指摘だけが context に戻る

ハイブリッド構造: executor は client (Claude Code) 側で動き、advisor 部分のみ server-side で実行。client は server_tool_use block を受け取って結果を context に統合します

呼び出しタイミング 3 パターン

タイミング 目的 例
着手前 計画の妥当性レビュー 「この方針で進める前に見て」
着手後 成果物の品質レビュー 「出来上がったものを確認して」
詰まったとき 方針の見直しレビュー 「別のアプローチはないか」

一番使うのは 着手後 — できたものを第三者に見てもらう

完了条件を明確にする (acceptance_criteria)

advisor に「何を基準に見てほしいか」を伝える技術

Before (基準なし):

「このレポート、見てください」

After (基準あり):

「このレポートを以下の基準で見てください:
- データの鮮度 (最新データを使っているか)
- 数字の正確性 (計算に間違いがないか)
- 結論と根拠の整合性 (飛躍がないか)」

基準を渡すと、レビューの精度が上がる

任意発動 vs 強制発動 — advisor と hook の使い分け

advisor hook (ch07 で解説)
発動 AI が自律判断 (skip あり) 強制発動 (skip なし)
確実性 確実ではない 確実
向いている場面 軽いレビュー・方針確認 絶対に通さないといけない検査

「advisor は AI が skip したら結局確認されないじゃないか」
→ その通り。だからこそ ch07 で hook (強制発動) を学ぶ

使い分けの指針:
- 確実性が要る品質ゲート → hook (ch07)
- あればよい軽いレビュー → advisor

※ hook は AI のライフサイクルイベント (セッション開始 / プロンプト送信 / ツール実行前後 / 圧縮前 / 終了等) に任意の処理を差し込む Claude Code 公式機能 (全 29 種類)。ch07 で代表的な hook を抜粋して解説します

実例 — 営業の提案書レビュー

現象

  • AI に提案書を作成させた
  • 競合製品のスペック比較表の数字が 2 年前のもの だった
  • 提出先から「このデータ、古くないですか」と指摘

advisor で解決

> 提案書が完成しました。advisor で見てください。
> 基準: データの鮮度、数字の正確性

→ 別の AI が「競合スペック表の数字が、調べた範囲と一致しません」と指摘
→ 提出前に修正できた

実例 — PM の仕様書レビュー

現象

  • AI に仕様書ドラフトを書かせて、そのままエンジニアに共有
  • 後で「非機能要件が一切書かれていない」と指摘 → 手戻り

advisor で解決

> 仕様書ドラフトができました。advisor で見てください。
> 基準: 非機能要件の記載 (セキュリティ/性能/可用性)

→ 別の AI が「可用性要件 (SLA) の記載がありません」と指摘
→ エンジニアに渡す前に修正

ここまでの振り返り

  • AI の出力を盲信すると、数字の間違い・存在しない引用・隠れバグに気づかない
  • 「自分の AI に確認」は同じ偏りだから意味がない
  • 別の AI に見てもらうには 3 つの方法がある (サブエージェント / advisor / 別チャット)
  • context 流入量 で選ぶ → advisor は指摘だけが戻る (ch04「長文を作らない」原則と整合)
  • advisor は AI が skip することもある — 確実な品質ゲートではない
