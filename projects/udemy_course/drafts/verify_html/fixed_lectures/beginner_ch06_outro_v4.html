3階層で振り返る初級編

前章で L3 ハーネスを実装した今、3 階層全体を一枚絵で見直してみます

この講座は 3つの階層 でAI開発の基礎を構築しました:

階層 名称 対応章 役割
L1 プロンプト 第1章 AIに指示を確実に伝える
L2 コンテキスト 第2〜4章 プロジェクト知識をAIに共有する
L3 ハーネス 第5章 hooks(イベント割込)/skills(オンデマンド業務道具)/sub-agents(専門役割を担う子エージェント) 等でAIの動作を制御・拡張する周辺基盤

Anthropic 公式の業界標準定義: Agent = Model + Harness (Engineering Blog 2025-11「Effective harnesses for long-running agents」)
この 3 層が揃って、AI は「理解し・文脈を持ち・安全に動く」開発パートナーになる

到達点1: プロンプト [L1] — AIに確実に伝える

何を学んだか:
- 構造化プロンプトの書き方（入力・処理・出力の3要素）
- Before/Afterで確認した「伝わる指示」と「伝わらない指示」の差

何が解決できたか:
- AIへの指示が 毎回ブレる問題 を解決
- 同じプロンプトで 再現性のある結果 を得られる

プロンプト = AIとのコミュニケーションの「基礎言語」

到達点2: コンテキスト [L2] — プロジェクト知識の共有

何を学んだか:
- CLAUDE.md（第2章）— AIに読ませるプロジェクトの「説明書」
- .md 仲間たち（第3章）— CLAUDE.md と Skills (Commands は Skills の旧形式・公式マージ済み) の二系統と、CLAUDE.md 内の @path 取り込み構文
- 歴史と意義（第4章）— なぜ3階層が必要なのか

何が解決できたか:
- AIに毎回前提条件を説明する手間が ゼロに
- 設定ファイル1つでAIの振る舞いが 自動で変わる

コンテキスト = AIに「プロジェクトの文脈」を永続的に与える仕組み

到達点3: ハーネス [L3] — AIを制御・拡張する周辺基盤

何を学んだか:
- hookの仕組み — AIの ライフサイクル(起動 → 指示受付 → ツール実行 → 終了までの一連の動作) 上のイベント (公式 29 種類) に任意の処理を発火させる仕組み
- PreToolUse Hook でコマンド実行前に検査・ブロック
- 前章復習: INPUT=$(cat) で stdin (標準入力) から JSON を受取り、危険検知時は exit 2 + stderr (標準エラー出力) にブロック理由を出力すれば、AI が理由を理解して別案を出す

何が解決できたか:
- rm -rf や sudo 等の 危険操作を自動検知
- AIが勝手にファイルを消すリスクを 排除

ハーネス = AI モデルを「実際に仕事を完遂する work engine」に変える周辺基盤
安全装置はその中の代表的な一用途

初級編で解決した3つの課題

課題 解決策 階層 学んだ章
AIへの指示が毎回ブレる 構造化プロンプト L1 第1章
AIに毎回ルールを説明する手間 CLAUDE.md + Skills L2 第2〜3章
AIが危険な操作をする不安 PreToolUse Hook L3 第5章

この3つが揃えば、Claude Code は「安心して使える開発パートナー」になる

中級編のプレビュー

中級編では、今日学んだ3階層を 深掘り し、新しい技術と運用を加えていきます

項目 初級編 (今) 中級編 (次)
所要時間 約 3 時間 / 7 章 約 13 時間 / 12 章（1日30分×4週間程度）
L1 プロンプト 基礎の構造化 高度なテンプレート化
L2 コンテキスト 30行ミニマム 永続化・マルチプロジェクト
L3 ハーネス 1本の入門 PreToolUse Hook 18本の実装例で主要カテゴリ (ファイル保護・コマンド検査・通知など) を学ぶ
MCP 今回未対応 自前構築 + 永続化バックエンド

MCP（Model Context Protocol）= LLM と外部ツール・データソース・API を繋ぐ オープンソース標準プロトコル。Claude Code を数百の外部ツールに接続するための公式仕様。claude-mem のような「記憶を拡張する MCP サーバー」はその一用途例にすぎず、課題追跡・監視データ分析・DB クエリ・デザイン統合など幅広い領域で使われている
