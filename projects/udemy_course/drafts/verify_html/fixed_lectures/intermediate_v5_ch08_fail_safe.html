前章 (ch07) からの繋がり

ch07 で学んだこと:
- hook = AI のライフサイクルイベント (公式 29 種類) に処理を差し込む仕組み
- 代表 4 つ (SessionStart / PreToolUse / PostToolUse / UserPromptSubmit) を習得
- hook 引数は stdin JSON 一択 ($1・環境変数は公式仕様に無い)
- done_gate.sh で advisor (任意発動) を hook で間接強制 — Anthropic 公式概念の「ハーネス」(Agent = Model + Harness) の典型的な実装パターン

ch08 で扱う 2 つの不足を埋める:

hook で自動化した → でも「黙って失敗」は止められない    → 段階確認 / watcher / dry-run
hook で発火させる → でも「何を止めるか」が定義されていない → 禁止操作リスト + Permissions モデル

本章は ch07 hook の 「中身」と「止め方」 を整える章。最後に「実行時 BLOCK プロトコル」の公式仕様も押さえます。

この章で学ぶこと

  1. 「黙って失敗」がなぜ起きるのか、その 3 パターンを説明できる
  2. 段階確認で各工程の終わりにチェックポイントを置ける
  3. 検知の仕組み (watcher) で異常を自動発見できる
  4. 禁止操作リスト + 公式 Permissions モデル で「絶対にやってはいけないこと」を強制できる
  5. PreToolUse BLOCK 制御プロトコル (公式仕様) でコピペ可能な阻止ロジックを書ける

「完了しました」— 本当に完了していますか?

AI がコードを書く際、エラーが起きても無視する書き方（Python では try/except と呼ばれます）があります。エラーを捕捉して握り潰し、「完了」と報告 — 実際にはデータが壊れていた、という経験はありませんか?

  • エラー時に代替値を返して誤魔化す → 何が起きたか痕跡が残らない
  • 後日、同じエラーが起きても原因調査が不可能
  • 「そのコマンド使うな」と指示したのに、AI が普通に実行して本番を壊した

「黙って失敗」の 3 パターン

パターン 具体例 何が問題か
① 例外の握り潰し エラーが起きても無視する構文を書く エラーが完全に消える
② 代替値による隠蔽 エラー時にダミーデータを返す 失敗が成功に見せかけられる
③ 条件分岐の漏れ 想定外の入力で無言の不正動作 誰も気づかない

どのパターンも「失敗が見えない」という共通点があります。見えないものは直しようがありません。

解決策① — 段階確認で関所を置く

各工程の終わりにチェックポイントを置き、異常を検知したら即座に停止・報告する

データ取得 → [検証: 件数 > 0 ?]
                    ↓ NG → 即STOP + 報告
                    ↓ OK
データ前処理 → [検証: フォーマット正常 ?]
                    ↓ NG → 即STOP + 報告
                    ↓ OK
レポート出力 → 完了
  • 各工程 = 1 つの責任 に分割
  • 関所 = 「この条件を満たさなければ次に進めない」という強制停止点
  • 関所で止まれば、原因は直前の工程に絞り込める

段階確認の具体例 — 週次レポート生成

困りごと: AI に週次レポートを生成させたら、過去データの取得に失敗したのに代替値を入れて「完了」と報告。気づかずに上司に提出してしまった

Before (関所なし):

データ取得 → レポート生成 → 「完了しました」
                ↑ 失敗を代替値で隠蔽

After (関所あり):

データ取得 → [Gate: 取得件数 > 0] → レポート生成 → [Gate: 前週差分が妥当] → 出力
  • 関所 1: データが 0 件なら即 STOP — 「取得失敗」と報告
  • 関所 2: 前週比が異常なら確認待ち — 「データ異常の可能性」と報告

解決策② — 検知の仕組み (watcher)

段階確認は「その場で検知」する仕組み。watcher は「後から異常を見つける」仕組み

  • cron (スケジュール実行) や hook で定期的に監視 — 段階確認が届かないケース (全体クラッシュ・外部サービスダウン等) も捕捉
  • Claude Code では hook と組み合わせて、バッチ処理後に自動で結果をチェックさせることができます
  • 異常パターンを自動検出:
  • 終了コードが 0 以外（プログラムが正常終了すると 0、異常終了すると別の番号を返す仕組みです）
  • 出力ファイルが存在しない
  • 処理件数が前回と大きく乖離 (例: 前日比 50% 以上の変動)
  • ログにエラーメッセージが含まれる
毎朝6:00 バッチ実行
  → watcher が「前日比-70%」を検知
  → 即座に通知: "⚠ 件数異常: 昨日1000件 → 今日300件"

解決策③ — --dry-run で安全に試す

本番実行の前に、結果だけを確認する — 2 段階で確実性を担保する技術

フラグ 動作 用途
--dry-run 実行のプレビューのみ表示 (実際の変更なし) 「何が起きるか」を事前確認
--no-confirm 確認なしで即座実行 信頼済みの定型処理向け

運用フロー:

  1. まず --dry-run で結果を確認
  2. 想定通りなら本番実行
  3. 想定と違えば原因を調査
$ python import_data.py --dry-run
→ "取り込む予定: 1,247件 / フォーマット: CSV / 先頭: 2026-01-15"
# ← AIに「--dry-run モードを作って」と頼めば、この機能を実装してくれます
# ← これを見てOKなら本番実行

解決策④ — 禁止操作リストで「止める」

指示で「使うな」と書いても、AI は実行時に忘れることがあります。「仕組み」で止める

禁止操作リスト = 「絶対に実行してはいけない操作」をリスト化して明文化したもの

ID 禁止操作 理由
D001 rm -rf / OS 全体を破壊
D002 本番データベースへの直接変更 データ消失の危険
D003 --force 付きのリモートへの反映 他の人の作業を上書き
D004 確認なしの全ファイル削除 取り返しがつかない
D005 管理者権限での実行 権限の不当な昇格

⚠️ 重要: 「禁止操作リスト (D001-D005)」という名前と ID 体系は 本リポジトリ独自の設計パターン名であり、Claude Code 標準機能ではありません。実装の仕組み自体 (PreToolUse hook で実行直前に検査して BLOCK する) は公式機能です。

禁止操作リストの具体例 — 本番デプロイ

困りごと: 「本番 DB の DROP TABLE は禁止」と指示したのに、AI が開発環境だと思って本番で実行

Before (指示のみ):

CLAUDE.md に「本番DBを触るな」と書いた
→ AI が文脈を勘違い → 本番で DROP TABLE を実行

After (仕組みで阻止):

PreToolUse hook が SQL文を検査
→ "DROP" / "TRUNCATE" / 本番ホスト名 を含む SQL を即ブロック
→ 「禁止操作が検出されました: D002 本番DBへの直接変更」
  • 指示は AI が忘れることがある
  • 仕組みは忘れない — 実行の瞬間に強制阻止

公式 Permissions モデルとの併用 (2 軸構造)

禁止操作リスト (本講座独自) と並んで、Claude Code には公式の Permissions モデルがあります。両者は併用する設計です。

Permissions は 2 軸構造:

軸 種類 例
軸 1: 権限モード (6 種) default / acceptEdits / plan / auto / dontAsk / bypassPermissions セッション全体の挙動を制御
軸 2: ツール個別ルール (3 種) Allow / Ask / Deny 個別ツール・パターンに対する判定
  • 評価順: deny → ask → allow の順で判定
  • 拡張: PreToolUse hook で実行時拡張 も可能 (= 本章の禁止操作リストの位置付け)
[公式 Permissions]      [本講座独自の禁止操作リスト]
  Deny ルール             D001-D005 を PreToolUse hook で検査
  (静的・設定ファイル)     (動的・実行時のパターンマッチ)
        ↓                          ↓
       両者を組み合わせて「絶対にやらせない」を実現

ch07 との連動 — hook が禁止操作の「実行時強制力」を担保する

前章で学んだ PreToolUse hook（AI がツールを実行する直前に発動する仕組み）が、禁止操作リストの実行時の強制力を担う

┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│ AIがコマンド構築  │ ──→ │ PreToolUse hook  │ ──→ │ 禁止操作リストと  │
│ "DROP TABLE ..."  │     │ がコマンドを検査  │     │ 照合 → 一致!     │
└──────────────────┘     └──────────────────┘     └────────┬─────────┘
                                                           │
                                                           ▼
                                                  ┌──────────────────┐
                                                  │ 実行を BLOCK     │
                                                  │ 「禁止操作 D002」 │
                                                  └──────────────────┘
  • ch07: hook で「いつ検査するか」を制御 (発火タイミング)
  • ch08: 禁止操作リストで「何を禁止するか」を定義 (検査の中身)
  • この 2 つの組み合わせで、失敗を見逃さない仕組みが完成

PreToolUse BLOCK 制御プロトコル (★ 公式仕様)

PreToolUse hook で実行を阻止するには、公式仕様で 2 通りの書き方が決まっています。コピペで動かないと意味がないので、両方を覚えてください。

方式 A: exit code 2 + stderr

#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

if echo "$COMMAND" | grep -qE 'DROP TABLE|TRUNCATE'; then
  echo "BLOCK: D002 本番DBへの直接変更" >&2
  exit 2          # ← exit code 2 で阻止
fi
exit 0

方式 B: stdout に JSON

cat <<'EOF'
{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"D002 本番DBへの直接変更"}}
EOF
  • 方式 A: 簡便。stderr のメッセージが AI へのフィードバックに使われる
  • 方式 B: 構造化。permissionDecision: "deny" で明示的に拒否

4 つの解決策 — まとめ

解決策 何を防ぐか 仕組み
段階確認 工程途中の異常を見逃す 各工程の終わりに関所を置く
watcher 事後に気づく遅延 定期的な異常パターンの自動検知
--dry-run 確認なしの本番実行 事前プレビュー → OK なら本番
禁止操作リスト + Permissions AI が禁止操作を実行 PreToolUse hook + 禁止パターン定義 / 公式 Allow-Ask-Deny

この 4 つを組み合わせることで、「黙って失敗」も「止まらない暴走」も防げる

確認問題

Q1: AI がエラーを無視する書き方をして「完了しました」と報告しました。実際には処理が失敗しています。これは「黙って失敗」のどのパターンですか?

Q2: 段階確認 (チェックポイント) を設計するとき、各関所では何を確認すべきですか?

Q3: 禁止操作リストを「指示」ではなく「仕組み」として機能させるには、前章のどの技術と組み合わせる必要がありますか? また、PreToolUse hook で実際に BLOCK するには公式仕様でどう書きますか?

A1: パターン①「例外の握り潰し」です。エラーが起きても無視する構文で処理され、何も記録されず、結果として失敗が完全に隠蔽されます。

A2: 「その工程の出力が、次の工程の入力として妥当か」を確認します。例: データ取得後なら「件数が 0 より多いか」、前処理後なら「フォーマットが正常か」など、工程ごとの妥当性条件を定義します。

A3: 前章で学んだ PreToolUse hook と組み合わせます。hook が AI の実行直前にコマンドを検査し、禁止操作リストと照合して、一致すれば実行をブロックします。BLOCK の公式仕様は 2 通り: ① exit code 2 で stderr に理由を出力する方式、② stdout に {"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"..."}} JSON を返す方式。引数受取は stdin JSON (INPUT=$(cat); jq -r ...) 一択で、$1 や $CLAUDE_TOOL_INPUT は公式仕様に存在しません。指示は忘れることがありますが、仕組みは忘れません。
