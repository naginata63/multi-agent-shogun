CLAUDE.mdとは — AIに読ませるREADME

🪞 「新しいプロジェクトに入った時、最初に何を読みますか?」

  • 新しいプロジェクト → まず README.md を読む
  • AI も同じ → CLAUDE.md を読む
  • CLAUDE.md = AI に読ませるプロジェクトの説明書

読込挙動 (Claude Code 公式):
- 新しい会話/セッション開始時に自動読込 (プロセス再起動は不要)
- /clear や /compact の 後もディスクから再読込される
- 一度書けば 毎セッション同じルールが効く

Before (CLAUDE.md なし) → AI が Jest でテスト生成 (本当は pytest) / 2 スペース (本当は 4 スペース) / 毎回前提説明を繰り返す
After (CLAUDE.md あり) → AI が pytest を選ぶ / 4 スペースで書く / 指示なしで正しい振る舞い

📁 CLAUDE.md は 4 階層 — 「連結」して読込

階層 配置パス 何のため
① 管理ポリシー OS 配下 (例: /Library/Application Support/ClaudeCode/CLAUDE.md) 組織レベル
② ユーザー指示 ~/.claude/CLAUDE.md 自分の全プロジェクト共通
③ プロジェクト指示 ./CLAUDE.md または ./.claude/CLAUDE.md このプロジェクト固有
④ ローカル ./CLAUDE.local.md 個人用 (.gitignore 推奨)

重要な挙動:
- ディレクトリツリーを 上方向に歩いて発見された全 CLAUDE.md を「連結」 (上書きではない)
- 矛盾時は「後勝ち」 = 後に読まれるプロジェクト指示が従われやすい
- 例: ② に「インデント 4 スペース」、③ に「インデント 2 スペース」と書いた場合 → ③ が従われやすい (連結はされるが、矛盾箇所だけ後勝ち)
- サブディレクトリの CLAUDE.md は オンデマンド読込

最初は ③ プロジェクト指示 (./CLAUDE.md) 1 つだけで十分。②③④ の使い分けは第 3 章で扱います

5 セクション構成 — 30 行で書ける

CLAUDE.md は以下の 5 セクションで構成するのが基本:

# セクション 役割 例
1 使用言語・FW (フレームワーク) 何を使っているか Python 3.12 / FastAPI
2 コーディング規約 スタイルの統一 4 スペース / 型ヒント必須
3 よく使うコマンド ビルド・テスト・実行 pytest / uvicorn
4 禁止事項 やってはいけないこと rm -rf 禁止 / main に push 禁止
5 プロジェクト構造 ディレクトリの意味 src/ = 本体 / tests/ = テスト

これだけで 約 30 行。完璧を目指さない — 30 行の CLAUDE.md は 0 行の何倍も価値がある

30 行ミニマム版の例

Python の例ですが 中身が読めなくても構いません。JS でも Go でも構造は同じ。5 セクション という構造だけ見てください

# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM (データベースを扱いやすくする仕組み): SQLAlchemy 2.0

## コーディング規約
- インデント: 4 スペース
- 型ヒント必須 (変数の種類を明示する書き方)
- 関数の説明書き (docstring) は Google 形式

## よく使うコマンド
- テスト: pytest tests/
- 起動: uvicorn app.main:app --reload
- コード検査 (lint): ruff check .

## 禁止事項
- 本番 DB の直接操作禁止
- main ブランチへの直接 push 禁止

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = データベースの変更履歴 (マイグレーション)

保存 3 ステップ: ① プロジェクトのルートを開く ② CLAUDE.md という名前で保存 ③ Claude Code で新しい会話/セッションを開始 → 自動的に読み込まれる (プロセス再起動不要)

セクション解説 — 何をどう書くか

💡 ここから先は Python 固有の用語も出てきます。「そういう設定項目がある」程度で OK・ご自身の言語に置き換えて読んでください

① 使用言語・FW: 言語のバージョンまで書く → AI が使える新機能を正しく判断
- 例: 「Python 3.12」と書けば PEP 701 で追加された f-string クォート再利用 — f"playlist: {", ".join(items)}" のように外側と内側で同じクォートを再利用できる書き方 (3.11 以前ではクォート衝突でエラーだった) — が使えると判断する

② コーディング規約: インデント幅・命名規則・リンター設定 → AI 生成コードのスタイルが揃いレビュー指摘が減る

③ よく使うコマンド: テスト・ビルド・Lint のコマンド → AI が修正後に確認実行できる (Python は pytest tests/, JS は npm test, Go は go test ./...)

④ 禁止事項: rm -rf 禁止 / 本番 DB 直接操作禁止 / main 直 push 禁止 → AI が守ろうとするソフトな指示 (AI に対する「お願い」レベル。技術的に実行をブロックする強制力は L3 hook が担う・第 5 章で扱う)

⑤ プロジェクト構造: src/ tests/ 等のディレクトリの役割 → AI がファイルを置く場所を間違えない

よくある失敗 3 選 + 書く時のポイント 5 つ

❌ 失敗 3 選:
1. 長すぎる (100 行超) → AI が重要な部分を見落とす → 30 行以内に収める
2. 曖昧な表現 (「きれいなコード」) → AI の解釈がバラバラ → 「4 スペース」のように具体的に
3. 矛盾する指示 (TypeScript と JavaScript 両方指定) → AI が混乱 → 一貫性を確認してから保存

✍️ 書く時のポイント 5 つ:
1. 短く始める — 5 セクション 30 行で十分
2. 具体的に書く — 「きれいに」ではなく「4 スペース」
3. 禁止事項を必ず入れる — AI に守ってほしいことの最低限
4. 随時アップデート — 変更があったら追記 (30 行超えそうなら古いルール削る)
5. AI の振る舞いを見て調整 — 期待と違ったら追記

完璧を目指さない — 30 行の CLAUDE.md は 0 行の何倍も価値がある

Before / After 検証 — 3 つの効果

① テストフレームワーク選択:
- Before: AI が Jest でテスト生成 (本当は pytest)
- After: AI が pytest でテスト生成

② インデントスタイル:
- Before: AI が 2 スペースで生成 (本当は 4 スペース)
- After: AI が 4 スペースで生成

③ セッションを跨ぐ記憶:
- Before: 毎回「このプロジェクトは Python です」と説明
- After: 新しい会話を開始するだけで自動理解 (/clear 後もディスクから再読込)

30 行書くだけで AI の振る舞いが劇的に変わる — これが L2 コンテキストの最初の成果

📚 まとめ + 確認問題

この章で学んだこと:
1. CLAUDE.md = AI に読ませるプロジェクトの説明書 (3 階層モデルの L2 コンテキスト)
2. 新しい会話/セッション開始時に自動読込 (プロセス再起動不要・/clear 後もディスクから再読込)
3. 4 階層構造 (管理 / ユーザー / プロジェクト / CLAUDE.local.md) を「連結」して読込・矛盾時は「後勝ち」
4. 5 セクション 30 行 のミニマム構成で十分・まず ./CLAUDE.md 1 つから
5. 禁止事項は AI へのソフトな指示・技術的強制ブロックは L3 hook (第 5 章) が担う
6. あり/なしで AI の 振る舞いが劇的に変わる

確認問題:
- Q1: CLAUDE.md の主な役割は?

解答AI にプロジェクトのルールを教える「説明書」。新しい会話/セッション開始時に自動読込され、毎セッションで適用される (プロセス再起動不要)。

- Q2: ミニマム構成の 5 セクションを答えよ
解答使用言語・FW / コーディング規約 / よく使うコマンド / 禁止事項 / プロジェクト構造

- Q3: 「きれいなコードを書いて」と書くべきでない理由は?
解答曖昧な表現は AI の解釈がバラバラになるから。「4 スペース」「型ヒント必須」のように具体的に書く。

第 3 章: CLAUDE.md の仲間たち (約 30 min)

CLAUDE.md 以外にも AI に渡せる .md ファイル群がある。CLAUDE.md (プロジェクト全体) から特定用途のファイルへ広げていく

次章で扱う主なファイル:
- .claude/skills/<name>/SKILL.md — 業務領域のワークフロー定義 (必要時にオンデマンド読込)
- .claude/agents/<name>.md — サブエージェント (役割別の専門担当)
- @path imports — CLAUDE.md 内から他ファイルを取り込む構文

📌 公式仕様の補足: commands と skills は同じ機能の 2 つの書き方 (旧/新)・公式が 2026 年 1 月にマージ。詳細は次章 — .claude/commands/<name>.md は legacy 互換として残り、新規は .claude/skills/<name>/SKILL.md のディレクトリ形式が推奨
