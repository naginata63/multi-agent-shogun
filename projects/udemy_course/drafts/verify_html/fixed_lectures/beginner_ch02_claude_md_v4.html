CLAUDE.mdとは — AIに読ませるREADME

🪞 「新しいプロジェクトに入った時、最初に何を読みますか？」

  • 新しいプロジェクト → まず README.md を読む
  • AIも同じ → CLAUDE.md を読む
  • CLAUDE.md = AIに読ませるプロジェクトの説明書

Claude Codeは 新しい会話/セッションを開始すると自動的に CLAUDE.md を読み込み、プロジェクトのルールを理解する。/clear や /compact の後もディスクから再読込される (プロセスの再起動は不要)

📁 CLAUDE.md は階層構造 — 4階層を「連結」して読込

階層 配置パス 何のため
① 管理ポリシー OS 配下 (例: /Library/Application Support/ClaudeCode/CLAUDE.md) 組織レベル
② ユーザー指示 ~/.claude/CLAUDE.md 自分の全プロジェクト共通
③ プロジェクト指示 ./CLAUDE.md または ./.claude/CLAUDE.md このプロジェクト固有
④ ローカル ./CLAUDE.local.md 個人用 (.gitignore 推奨)

重要な挙動 (Claude Code 公式):
- ディレクトリツリーを 上方向に歩いて発見された全 CLAUDE.md を「連結」 (上書きではない)
- 矛盾時は 後に読まれるプロジェクト指示が従われやすい (上書きではなく「後勝ち」)
- サブディレクトリの CLAUDE.md は オンデマンド読込

最初は ③ プロジェクト指示 (./CLAUDE.md) 1つだけで十分。②③ の使い分けは第3章で詳しく扱います

なぜCLAUDE.mdが必要なのか

CLAUDE.mdなしの会話:
- 「このプロジェクトはPythonです」→ 次の会話で忘れる
- 「テストはpytestで実行して」→ 毎回言わないといけない
- 「TypeScriptのstrictモードです」→ また忘れる

CLAUDE.mdありの会話:
- 一度書けば 毎セッション自動で読み込まれる
- プロジェクトのルールを AIが自動で守る
- 同じ前提説明の 繰り返しがゼロに

Before / After — 効果の実感

Before (CLAUDE.md なし):
- AI「テストを書きました」 → Jestで書いてる (プロジェクトはpytest)
- AI「コードを修正しました」 → 2スペースインデント (プロジェクトは4スペース)
- 毎回指示を繰り返す必要がある

After (CLAUDE.md あり):
- AI「pytestでテストを書きました」 → 自動で適切なツールを選択
- AI「4スペースインデントで修正しました」 → コーディング規約を遵守
- 指示なしで正しい振る舞い

5セクション構成 — 30行で書ける

CLAUDE.mdは以下の5セクションで構成するのが基本:

# セクション 役割 例
1 使用言語・FW（フレームワーク） 何を使っているか Python 3.12 / FastAPI
2 コーディング規約 スタイルの統一 4スペース / 型ヒント必須
3 よく使うコマンド ビルド・テスト・実行 pytest / uvicorn
4 禁止事項 やってはいけないこと rm -rf 禁止 / mainにpush禁止
5 プロジェクト構造 ディレクトリの意味 src/ = 本体 / tests/ = テスト

これだけで 約30行。最初はこれだけで十分

30行ミニマム版の例

以下はPythonの例ですが、中身が読めなくても構いません。この例はPythonですが、JavaScriptでもGoでも構造は同じです。セクションの構成（5つ）が重要で、中身の技術用語はご自身の使う言語に置き換えてください。構造（5セクション・箇条書き）だけ見てください。

# CLAUDE.md

## 使用言語・フレームワーク
- Python 3.12 / FastAPI
- ORM（データベースを扱いやすくする仕組み）: SQLAlchemy 2.0

## コーディング規約
- インデント: 4スペース
- 型ヒント必須（変数の種類を明示する書き方）
- 関数の説明書き（docstring）はGoogle形式

## よく使うコマンド
- テスト: pytest tests/
- 起動: uvicorn app.main:app --reload
- コード検査（lint）: ruff check .

## 禁止事項
- 本番DBの直接操作禁止
- mainブランチへの直接push禁止

## プロジェクト構造
- app/ = アプリ本体
- tests/ = テスト
- migrations/ = データベースの変更履歴（マイグレーション）

保存の手順 (3ステップ):
1. プロジェクトの一番上のフォルダ（ルート）を開く
2. CLAUDE.md という名前でファイルを作成し、上記を貼り付けて保存
3. Claude Code で 新しい会話/セッションを開始 → 自動的に読み込まれる (プロセス再起動は不要・/clear 後もディスクから再読込される)

💡 確認方法: 新しい会話を始めて「CLAUDE.mdの内容を教えて」と聞けば、正しく読み込まれているかすぐ確認できます

セクション詳細 — 使用言語・FW と コーディング規約

💡 ここから先はPython固有の用語が出てきます。「そういう設定項目があるんだな」程度に読み流してOK です。ご自身の言語に置き換えて考えましょう。

使用言語・フレームワーク:
- 言語のバージョンまで書く → AIが使える機能を正しく判断
- 例: 「Python 3.12」と書けば、最新の文法機能（例: 値を代入しつつ比較できる := 演算子）を使えると判断

コーディング規約:
- インデント幅、命名規則、リンター設定を書く
- AIが生成するコードの スタイルが統一される
- コードレビューの指摘事項が減る

「4スペース」と書くだけで、AIは常に4スペースでインデントする

セクション詳細 — よく使うコマンド と 禁止事項

よく使うコマンド:
- テスト、ビルド（プログラムを動かせる形に変換すること）、Lint（コードの問題を自動検査すること）の実行コマンドを書く
- AIがコード修正後に テストを実行して確認 する（実行前に人間へ確認することもあります）

プロジェクト テストコマンド
Python pytest tests/
JavaScript npm test
Go go test ./...

禁止事項:
- 危険な操作 (rm -rf 等) の禁止
- 本番環境への直接アクセス禁止
- AIが 実行前に止まる 安全装置

効果の検証 — 3つのBefore / After

① テストフレームワークの選択:
- Before: AIがJestでテストを生成 (プロジェクトはpytest)
- After: AIがpytestでテストを生成

② インデントスタイル:
- Before: AIが2スペースでコード生成 (プロジェクトは4スペース)
- After: AIが4スペースでコード生成

③ セッションを跨ぐ記憶:
- Before: 毎回「このプロジェクトはPythonです」と説明
- After: 新しい会話を開始するだけで自動で理解

よくある書き方の失敗 3選

❌ 失敗1: 長すぎる
- 100行以上のCLAUDE.md → AIが重要な部分を見落とす
- 解決: 30行以内に収める。詳細は別ファイルに

❌ 失敗2: 曖昧な表現
- 「きれいなコードを書いて」→ AIの解釈がバラバラ
- 解決: 具体的な規約を書く (例: 「4スペース」)

❌ 失敗3: 矛盾する指示
- 「TypeScriptを使って」+ 「JavaScriptで書いて」→ AIが混乱
- 解決: 一貫性を確認してから保存する

CLAUDE.mdを書く時のポイント

  1. 短く始める — 最初は5セクション30行で十分
  2. 具体的に書く — 「きれいに」ではなく「4スペース」
  3. 禁止事項を必ず入れる — 安全性の最低限
  4. 随時アップデート — プロジェクトの変更点があったら追記する（30行を超えそうなら、古いルールを見直して削る）
  5. AIの振る舞いを見て調整 — 期待と違ったら追記する

完璧を目指さない — 30行のCLAUDE.mdが、0行の何倍も価値がある

まとめ — この章で学んだこと

  1. CLAUDE.md = AIに読ませるプロジェクトの説明書 (3階層モデルの L2 コンテキスト)
  2. 新しい会話/セッション開始時に自動読み込み → 毎回の説明が不要に (プロセス再起動不要)
  3. 4階層構造 (管理 / ユーザー / プロジェクト / ローカル) を 「連結」 して読込
  4. 5セクション30行 のミニマム構成で十分 — まずは ./CLAUDE.md 1つから
  5. あり/なしでAIの 振る舞いが劇的に変わる
  6. 長すぎ・曖昧・矛盾の 3つの失敗 を避ける

確認問題

Q1: CLAUDE.mdの主な役割は何か？

解答AIにプロジェクトのルールを教える「説明書」として機能する。新しい会話/セッションの開始時に自動で読み込まれ、毎セッションでルールが適用される (プロセス再起動は不要)。

Q2: ミニマム構成の5セクションをすべて答えよ。

解答使用言語・フレームワーク、コーディング規約、よく使うコマンド、禁止事項、プロジェクト構造

Q3: CLAUDE.mdで「きれいなコードを書いて」と書くべきでない理由は何か？

解答曖昧な表現はAIの解釈がバラバラになるから。「4スペース」「型ヒント必須」のように具体的な規約を書くべき。

次のレクチャー予告

第3章: CLAUDE.mdの仲間たち — commands / skills / agents (約 30 min)

CLAUDE.md以外にもAIに渡せる .md ファイル群があります。CLAUDE.md (プロジェクト全体) から特定用途のファイルへ広げる流れを学びます。

  • .claude/commands/<name>.md — 自分用のスラッシュコマンド (定型操作の保存)
  • .claude/skills/<name>/SKILL.md — 業務領域のワークフロー定義 (必要時にオンデマンド読込)
  • .claude/agents/<name>.md — サブエージェント (役割別の専門担当)
  • @path imports — CLAUDE.md 内から他ファイルを取り込む構文

CLAUDE.md がプロジェクト全体の説明書なら、これらは 特定用途の補強パッケージ
