## 2. 章別構成 (ストーリー + 解説アウトライン + 学習目標)

### ch00 序章: AI開発とはなにか — エンジニアリングの新しい形 (L1: Remember)

**所要**: 15 min / **無料プレビュー**

#### ストーリー (Emotional Hook)
> 「2022年、ChatGPTが登場して世界が変わりました。しかし『生成AI』は単なるチャットボットではありません。AIがコードを読み、書き、実行する——この新しい開発の形を『AIエンジニアリング』と呼びます。15分で、その全体像を掴みましょう。」

- 受講者の「これだ！」ポイント: 「AI開発＝チャットで質問するだけ」という誤解が解け、AIが自律的にコードを操作する世界観が見える瞬間。

#### 学習目標
- 生成AIとは何か（従来のAIとの違い）を説明できる
- AIエンジニアリングとは何かを説明できる（軽い命名・詳細は中級編）
- Claude Codeがどのようなツールか概要を理解する
- 本講座の3階層モデル（L1/L2/L3）の全体像を把握する

#### 解説アウトライン
1. 生成AIとは: テキスト・コード・画像を「生成」するAIの定義
2. 従来のAI（ルールベース・検索）との違い: 「探す」から「作る」への転換
3. AIエンジニアリングとは: AIが自律的にコードを読み・書き・実行する新しい開発の形
4. Claude Codeの立ち位置: CLI特化 × Anthropic製 × 開発者向け
5. 本講座の3階層モデル: プロンプト(L1) → コンテキスト(L2) → ハーネス(L3)
6. 本講座のゴール: 3つの学習到達点のプレビュー

### ch01 プロンプト基礎 — AIに確実に伝える技術 [L1: Apply]

**所要**: 30 min / Bloom: L3 Apply
**参考**: 中級編 第1章の圧縮版（完全な代替ではない）

#### ストーリー (Emotional Hook)
> 「AIに伝わればいいと思っていませんか？ 実は、伝わるプロンプトと再利用できるプロンプトは別物です。この章では、1回書いて終わる指示を、何度でも使える形に変えます。」

- 受講者の「これだ！」ポイント: 自分の適当な指示が構造化される瞬間。同じタスクを2回目に実行した時の違い（再現性）をスライド比較で解説。

#### 解説アウトライン
1. プロンプトの3要素: 入力 / 処理 / 出力形式
2. 構造化プロンプトの書き方: 各要素の役割と書き方
3. Before/After比較: 非構造化プロンプト vs 構造化プロンプトの出力差
4. 再現性の確認: 同じプロンプトを2回実行した結果の安定性
5. よくある失敗パターン3選: 曖昧な指示 / 出力指定なし / コンテキスト不足

#### 学習目標
- 構造化プロンプトの書き方を理解する
- 入力・処理・出力の3要素に指示を分解できる
- 同じプロンプトで再現性のある結果を得る仕組みを理解する

### ch02 CLAUDE.md ミニマム — プロジェクトの「説明書」 [L2: Apply]

**所要**: 30 min / Bloom: L3 Apply
**参考**: 中級編 第3章の圧縮版（完全な代替ではない）

#### ストーリー (Emotional Hook)
> 「新しいプロジェクトに入った時、最初にREADME.mdを読みますよね？CLAUDE.mdは、AIに読ませるREADMEです。30行書くだけで、AIがプロジェクトのルールを理解します。」

- 受講者の「これだ！」ポイント: CLAUDE.mdなしで「テストを書いて」と頼んだ時と、ありで頼んだ時の結果の違いをスライドで対比。コーディング規約や使用ライブラリをAIが勝手に守る仕組みを解説。

#### 解説アウトライン
1. CLAUDE.mdとは: AIに読ませるプロジェクトREADME
2. 5セクション構成の解説: 使用言語/フレームワーク / コーディング規約 / よく使うコマンド / 禁止事項
3. 30行ミニマム版の例示: 実際のサンプルコードをスライドで解説
4. 効果の測定: あり/なしでAIの振る舞いが変わる具体的な例
5. よくある書き方の失敗: 長すぎる / 曖昧 / 矛盾する指示

#### 学習目標
- CLAUDE.mdの役割（AIへのプロジェクト説明書）を理解する
- 最低限の構成（5セクション・30行）でCLAUDE.mdを書ける
- あり/なしでAIの振る舞いが変わる理由を説明できる

### ch03 .md仲間たち — コンテキストを深める設定ファイル群 [L2: Understand + Apply]

**所要**: 30 min / Bloom: L2-L3
**参考**: v3にない新規章

#### ストーリー (Emotional Hook)
> 「CLAUDE.mdは『メインの説明書』でした。でもAIには、もっとたくさんの『説明書』があります。エージェントの行動指針、ワンクリックコマンド、再利用可能なスキル——これらの.md仲間たちが、AIを本格的な開発パートナーに変えます。」

- 受講者の「これだ！」ポイント: CLAUDE.mdだけだと思っていた設定が、実は6つ以上のファイル群で支えられていることに気づく瞬間。「全部書かないといけないの？」→「いいえ、必要なものだけ」の安心感。

#### 解説アウトライン
1. .mdファイル群の全体像: 6つの設定機構をマップ図で紹介
2. **CLAUDE.md** (復習): プロジェクト固有ルール — すべての基盤
3. **agents.md**: エージェント別の行動指針 — 役割ごとにAIの人格を切り替える仕組み
4. **commands/**: ワンクリックコマンド（スラッシュコマンド）— よく使う操作を定型化
5. **skills/**: 再利用可能なワークフロー — 複数ステップの作業を1コマンドにまとめる
6. **imports/**: 外部ファイルの取り込み — 共通設定をプロジェクト間で共有する仕組み
7. **settings.json / settings.local.json**: JSON設定ファイル — .md以外の設定の入り口
8. どれから始めるか: 初心者はCLAUDE.md → commands → skills の順で拡張

#### 学習目標
- Claude Codeの.md設定ファイル群を6種類以上挙げられる
- 各ファイルの役割（CLAUDE.md/agents.md/commands/skills/imports/global settings）を説明できる
- 初心者がどの順番で設定ファイルを導入すべきか判断できる

### ch04 歴史と意義 — なぜ3階層なのか [L2: Understand + L4: Analyze]

**所要**: 25 min / Bloom: L2-L4
**参考**: v3にない完全新規章

#### ストーリー (Emotional Hook)
> 「『なぜプロンプトとCLAUDE.mdとhookの3つが必要なの？ 1つじゃダメなの？』この問いに答えるために、ソフトウェア開発の歴史を振り返ります。DevOps、SRE、Platform Engineering——そしてAI Engineering。各時代が解決した課題を見ると、3階層の必然性が見えてきます。」

- 受講者の「これだ！」ポイント: バラバラに学んでいた3階層（L1/L2/L3）が、歴史の流れの中で「なぜこの順番で、なぜこの構成なのか」が一気に繋がる瞬間。

#### 解説アウトライン
1. **ソフトウェア開発の進化**: 手作業 → スクリプト → CI/CD → プラットフォーム → AI開発
2. **DevOps (2009〜)**: 開発と運用の統合 — 「文化とツールの両方」が必要だった教訓
3. **SRE (2016〜)**: 信頼性エンジニアリング — 「システムに信頼性を組み込む」発想の転換
4. **Platform Engineering (2020〜)**: 開発者体験の向上 — 「プラットフォームが開発者を守る」モデル
5. **AI Engineering (2024〜)**: AI開発エンジニアリング — 「AIに文脈を与え、AIを制御する」新パラダイム
6. **なぜ3階層か**:
   - L1 プロンプト = 人間の意図の伝達（DevOpsの「文化」に相当）
   - L2 コンテキスト = プロジェクト知識の永続化（Platformの「自己サービス」に相当）
   - L3 ハーネス = AIの動作制御（SREの「信頼性」に相当）
7. **Claude Codeが解決したこと**: CLIでAIに文脈を与え、安全に自律動作させる統合環境
8. **3階層の相互関係**: L1だけでは再現性がなく、L2だけでは制御できず、L3だけでは文脈がない — 3つ揃って初めて完成

#### 学習目標
- DevOps → SRE → Platform Engineering → AI Engineering の系譜を説明できる
- 3階層モデル（プロンプト/コンテキスト/ハーネス）がなぜ必要か理由を言える
- 各階層が欠けた場合に何が起きるかを分析できる
- Claude CodeがAI開発においてどのような課題を解決したか説明できる

### ch05 hookの入門 — AIの安全装置を作る [L3: Apply]

**所要**: 30 min / Bloom: L3 Apply
**参考**: 1本だけ解説。18本の全網羅は中級編で。

#### ストーリー (Emotional Hook)
> 「AIに『このファイルを消して』と言われたら、どうしますか？ hookは、AIが危険な操作をする前に立ち止まらせる安全装置です。たった1本で、あなたのコードを守れます。」

- 受講者の「これだ！」ポイント: hookなしで `rm -rf` を含むコマンドが実行される恐怖 → hookありで「STOP: 危険な操作を検知しました」とブロックされる安心感。スライドでこの差を1発で解説。

#### 解説アウトライン
1. hookとは: AIのライフサイクルイベントに反応する仕組み
2. 7つのイベント種類: SessionStart / PreToolUse / PostToolUse / Stop 等
3. PreToolUse hookの仕組み解説: コマンド実行前に検査 → 危険パターンをブロック
4. 設定ファイルの構造: `.claude/settings.json` の hook エントリ解説
5. ブロック例の解説: `rm -rf` / `sudo` / `DROP TABLE` 等の検知パターン
6. hookの拡張性: より高度な18本のhookは中級編で解説

#### 学習目標
- hookとは何か（AIのライフサイクルイベントに反応する仕組み）を理解する
- PreToolUse hook の仕組みと設定方法を理解する
- AIの危険操作を自動検知・ブロックする仕組みを説明できる
