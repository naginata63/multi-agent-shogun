## Q1: 実バグ分析+修正案
- 開始時刻: 2026-04-16 10:42:47

- advisor応答要約: コンテキスト不足のため自力分析。FAISS mmapの制約とBigQuery移行の妥当性を確認。

### 回答

**対象ファイル**: `projects/dozle_kirinuki/scripts/subtitle_semantic_index.py`

**現状確認**: 現行コードはBigQuery版（FAISS廃止済み）。docstringに「ローカルFAISSインデックス廃止」の記録あり。FAISS/mmap関連コードは現行版に存在しない。

#### 原因分析

mmap化で検索が壊れる典型的な原因を、コード構造から逆推定する:

**1. FAISS IVF系インデックスのmmap非対応（最も可能性高い）**

FAISSの`read_index(path, faiss.IO_FLAG_MMAP)`は`IndexFlatIP`や`IndexFlatL2`では正常動作するが、`IndexIVFFlat`や`IndexIVFPQ`ではinverted listsのmmap読み込みに既知の問題がある。検索結果が空またはランダムになる。

**2. in-place正規化との競合**

`faiss.normalize_L2()`はin-placeで配列を書き換える。mmap（読み取り専用）された配列に対しては`SIGBUS`または暗黙の書き込み失敗を起こす。コードに正規化処理がある場合、これが直接の原因。

**3. numpy mmap_mode='r' の制約**

元のFAISS版で`np.load(file, mmap_mode='r')`を使っていた場合、返される`np.memmap`オブジェクトはread-only。FAISSが内部で配列を変更しようとすると失敗する。

#### 修正案

**案A（推奨＝現行コードの選択）: BigQuery版への完全移行**
- BigQuery `VECTOR_SEARCH()`がmmap問題を完全回避
- スケーラビリティと信頼性が向上
- 現行コードが既にこの方針

**案B: FAISS mmapを正しく使う場合**
```python
# 修正: IO_FLAG_MMAPを削除して通常読み込みに
index = faiss.read_index(path)

# または IndexFlat 系を使用（mmap安全）
index = faiss.IndexFlatIP(dimension)

# numpy mmap の修正
embeddings = np.load("embeddings.npy").astype(np.float32)
# または copy-on-write モード:
embeddings = np.load("embeddings.npy", mmap_mode="c")
```

**案C**: IO_FLAG_MMAPを使う場合は、IndexFlat系（mmap安全なタイプ）のみに限定し、IVF系は避ける。

#### 根本原因まとめ

FAISSのIVF系インデックスを`IO_FLAG_MMAP`で開くとinverted listsの読み込みが不完全になり、`search()`が不正結果を返す。これはFAISS既知の制約。現行のBigQuery移行が正しい解決策。

#### 現在の責務分離の設計意図

| スクリプト | 役割 | ライフサイクル | 呼び出し元 |
|-----------|------|-------------|-----------|
| inbox_write.sh | 郵便局員（手紙を投函） | 1ショット実行→即終了 | 全エージェント |
| inbox_watcher.sh | 郵便配達員（新着通知） | 長期稼働デーモン | 起動時のみ起動 |

設計意図は明確にUnix哲学（1ツール1機能）に従っている:

1. **write** = メッセージ永続化（flock排他＋原子書込tmp+rename）＋副作用（cmd status自動更新、task done自動更新）
2. **watcher** = ファイル変更検知（inotifywait/fswatch）＋エスカレーション（nudge→Escape→/clear）＋自己監視（Phase 1/2/3）＋ビジー/アイドル検知

#### 統合した場合のメリット

1. **コード重複削減**: YAML読み書き、ロック取得、agent_id解決などの共通処理が1箇所に集約
2. **設定の一元管理**: INBOX path、lockfile path、escalation timer等の定数を1ファイルで管理
3. **デバッグ容易性**: 「書き込み→検知→通知」のフローを1ファイルで追跡可能

#### 統合した場合のデメリット（致命的）

1. **ライフサイクル不一致**: writeは1ショット、watcherはデーモン。1プロセスに両方詰めると「daemonモード」と「oneshotモード」の分岐が必要になり、むしろ複雑化する
2. **信頼性低下**: 統合プロセスがクラッシュすると書き込みも監視も両方死ぬ。現在はwatcherが死んでもwriteは動く（メッセージは届く、再起動で回復可能）
3. **テスト困難性**: writeの単体テストは簡単（`inbox_write.sh target "msg"`）。watcherはinotifywait mockが必要。統合するとテスト粒度が粗くなる
4. **マルチエージェント競合**: 各エージェントが独立したwatcherプロセスを持つ。writeは全エージェントから呼ばれる。統合すると「どのエージェントのwatcherがwriteも担当するか」の調整が必要
5. **1262行→1500行+**: inbox_watcher.shは既に1262行。write機能（255行）を統合すると1500行超となり、保守性が著しく低下

#### 推奨: 統合しない（現在の分離維持）

**根拠**: Unix哲学「1ツール1機能」が正しく機能している。分離によって以下が実現されている:
- **部分障害耐性**: watcherクラッシュ→writeは動く。write失敗→watcherは次回タイムアウトで検知
- **独立テスト**: 各スクリプトを独立してテスト可能
- **並行安全性**: 複数エージェントからの同時writeがflockで安全。watcherは各エージェント1プロセス

#### 改善提案（統合せずに共通化）

統合ではなく、共通部分を抽出して保守性を向上:

```bash
# lib/inbox_common.sh — 共通関数ライブラリ（新規作成）
# inbox_write.sh / inbox_watcher.sh 双方からsource

INBOX_BASE="${SCRIPT_DIR}/queue/inbox"
MAX_MESSAGES=50

get_inbox_path() { echo "${INBOX_BASE}/${1}.yaml"; }
get_lock_path() { echo "${INBOX_BASE}/${1}.yaml.lock"; }

acquire_lock() {
    local lockfile="$1"
    if command -v flock &>/dev/null; then
        exec 200>"$lockfile"
        flock -w 5 200 || return 1
    else
        local d="${lockfile}.d" i=0
        while ! mkdir "$d" 2>/dev/null; do
            sleep 0.1; i=$((i+1)); [ $i -ge 50 ] && return 1
        done
    fi
}

yaml_safe_load() {
    # inbox YAMLの読み込み（存在チェック＋空ファイル対応）
    local path="$1"
    [ -f "$path" ] || { echo "{}"; return; }
    python3 -c "import yaml,sys; print(yaml.safe_load(open(sys.argv[1])) or {})" "$path"
}
```

これにより:
- 重複コード（lock取得、path生成）を1箇所に集約
- 各スクリプトの本体は変更なし
- 既存のテストに影響なし

- 所要時間: 480秒

### 回答

**問題**: ffmpeg concatで44100Hz音声と48000Hz音声を混在させると、音ズレ・ノイズ・音飛びが発生する。

#### コードベース内の現状（実ファイル確認済み）

| ファイル | サンプルレート | 用途 |
|---------|--------------|------|
| detect_scene_changes.py:163 | 44100Hz | 音声抽出（PCM） |
| ffmpeg_ops.py:76,483,561 | 44100Hz | 無音生成（anullsrc） |
| generate_manga_short.py:2294 | 44100Hz | 無音生成 |
| Demucs出力（一般的） | 44100Hz | ボーカル分離 |
| YouTube DL（一般的） | 48000Hz または 44100Hz | 元動画音声 |
| WhisperX処理 | 16000Hz | 文字起こし（使用後は破棄） |

**問題発生パターン**:
1. YouTube DL → 48000Hz動画 + ffmpeg_ops無音（44100Hz）をconcat → 音ズレ
2. Demucs出力（44100Hz）+ 無音埋め（44100Hz）→ 一致する場合もあるが、元動画が48000Hzの場合に不一致
3. 異なるソースからのクリップ結合時にレート不一致

#### 根本対策設計

**方針**: 全音声をconcat前に統一レート（48000Hz推奨）に正規化する。44100HzはCD規格だが、YouTube/Web配信では48000Hzが標準。

**実装案**:

```python
# 案1: ffmpeg_ops.pyの共通関数にリサンプル追加
STANDARD_SAMPLE_RATE = 48000  # 統一レート

def ensure_sample_rate(input_path: str, output_path: str, target_sr: int = STANDARD_SAMPLE_RATE) -> str:
    """音声ファイルのサンプルレートを統一する。既に一致する場合はコピー不要。"""
    import subprocess
    # 現在のサンプルレートを取得
    probe = subprocess.run(
        ["ffprobe", "-v", "quiet", "-show_entries", "stream=sample_rate",
         "-of", "csv=p=0", input_path],
        capture_output=True, text=True
    )
    current_sr = int(probe.stdout.strip()) if probe.stdout.strip() else 0

    if current_sr == target_sr:
        return input_path  # 変換不要

    # リサンプル（GPU NVENC不要。音声はCPU処理が十分高速）
    subprocess.run([
        "ffmpeg", "-y", "-i", input_path,
        "-ar", str(target_sr),
        "-c:a", "aac", "-b:a", "192k",
        output_path
    ], check=True)
    return output_path
```

def validate_concat_inputs(input_paths: list[str]) -> bool:

    rates = set()

# AIエージェント7体で動画編集した話

AIが「自律的に動くエージェント」になる——。2025年末、そんな技術を目にした時は「またAIの誇大広告だろう」と思いました。でも実際に作ってみたら、思ったよりすごかった。そして思ったより大変だった。

## きっかけは「手が足りない」

ドズル社というゲーム実況グループの切り抜き動画をYouTubeに投稿しています。やることは単純です。長時間の配信アーカイブから面白い場面を見つけて、ショート動画に編集してアップする。

シンプルな作業です。でも、やることが多い。

1. 1〜2時間の動画から面白いシーンを探す
2. 音声を分離して文字起こしする
3. 話者が誰かを特定する
4. 字幕を付ける
5. サムネイルを作る
6. YouTubeにアップロードする

これを手作業でやると、1本のショート動画に2〜3時間かかる。1日の配信アーカイブから5本作ろうとすると、それだけで1日が終わる。

「AIにやらせれば早いんじゃない？」——そう思って、AIエージェント7体のチームを作りました。

## 7体のエージェント、役割分担

システムは戦国武将になぞらえています。将軍（全体指揮）、家老（タスク管理）、軍師（品質チェック）、そして足軽1〜4号（実働部隊）。各エージェントが独立したAIプロセスで、メールボックス方式で連携します。

**将軍**は人間（私です）の意図をコマンドに変換して家老に伝える。
**家老**はコマンドを受け取って、具体的なタスクに分解し、足軽に割り当てる。
**足軽**は実際にコードを書いたり、スクリプトを実行したりする。
**軍師**は足軽の成果物を品質チェックして、ダメ出しする。

要するに、小さな会社のような階層構造をAIで作ったわけです。

## 最初の壁：指示が伝わらない

一番苦労したのは、エージェント間のコミュニケーションです。

ある時、足軽に「YouTubeに動画をアップロードして」と指示したら、アップロードはしたものの、説明欄が空っぽだったことがありました。アップロード自体は成功している。でも「説明欄も書いて」という暗黙の前提が伝わっていなかった。

こういう「人間なら言わなくてもわかること」が、AIには伝わらないんです。

対策として、タスクの指示書（YAMLファイル）に「完了条件」を必ず書くようにしました。「動画のアップロードが完了していること。説明欄にタイトル・ハッシュタグ・関連リンクが含まれていること」みたいに、全てを明示的に書く。最初は面倒でしたが、結果的にバグも減りました。

## 2番目の壁：同じ間違いを繰り返す

エージェントチームを1ヶ月ほど運用して気づいたのは、「一度起きたミスが、別のエージェントでも起きる」ということです。

例えば、ffmpegで動画を変換する時、CPUの遅いエンコード方式を使っていて、32分の動画の変換に3時間半かかったことがありました。GPUを使えば数分で終わるのに。この問題を足軽1号に修正させたのに、翌週に足軽3号が同じ間違いをした。

原因は「知見がエージェント間で共有されていない」ことでした。そこで、教訓を記録する仕組みを作りました。ミスをしたらその原因と対処法を専用ファイルに書き込み、全エージェントが参照できるようにした。人間の組織でいう「社内Wiki」のAI版です。

## 3番目の壁：指示が矛盾していることに気づかない

これが一番面白かった（そして一番悔しかった）失敗です。

あるテストで「前回と全く同じ5問を解かせて」と将軍に指示したんです。でも将軍の指示書には「前回と同じ5問」と書いてある一方で、別の箇所に「問題は現在のコードベースに合わせて更新済み」と書いてあった。

足軽は「更新済み」と書いてある方に従って、5問中3問を勝手に差し替えてしまった。「同一」と「更新」が矛盾しているのに、誰も気づかなかったんです。将軍が書いた指示なのに。

この件で学んだのは、「AIは矛盾に気づかない」ということです。人間なら「同じって言ってるのに更新済みってどういうこと？」と聞き返すけど、AIは黙って解釈して進めてしまう。

## 成果

運用開始から3ヶ月で、エージェントチームは以下をこなしました：

- 約200のコマンドを処理
- 切り抜き動画のパイプラインを自動化（YouTube DL→音声分離→文字起こし→話者識別→字幕→サムネイル→アップロード）
- 漫画風ショート動画の半自動生成パイプラインを構築
- セキュリティチェック機構（スクリプト実行のブロックリスト）を自ら実装

手作業に比べて、動画1本あたりの制作時間は2〜3時間から15〜30分に短縮されました。品質は……まだ人間の手直しが必要ですが、8割方は自動でいけています。

## 感想

AIエージェントチームを作って一番驚いたのは、「AI同士のコミュニケーション設計」が一番難しいということです。AIの技術力（コードを書く能力）よりも、AI同士がどう情報を伝えるか、どう矛盾を検知するか、どう知見を共有するか——こういう組織設計の課題に一番時間をかけました。

「AIに仕事を任せる」というのは、単にプロンプトを投げることではありません。仕事の流れを設計して、ミスを検知する仕組みを作って、教訓を蓄積する組織を作ること。人間の組織運営と根本的には同じなんだなと思いました。

AIエージェントの技術は進化が速いので、半年後にはもっと簡単になっているかもしれません。でも「組織設計」の部分は、技術がどう変わっても変わらない課題なんじゃないかと思います。
