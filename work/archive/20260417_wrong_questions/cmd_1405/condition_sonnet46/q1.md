# Q1: 実バグ分析 — pretool_check.sh blacklistチェックのすり抜け問題

- 終了時刻: 2026-04-17 12:23:14

## 問題文

pretool_check.shのblacklistチェックで、パス付きスクリプト名（python3 /full/path/to/banned.py）がすり抜ける問題の原因分析+修正案

---

## 原因分析

### コードの確認

`pretool_check.sh` の blacklist チェック部分（〜156〜205行）を精査する。

**ファイル名抽出パターン（158行目）**:

```bash
PY_FILES=$(echo "$COMMAND" | grep -oE '[^/[:space:]]+\.py' | grep -v '__pycache__')
```

この正規表現 `[^/[:space:]]+\.py` は「`/`とスペース以外の連続文字 + `.py`」にマッチする。
`python3 /full/path/to/banned.py` に対しては `banned.py`（ベースネームのみ）が抽出される。

**blacklist照合（163〜167行目）**:

```bash
for PY_FILE in $PY_FILES; do
  if [[ -f "$BLACKLIST" ]] && grep -qxF "$PY_FILE" "$BLACKLIST"; then
    echo "BLOCKED: ${PY_FILE} はblacklistに登録された使用禁止スクリプトです。" >&2
    exit 2
  fi
done
```

`grep -qxF` は `-x`（行全体一致）+ `-F`（固定文字列）による完全一致照合。

---

### バグ1（主要）: blacklist登録形式と抽出形式の不一致

抽出側は常にベースネームを返すが、blacklist の登録形式が制約されていない。

| blacklist登録形式 | 実行コマンド | 抽出値 | 照合結果 |
|---|---|---|---|
| `banned.py` | `python3 banned.py` | `banned.py` | BLOCKED ✓ |
| `banned.py` | `python3 /full/path/to/banned.py` | `banned.py` | BLOCKED ✓ |
| `/full/path/to/banned.py` | `python3 /full/path/to/banned.py` | `banned.py` | **すり抜け** ✗ |
| `scripts/banned.py` | `python3 scripts/banned.py` | `banned.py` | **すり抜け** ✗ |

現在の `config/script_blacklist.txt` はベースネーム登録なので実害はないが、パス付き登録をした瞬間に機能不全になる「暗黙の前提」が存在する。設計としてこの前提は文書化されていない。

---

### バグ2（副次）: Python フラグ前置でのすり抜け

検出フィルタの正規表現（156〜157行目）:

```bash
if echo "$COMMAND" | grep -qE 'python3[[:space:]]+[^-][^[:space:]]*\.py' && \
   ! echo "$COMMAND" | grep -qE 'python3[[:space:]]+-[cm]|venv/|pip'; then
```

`[^-]` は「python3 直後の最初のトークンがハイフンで始まらないこと」を要求する。
→ `python3 -u /full/path/to/banned.py` の場合、`[^-]` が `-` に失敗し、**ブロック全体がスキップ**される。

意図的なバイパス手法として悪用できる:
```bash
python3 -u /path/to/banned.py       # -u: unbuffered
python3 -W ignore banned.py         # -W: warnings
python3 -O banned.py                # -O: optimize
```

---

### バグ3（潜在）: 引用符によるパス変形

```bash
python3 "/full/path/to/banned.py"
```
このケースで `grep -oE '[^/[:space:]]+\.py'` は `banned.py"` を抽出する可能性がある（末尾の `"` が `\.py` の後ろに来るため、実際は `banned.py` がマッチする。ただし引用符の位置によっては誤抽出リスクがある）。

---

### 結論

**主原因**: blacklist 登録形式（フルパス可）と抽出形式（ベースネーム固定）の不整合。暗黙の前提に依存した設計。  
**副原因**: Python フラグが先行するコマンドの検出漏れ。`[^-]` チェックが最初の引数のみを見ている。

---

## 修正案

### 修正案1（推奨）: 照合時のベースネーム正規化

blacklist の各エントリをベースネームに正規化してから比較。登録形式に依存しなくなる。

```bash
for PY_FILE in $PY_FILES; do
  if [[ -f "$BLACKLIST" ]]; then
    PY_BASENAME="${PY_FILE##*/}"
    while IFS= read -r bl_entry; do
      [[ -z "$bl_entry" || "$bl_entry" == \#* ]] && continue
      bl_base="${bl_entry##*/}"
      if [[ "$PY_BASENAME" == "$bl_base" ]]; then
        echo "BLOCKED: ${PY_BASENAME} はblacklistに登録された使用禁止スクリプトです。" >&2
        echo "config/script_blacklist.txt を参照。代替スクリプトを使用してください。" >&2
        exit 2
      fi
    done < "$BLACKLIST"
  fi
done
```

### 修正案2（補完）: フラグ付き実行の検出強化

```bash
# 変更前
if echo "$COMMAND" | grep -qE 'python3[[:space:]]+[^-][^[:space:]]*\.py'

# 変更後: フラグをスキップして .py を探す（任意個のフラグ対応）
if echo "$COMMAND" | grep -qP 'python3\b.*\b\S+\.py\b' && \
   ! echo "$COMMAND" | grep -qE 'python3[[:space:]]+-[cm]|venv/|pip'; then
```

### 修正案3（運用）: blacklist登録ルールの明文化

最小コストで即実施できる再発防止策。`config/script_blacklist.txt` にコメント追記:

```
# 登録ルール: ベースネームのみで登録すること（パスを含めない）
# OK: generate_odai_panels.py
# NG: scripts/generate_odai_panels.py / /full/path/to/generate_odai_panels.py
generate_odai_panels.py
generate_qnly_agree.py
```
