## 問題文

複数ファイルリファクタリング設計: inbox_write.sh + inbox_watcher.sh の統合案設計

---

## 現状分析

### inbox_write.sh の責務

1. **メッセージ書き込み**: YAML inbox ファイルへの flock 排他書き込み
2. **cmd status 自動更新**: `task_assigned` 型でcmd pending→in_progress 更新
3. **task done 自動更新**: `report_completed` 型でtask assigned→done 更新
4. **オーバーフロー保護**: 50件超え時のメッセージ間引き
5. **inotifywait 向け touch**: atomic rename 後に touch でイベント強制発火

### inbox_watcher.sh の責務

1. **ファイル変更監視**: `inotifywait` でインボックス変更を検知
2. **未読カウント確認**: unread メッセージ数を確認
3. **wake-up シグナル送信**: `tmux send-keys` でエージェントを起動
4. **エスカレーション**: 0〜2分→通常nudge / 2〜4分→Escape + nudge / 4分→/clear 送信
5. **busy/idle 判定**: agent_status.sh に基づきスキップ判定

### 問題点・設計上の課題

1. **責務の重複**: inbox_write.sh と inbox_watcher.sh は両者ともメッセージ関連インフラだが、インタフェースが分散している。新しいエージェントを追加する際に両ファイルを修正する必要がある。

2. **cmd status 更新ロジックの散在**: inbox_write.sh 内の `_update_cmd_status()` と `_update_task_done()` は YAML の直接操作を含む副作用であり、「メッセージ書き込み」という主責務から逸脱している。

3. **エラー伝播の非対称性**: inbox_write.sh は「メッセージ書き込み失敗=致命的」として3回リトライするが、status 更新失敗は `return 0` で握り潰す。挙動の一貫性がない。

4. **cli_adapter.sh への依存が暗黙的**: inbox_watcher.sh は `source lib/cli_adapter.sh` を実行するが、失敗してもサイレントに続行する。エラー診断が困難。

5. **Python 3 依存**: inbox_write.sh の YAML 書き込みは `.venv/bin/python3` に依存している。venv が壊れた場合にメッセージング全体が停止する。

---

## 統合設計案

### 設計方針

- **責務の明確化**: 「書く」「届ける」「状態更新する」を3つの独立したレイヤーに分離
- **後方互換性維持**: 既存の呼び出し形式 `bash scripts/inbox_write.sh <target> <content> [type] [from]` を維持
- **単一の lib ファイルで共通ロジックを管理**

---

### 提案アーキテクチャ

```
lib/
  inbox_core.sh        ← 新規: 共通ロジック（ロック・YAML操作・型定義）
  agent_status.sh      ← 既存: busy/idle判定
  cli_adapter.sh       ← 既存: CLI種別判定

scripts/
  inbox_write.sh       ← 薄くなる: lib/inbox_core.sh を source して書き込みのみ
  inbox_watcher.sh     ← 薄くなる: lib/inbox_core.sh を source して監視のみ
```

# lib/inbox_core.sh — inbox システム共通ロジック

REPO_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"

# --- 定数 ---
MAX_INBOX_MESSAGES=50
INBOX_DIR="${REPO_DIR}/queue/inbox"
TASK_DIR="${REPO_DIR}/queue/tasks"
CMD_YAML="${REPO_DIR}/queue/shogun_to_karo.yaml"

# --- ロック取得/解放 ---
inbox_acquire_lock() {
    local lockfile="$1"
    if command -v flock &>/dev/null; then
        exec 200>"$lockfile"
        flock -w 5 200 || return 1
    else
        local lock_dir="${lockfile}.d"
        local i=0
        while ! mkdir "$lock_dir" 2>/dev/null; do
            sleep 0.1
            ((i++)) && [ $i -ge 50 ] && return 1
        done
    fi
}

inbox_release_lock() {
    local lockfile="$1"
    if command -v flock &>/dev/null; then
        exec 200>&-
    else
        rmdir "${lockfile}.d" 2>/dev/null

}

### inbox_write.sh（リファクタ後）

```bash
#!/usr/bin/env bash
# inbox_write.sh — メッセージ書き込み（薄いラッパー）
set -e
source "$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)/lib/inbox_core.sh"

TARGET="$1" CONTENT="$2" TYPE="${3:-wake_up}" FROM="${4:-unknown}"
[ -z "$TARGET" ] || [ -z "$CONTENT" ] && { echo "Usage: inbox_write.sh <target> <content> [type] [from]" >&2; exit 1; }

INBOX="${INBOX_DIR}/${TARGET}.yaml"
[ ! -f "$INBOX" ] && { mkdir -p "$(dirname "$INBOX")"; echo "messages: []" > "$INBOX"; }

MSG_ID="msg_$(date +%Y%m%d_%H%M%S)_$(head -c 4 /dev/urandom | xxd -p)"
TIMESTAMP=$(date "+%Y-%m-%dT%H:%M:%S")

# 書き込み（3リトライ）
for attempt in 1 2 3; do
    if inbox_acquire_lock "${INBOX}.lock"; then
        inbox_write_yaml "$INBOX" "$MSG_ID" "$FROM" "$TIMESTAMP" "$TYPE" "$CONTENT" && STATUS=0 || STATUS=1
        [ $STATUS -eq 0 ] && touch "$INBOX" 2>/dev/null
        inbox_release_lock "${INBOX}.lock"
        if [ $STATUS -eq 0 ]; then
            inbox_update_cmd_status "$TYPE" "$CONTENT"
            inbox_update_task_done "$TYPE" "$CONTENT" "$FROM"
            exit 0
        fi
    fi
    [ $attempt -lt 3 ] && sleep 1
done
exit 1
```

### リファクタリングのメリット

| 観点 | 現状 | 統合後 |
|---|---|---|
| 共通ロジック | 2ファイルに重複 | `lib/inbox_core.sh` に一元化 |
| 新エージェント追加 | write + watcher 両方修正 | `inbox_core.sh` のみ修正 |
| テスタビリティ | __INBOX_WATCHER_TESTING__ フラグ | 関数ごとに独立テスト可 |
| Python 依存の局所化 | write.sh 内に混在 | `inbox_yaml_write.py` に集約 |
| status 更新の副作用 | write.sh 内に隠れている | `inbox_core.sh` に明示的に存在 |

---

### 移行手順

1. `lib/inbox_core.sh` を新規作成し、既存の `_update_cmd_status()` / `_update_task_done()` / ロックロジックを移植
2. `inbox_write.sh` を `lib/inbox_core.sh` を source する薄いラッパーに書き換え
3. `inbox_watcher.sh` の共通ロジックを `lib/inbox_core.sh` に移動
4. 既存の呼び出し形式（`bash scripts/inbox_write.sh <target> <content> [type] [from]`）はそのまま維持
5. テスト: `__INBOX_WATCHER_TESTING__=1` 相当の unit test で各関数を検証

### 注意点

- `inbox_watcher.sh` の inotifywait ループ・エスカレーション・tmux send-keys ロジックは複雑なため、統合より「lib への共通化」にとどめる。全統合（1ファイル化）は over-engineering。
- Python venv 依存の解消（bash native YAML 書き込み）は別 cmd で検討。
