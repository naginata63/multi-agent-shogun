## 問題の根本原因

### 原因1（主要）: 504 が RETRYABLE_ERRORS に含まれていない

```python
RETRYABLE_ERRORS = ("429", "RESOURCE_EXHAUSTED", "503")
```

`504 DEADLINE_EXCEEDED` は `RETRYABLE_ERRORS` に `"504"` も `"DEADLINE_EXCEEDED"` も含まれていない。

結果: 504 エラーが発生すると `else: return result` または `else: return {"error": error_str}` で即失敗扱いになり、リトライされない。

### 原因2: タイムアウト設定とVertex APIの画像生成時間の不一致

```python
_http_opts = types.HttpOptions(timeout=120_000)  # 2分
```

Imagen 3 は高品質プロンプトで 60〜180 秒かかることがある。2分タイムアウトは画像生成の「ギリギリ」であり、サーバー側の遅延（GCS アクセス・複数パネル並列負荷等）で容易に超過する。

504 は「サーバーが Vertex API バックエンドから期限内に応答を取得できなかった」ことを示す。つまり:
- クライアント timeout (120s) < Vertex サーバー内部処理時間 → 504

### 原因3: exponential backoff の初回 wait が長すぎる

```python
BASE_WAIT = 120  # 秒
```

429/503 に対して120秒待ちは適切だが、504 の場合は「一時的な過負荷」が多く、10〜30秒後にリトライすれば成功することが多い。504 専用の短い backoff が必要。

### 原因4: 全パネルを逐次処理している可能性

generate_panel_with_retry() を パネルごとに呼び出す構造の場合、失敗パネルが全体をブロックする。また、逐次呼び出しは1回の API リクエストが失敗すると後続パネルも遅延する。

---

## 根本対策設計

### 対策1（必須・即実施）: RETRYABLE_ERRORS に 504 と DEADLINE_EXCEEDED を追加

```python
# 変更前
RETRYABLE_ERRORS = ("429", "RESOURCE_EXHAUSTED", "503")

# 変更後
RETRYABLE_ERRORS = ("429", "RESOURCE_EXHAUSTED", "503", "504", "DEADLINE_EXCEEDED")
```

これだけで「504 が出てもリトライされない」問題は解消する。

### 対策2（推奨）: タイムアウト値を画像生成の実測に合わせて延長

```python
# 変更前
_http_opts = types.HttpOptions(timeout=120_000)  # 2分

# 変更後
_http_opts = types.HttpOptions(timeout=300_000)  # 5分
```

Imagen 3 の公式推奨待機時間（最大 180〜240 秒）+ バッファ。5分は過剰ではない。
合わせて `generate_panel()` 内の subprocess timeout も見直す（現在 `timeout=60` が散在）。

### 対策3（推奨）: エラー種別ごとの backoff 分岐

```python
def _get_retry_wait(error_str: str, attempt: int) -> float:
    """エラー種別に応じた backoff 時間を返す。"""
    jitter = random.uniform(0, 5)
    if "429" in error_str or "RESOURCE_EXHAUSTED" in error_str:
        # レート制限: 長めのwait
        return min(BASE_WAIT * (2 ** attempt), MAX_WAIT) + jitter
    elif "504" in error_str or "DEADLINE_EXCEEDED" in error_str:
        # タイムアウト: 短めのwait（短期的な過負荷対策）
        return min(30 * (2 ** attempt), 120) + jitter
    elif "503" in error_str:
        # サービス一時不可: 中程度のwait
        return min(60 * (2 ** attempt), 300) + jitter
    else:
        return min(BASE_WAIT * (2 ** attempt), MAX_WAIT) + jitter
```

### 対策4（推奨）: 504 発生時の per-panel ステータス管理とスキップ再開

504 が多発する環境では、パネルごとにステータスファイルを書き出し、中断後に再開できる仕組みが重要:

```python
PANEL_STATUS_FILE = panels_json_dir / "generation_status.json"

def load_panel_status(panels_json_dir: Path) -> dict:
    """既存のパネルステータスを読み込む（再開用）。"""
    try:
        return json.loads(PANEL_STATUS_FILE.read_text())
    except (FileNotFoundError, json.JSONDecodeError):
        return {}

def save_panel_status(panels_json_dir: Path, panel_id: str, status: str):
    """パネルごとのステータスをファイルに保存。"""
    data = load_panel_status(panels_json_dir)
    data[panel_id] = {"status": status, "timestamp": time.time()}
    PANEL_STATUS_FILE.write_text(json.dumps(data, ensure_ascii=False, indent=2))
```

使用例:
```python
for panel in panels:
    existing = panel_status.get(panel["id"], {})
    if existing.get("status") == "success":
        print(f"  Skip {panel['id']} (already done)")
        continue
    result = generate_panel_with_retry(client, panel, ...)
    save_panel_status(panels_json_dir, panel["id"], result["status"])
```

`--skip-gen` フラグと組み合わせることで、504 多発後の部分再試行が可能になる。

### 対策5（オプション）: Vertex AI のリージョン・モデルのフォールバック

504 は特定リージョンや特定モデルバージョンに集中することがある。

```python
VERTEX_LOCATIONS_FALLBACK = ["global", "us-central1", "europe-west4"]

def create_client_with_fallback():
    """フォールバック付きクライアント生成。"""
    for location in VERTEX_LOCATIONS_FALLBACK:
        try:
            client = genai.Client(
                vertexai=True, project=VERTEX_PROJECT, location=location,
                http_options=types.HttpOptions(timeout=300_000)
            )
            # 軽量テストリクエストで生存確認
            return client
        except Exception:
            continue
    raise RuntimeError("All Vertex AI locations failed")
```

## 補足: 504 の種類と対処の違い

| パターン | 原因 | 対処 |
|---|---|---|
| クライアント側タイムアウト | `http_options.timeout` が短すぎる | タイムアウト延長（対策2） |
| サーバー側タイムアウト | Vertex API バックエンド過負荷 | リトライ + shorter backoff（対策1+3） |
| ネットワーク切断 | NW 不安定 | リトライ（対策1で対応済み） |
| クォータ超過による遅延 | 同時リクエスト過多 | バッチサイズ削減・並列度制限 |
