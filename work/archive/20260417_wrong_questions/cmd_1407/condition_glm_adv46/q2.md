### 回答

#### 現在の責務分離の設計意図

**inbox_write.sh**（255行）:
- メッセージ永続化: YAMLファイルへのatomic write（flock + tmpfile + rename）
- 副作用: cmd status自動更新（pending→in_progress）、task done自動更新（assigned→done）
- オーバーフロー保護: 50件上限（未読優先保持）
- 自己完結: 呼び出し側は `inbox_write.sh target "message" type from` だけで配信完了

**inbox_watcher.sh**（1263行）:
- イベント検知: inotifywait/fswatch でファイル変更を監視
- 起動シグナル配信: tmux send-keys で`inboxN`通知
- エスカレーション: 未読放置時の段階的対応（Phase1: nudge → Phase2: Escape+nudge → Phase3: /clear）
- CLI種別対応: claude/codex/copilot/kimi/geminiの差分吸収
- セルフウォッチ対応: エージェント自身のinotifywait検出 → nudge不要判定

設計意図は**Writer/Watcher分離**。Writerは純粋なデータ操作、Watcherはインフラ層（プロセス管理・シグナル配信）。

#### 統合した場合のメリット

1. **デプロイ簡素化**: 2スクリプト→1スクリプトで依存管理が容易
2. **設定一元化**: CLI種別・エスカレーション閾値・ロック機構の設定が1箇所に集約
3. **テスト容易性**: 統合テストが単一プロセスで完結

#### 統合した場合のデメリット（致命的）

1. **単一障害点**: watcherプロセスが落ちると書き込みも配信も停止。現在はwriteが独立しているためwatcher障害でもメッセージ保存は可能
2. **リソース競合**: write（同期的）とwatch（イベントループ）が同じプロセスで競合。inotifywaitのブロック中にwrite要求が来るとデッドロックリスク
3. **責務過多**: 1263+255=1518行の単一スクリプトは保守困難
4. **呼び出し側への影響**: 現在`bash inbox_write.sh`は瞬時終了。統合するとwatcherループのオーバーヘッドが発生

#### 推奨: 統合しない

Writer/Watcher分離はUnix設計原則（do one thing well）に合致。現在の設計は意図的で合理的。

改善すべき点:
1. **共通設定の外部化**: CLI種別・閾値を`config/inbox_settings.sh`に外出し（DRY原則）
2. **共通ライブラリ整理**: `lib/agent_status.sh`の重複インクルード解消
3. **テストカバレッジ向上**: inbox_write.shにはbatsテストがあるがwatcher.shにはない

#### もし統合する場合の設計案（参考）

```bash
#!/bin/bash
# inbox_manager.sh — 統合版
MODE="${1:-daemon}"

case "$MODE" in
  write)
    shift
    _do_write "$@"  # inbox_write.shの全機能
    ;;
  daemon)
    AGENT_ID="$2" PANE_TARGET="$3"
    _do_watch "$AGENT_ID" "$PANE_TARGET"  # watcherメインループ
    ;;
esac
```
