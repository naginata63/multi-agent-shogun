# Q2: inbox_write.sh + inbox_watcher.sh の統合案設計

## 現状分析

### inbox_write.sh（255行）
- **役割**: メッセージの永続化（inbox YAMLへの原子書き込み）
- **機能**:
  - flock/mkdir による排他ロック
  - Python subprocess でYAML追記（原子書き込み: tmp → os.replace）
  - オーバーフロー保護（max 50件、未読優先保持）
  - `cmd_new` 時のRAGフック起動
  - `task_assigned` 時の shogun_to_karo.yaml status自動更新
  - `report_done` 時の task YAML status自動更新

### inbox_watcher.sh（1263行）
- **役割**: メッセージ変更検知 → エージェント起床シグナル配信
- **機能**:
  - inotifywait/fswatch によるイベント駆動監視
  - summary-first 高速パス（未読件数のみ取得）
  - エスカレーション（0-2分: 通常nudge → 2-4分: Escape+nudge → 4分+: /clear）
  - CLI種別別コマンド変換（claude/codex/copilot/kimi/gemini）
  - busy/idle 検出（flag file + pane直接確認）
  - self-watch 検出（エージェント自身のinotifywait認識）
  - auto-recovery（/clear後のtask_assigned自動投入）
  - metrics収集（read_count, bytes_read, estimated_tokens）

### 結合度
- write → watcher: ファイルシステム経由のみ（`touch` でinotifyイベント補助）
- watcher → write: 依存なし（一方通行）
- 共通データ: inbox YAML形式のみ

## 統合案: inbox_manager.sh（単一プロセス）

### 設計方針

1. **write を関数化**: inbox_write の全ロジックを関数として統合
2. **watcher メインループを維持**: イベント駆動 + タイムアウトフォールバック
3. **共有状態をメモリ化**: flock → プロセス内変数（ただし外部呼び出し用にYAML I/Fは残す）
