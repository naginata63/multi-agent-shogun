# Q1: subtitle_semantic_index.pyのmmap化で検索が壊れた件の原因分析+修正案

## 原因分析

現在の `subtitle_semantic_index.py` は既にBigQuery版に移行済み（docstringに「ローカルFAISSインデックス廃止」と明記）。mmap化で検索が壊れたのは旧FAISSローカル版の問題である。

### 根本原因（3層）

1. **ゼロベクトル混入によるインデックス汚染**
   - `embed_texts()`（L108-149）でAPI呼び出しが失敗した場合、フォールバックとして `[0.0] * EMBED_DIM`（ゼロベクトル）が返される（L143-144, L146-148）
   - 旧FAISS版ではこれらのゼロベクトルがそのままインデックスに登録されていた
   - FAISSのL2/Cosine検索はゼロベクトルに対して距離0（完全一致）を返すため、検索結果の上位がゼロベクトルチャンクで埋まり、実質的な検索機能が壊灭
   - **現在のBigQuery版でも対策済み**: `is_zero_vector()` チェック（L152-156）→ BQ投入前スキップ（L312-315）→ `health` コマンドでゼロベクトル検出（L419-466）

2. **mmap（メモリマップトファイル）の読み書き競合**
   - FAISSインデックスを `faiss.write_index()` で保存 → `faiss.read_index(filename, io_flags=faiss.IO_FLAG_MMAP)` でmmap読み込みする構成だったと推測
   - `build`/`update` 実行中にインデックスファイルを書き換えると、mmapで読み込んでいるプロセスがセグフォまたは古いデータを参照
   - Linuxのmmap semanticsでは、ファイルサイズが変更された場合の動作は未定義（POSIX規格）

3. **チャンクIDの不一致**
   - 旧版では `global_chunk_id` を変数で管理（L289参照可能）。`build` → `update` の間にインデックスを再構築すると、チャンクIDとインデックス位置の対応がずれ、検索結果の `chunk_id` が間違ったチャンクを指す

## 修正案（現在のBigQuery版で採用されている解決策）

| 問題 | BigQuery版での解決策 | 該当コード |
|------|---------------------|-----------|
| ゼロベクトル汚染 | `is_zero_vector()` で検出 → BQ投入前スキップ + healthチェックコマンド | L152-156, L302-315, L419-466 |
| 読み書き競合 | BigQueryに移行（サーバー管理のストレージ、トランザクショナルなINSERT） | `bq_insert_rows()` L237-253 |
| チャンクID不一致 | BigQueryの自動採番 + レジストリ管理 | L316-317, L333-339 |
| 検索品質 | `VECTOR_SEARCH()` + COSINE距離でセマンティック検索 | L371-387 |

### もしFAISS版を維持する場合の修正案

1. **ゼロベクトルガード**: embedding生成後に `is_zero_vector()` で検出し、インデックス登録をスキップ
2. **Read-Write分離**: 書き込み時は一時ファイル（`index.tmp`）に書き出し、完了後に `os.replace()` でアトミックリネーム。読み込み側は常に完成したインデックスを参照
3. **バージョン管理**: インデックスファイルにバージョン番号を付与（`index_v3.faiss`）。mmapハンドルをバージョン番号で管理し、古いバージョンを参照中のプロセスがあればgracefulに切り替え
