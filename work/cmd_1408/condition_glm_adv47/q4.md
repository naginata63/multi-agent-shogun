# Q4: ffmpegサンプルレート不一致問題（44100Hz vs 48000Hz concat）の根本対策設計

## 問題概要

複数音声ファイルをffmpegで連結（concat）する際、サンプルレートが混在（44100Hzと48000Hz）していると、以下の問題が発生する:

1. **音ズレ**: concat demuxerはサンプルレート変換を行わないため、44100Hzの区間が48000Hzとして扱われ、ピッチと速度がずれる
2. **ノイズ**: 連結境界で波形の不連続が生じ、クリックノイズが発生
3. **長さ不一致**: 動画（48000Hz）と音声（44100Hz）のmux時に同期が崩れる

## 根本原因

ffmpegのconcatには3つの方式があり、それぞれサンプルレートの扱いが異なる:

| 方式 | サンプルレート扱い | 問題 |
|------|-------------------|------|
| concat protocol (`file1.mp3|file2.mp3`) | 最初のファイルに統一 | 2番目以降が音ズレ |
| concat demuxer (`-f concat -i list.txt`) | 変換なし（混在そのまま） | 再生時にノイズ |
| filter_complex (`[0][1]concat=n=2`) | 共通フォーマットに自動変換 | 安全だが全ストリームを再エンコード |

## 根本対策設計

### 方針: 「入力正規化 → 連結」の2段構え

全ての音声を連結**前**に統一フォーマットに正規化する。連結後の変換に頼らない。

### 実装: normalize_audio_before_concat()

```python
import subprocess
from pathlib import Path

# プロジェクト標準フォーマット
TARGET_SAMPLE_RATE = 48000
TARGET_CHANNELS = 1  # モノラル（音声認識用）
TARGET_CODEC = "pcm_s16le"  # WAV（Demucs入力用）

def normalize_audio(input_path: Path, output_path: Path,
                    sample_rate: int = TARGET_SAMPLE_RATE,
                    channels: int = TARGET_CHANNELS) -> Path:
    """音声ファイルを統一フォーマットに正規化"""
    cmd = [
        "ffmpeg", "-y", "-i", str(input_path),
        "-ar", str(sample_rate),      # サンプルレート統一
        "-ac", str(channels),         # チャンネル数統一
        "-c:a", TARGET_CODEC,         # コーデック統一
        "-af", "aresample=flags=res:cutoff=1.0",  # 高品質リサンプリング
        str(output_path)
    ]
    subprocess.run(cmd, check=True, capture_output=True)
    return output_path
```

### 適用ポイント（3箇所）

既存コードのサンプルレート問題が潜む箇所（実コードベースでの確認推奨）:

1. **DL直後の音声抽出**: yt-dlpでDLした動画から音声抽出時に `-ar 48000` を付与
2. **Demucs出力の正規化**: Demucsは入力のサンプルレートを維持するが、複数トラックを連結する場合は事前正規化
3. **concat前のバリデーション**: concat実行前に全入力ファイルのサンプルレートをffprobeで検証

### バリデーション関数

```python
import json

def validate_sample_rates(audio_files: list[Path],
                          expected_rate: int = TARGET_SAMPLE_RATE) -> list[str]:
    """連結前のサンプルレート検証"""
    mismatches = []
    for f in audio_files:
        result = subprocess.run(
            ["ffprobe", "-v", "quiet", "-print_format", "json",
             "-show_streams", str(f)],
            capture_output=True, text=True
        )
        info = json.loads(result.stdout)
        for stream in info.get("streams", []):
            if stream.get("codec_type") == "audio":
                rate = int(stream.get("sample_rate", 0))
                if rate != expected_rate:
                    mismatches.append(f"{f.name}: {rate}Hz (expected {expected_rate})")
    return mismatches
```

### 運用ルール

1. **48000Hz統一**: プロジェクト全体のサンプルレートを48000Hzに固定。44100Hzの入力は即座にリサンプリング
2. **正規化のタイミング**: 「DL直後」または「concat直前」のどちらかで必ず実行
3. **CLAUDE.md記載**: ffmpeg Encoding Ruleに「音声サンプルレート: 48000Hz統一（-ar 48000必須）」を追加すべき
4. **CI検証**: 音声を含む成果物のサンプルレートを自動チェック（`ffprobe` + assert）

### なぜ concat filter で解決しないのか

`filter_complex`の`concat`は自動的にサンプルレートを統一するが:
- 全ストリームの再エンコードが必要（GPU/NVENCは音声に無関係 = CPU負荷）
- 長尺動画（30分+）では再エンコードに数分かかる
- メモリ使用量が入力ファイル数に比例して増大

入力段階での正規化は一回きりの処理で済み、concat demuxer（コピー方式）の高速性を活かせる。
