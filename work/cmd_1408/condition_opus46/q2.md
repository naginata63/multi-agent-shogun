## 統合設計

### 設計方針

2つのスクリプトは**責務が明確に分離**されており、安易に1ファイルにマージすべきではない：
- inbox_write.sh = **同期的なメッセージ書き込みAPI**（呼び出し元は全エージェント）
- inbox_watcher.sh = **非同期デーモン**（各エージェントに1プロセス常駐）

統合するなら、**共通ロジックの抽出** + **明確なレイヤー分離**が鍵。

### 統合による改善ポイント

| 現状の問題 | 統合後の解決 |
|-----------|-------------|
| inbox_writeとwatcherでYAML読み書きロジックが重複 | inbox_ops.shに一本化 |
| watcher内のPython埋め込みが3箇所（get_unread_count, get_unread_info, enqueue_recovery）に散在 | inbox_ops.shのPython関数に集約 |
| flock取得パターンの重複（write: 3リトライ、watcher: flock+fallback）| inbox_ops.shで統一ロックAPI |
| cli_adapter.shの依存がwatcher側にしかない | daemon層のみに限定（明確な依存方向） |
| テスト困難（1263行のmonolith） | 各Layer単体でテスト可能 |

### 移行計画

1. **Phase 1**: lib/inbox_ops.sh を作成し、inbox_write.shのロジックを移植。inbox_write.shはinbox_ops.shを呼ぶ薄いラッパーに。**後方互換性維持**（既存の呼び出し元は変更不要）
2. **Phase 2**: inbox_watcher.shのget_unread_info/get_unread_count/enqueue_recovery_task_assignedをinbox_ops.sh呼び出しに切替
3. **Phase 3**: inbox_watcher.shをinbox_daemon.shにリネーム。テストコード（bats）の更新

### リスク評価

- **低リスク**: Phase 1（inbox_write.shのリファクタリング、外部IF不変）
- **中リスク**: Phase 2（watcher側のPython埋め込み切替、flock挙動の微妙な差異に注意）
- **高リスク**: Phase 3（デーモンリネーム、全shutsujin_departure.sh起動パスの更新必要）

### 統合しない方がよい部分

- **エスカレーションロジック**: daemon固有。inbox_ops.shに入れるべきでない
- **CLI種別判定・send-keys送信**: daemon固有のUI操作層
- **inotifywait/fswatchメインループ**: daemon固有のイベントループ

これらはinbox_daemon.sh内に留めるべき。無理に共通化すると、inbox_write.sh経由の同期呼び出しにデーモン依存が入り込み、単体テストが困難になる。

## 結論

inbox_write.shとinbox_watcher.shの統合は、**共通操作のライブラリ化（lib/inbox_ops.sh）** + **各エントリポイントのスリム化**が最適解。1ファイルにマージするのではなく、3層に分離して共通ロジックを中間層に抽出する。後方互換性を維持しながら段階的に移行でき、テスタビリティも大幅に向上する。
