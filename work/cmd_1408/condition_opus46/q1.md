# Q1: subtitle_semantic_index.pyのmmap化で検索が壊れた件の原因分析+修正案

## 問題の原初コンテキスト

この問題は「subtitle_semantic_index.pyのmmap化で検索が壊れた」というものだが、現在のコードを読むと、**mmap化はすでに完全に撤去されている**。現行のsubtitle_semantic_index.pyはBigQuery版に移行済みであり、ローカルFAISSインデックスもmmapも一切使用していない。

## 原因分析

mmapを使ったFAISSインデックスで検索が壊れる典型的な原因は以下の通り：

### 1. mmapモードとインデックス構造の不整合

FAISSの`faiss.read_index(path, faiss.IO_FLAG_MMAP)`でmmap読み込みする場合、インデックスの種類によっては正しく動作しない。具体的には：

- **IVF系インデックス**（`IndexIVFFlat`等）: mmapモードではnprobe探索時にメモリアクセスパターンがランダムになり、ページフォルトが大量発生。検索速度が著しく低下するだけでなく、probeされないセルが存在するため**検索結果が欠落**する。
- **Flat系インデックス**（`IndexFlatL2`等）: 全ベクトルに対する総当たり検索のため、mmapでも理論上は正しい結果が得られるが、大量のページフォルトでI/O待ちが発生し実用に耐えない。

### 2. ファイルの部分的なmmap化による破損

mmapでファイルをマッピングした状態で、別プロセス（build/updateコマンド）がインデックスファイルを上書きすると：

- mmapされたメモリ領域が新ファイルの内容と一致しなくなる
- 旧ファイルのinodeが解放されず、mmapは古いデータを参照し続ける
- 結果として、検索クエリに対して**古い（または破損した）ベクトルが返る**

### 3. エンディアン・アライメント問題

mmapはファイルのバイト列をそのままメモリにマッピングするため、ベクトルデータの書き込み時のバイトオーダーと読み込み時のバイトオーダーが一致していないと、浮動小数点数の値が化ける。NumPyのndarray.tofile()とnp.fromfile()の組み合わせなら問題ないが、FAISSの内部フォーマットを直接mmapすると、ヘッダオフセットの計算ミスでベクトル領域がずれる。

## 推定される具体的バグシナリオ

元のコード（ローカルFAISS版、現存しない）では以下の流れだったと推測：

```
1. build: merged_*.jsonからchunk化 → embedding生成 → FAISSインデックスに追加 → faiss.write_index(path)
2. search: faiss.read_index(path, faiss.IO_FLAG_MMAP) → query embedding → search
3. update: 新規merged_*.jsonを追加 → FAISSインデックスにadd → faiss.write_index(path) ← 同じpath
```

ステップ3でファイルを上書きした瞬間、ステップ2のmmapされたインデックスが無効化される。FAISSのwrite_indexは`tmp+rename`パターンを使わないため、書き込み中に読み込みが走ると中途半端なデータが返る。

## 修正案

### 案A: mmapを使わない（現行コードが採用した解決策）

現行のBigQuery版では、ローカルにベクトルを保持せず、検索時はBigQuery VECTOR_SEARCH() APIを呼び出すのみ。mmapの問題を根本的に解消している。

**メリット**: mmapの問題が完全に消える。スケーラビリティも向上。
**デメリット**: 検索のたびにBigQueryへのAPI呼び出しが発生（レイテンシ増加、コスト発生）。

### 案B: mmapを使いつつ安全化する（仮にローカルFAISSに戻す場合）

1. **Copy-on-Read**: mmapではなく`faiss.read_index(path)`（通常読み込み）を使い、全データをメモリに載せる
2. **Atomic Update**: インデックス更新時は`tmp_path`に書き出してから`os.replace(tmp_path, path)`でアトミックに差し替え
3. **ファイルロック**: `fcntl.flock()`でbuild/update中はsearchをブロック
4. **インデックス種別の固定**: `IndexFlatIP`（内積）に統一し、IVF系を避ける（チャンク数が数万レベルなら全探索でも十分高速）

### 案C: ハイブリッド（推奨：仮にローカルに戻す場合）

- 検索頻度が低い（1日数回）なら、mmap不要で毎回`read_index`
- 常時起動のサービスとして検索する場合は、インデックスをメモリに常駐させ、更新はUNIXシグナル（SIGHUP）で再読み込みトリガー

## 結論

mmapによる検索破損は**ファイル更新時のrace condition**と**FAISSインデックス種別とmmapの相性**が複合した問題であり、現行コードではBigQuery移行によって根本的に解消されている。ローカルFAISSに戻す必要がない限り、現行アーキテクチャが最適解。
