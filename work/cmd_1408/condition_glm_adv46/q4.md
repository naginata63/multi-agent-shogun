# Q4: ffmpegサンプルレート不一致問題（44100Hz vs 48000Hz concat）の根本対策設計

## 問題の概要

ffmpegで複数音声ファイルをconcatする際、ファイル間でサンプルレートが不一致（44100Hz vs 48000Hz等）の場合、以下の問題が発生する:

1. **音声の途切れ・ノイズ**: concatフィルタがサンプルレート変換を行わず、切り替わり点でグリッチが発生
2. **再生速度の変化**: 44100Hzのデータを48000Hzとして扱うと、音程が高くなり再生時間が短くなる
3. **concatフィルタのエラー**: 一部のffmpegバージョンでは異なるサンプルレートのストリームをconcatするとエラー

## 根本原因

ffmpegのconcatフィルタは、入力ストリームのパラメータ（サンプルレート、チャンネル数、サンプルフォーマット）が**完全に一致することを前提**としている。不一致がある場合、最初のセグメントのパラメータに合わせて以降のセグメントを無理やり結合するため、音質劣化やノイズが生じる。

## 根本対策設計

### 方針: 「結合前に統一」ではなく「結合時に自動変換」

結合前の全ファイルを事前にリサンプリングするアプローチは、一時ファイルの管理コストと処理時間がかかる。ffmpegのフィルタグラフ内でリサンプリングを完結させる。

### 設計: concat+aformat フィルタグラフ

```
[0:a]aresample=48000,asetnsamples=n=48k:p=0[a0];
[1:a]aresample=48000,asetnsamples=n=48k:p=0[a1];
[2:a]aresample=48000,asetnsamples=n=48k:p=0[a2];
[a0][a1][a2]concat=n=3:v=0:a=1[out]
```

### 実装案: safe_audio_concat() 関数

```python
import subprocess
import json

def get_audio_info(input_path: str) -> dict:
    """ffprobeで音声ストリームのパラメータを取得"""
    result = subprocess.run(
        ["ffprobe", "-v", "quiet", "-print_format", "json",
         "-show_streams", "-select_streams", "a", input_path],
        capture_output=True, text=True
    )
    streams = json.loads(result.stdout).get("streams", [])
    if not streams:
        return {"sample_rate": 0, "channels": 0, "codec_name": ""}
    s = streams[0]
    return {
        "sample_rate": int(s.get("sample_rate", 0)),
        "channels": int(s.get("channels", 0)),
        "codec_name": s.get("codec_name", ""),
    }

def safe_audio_concat(input_files: list[str], output_path: str,
                      target_rate: int = 48000, target_channels: int = 2,
                      codec: str = "aac", bitrate: str = "192k"):
    """
    サンプルレート不一致を自動解決して音声concatする。
    
    全入力をtarget_rate/target_channelsに統一してからconcatする。
    一時ファイル不要（フィルタグラフ内で完結）。
    """
    n = len(input_files)
    if n == 0:
        raise ValueError("No input files")
    if n == 1:
        # 1ファイルならリサンプリングのみ
        subprocess.run([
            "ffmpeg", "-y", "-i", input_files[0],
            "-c:a", codec, "-b:a", bitrate,
            "-ar", str(target_rate), "-ac", str(target_channels),
            output_path
        ], check=True)
        return

    # フィルタグラフ構築
    filter_parts = []
    concat_inputs = []
    
    for i in range(n):
        # 各入力を統一パラメータにリサンプリング
        filter_parts.append(
            f"[{i}:a]aresample={target_rate},"
            f"aformat=sample_rates={target_rate}:channel_layouts=stereo,"
            f"asetnsamples=n={target_rate}:p=0[a{i}]"
        )
        concat_inputs.append(f"[a{i}]")
    
    # concat
    concat_filter = "".join(concat_inputs) + f"concat=n={n}:v=0:a=1[out]"
    filter_complex = ";".join(filter_parts) + ";" + concat_filter
    
    # ffmpegコマンド構築
    cmd = ["ffmpeg", "-y"]
    for f in input_files:
        cmd.extend(["-i", f])
    cmd.extend([
        "-filter_complex", filter_complex,
        "-map", "[out]",
        "-c:a", codec, "-b:a", bitrate,
        output_path
    ])
    
    subprocess.run(cmd, check=True)
```

### テスト計画

1. **正常系**: 同一レート（48000Hz同士）のconcat → 従来と同一結果
2. **不一致系**: 44100Hz + 48000Hz の混在concat → グリッチなし
3. **極端系**: 22050Hz + 48000Hz + 96000Hz → 全て48000Hzに統一
4. **単一ファイル**: 1ファイルのみ → リサンプリングのみ実行
5. **NVENC環境**: 映像ストリームなし（音声のみ）でNVENC不要であることを確認

### パフォーマンス考慮

- `aresample`フィルタはCPU処理。音声のみなのでリアルタイムの10-50倍速
- NVENCは映像エンコードのみ使用（CLAUDE.mdルール準拠）
- 一時ファイル不要のため、ディスクI/Oを最小化

## 結論

根本対策は **concat前にaresample+aformatで全ストリームのパラメータを統一する** こと。既存の`ffmpeg -f concat`直接使用を`safe_audio_concat()`関数に置き換えることで、サンプルレート不一致による音質劣化を構造的に防止できる。ffmpeg_ops.pyの3箇所のハードコード箇所（L76, L483, L561）が修正対象。
