# Q4: ffmpegサンプルレート不一致問題（44100Hz vs 48000Hz concat）の根本対策設計

---

## 問題の詳細

YouTube動画素材（44100 Hz）とゲーム/配信素材（48000 Hz）を concat する際に発生する問題：

1. **concatデマクサ使用時**: `concat` はデフォルトで全入力のオーディオ仕様が一致することを要求。44100 vs 48000 の混在でエラーまたはサイレント破損
2. **filter_complex concat 使用時**: ffmpegが自動リサンプルを試みるが、サンプル単位の丸め誤差により **音声タイミングドリフト** が発生（長尺映像で顕著）
3. **症状**: 映像と音声のズレが徐々に蓄積（数秒〜数十秒）、クリップ境界でポップノイズ発生

---

## 根本原因の技術的説明

### サンプル数の非整合問題

44100 Hz を 48000 Hz にリサンプルする際：

```
48000 / 44100 = 160/147（有理数だが簡単な分数にならない）
```

`aresample` フィルタはデフォルトで `linear` リサンプラーを使用し、各フレームで端数処理が発生。
コンテナがPTSをサンプル単位で管理しているため、**リサンプル後のサンプル数 × 出力レート ≠ 入力期間** になる場合がある。

### ffmpegのffmpeg_ops.py における現状の問題（未修正箇所）

codebaseの `ffmpeg_ops.py` L76, L483, L561 では `-ar` オプション指定なし、または入力レートをそのまま継承している可能性が高い。

---

## 根本対策の設計

### 対策1: パイプライン入口での正規化（最推奨）

**考え方**: concat前に全クリップを48000Hzに統一する「正規化ステップ」を設ける。

```python
# ffmpeg_ops.py に追加するユーティリティ関数
def normalize_sample_rate(input_path: str, output_path: str, target_rate: int = 48000) -> bool:
    """入力ファイルのサンプルレートを確認し、target_rateと異なる場合のみ変換する"""
    import subprocess, json

    # ffprobeでサンプルレートを確認
    probe = subprocess.run([
        "ffprobe", "-v", "quiet", "-print_format", "json",
        "-show_streams", "-select_streams", "a:0", input_path,
    ], capture_output=True, text=True)
    
    info = json.loads(probe.stdout)
    streams = info.get("streams", [])
    if not streams:
        return False
    
    current_rate = int(streams[0].get("sample_rate", 0))
    if current_rate == target_rate:
        # 変換不要 → コピー（またはシンボリックリンク）
        import shutil
        shutil.copy2(input_path, output_path)
        return True

    # NVENCは音声エンコードに関係しないのでCPUで十分速い
    result = subprocess.run([
        "ffmpeg", "-i", input_path,
        "-c:v", "copy",           # 映像はコピー（再エンコード不要）
        "-c:a", "aac",
        "-ar", str(target_rate),  # サンプルレート正規化
        "-b:a", "192k",
        "-af", f"aresample={target_rate}:resampler=soxr",  # SOXRリサンプラー（高品質）
        "-y", output_path,
    ], capture_output=True)
    
    return result.returncode == 0
```

**正規化タイミング**:
```
[DL完了] → [Demucs分離] → [正規化: normalize_sample_rate()] → [concat] → [エンコード]
                                        ↑
                               ここで全クリップを48000Hzに統一
```

---

### 対策2: filter_complex concat での明示的リサンプル

concat時に毎回 `aresample` フィルタを明示的に挟む：

```bash
# 2クリップの例
ffmpeg \
  -i clip1.mp4 \
  -i clip2.mp4 \
  -filter_complex "
    [0:a]aresample=48000:resampler=soxr[a0];
    [1:a]aresample=48000:resampler=soxr[a1];
    [0:v][a0][1:v][a1]concat=n=2:v=1:a=1[v][a]
  " \
  -map "[v]" -map "[a]" \
  -c:v h264_nvenc -preset p4 \
  -c:a aac -ar 48000 -b:a 192k \
  output.mp4
```

**ポイント**: `resampler=soxr` を明示することで線形補間より高品質（位相誤差が少ない）なリサンプリングになる。

---

### 対策3: ffmpeg_ops.py への全体的な -ar 48000 明示追加

現在の問題箇所（L76, L483, L561）に対して：

```python
# 変更前（問題のあるパターン）
subprocess.run(["ffmpeg", "-i", input, "-c:a", "aac", output])

# 変更後（根本対策）
subprocess.run(["ffmpeg", "-i", input, 
                "-c:a", "aac", "-ar", "48000", "-b:a", "192k",
                output])
```

すべての出力に `-ar 48000` を明示することで、入力側のサンプルレートに関わらず出力が統一される。

## 正規サンプルレートの選択理由

**48000 Hz を選ぶ理由**:
- YouTube、放送、ゲームキャプチャの標準（48000 Hz = SMPTE 標準）
- 動画コンテナ（MP4/MKV）のデフォルト
- 44100 Hz は音楽CD由来（音楽メディア専用）

ドズル社切り抜きチャンネルの素材は主にゲーム配信（48000 Hz）のため、**48000 Hz 統一が自然**。
