## mmap化で検索が壊れた根本原因

### 原因1: numpy.memmap の「ファイルサイズ固定」問題（最有力）

旧実装では embedding vectorを `.npy` ファイルに保存し、検索時に `numpy.load(path, mmap_mode='r')` で読み込んでいたと推定される。

`numpy.memmap` は **ファイルをオープンした時点のサイズに固定されたビューを作る**。

```python
# 問題のある旧コード（推定）
index = np.load("embeddings.npy", mmap_mode="r")  # この時点のサイズで固定
```

`cmd_build` でチャンクを追記した場合：
- 以前は `np.save("embeddings.npy", updated_array)` のような処理をしていた
- しかし既存の mmap ビューはオープン時のファイルサイズのみを参照
- **新規追加ベクトルが検索に含まれない** → サイレントな検索漏れ

---

### 原因2: アトミックリプレース（os.replace）との相性問題

現コードの `bq_insert_rows` → `save_registry` パターンは `atomic write (tmp + rename)` を使用している。

旧コードで同様に `os.replace(tmp_path, index_path)` を使っていた場合：

- `os.replace` は **ファイルシステム上のinode（番号）を新しいものに置き換える**
- 既存の mmap は **古いinode（削除マーク付き）を参照し続ける**
- Linux ではファイル削除後もinode参照が残る間はデータが読める（ゾンビinode）
- しかし **次の書き込みサイクルでinode再利用されると化ける**

```
旧挙動:
  build → os.replace → 新inode/新embeddings
  search（mmap）  → 旧inode参照 → 古いベクトルで検索 → 意味のない結果
```

---

### 原因3: EMBED_DIM変更時の次元不一致

`EMBED_DIM = 3072`（現在値）は `gemini-embedding-2-preview` の出力次元。

もし旧モデル（例: `text-embedding-004` の 768次元 や `embedding-001` の 1536次元）でビルドされたインデックスを新モデルのクエリベクトルで検索すると：

- `numpy.dot(query_3072, old_index_1536)` → shape mismatch → エラー or 無意味な結果
- FAISS の場合は `d_in != d_out` で例外か静かに壊れた結果を返す

---

## 検索が壊れた際の症状

| 症状 | 対応する原因 |
|------|-------------|
| 常に同じ古い結果が返る | 原因1（mmap固定ビュー）|
| 検索結果が突然変わった / 再現しない | 原因2（inode置き換え競合）|
| 形状不一致エラー or スコア全部0 | 原因3（次元変更）|
| build直後は正常・再起動後に壊れる | 原因1+2の複合 |

## 修正案

### 修正案A: BigQuery移行（既に実施済 — 最終的な正解）

現コードがそれ。`bq_insert_rows` → `VECTOR_SEARCH()` の設計により：
- ローカルファイルのmmap問題が根本解消
- アトミック書き込み不要（BigQueryはトランザクション管理）
- 次元変更時はテーブルスキーマ変更だけで対応可能

✅ **既に実装済み。現コードは問題なし。**

---

### 修正案B: ローカルfallback実装が必要な場合の安全な設計

BigQueryが使えない環境でローカルインデックスを保持する場合：

```python
# NG: mmapで開く
index = np.load("embeddings.npy", mmap_mode="r")

# OK: 通常ロード（毎回ファイルをメモリに読み込む）
index = np.load("embeddings.npy")
```

インデックス更新時：
```python
# NG: インプレース追記（mmapビューとサイズ不整合）
with open("embeddings.npy", "ab") as f:
    np.save(f, new_vectors)

# OK: 全部読み込んで全部書き直す
old = np.load("embeddings.npy")
merged = np.concatenate([old, new_vectors])
tmp_path = "embeddings.npy.tmp"
np.save(tmp_path, merged)
os.replace(tmp_path, "embeddings.npy")  # アトミック置換（mmapは新規オープンで対応）
```

検索側は **毎回 `np.load(path)`（mmapなし）で読む**。キャッシュが必要なら `functools.lru_cache` + ファイルの `mtime` をキャッシュキーにする。

## まとめ

mmap化で検索が壊れた最大の原因は「**numpy.memmap の固定ビュー + アトミック置換によるinode変更の競合**」。
`mode='r'` のmmapは更新を見えなくする。`os.replace` はinodeを変えるためmmapビューが古いデータを指す。

現コードは **BigQueryへの移行によってこの問題を根本解決している**。追加の修正は不要。
