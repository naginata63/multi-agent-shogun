# Q4: ffmpeg サンプルレート不一致問題（44100Hz vs 48000Hz concat）の根本対策設計

## 1. 現象と再現条件

### 1.1 直接原因

ffmpeg の `concat` demuxer（`-f concat -i list.txt -c copy`）は **すべての入力ストリームの codec 属性（codec_id / sample_rate / channel_layout / timebase / video resolution / pixel_format）が完全一致していること** を要求する。これが崩れると以下のいずれかが発生する：

1. **サイレント失敗**: concat 成功するがAAC フレーム境界が崩壊 → 特定箇所で音が途切れる・ノイズが乗る
2. **明示的エラー**: `Non-monotonous DTS` / `Failed to update timestamps` で停止
3. **長さ破壊**: sample_rate 違いが duration 計算に反映されず、A/V 同期が累積的にズレる（30秒クリップ × 10本の連結で2〜3秒のドリフト）

### 1.2 本プロジェクトでの再現源（実コード）

`projects/dozle_kirinuki/scripts/ffmpeg_ops.py` に **ハードコードされた `anullsrc=r=44100:cl=stereo`** が3箇所：

| 行 | 生成物 | サンプルレート |
|----|--------|----------------|
| L76 | タイトルカード（静止画＋無音） | **44100** |
| L483 | 黒フレーム＋silence（トランジション用） | **44100** |
| L561 | `_concat_with_black` のフォールバック黒フレーム | **44100** |

一方、YouTube からダウンロードした元動画の AAC 音声は **48000Hz が主流**（YouTubeのコンテナ/エンコード規定）。Demucs 分離出力・WhisperX 抽出も 44.1k/48k が混在する。

これらを `concat` demuxer に投入すると失敗。実害シナリオ：

- タイトルカード（44.1k）＋本編クリップ（48k）＋黒フレーム（44.1k）＋次クリップ（48k） → 4境界で破綻
- ハイライト動画のエンディング黒フレーム直前でブツ切れ

### 1.3 見落とされる理由

- ffmpeg が `-c copy` 時は **警告だけで処理を続行する** ケースが多く、CI/E2E で気付かない
- `ffprobe` で stream 単位の sample_rate を全部確認する習慣がないと、問題ファイルに辿り着けない
- 聴感で気付くのはイヤホンでモニタした時のみ（スマホスピーカーでは隠蔽される）

## 2. 設計の分岐点

根本対策は3方向ある。単に「48kHzに揃える」だけでは設計として弱いので、トレードオフを示す。

### 案A: 「**プロジェクト規定サンプルレートを固定**し、全生成 CLI で強制適用」

ffmpeg 呼出を通るすべての音声生成箇所で `-ar <RATE>` を明示。既存コードの `anullsrc=r=XXXXX` を含め単一の定数から引く。

```python
# projects/dozle_kirinuki/scripts/audio_constants.py（新規 .py は禁止なので既存 ffmpeg_ops.py 冒頭に置く）
AUDIO_SAMPLE_RATE = 48000   # プロジェクト統一（YouTube 互換・Demucs 互換）
AUDIO_CHANNEL_LAYOUT = "stereo"
AUDIO_CODEC = "aac"
AUDIO_BITRATE = "192k"

# 黒フレーム生成
["-f", "lavfi", "-i", f"anullsrc=r={AUDIO_SAMPLE_RATE}:cl={AUDIO_CHANNEL_LAYOUT}"]
# 最終エンコード
["-c:a", AUDIO_CODEC, "-b:a", AUDIO_BITRATE, "-ar", str(AUDIO_SAMPLE_RATE)]
```

- **利**: 実装最小（ffmpeg_ops.py 3箇所＋最終エンコード1箇所の5行差分）。新規 .py 不要（既存ファイル内定数化のみ）
- **害**: 外部からの入力（元動画、外注音源、SE素材）が 44.1k だった時に暗黙リサンプリングが走る。リサンプリング品質で音質劣化（soxr が使える環境なら軽微）
- **前提**: YouTube アップ先との整合性確認済み（48k は YouTube 推奨）

### 案B: 「**入力側正規化パイプライン**: 全入力を concat 前に 48k aac -ar 48000 へ normalize」

`_concat_with_black` や本編連結の手前に **pre-normalize ステップ** を挟む。

```python
def _normalize_audio_for_concat(input_path: str, out_path: str,
                                 target_rate: int = 48000) -> None:
    """concat 前の共通フォーマット化。video は copy、audio だけ再エンコ。"""
    cmd = [
        "/usr/bin/ffmpeg", "-y",
        "-i", input_path,
        "-c:v", "copy",
        "-ar", str(target_rate), "-ac", "2",
        "-c:a", "aac", "-b:a", "192k",
        "-movflags", "+faststart",
        out_path,
    ]
    subprocess.run(cmd, check=True)
```

- **利**: 入力源の多様性を吸収できる。案A より堅牢。 リサンプリング地点が明示的 → ログで追跡可能
- **害**: concat のたびに audio 再エンコ → 処理時間増（1時間動画で +30〜60秒 × クリップ数）。NVENC は映像だけ加速、AAC は CPU
- **緩和**: 正規化済みフラグをファイル命名規則で表現（`.48k.mp4`）→ 2回目以降スキップ

### 案C: 「**concat 前に ffprobe で検査→差分のみリサンプル**（ハイブリッド）」

```python
def _probe_sample_rate(path: str) -> int:
    r = subprocess.run(
        ["/usr/bin/ffprobe", "-v", "error", "-select_streams", "a:0",
         "-show_entries", "stream=sample_rate", "-of", "default=nk=1:nw=1", path],
        capture_output=True, text=True, check=True,
    )
    return int(r.stdout.strip() or 0)

def _align_sample_rates(paths: list, target: int = 48000) -> list:
    out = []
    for p in paths:
        sr = _probe_sample_rate(p)
        if sr == target:
            out.append(p)
            continue
        aligned = p.replace(".mp4", f".{target}.mp4")
        _normalize_audio_for_concat(p, aligned, target)
        out.append(aligned)
    return out
```

`_concat_with_black` 冒頭で `clip_paths = _align_sample_rates(clip_paths)` を呼ぶ。

- **利**: すでに 48k のファイルを不要再エンコしない（CPU節約）。運用の可視化が最も良い
- **害**: 追加の ffprobe コールでわずかに遅い（数十ms/ファイル）
- **推奨**: 実運用ではコレ

## 3. 推奨採択: **案A + 案C のハイブリッド**

**理由**:
- 案A で **生成側（内部で作る音声）を 48k に固定** → 将来のコードが増えても不整合が出ない予防線
- 案C で **concat 直前の入力側を保険的にチェック** → 外部素材（手動で置かれた SE、外部音源）の混入にも対応

両方採用しても差分実装量は 60行以下に収まる。既存ファイル（ffmpeg_ops.py）への追記のみで完結。

## 6. 見落としがちな付随課題（実害優先で指摘）

1. **SE 素材の多くが 44.1k** — 漫画ショートの効果音ファイル（S1/S2/S3 等）が 44.1k で保存されている可能性大。`_mix_se_into_video` 内の `amix` 前で SE を 48k に aresample で揃える必要がある。具体箇所: L515 の `adelay` の前に `,aresample=48000` を挿入

2. **Demucs 出力が 44.1k** — Demucs 4系のデフォルトサンプルレートは 44.1k。ボーカル分離後のクリップを直接 concat すると同じ問題再発。Demucs 呼出し側で `--out-rate 48000` 相当の後処理を足すか、案C の検査が必ず拾う設計にすること

3. **Gemini TTS / 音声合成の出力** — もし将来 TTS を追加する場合、出力サンプルレートが 22.05k / 24k と低い可能性がある。`_align_sample_rates` は必ず整数比のアップサンプリングでも機能することを保証（soxr リサンプラ推奨: `-af "aresample=48000:resampler=soxr:precision=28"`）

4. **音量正規化との衝突** — `loudnorm` や `dynaudnorm` フィルタは内部で出力サンプルレートを変えることがある。既存コードに loudnorm があれば、その後段に `aresample=48000` を明示すべき

## 7. 結論

**根本対策 = 「プロジェクト内生成音声を 48kHz 固定（案A）＋concat 直前に probe→正規化する保険（案C）」**

既存 ffmpeg_ops.py の3箇所の `r=44100` 差替え・定数化・`_align_sample_rates` 追加で実装完了。リスクは AAC 再エンコ時間（+数十秒×クリップ数）のみ。新規 .py 不要。

将来の SE/Demucs 追加時も、案C のガードが `_concat_with_black` を通る限り自動で吸収される。
