- **YAMLロック取得**: 両ファイルで `flock` + `mkdir` フォールバック実装（`inbox_write.sh` L33-57, `inbox_watcher.sh` L315, L425）。ロジックが散在し3箇所に複製。

- **inbox初期化**: `inbox_write.sh` L24-27, `inbox_watcher.sh` L44-47。

### 1.3 責務境界の漏れ

1. **writer 側に「ドメインロジック」が漏出**: `inbox_write.sh` が shogun_to_karo.yaml や tasks/*.yaml を書き換えている。これは「メッセージ配送」ではなく「ワークフロー状態機械」の仕事。本来 watcher（もしくは専用 state-machine daemon）が担うべき。
2. **watcher が「メッセージ書込み」も行う**: `enqueue_recovery_task_assigned` で inbox YAML に自己生成メッセージを append する（watcher→writer 逆流）。結果、writer / watcher のどちらがインボックス書込みの唯一の起点か曖昧。
3. **CLI種別 dispatch が watcher に集中**: 1262行の watcher 内部に claude/codex/copilot/kimi/gemini の分岐が散在（L513-559, L644-699, L831-836, L895, L903, L929-933, L1139-1141）。adapter 抽象がすでにあるのに完全活用されていない。

### 1.4 スキーマが暗黙

メッセージの型（`read: bool`, `type: str`, `content: str`, `id`, `timestamp`, `from`）は両ファイルに分散記述。`type` の有効値リスト（`task_assigned`, `clear_command`, `model_switch`, `cli_restart`, `cmd_new`, `report_received`, `report_done`, `report_completed`, `blocked`, `wake_up`, `advisory`, `qc_result` etc.）は一覧化されておらず、コード横断 grep でしか把握できない。

## 2. 統合設計 — 4層アーキテクチャ

```
┌──────────────────────────────────────────────────────────────────┐
│ Layer 4: Presentation  │  inbox_write.sh / inbox_tail.sh (CLI)   │
├──────────────────────────────────────────────────────────────────┤
│ Layer 3: Daemon        │  inbox_watcher (event loop + escalation)│
├──────────────────────────────────────────────────────────────────┤
│ Layer 2: Domain        │  cmd_state_machine.py (side-effects)    │
│                        │  cli_adapter.py (CLI-specific send)     │
├──────────────────────────────────────────────────────────────────┤
│ Layer 1: Core          │  inbox_lib.py (schema/lock/atomic I/O)  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.1 Layer 1 — `scripts/inbox_lib.py`（新規・共通ライブラリ）

```python
# scripts/inbox_lib.py — Single Source of Truth for inbox operations
from pathlib import Path
from dataclasses import dataclass, asdict
from contextlib import contextmanager
from enum import Enum
import fcntl, os, tempfile, yaml, uuid, datetime

class MessageType(str, Enum):
    TASK_ASSIGNED = "task_assigned"
    CLEAR_COMMAND = "clear_command"
    MODEL_SWITCH  = "model_switch"
    CLI_RESTART   = "cli_restart"
    CMD_NEW       = "cmd_new"
    REPORT_DONE   = "report_done"        # alias: report_completed
    REPORT_RECEIVED = "report_received"
    BLOCKED       = "blocked"
    ADVISORY      = "advisory"
    QC_RESULT     = "qc_result"
    WAKE_UP       = "wake_up"

@dataclass
class InboxMessage:
    id: str
    from_: str   # serialized as "from"
    timestamp: str
    type: str
    content: str
    read: bool = False

    @classmethod
    def create(cls, from_: str, type_: str, content: str) -> "InboxMessage":
        now = datetime.datetime.now().replace(microsecond=0).isoformat()
        rand = os.urandom(4).hex()
        return cls(
            id=f"msg_{now.replace(':','').replace('-','').replace('T','_')[:15]}_{rand}",
            from_=from_, timestamp=now, type=type_, content=content, read=False
        )
    def to_yaml(self) -> dict:
        d = asdict(self); d["from"] = d.pop("from_"); return d

@contextmanager
def inbox_lock(path: Path, timeout: float = 5.0):
    """Cross-platform file lock (flock on Linux, mkdir on macOS).
       Yields a context where the inbox can be read/written atomically."""
    lock = path.with_suffix(path.suffix + ".lock")
    if hasattr(fcntl, "flock"):
        with open(lock, "w") as lf:
            fcntl.flock(lf, fcntl.LOCK_EX)
            try: yield
            finally: fcntl.flock(lf, fcntl.LOCK_UN)
    else:
        ldir = path.with_suffix(path.suffix + ".lock.d")
        # ...mkdir polling fallback...
        yield

def read_inbox(path: Path) -> dict:
    if not path.exists(): return {"messages": []}
    with open(path) as f: return yaml.safe_load(f) or {"messages": []}

def write_inbox_atomic(path: Path, data: dict):
    fd, tmp = tempfile.mkstemp(dir=path.parent, suffix=".tmp")
    with os.fdopen(fd, "w") as f:
        yaml.safe_dump(data, f, default_flow_style=False,
                       allow_unicode=True, sort_keys=False)
    os.replace(tmp, path)
    # Force inotify close_write event
    path.touch()

def append_message(path: Path, msg: InboxMessage, max_keep=50):
    with inbox_lock(path):
        data = read_inbox(path)
        data.setdefault("messages", []).append(msg.to_yaml())
        # Overflow protection
        msgs = data["messages"]
        if len(msgs) > max_keep:
            unread = [m for m in msgs if not m.get("read")]
            read   = [m for m in msgs if m.get("read")]
            data["messages"] = unread + read[-30:]
        write_inbox_atomic(path, data)

def read_unread(path: Path) -> list[dict]:
    with inbox_lock(path):
        data = read_inbox(path)
        return [m for m in data.get("messages", []) if not m.get("read")]

def mark_read(path: Path, msg_ids: set[str]):
    with inbox_lock(path):
        data = read_inbox(path)
        for m in data.get("messages", []):
            if m.get("id") in msg_ids: m["read"] = True
        write_inbox_atomic(path, data)
```

**効果**: 両shellスクリプトの Python ヒアドキュメントを全て削除可能。テスタブル（pytest）。

# scripts/cmd_state_machine.py
class CmdStateMachine:
    """Observes inbox events and updates shogun_to_karo.yaml / tasks/*.yaml.
       Called by watcher on new message arrival, NOT by writer inline."""
    def on_message(self, msg: InboxMessage):
        if msg.type == "task_assigned":  self._mark_cmd_in_progress(msg)
        elif msg.type in ("report_done","report_completed"): self._mark_task_done(msg)

### 2.3 Layer 3 — `scripts/inbox_watcher.sh`（薄いイベントループ）

```bash
while true; do
    inotifywait -q -t 10 -e modify -e close_write "$INBOX" && rc=0 || rc=$?
    python3 -m inbox_watcher.process --agent "$AGENT_ID" --pane "$PANE_TARGET" \
            --trigger "${rc:+event}${rc:-timeout}"
done
```

`inbox_watcher/process.py` が:
1. `inbox_lib.read_unread()` で未読を取得
2. `CmdStateMachine.on_message()` を各メッセージに適用（writer から移管した副作用）
3. `EscalationEngine.decide()` が Phase1/2/3 を判定
4. `CLIAdapter.send_*()` で pane に送信
5. `inbox_lib.mark_read()` で特殊メッセージを既読化

## 4. 設計上の判断ポイント（トレードオフ）

### 案A: 本提案（4層分離 + Python集約）
- 利: テスタブル、責務明快、1262+255行 → 推定 900行（Python含む総計）
- 害: Python依存増（既に venv があるので追加コスト低い）、学習コスト中

### 案B: Bashのまま `lib/inbox_common.sh` に共通化
- 利: 現行技術スタック変えず最小リスク
- 害: YAML操作が依然 Python ヒアドキュメント、テスト困難が残る
- 推奨: P1〜P2のみ案Aで、P5以降は案Bに落とす折衷も現実的

### 案C: SQLite移行（message queue DBに置換）
- 利: ロック問題消滅、全文検索可、トランザクション
- 害: inotify 連携を別機構（LISTEN/NOTIFY 相当）にする必要。既存YAMLとの二重運用期間が発生
- 棄却: インフラ変更の重さが得られるメリットを超える

**推奨: 案A（P1〜P4まで）+ 案B折衷（P5〜）**。18ヶ月後の保守性を最優先。

## 5. 即時修正できる小粒バグ（設計統合とは別建て）

3. **`inbox_watcher.sh` L1062**: `stale_busy_limit=60` だが変数宣言コメントは「1 minute (shortened from 300s)」。将来誰かが 300s に戻したくなった時の経緯がコメントに残っていない。
4. **`inbox_write.sh` L205-211**: オーバーフロー保護で `unread + read[-30:]`。unread が31件以上あると50件上限を超える。`min(30, 50-len(unread))` 相当の数学を足すべき。
5. **`inbox_watcher.sh` L723**: `pgrep -f "inotifywait.*inbox/${AGENT_ID}.yaml"` は `grep` 的マッチで誤検出リスク。`^` アンカーと明示引数マッチに。

## 6. 結論

統合の核は **「writer は配送のみ。状態機械と副作用は consumer（watcher）に集約」**。これを達成すれば、inbox_write.sh は 8行に、watcher は Python 化で責務明確に縮む。Python 共通ライブラリ（`inbox_lib.py`）が SSoT となり、YAML ヒアドキュメントが消える。

移行は P1→P2→P3（シャドー）→P4 まで案A、P5 以降は家老の判断で案Bに落として漸進的に整備するのが現実解。
