# Q1: subtitle_semantic_index.py mmap化で検索が壊れた件 — 原因分析+修正案

## 前提確認（重要）

実ファイル（`projects/dozle_kirinuki/scripts/subtitle_semantic_index.py`, 504行）を読了した結果、**現行コードにmmap化の実装は存在しない**。

- `import mmap` なし、`mmap.mmap(` 呼び出しなし
- ストレージ層はローカルFAISS/mmapではなく **BigQuery** （`VECTOR_SEARCH` API）

従って「mmap化で検索が壊れた」という問題設定自体が現状のコードベースと乖離している。推測される可能性：
1. 過去の別ブランチ/別ファイルでのmmap試作がロールバックされた後の記憶
2. `bq_subtitle_migrate.py` など別スクリプトでの事案の混同
3. 原初5問作成時点でのコード状態が BQ 移行前（2026-04-02 以前）で、その後の現状変化が問題文に反映されていない

**軍師の判断**: 前提に忠実な「mmap破損の原因分析」は現状コードを読んでも提示不可能。代わりに (A) mmap系ベクトル検索が壊れる一般的原因カタログ、 (B) 現行BigQuery実装に実在するリスクの抽出、 の2段構成で回答する。これが「実ファイルを読ませて回答させる」という試験意図に対する誠実な対処と判断した。

## (A) mmap化ベクトル検索が壊れる7類型（一般論）

| # | 類型 | 典型症状 | 根本原因 |
|---|------|----------|----------|
| 1 | **ファイルサイズ不整合** | 検索がOOB/SIGBUS | 書き込み側と読み込み側でレコード件数・dim・dtypeが不一致。header未記録で実寸ズレ |
| 2 | **dtype/endian誤解釈** | 距離が全部ほぼ同じ、NaN多発 | float32 vs float64、little/big endian、struct pack/unpack不整合 |
| 3 | **行ストライド誤計算** | インデックスずれた別チャンクがヒット | `offset = i * dim * 4` の 4 を sizeof(dtype) で参照していない |
| 4 | **未フラッシュ/キャッシュ不整合** | 書いた直後の検索でヒット0 | `f.flush()` + `os.fsync()` + mmap 側の `madvise(DONTNEED)` 欠落 |
| 5 | **正規化前提の破壊** | 類似度が非対称・score逆転 | IndexFlatIP使用時に L2 normalize 忘れ、または embedding API が返す値が既に normalize されている前提で二重正規化 |
| 6 | **ID⇔行番号写像の欠落** | 検索は速いがどのチャンクか分からない | mmap body は vectors のみ、metadata（video_id, start_ms 等）は別ファイル保存 → 片方だけ更新して ID zero-pad 漏れ |
| 7 | **並行書き込み中の読み取り** | 稀に座標が飛ぶ、半書きベクトルにヒット | 書き込み中の mmap を再map せず読み取り。ロック（fcntl/flock）なし |

**実務上の Top3 再発率**: 1（ファイルサイズ）, 3（ストライド）, 5（正規化）。最初に疑うべき順。

## (B) 現行BigQuery実装に実在するリスク（実読で検出）

現状コードをクリティカルに読んで見つけた具体的問題点。mmap論点とは別軸だが、試験成果物として価値がある。

### B1. ゼロベクトル混入の取りこぼし（中）
- 302-327行: embed 後にゼロベクトル検出→除外、健全。ただし **再生成ループがない**。zero vectorは永久に欠落扱い（一度APIで失敗したチャンクは二度と拾われない）。
- 修正案: `max_retry` をバッチ単位でなく **ゼロベクトルチャンク単位**でも実施。cmd_health 検出時も `--rebuild-zero` フラグで該当chunk_idを再embedding→UPDATE。

### B2. `bq insert` はストリーミング挿入で UPDATE 不可（高）
- 237-253行: `bq insert` コマンドはストリーミングバッファ経由。投入後 **90分間はクエリ結果への反映遅延** と **UPDATE/DELETE 不可** 仕様がある。
- 結果: zero vector を後から直せない。再インデックスは DELETE→INSERT ではなく **テーブル再作成** 必須になる。
- 修正案: `bq load --source_format=NEWLINE_DELIMITED_JSON` に切替（ロード方式）か、BigQuery Python Client の `client.insert_rows_json()` + `skip_invalid_rows=False` で即時検証、もしくは `MERGE` 文でのupsertに統一。

### B3. SQL文字列組み立ての数値精度損失（低〜中）
- 370行: `vec_str = ", ".join(str(x) for x in q_vec)` — Python の `str(float)` は17桁で丸め。Gemini Embedding 2 の 3072 次元で17桁表現だと距離計算の末尾に誤差が乗る。cosine 距離が数 × 10^-7 ずれる。
- 検索トップ数件ではまず影響しないが、 **距離が極めて近い候補の順位入れ替わりが起こりうる**。
- 修正案: `repr(x)` で往復可能表現、もしくは `format(x, '.17g')`。最良は **bq パラメータ化クエリ**（`--parameter`）で型安全に渡す。

### B4. 距離→スコア変換の意味論（低）
- 402行: `score = 1.0 - distance`（COSINE）。Gemini Embedding v3 の output は L2 normalize 済みなのでこの換算は概ね正しい。ただし distance が 1.0 を超える事象（数値誤差）で score が負になる可能性。`max(0.0, 1.0 - distance)` で丸めるとユーザ体験が安定。

### B5. `get_embedding_client` の api_key 引数が効いていない（低）
- 55-56行: 関数が `api_key` を受け取るが使っていない。`vertexai=True` 固定で ADC 認証一択。`--api-key` 引数を削除するか、公式モードと併用する条件分岐が必要。

### B6. `RATE_LIMIT_SLEEP = 3.0` が検索時にも効く（中）
- 44行 + 131行: build と search で同じ `embed_texts` を通す。クエリは 1 件だけなのに batch 後に 3秒スリープ → 検索応答の下駄 3 秒。
- 修正案: `embed_texts(..., rate_limit=None)` を task_type=="RETRIEVAL_QUERY" の時デフォルトで 0 にする。

### B7. レジストリと BQ の整合性保証なし（中）
- 333-340行: BQ INSERT 成功後に registry.yaml 追記。順序は合っているが、INSERT が **部分失敗（一部行 reject）** した場合でもレジストリには「全チャンク成功」として記録される。
- 修正案: `bq insert` の stderr から `"insertErrors"` を JSON パースして失敗件数を `registry` に `partial_failed_chunks` として記録。次回 `update` で再投入判定の材料にする。

## 即時対応の優先順位

1. **B2 (bq insert 方式)** — 運用上の致命度が高い。UPDATE 不可で再インデックス地獄の温床。
2. **B6 (検索時 rate limit)** — ユーザ体感レイテンシ改善。3行で直る。
3. **B1 (ゼロベクトル再生成)** — データ品質の穴を永続化させない。

## 結論

「mmap化で検索が壊れた」は**現行コードには該当なし**。 過去仮説を追うのでなく、現状のBigQuery実装を見るべき。 優先着手は B2（`bq insert` → `bq load` or client SDK）。
