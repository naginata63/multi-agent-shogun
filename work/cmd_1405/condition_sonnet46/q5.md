# AIに漫画を描かせてYouTubeショートを量産した話

ゲーム実況YouTubeのチャンネル切り抜きを始めたのは去年のことだ。最初は自分で手を動かしていたが、ある時気づいた。

**「これ、全部AIにやらせられないか？」**

半年後、私は週に数本の漫画ショート動画を、ほぼ手放しで量産できるようになっていた。その仕組みを全部書く。

---

## きっかけ：切り抜き動画の限界

ゲーム実況の切り抜き動画は、大きく分けて2種類ある。

1. **トーク切り抜き**：面白い発言をそのまま切り出す
2. **漫画ショート**：面白い場面を漫画風のイラストで再現する

トーク切り抜きは作りやすいが、差別化が難しい。漫画ショートは手間がかかるが、独自性が出る。

問題は後者の「手間」だ。1本あたり、スクリプト作成・イラスト生成・動画編集で数時間かかる。これをどうにか自動化できないか、と考えた。

---

## パイプラインの全体像

最終的に行き着いたのは、こんな構成だ。

```
YouTube動画 → 文字起こし(Whisper) → 場面選定(Gemini) 
→ パネル構成(Claude) → 漫画生成(Gemini Imagen) 
→ 動画合成(ffmpeg) → YouTube非公開アップ → 殿（自分）確認
```

各段階で使うAIが違う。これは意図的だ。「テキスト→テキスト」はClaude、「画像生成」はGemini Imagen、「動画処理」はffmpeg（AIではないが）という役割分担が、コストと品質のバランス上ベストだと分かった。

---

## 漫画生成の実際

### パネル構成を作る

まず「どんな漫画にするか」を決める。これはClaudeが担当する。

配信の文字起こしデータを渡すと、面白い場面を選んで4〜6コマのパネル構成（JSON）を生成してくれる。

```json
{
  "panels": [
    {
      "id": "p01",
      "scene": "ドズルが突然大声を出す場面",
      "characters": ["dozle"],
      "dialogue": "「えっ、何これ！？」",
      "director_notes": "驚きの表情。口を大きく開けて"
    }
  ]
}
```

このJSONがすべての出発点になる。

### 絵を生成する

パネル構成が確定したら、Gemini ImagenでイラストのPNGを生成する。

ここが最初のハマりポイントだった。

**問題1：キャラがバラバラになる**

複数パネルで同じキャラクターを描いても、見た目がコマごとに変わってしまう。「ドズルはこの人です」という参照画像（ref_image）を毎回渡すことで解決した。

**問題2：ステーキが出てくる**

本当にそういうことが起きた。「驚く表情」というプロンプトで、なぜかステーキが画面に現れる。Geminiの画像生成は創造的すぎる。`director_notes` で細かく指示する必要がある。

**問題3：コスト爆発**

最初は気軽に「もっと良い絵が生成されるまで何度でも試そう」と回していたら、3,409枚生成して22,000円の請求が来た。今は「上限3セット」「まず1パネルだけ試す」というルールを決めている。

### 動画に仕上げる

生成したPNGを並べてffmpegでスライドショー動画を作る。音声は配信のBGMから抜き出したものを乗せる。縦型（1080×1920）に変換してYouTubeショートとして出力する。

ffmpegの設定で一点だけ注意がある。GPUエンコード（NVENC）を使わないと、32分動画の変換に3時間かかる。RTX 4060 Ti搭載のマシンなら数分で終わる。

## マルチエージェント化という発想

動画1本を作るのに、実は複数のAIエージェントが動いている。

私のシステムは「将軍（Shogun）」「家老（Karo）」「足軽（Ashigaru）」「軍師（Gunshi）」という役割分担になっている（なぜか戦国武将）。

- **将軍**：私に相当。全体方針を決めて指示を出す
- **家老**：タスクを分解して足軽に割り振る
- **足軽**：実際の作業（コード実行・API呼び出し）を担当
- **軍師**：品質チェック・戦略アドバイス

漫画ショートを1本作る場合、将軍（私）がコマンドを出すと、家老がタスクを細分化して複数の足軽に並列実行させる。私が見ているのは最終的な承認画面だけだ。

---

## やってみて分かったこと

### 良かったこと

**再現性が上がった**
手作業のときは「前回どうやったっけ」が頻発していた。パイプラインが固まると、同じクオリティを毎回出せる。

**深夜に勝手に動く**
寝ている間に生成タスクを走らせておける。朝起きたら完成している。

**改善のサイクルが速い**
「このプロンプトを変えたらどうなるか」を試すコストが下がった。人間が直接作業するより、AIに任せた方が試行回数を増やせる。

### 難しかったこと

**品質コントロール**
最終的には人間の目が必要だ。「それっぽい動画」は量産できるが、「面白い動画」を作るには人間の編集判断が入る。AIはスケールを支える道具であって、センスを置き換えるものではない。

**コスト管理**
API費用は油断すると跳ね上がる。上限設定とアラートは必須だ。

**デバッグの大変さ**
複数のAIが連携しているので、どこで問題が起きたかを特定するのが難しい。エラーハンドリングとログ設計に最初から力を入れておくべきだった。

## 数字で振り返る

- **Whisper文字起こし**：1時間の動画を10〜15分で処理
- **パネル構成生成**：1ケース30秒以内
- **Imagen画像生成**：1パネル60〜120秒（Vertex AI経由）
- **動画合成（ffmpeg NVENC）**：1本1〜3分
- **人間の作業時間**：コマンド発行+最終確認で30分未満/本

1本の漫画ショートを「ゼロから」作るのに、人間の実作業は30分程度まで落ちた。

---

## これから

まだ手放しにはなっていない部分がある。

**場面選定の感度**：Geminiが選ぶ「面白い場面」と、人間が感じる「面白さ」がズレることがある。評価モデルの導入を検討中。

**キャラ表情の制御**：現状は `director_notes` で文字指定しているが、表情差分画像のパターンをあらかじめ生成しておいて参照させる方式が良さそう。

**完全自動投稿**：今は非公開アップまでで止めて自分が確認している。品質が安定すれば、ここも自動化できる。

---

AIに漫画を描かせることは、2年前には「そのうちできるようになるかも」という話だった。今は実際に動いている。

面白いのは、「描かせる」だけでは全然ダメで、「どう描かせるか」の設計が全てだということ。プロンプトエンジニアリングとか、パイプライン設計とか、品質管理とか——そういう「AIを使う人間の仕事」はむしろ増えた気がする。

**AIが得意なのはスケールと一貫性。人間が得意なのは判断と感性。**

うまく組み合わせれば、1人でも「量産」と「品質」を両立できる。それがこの半年で学んだことだ。
