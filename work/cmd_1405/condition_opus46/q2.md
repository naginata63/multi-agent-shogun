## 問題文
複数ファイルリファクタリング設計: inbox_write.sh + inbox_watcher.sh の統合案設計

## 現状分析

### inbox_write.sh (255行)
- **責務**: メッセージをinbox YAMLに原子的に書き込む（flock排他ロック + Python YAML処理）
- **副作用1**: `_update_cmd_status()` — task_assigned時にshogun_to_karo.yamlのstatus: pending→in_progressに自動更新
- **副作用2**: `_update_task_done()` — report_done/report_completed時にtasks YAMLのstatus: assigned→doneに自動更新
- **副作用3**: `cmd_rag_hook.sh` — cmd_new時にRAGフック実行（バックグラウンド）
- **外部依存**: `.venv/bin/python3`（YAML処理）、`flock`（排他ロック）、`mawk`（status更新）

### inbox_watcher.sh (1263行)
- **責務**: inotifywaitによるinboxファイル変更検知→エージェントへのwake-up配信
- **機能群**:
  - エスカレーション3段階（通常nudge → Escape+nudge → /clear）
  - CLI種別別分岐（claude/codex/copilot/kimi/gemini）
  - Agent busy検知（flagファイル + pane直接検出）
  - nudgeスロットリング
  - 特殊コマンド処理（clear_command, model_switch, cli_restart）
  - メトリクス収集
  - Auto-recovery（/clear後のtask_assigned自動投入）
  - Context reset（task_assigned検出時の/clear自動送信）
  - Codex startup prompt送信
  - summary-first fast-path（unread_count軽量チェック）

## 統合の動機と目的

### 現在の問題点
1. **責務の重複と分散**: inbox_write.shがYAML書き込みだけでなくstatus自動更新まで行っている。これはwriterの責務を超えている
2. **inbox_watcher.shの肥大化**: 1263行に達し、エスカレーション・CLI分岐・busy検知・メトリクスなど多数の横断的関心事が1ファイルに詰まっている
3. **テスタビリティ**: inbox_watcher.shは`__INBOX_WATCHER_TESTING__`ガードがあるが、関数間の依存が深く単体テストが困難
4. **設定の散在**: エスカレーション閾値・nudgeクールダウン・CLI別挙動が環境変数とハードコードの混在

### 統合ではなく分離が正解

「統合」と題されているが、実際にはinbox_write.shとinbox_watcher.shは **書き込み（producer）** と **配信（consumer）** という異なる責務を持っており、1ファイルにマージすべきではない。真の課題は「watcher側の肥大化」と「write側の副作用過多」の整理である。

## 設計案: 3層アーキテクチャ

### Layer 1: inbox_core.sh（新規 — 共通基盤）
```
lib/inbox_core.sh (~80行)
├── INBOX_DIR, LOCKFILE パス定義
├── _acquire_lock() / _release_lock() — 排他ロック（flock/mkdir）
├── get_inbox_path() — agent_id → inbox YAMLパス
├── read_inbox_yaml() — Python YAML読み込み（共通）
├── write_inbox_yaml() — Python YAML書き込み（原子的rename）
└── get_unread_count() — unread件数取得（fast-path）
```

**根拠**: 現在inbox_write.shとinbox_watcher.shの両方がflock取得・YAML読み書きのロジックを重複して持っている。共通化により一貫性を保証する。

### Layer 2: inbox_write.sh（リファクタ — 書き込み専用）
```
scripts/inbox_write.sh (~60行, 現255行から大幅削減)
├── source lib/inbox_core.sh
├── main: 引数バリデーション → メッセージ構築 → write_inbox_yaml() → touch
└── ※ status自動更新・RAGフックはhook層へ移動
```

**変更点**:
- `_update_cmd_status()` → `lib/inbox_hooks.sh` に移動
- `_update_task_done()` → `lib/inbox_hooks.sh` に移動
- `cmd_rag_hook.sh` 呼び出し → `lib/inbox_hooks.sh` に移動
- inbox_write.sh本体は「メッセージをYAMLに追記する」だけに集中

## リスクと対策

| リスク | 影響 | 対策 |
|--------|------|------|
| sourceパス解決の失敗 | watcher全停止→全エージェントデッドロック | inbox_watcher.sh冒頭でsource失敗時にfallback単体動作モード |
| flock/YAML処理の互換性破壊 | メッセージ消失 | Phase 1でinbox_core.shを十分テストしてからPhase 2に進む |

| 移行中の本番稼働 | 中途半端な状態でエージェント群が動作 | 各Phase完了ごとにgit commit。問題発覚時はgit restoreで即時ロールバック |

## 統合案を採用しない理由（参考）

inbox_write.shとinbox_watcher.shを1ファイルにマージする案は以下の理由で不適:

1. **プロセスモデルの違い**: inbox_write.shは「1回呼び出し→即終了」のコマンド。inbox_watcher.shは「常駐デーモン」。ライフサイクルが根本的に異なる
2. **呼び出し元の多様性**: inbox_write.shは全エージェント（8+ プロセス）から並行呼び出しされる。マージするとデーモンのコードが全呼び出しに巻き込まれる
3. **障害分離**: 書き込みの失敗と配信の失敗は独立した障害。分離されていれば片方の障害がもう片方に波及しない

## まとめ

「統合」ではなく「責務分離によるリファクタリング」が正しいアプローチ。現在の2ファイル構成を6ファイルの3層アーキテクチャに再構成することで:

- inbox_write.sh: 255行→60行（76%削減）
- inbox_watcher.sh: 1263行→200行（84%削減）
- 合計行数: 1518行→670行（56%削減）
- テスタビリティ: 各lib/は独立してbatsテスト可能
- 可読性: 各ファイルが単一責務に集中
- 拡張性: 新CLIタイプ追加時はnudge.shのcase文追加のみ
