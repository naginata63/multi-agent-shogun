# Q1: 実バグ分析 — pretool_check.sh blacklistチェックのすり抜け問題

- モデル: claude-opus-4-6[1m]
- 開始時刻: 2026-04-17 11:54:02
- 終了時刻: 2026-04-17 11:54:07
- advisor呼び出し: 0回

## 問題文
pretool_check.shのblacklistチェックで、パス付きスクリプト名（python3 /full/path/to/banned.py）がすり抜ける問題の原因分析+修正案

## 原因分析

### 根本原因: ファイル名抽出パターンがベースネームのみを抽出する設計

pretool_check.sh 158行目のgrep抽出パターン:
```bash
PY_FILES=$(echo "$COMMAND" | grep -oE '[^/[:space:]]+\.py' | grep -v '__pycache__')
```

このパターン `[^/[:space:]]+\.py` は「スラッシュとスペース以外の文字列 + .py」にマッチする。つまり `/full/path/to/banned.py` からは `banned.py` だけが抽出される。

一方、blacklistファイルとの照合（164行目）:
```bash
if [[ -f "$BLACKLIST" ]] && grep -qxF "$PY_FILE" "$BLACKLIST"; then
```

`grep -qxF` は**完全一致**（`-x`: 行全体一致、`-F`: 固定文字列）で検索する。

### すり抜けシナリオの検証

**config/script_blacklist.txt に `banned.py` が登録されている場合:**
- `python3 banned.py` → `PY_FILES=banned.py` → blacklist照合 → **BLOCKED（正常）**
- `python3 /full/path/to/banned.py` → `PY_FILES=banned.py` → blacklist照合 → **BLOCKED（正常）**

実はこのケースでは**すり抜けない**。grepがベースネーム `banned.py` を正しく抽出し、blacklistにも `banned.py` として登録されているため、完全一致する。

### 真のすり抜けケース

問題が発生するのは以下のケース:

1. **blacklistにフルパスで登録されている場合**:
   - blacklist: `/full/path/to/banned.py`
   - コマンド: `python3 /full/path/to/banned.py`
   - 抽出: `banned.py`
   - 照合: `grep -qxF "banned.py"` vs `/full/path/to/banned.py` → **不一致 → すり抜け**

2. **コマンド内のパス表記バリエーション**:
   - `python3 ./scripts/banned.py` → 抽出: `banned.py` → blacklistに `./scripts/banned.py` があれば不一致
   - `python3 ~/scripts/banned.py` → 抽出: `banned.py` → 同上

3. **156行目の初段フィルタのすり抜け**:
   ```bash
   if echo "$COMMAND" | grep -qE 'python3[[:space:]]+[^-][^[:space:]]*\.py' && \
      ! echo "$COMMAND" | grep -qE 'python3[[:space:]]+-[cm]|venv/|pip'; then
   ```
   このパターンは `python3 /full/path/to/banned.py` にマッチする（`/` は `[^-]` に該当）ので、初段フィルタはすり抜けない。

### 結論: 脆弱性は「blacklist登録形式の不統一」にある

blacklistにベースネーム（`banned.py`）で登録されていればすり抜けないが、フルパスや相対パスで登録されている場合にすり抜ける。設計上、blacklistの登録形式とgrep抽出結果の形式が暗黙的に一致する前提になっている。

## 修正案

### 修正案1: 照合時にベースネーム正規化（推奨）

blacklist側もベースネームに正規化してから照合する:

```bash
for PY_FILE in $PY_FILES; do
  PY_BASENAME="${PY_FILE##*/}"  # パスからベースネーム抽出
  if [[ -f "$BLACKLIST" ]] && while IFS= read -r bl_entry; do
    bl_base="${bl_entry##*/}"
    [[ "$PY_BASENAME" == "$bl_base" ]] && exit 0
  done < "$BLACKLIST"; then
    echo "BLOCKED: ${PY_FILE} はblacklistに登録された使用禁止スクリプトです。" >&2
    exit 2
  fi
done
```

**利点**: blacklistの登録形式に依存しなくなる。フルパス登録でもベースネーム登録でも動作する。

### 修正案2: blacklist登録ルールの明文化（簡易）

blacklistにはベースネームのみ登録するルールを明文化:
```
# config/script_blacklist.txt
# 登録形式: ベースネームのみ（パスを含めない）
banned.py
gemini_speaker_pipeline.py
```

CLAUDE.mdまたはREADMEに「blacklistはベースネームで登録すること」を追記。

### 修正案3: 両方向の照合（防御的）

```bash
for PY_FILE in $PY_FILES; do
  if [[ -f "$BLACKLIST" ]]; then
    PY_BASENAME="${PY_FILE##*/}"
    # ベースネーム完全一致 OR 行末一致（パス付き登録対応）
    if grep -qxF "$PY_BASENAME" "$BLACKLIST" || grep -qF "/$PY_BASENAME" "$BLACKLIST"; then
      echo "BLOCKED: ${PY_BASENAME} はblacklistに登録された使用禁止スクリプトです。" >&2
      exit 2
    fi
  fi
done
```

### 推奨

修正案1（ベースネーム正規化照合）を採用し、合わせて修正案2（登録ルール明文化）も実施する。二重防御により、将来blacklistの登録形式が揺れても安全性が保たれる。

## 補足: 追加の脆弱性

- **引用符によるバイパス**: `python3 "/full/path/to/banned.py"` の場合、grepパターンが引用符内のパスを正しく扱うかは未検証。現状の `[^/[:space:]]+\.py` は引用符を含まないため `banned.py"` が抽出される可能性がある（末尾の `"` が `.py` の後に付く）。ただし実際は `.py` にマッチした時点で抽出されるため、`banned.py` が正しく取れる。
- **複数スクリプト実行**: `python3 a.py && python3 banned.py` の場合、`PY_FILES` に両方抽出されるが、163行目のforループで各ファイルを個別チェックするため正常動作する。
