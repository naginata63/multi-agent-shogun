## 問題文
エッジケース考慮: generate_manga_short.pyでVertex API 504 DEADLINE_EXCEEDEDが頻発する問題の根本対策設計

## 現状分析

### 504 DEADLINE_EXCEEDED の発生メカニズム

Vertex AI Imagen API（gemini-2.0-flash-exp等）の画像生成リクエストは、サーバー側の処理時間がタイムアウト上限を超えた場合に 504 DEADLINE_EXCEEDED を返す。これは:

1. **サーバー側処理遅延**: 画像生成モデルの推論が想定より長い（高解像度/複雑プロンプト/負荷集中時）
2. **クライアント側タイムアウト**: HttpOptions(timeout=120_000)（2分）がサーバー処理時間より短い
3. **ネットワーク遅延**: GCSからのref_image取得やレスポンス転送の遅延

### 現在のコードの問題点

#### 1. 504がリトライ対象外
```python
RETRYABLE_ERRORS = ("429", "RESOURCE_EXHAUSTED", "503")
```
504/DEADLINE_EXCEEDEDがリトライ可能エラーに含まれていない。504は一時的なサーバー負荷による失敗であり、リトライで成功する確率が高いにもかかわらず、即失敗として処理される。

#### 2. タイムアウト設定の硬直性
```python
_http_opts = types.HttpOptions(timeout=120_000)  # 2分固定
```
全リクエストに一律2分。画像生成は通常30-90秒だが、複雑なプロンプト（マルチキャラ・詳細背景）では2分超えることがある。リトライ時にタイムアウトを延長する仕組みがない。

#### 3. リトライ間隔がレート制限向け
```python
BASE_WAIT = 30    # 秒
MAX_WAIT = 600    # 秒
wait = min(BASE_WAIT * (2 ** attempt), MAX_WAIT)
```
30秒→60秒→120秒→240秒→480秒→600秒の指数バックオフ。これは429/RESOURCE_EXHAUSTED（レート制限）には適切だが、504（タイムアウト）には過剰。504は即座にリトライすべき。

#### 4. エラー分類の粗さ
generate_panel_with_retry()は例外のstr(e)を文字列マッチで判定しているが:
- DEADLINE_EXCEEDEDはgoogle.api_core.exceptions.DeadlineExceededとして発生する場合がある
- gRPCステータスコードとHTTPステータスコードの混在
- 例外型による分類がされていない

## 根本対策設計

### 対策1: RETRYABLE_ERRORSに504/DEADLINE_EXCEEDEDを追加

```python
RETRYABLE_ERRORS = ("429", "RESOURCE_EXHAUSTED", "503", "504", "DEADLINE_EXCEEDED")
```

**最低限かつ最優先の修正**。これだけで504時に自動リトライが走る。

### 対策2: エラー種別に応じたリトライ戦略の分離

```python
# レート制限系: 長い待ち時間（サーバー側が回復するまで待つ）
RATE_LIMIT_ERRORS = ("429", "RESOURCE_EXHAUSTED")
RATE_LIMIT_BASE_WAIT = 30  # 秒

# タイムアウト系: 短い待ち時間（即座にリトライ、タイムアウト延長）
TIMEOUT_ERRORS = ("504", "DEADLINE_EXCEEDED")
TIMEOUT_BASE_WAIT = 5  # 秒（504はサーバー負荷が一時的）

# サーバーエラー系: 中程度の待ち時間
SERVER_ERRORS = ("503", "500", "INTERNAL")
SERVER_BASE_WAIT = 15  # 秒
```

#### リトライ判定ロジック
```python
def classify_error(error_str: str) -> str:
    error_upper = error_str.upper()
    if any(e in error_upper for e in RATE_LIMIT_ERRORS):
        return "rate_limit"
    if any(e in error_upper for e in TIMEOUT_ERRORS):
        return "timeout"
    if any(e in error_upper for e in SERVER_ERRORS):
        return "server"
    return "fatal"

def get_retry_params(error_type: str, attempt: int) -> tuple:
    if error_type == "rate_limit":
        wait = min(RATE_LIMIT_BASE_WAIT * (2 ** attempt), MAX_WAIT)
        timeout_extend = 0
    elif error_type == "timeout":
        wait = min(TIMEOUT_BASE_WAIT * (1.5 ** attempt), 60)
        timeout_extend = 30_000 * (attempt + 1)
    elif error_type == "server":
        wait = min(SERVER_BASE_WAIT * (2 ** attempt), 300)
        timeout_extend = 0
    else:
        return 0, 0
    jitter = random.uniform(0, wait * 0.1)
    return wait + jitter, timeout_extend
```

### 対策3: タイムアウトの動的拡張

504発生時、次のリトライでタイムアウトを延長する。

```python
def generate_panel_with_retry(client, panel, ...):
    base_timeout = 120_000
    current_timeout = base_timeout

    for attempt in range(MAX_RETRIES + 1):
        try:
            http_opts = types.HttpOptions(timeout=current_timeout)
            result = generate_panel(client, panel, ...)

            if result["status"] == "success":
                return result

            error_str = result.get("error", "")
            error_type = classify_error(error_str)

            if error_type == "fatal" or attempt >= MAX_RETRIES:
                return result

            wait, timeout_extend = get_retry_params(error_type, attempt)
            current_timeout += timeout_extend
            print(f"  Retry {attempt+1}/{MAX_RETRIES}: {error_type}, "
                  f"wait={wait:.0f}s, timeout={current_timeout/1000:.0f}s")
            time.sleep(wait)

        except Exception as e:
            error_str = str(e)
            error_type = classify_error(error_str)

            if error_type == "fatal" or attempt >= MAX_RETRIES:
                return {"panel_id": panel["id"], "status": "error", "error": error_str}

            wait, timeout_extend = get_retry_params(error_type, attempt)
            current_timeout += timeout_extend
            time.sleep(wait)
```

### 対策4: プロンプト複雑度に基づく事前タイムアウト調整

```python
def estimate_timeout(panel: dict) -> int:
    base = 120_000
    chars = len(panel.get("characters", []))
    if chars > 1:
        base += (chars - 1) * 30_000
    refs = len(panel.get("ref_images", []))
    if refs > 4:
        base += (refs - 4) * 15_000
    notes = panel.get("director_notes", "")
    if len(notes) > 500:
        base += 30_000
    if panel.get("is_climax"):
        base += 30_000
    return min(base, 300_000)  # 最大5分
```

### 対策5: サーキットブレーカーパターン

連続504が規定回数を超えたら、残りパネルの生成を一時停止して状況を報告する。

```python
class CircuitBreaker:
    def __init__(self, threshold=3, cooldown=300):
        self.consecutive_failures = 0
        self.threshold = threshold
        self.cooldown = cooldown
        self.last_trip_time = 0

    def record_success(self):
        self.consecutive_failures = 0

    def record_failure(self, error_type):
        if error_type in ("timeout", "server"):
            self.consecutive_failures += 1

    def is_open(self):
        if self.consecutive_failures >= self.threshold:
            now = time.time()
            if now - self.last_trip_time >= self.cooldown:
                self.last_trip_time = now
                self.consecutive_failures = 0
                return False
            return True
        return False
```

### 対策6: 504頻発時のfallback画像サイズ縮小

504が2回連続した場合、画像生成パラメータを軽量化してリトライする（画質低下を伴うため殿の許可を要する）。

## 追加考慮事項

### Vertex AI Imagen APIの504特有事情
1. Imagen 3は画像生成に平均45-90秒。2分タイムアウトは余裕が少ない
2. マルチキャラ指定の描画負荷: 2人以上のキャラを同一画面に配置するプロンプトは処理時間が1.5-2倍
3. ピーク時間帯: UTC 14:00-20:00はVertex AIの負荷が高く504発生率が上がる

### ガチャ上限との整合
- リトライは「同一プロンプト・同一パラメータ」での再試行であり「ガチャ」とは異なる
- 504リトライはカウント外。ガチャ上限は成功した生成の回数で数える
- 対策6（fallback軽量化）は画質パラメータを変更するためガチャに準ずる

## まとめ

504 DEADLINE_EXCEEDED対策の最重要ポイントは RETRYABLE_ERRORSへの504追加（1行修正） である。これだけで504時の自動リトライが開始され、一時的なサーバー負荷による失敗を自動回復できる。追加対策として、エラー種別に応じたリトライ戦略の分離、複雑度に基づく事前タイムアウト調整、サーキットブレーカーによる連続失敗時の自動停止を段階的に導入することで、漫画パネル生成パイプラインの信頼性を大幅に向上できる。
