# AIに漫画を描かせてYouTubeショートを量産した話

## はじめに

YouTubeショートで漫画動画を毎日投稿したい——でも漫画を描くのは時間がかかる。

そこで私は「AIに漫画を描かせて、それをショート動画にしてYouTubeに上げる」という実験を始めた。結果として、1本あたり30分程度で漫画ショートが作れるようになった。

この記事では、実際のワークフローと試行錯誤の記録を共有する。

## 1. なぜ「AI漫画ショート」なのか

YouTubeショートは60秒以内の縦型動画。漫画形式のコンテンツは視覚的にわかりやすく、エンタメとして需要がある。

手描きだと1コマ30分〜1時間。4コマ漫画で2〜4時間かかる。これをAI画像生成に置き換えることで、制作時間を大幅に短縮できる。

## 2. 全体ワークフロー

実際の制作フローは以下の5ステップ:

1. **候補選定**: YouTube動画から面白いシーンを選ぶ
2. **セリフ抽出**: 音声認識でセリフを文字起こし
3. **構成作成**: 漫画パネル構成JSONを作る
4. **画像生成**: Gemini APIで各パネルの画像を生成
5. **動画合成**: ffmpegで縦型動画に合成→YouTube非公開アップ

この全工程をCLIツールとシェルスクリプトで半自動化している。

## 3. AI画像生成の実際

### 使用ツール

- **Gemini 2.0 Flash**（画像生成API）: 各パネル画像の生成
- **Claude Code**（CLI）: プロンプト設計と品質管理
- **ffmpeg**（NVENC）: 動画エンコード

### プロンプト設計のコツ

漫画パネルの画像生成では、プロンプトの設計が品質を左右する。私が気をつけているポイント:

- **黒白指定**: 「monochrome manga style」で統一
- **コマ割り禁止**: 「single panel, no panel borders」を指定しないと、AIが勝手にコマ割りしてしまう
- **キャラ一貫性**: キャラクターの外見特徴を毎回同じプロンプトに含める
- **吹き出し禁止**: セリフは後から合成するので「no speech bubbles」を指定

### ガチャ問題と対策

AI画像生成は「ガチャ」——同じプロンプトでも毎回違う結果になる。

最初は全パネルを一度に生成していたが、1枚でも手や顔が崩れると全部作り直しになっていた。今は:

1. **まず1パネルだけ試し生成**
2. 問題なければ残りを生成
3. NGパネルだけ個別に再生成（最大3回）

この「1パネル先行情報」だけで、API呼び出し回数を半分以下に抑えられた。

## 4. 動画化の工程

画像が揃ったら、ffmpegで動画に合成する:

1. パネル画像→PNG連番
2. 音声（元動画から抽出 or TTS）を結合
3. ffmpegで縦型（1080x1920）動画に変換
4. YouTube APIで非公開アップロード

エンコードはNVENC（GPU）を使う。CPUエンコード（libx264）だと30分動画に3時間以上かかるが、NVENCなら数分で完了。

## 5. ハマりどころと対策

### 手が6本になる問題

AI漫画の定番。プロンプトで「normal 5 fingers each hand」と指定しても100%防げない。

**対策**: 試し生成で確認→NGなら再生成。3回ガチャってダメなら諦めて構図を変える。

### キャラクターの一貫性

同じキャラがパネルごとに別人になる問題。

**対策**: reference image（参考画像）を生成時に渡す。Gemini APIの画像入力を使って、最初のパネル画像を後続パネルの参考にする。

### Vertex API 504エラー

画像生成APIが頻繁にタイムアウトする問題。

**対策**: チェックポイント機能で完了済みパネルをスキップ。再実行時に最初からやり直さなくて済むようにした。

### 吹き出し内の文字化け

AIが吹き出し内に謎の文字を生成する問題。

**対策**: そもそもAIに吹き出しを描かせない。セリフは後から別レイヤーで合成する。

## 6. コストの話

### Gemini API無料枠の罠

Gemini APIには無料枠があるが、ガチャ運用で調子に乗っているとあっという間に超過する。

**実例**: ある時、プロンプトを調整しながら3,409枚生成したら、無料枠を超過して22,000円の請求が来た。

**現在の対策**:
- 同一パターンは最大3回まで
- 1パネル試し打ち→確認→残り生成
- GCPで月額上限設定（予算アラート付き）

### コスト試算（1本あたり）

| 項目 | コスト |
|------|--------|
| 画像生成（4-6パネル） | 0円（無料枠内）〜 数十円 |
| YouTube API | 0円 |
| ffmpeg（GPU） | 電気代のみ |
| 音声認識 | 0円（WhisperX ローカル） |
| **合計** | **0円〜数十円** |

無料枠内に収めれば、実質ゼロコストで運用可能。

## 7. 成果と今後

### 現在の成果

- 1本あたり制作時間: **30分〜1時間**（手描きの1/10以下）
- 品質: 「AI感」はあるが、ショート動画としては十分視聴に耐えるレベル
- 運用: CLIから一撃で非公開アップまで完了

### 今後の課題

1. **キャラクターの一貫性向上**: より高品質なreference画像の管理
2. **セリフ合成の自動化**: 吹き出し位置の自動判定
3. **投稿の自動化**: タイトル・説明欄・サムネイルの自動生成
4. **品質の安定化**: ガチャ成功率の向上（現在7-8割）

### まとめ

AIに漫画を描かせるのは、まだ「手描きの代替」ではない。しかし「ショート動画用のエンタメコンテンツ」としては十分に実用レベルに達している。

重要なのは、AIの弱点（手、一貫性、文字）をワークフロー側でカバーすること。そしてコスト管理を怠らないこと。

「AIに任せきり」ではなく「AIの得意なことを活かしつつ、人は確認する」という体制が、現時点での最適解だと思う。
