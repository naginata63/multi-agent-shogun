# Q1: pretool_check.sh blacklist パス付きスクリプト名すり抜け問題 — 原因分析+修正案

advisor model: claude-opus-4-6[1m]
実測開始時刻: 2026-04-17T13:26:19

## 問題の概要

`pretool_check.sh` L149-206 のスクリプトblacklistチェックにおいて、`python3 /full/path/to/banned.py` のようにフルパス指定で実行された場合、blacklistチェックをすり抜ける脆弱性が存在する。

## 原因分析

### 抽出ロジック（L158）

```bash
PY_FILES=$(echo "$COMMAND" | grep -oE "[^/[:space:]]+.py" | grep -v "__pycache__")
```

正規表現 `[^/[:space:]]+.py` はスラッシュと空白を含まない .py 文字列をマッチさせる。

`python3 /full/path/to/banned.py` 入力時:
- `[^/[:space:]]+` は /\ を拒否 → パス全体には不マッチ
- しかし `banned.py` 部分だけが部分マッチする

**実際の挙動**: `grep -oE` は各位置で最長マッチを探す。パス内の "/" 直後から `banned.py` がマッチ。**フルパスでもbasename部分を正しく抽出可能**。

### 真の脆弱性: パターンマッチの前提条件（L156-157）

```bash
if echo "$COMMAND" | grep -qE "python3[[:space:]]+[^-][^[:space:]]*.py" && \
   ! echo "$COMMAND" | grep -qE "python3[[:space:]]+-[cm]|venv/|pip"; then
```

`python3 /full/path/to/banned.py` → 空白の次が "/"（"-" ではない）→ `[^-]` にマッチ → チェック対象になる。

### すり抜け可能な5パターン

1. **コマンド置換**: `python3 $(cat config/banned_script.txt)` — grep時点で未展開 → `.py` にマッチせずスルー
2. **環境変数**: `python3 $SCRIPT` — 展開されず `.py` 終端なし → マッチ回避
3. **バックスラッシュ区切り（WSL2）**: `python3 "C:\path\to\banned.py"` — \ 含むためblacklist `grep -qxF` と不一致
4. **シンボリックリンク**: `python3 ./symlink_to_banned.py` — symlink名 ≠ blacklist名
5. **引用符付き**: `python3 "banned.py"` → `[^[:space:]]*` が引用符を含む → `.py"` となり `grep -qxF` 不一致の可能性

## 修正案

### 修正1: basename抽出の明示化（L158）

```bash
# 修正後: basename確実抽出 + 引用符除去
PY_FILES=$(echo "$COMMAND" | python3 -c "
import sys, re, os
cmd = sys.stdin.read()
for m in re.finditer(r"python3\s+([^\s;|&]+.py)", cmd):
    path = m.group(1).strip(chr(34) + chr(39))
    print(os.path.basename(path))
" 2>/dev/null | grep -v __pycache__ | sort -u)
```

### 修正2: 変数・コマンド置換検出（L156の後に追加）

```bash
if echo "$COMMAND" | grep -qE "python3[[:space:]]+" && \
   echo "$COMMAND" | grep -qE "\$[{(]|\`"; then
  echo "WARNING: 変数/コマンド置換経由のpython3実行。blacklist回避の可能性。" >&2
fi
```

### 修正3: シンボリックリンク解決（L162-169ループ内）

```bash
for PY_FILE in $PY_FILES; do
  RESOLVED="$PY_FILE"
  for SEARCH_DIR in "$REPO_DIR/scripts" "$REPO_DIR/tools"; do
    if [[ -L "${SEARCH_DIR}/${PY_FILE}" ]]; then
      REAL=$(readlink -f "${SEARCH_DIR}/${PY_FILE}" 2>/dev/null)
      RESOLVED=$(basename "$REAL" 2>/dev/null || echo "$PY_FILE")
    fi
  done
  if [[ -f "$BLACKLIST" ]] && grep -qxF "$RESOLVED" "$BLACKLIST"; then
    echo "BLOCKED: ${RESOLVED} はblacklistに登録された使用禁止スクリプトです。" >&2
    exit 2
  fi
done
```

## 優先度評価

| 修正 | 優先度 | 理由 |
|------|--------|------|
| 修正1: basename明示化 | 中 | 引用符付きパスでの誤抽出防止。現状でもフルパスbasenameは抽出される |
| 修正2: 変数展開警告 | **高** | \$() でblacklist完全回避可能 |
| 修正3: symlink解決 | 低 | 足軽がsymlinkを使うユースケースほぼなし |

## 結論

パス付きスクリプト名のすり抜けは現行実装で大部分は発生しない。`grep -oE "[^/[:space:]]+.py"` がフルパスのbasenameを正しく抽出するため。

真のリスクは**コマンド置換・環境変数経由の実行**にあり、grep時点で静的判定不可能なためblacklist全体をすり抜ける。修正2（変数展開警告）を最優先とする。
