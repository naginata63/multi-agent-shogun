違いは「結果の確認方法」を指定してるかどうか。
検証手段を伝えると、Claude Code自身が「これで合ってるかな？」と内部でチェックするようになり
ます。 出力の形式が決まってるので、曖昧な回答になりにくい。
人間に仕事を頼むときと同じですよね。

Codeも全く同じです。

パターン1: ファイル検索

Before/Afterがあると、何をどう変えたか一目瞭然。 「勝手に変えられた」という事故も防げます。
パターン3: データ集計
出力形式を「表」と指定するのがポイント。 表になっていれば、数字の確認がしやすい。

あと、おまけのテクニックを1つ。
プロンプトの最後に 「わからないことがあったら聞いて」 と付けると、勝手に暴走しなくなります。
曖昧な部分があれば、実行前に確認してくれる。 これだけで事故率がかなり下がるので、ぜひ試して
みてください。
まとめ
プロンプトのコツは「検証手段を与える」。これだけ。
難しいプロンプトエンジニアリングは要りません。 「やって」を「やって、結果はこう見せて」に変
えるだけで、回答の精度がガラッと変わります。
