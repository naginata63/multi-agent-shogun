コンパクションって何？
【画像: コンパクションの仕組み——会話が長くなると古いやり取りが要約・圧縮され、詳細な指示が
消えていく図。キャラが「あれ？さっきの指示どこいった？」と困惑している (01-what-is-
compaction.png)】
AIには「一度に覚えていられる量」に上限があります。
イメージとしては、作業デスクの広さ。
会話が短いうちは、デスクに余裕がある。 指示書も資料も全部広げておける。
でも会話が長くなると、デスクが書類で埋まっていく。
で、上限に達すると、AIが自動的に「古い会話を要約して圧縮する」んです。
これがコンパクション。
問題は、この圧縮のとき細かいニュアンスが消えること。
「丁寧語で話して」は残るかもしれない。 でも「ただし記事の後半ではカジュアルにしてOK」みたい
な例外ルールは、圧縮のときに削られやすい。

「ポインタ方式」という発想
【画像: 図書館の目録カード方式の図。左に目録カード「記事を書く → content/ → voice.md」。中央
に矢印。右に本棚から1冊だけ取り出しているキャラ。吹き出しで「全部持ち歩かなくていいんだ」
(03-pointer-method.png)】
で、どうしたかというと。
CLAUDE.mdには「何を知っているか」ではなく「どこを見ればいいか」だけ書くことにしました。
これを私は「ポインタ方式」と呼んでます。
図書館で本を探すとき、目録カードを見ますよね。
目録カードに本の内容が全部書いてあるわけじゃない。 「この本は3階のB棚にあります」という場所
の案内だけが書いてある。
CLAUDE.mdも同じ。
たとえば、記事を書くときの口調ルールはけっこう長い。 文の長さ、使ってはいけないフレーズ、冒
頭の書き方、タイトルの作り方...全部合わせると100行くらいになる。
これを全部CLAUDE.mdに書いていたら、それだけで100行消費。
でもポインタ方式なら、こう書くだけで済む。
たった1行。
AIは「記事を書いて」と言われたら、この1行を見て、必要なときだけ writing-style.md を読みに行
く。
ふだんは1行分のスペースしか使わない。 でも必要なときは100行分の情報にアクセスできる。
コンパクションで消えるのは「詳細」。「参照先」は短いから消えにくい。
記事を書くとき → writing-style.md を読む

実際の設計パターン——「ルーター」で情報を振り
分ける
【画像: 6つのモジュール構成図。中央にBRAIN.md（ルーター）があり、そこから6方向に矢印が出て
いる。各矢印の先に「自分は誰か？→ identity/」「何を作るか？→ content/」「何を知っているか？→
knowledge/」「誰を知っているか？→ network/」「何をやるか？→ operations/」「どう自動化するか？
→ agents/」とテキストラベル付きの箱がある (04-router-pattern.png)】
ポインタ方式をもう少し体系的にやると、こうなります。
私が今使っている構成は、3段階。
第1段階: CLAUDE.md（入口）
ここには「絶対ルール」と「参照先の一覧」だけを書く。 300行以内が目安。私のは今288行。
第2段階: BRAIN.md（ルーター）
「何の作業をするか」によって、どのフォルダを見るかを振り分ける役割。
私の場合、情報を6つの「問い」で分類しています。
ここから先は有料部分です
自分は誰か？       → identity/（性格・口調）
何を作るか？       → content/（コンテンツ戦略）
何を知っているか？ → knowledge/（学習記録・書籍）
誰を知っているか？ → network/（人脈・連絡先）

<!-- page 6 -->

たとえば「記事を書いて」と言われたら。
AIはまず BRAIN.md を見る。 → ルーティングルールに「記事を書く → content/ で戦略確認 →
identity/voice.md でトーン確認」と書いてある。 → 2つのフォルダだけ読めばいい。他の4モジュー
ルはスキップ。
こうやって、必要な情報だけを段階的に辿る。
全部を最初から読み込むんじゃなくて、作業内容に応じて必要なファイルだけ読む。
第3段階: 各フォルダのMODULE.md（案内板）
各フォルダにも、中の案内図を置いてあります。
たとえば identity/ フォルダに入ると、MODULE.md にこう書いてある。
AIはこれを見て、「今は口調の話だから voice.md だけ読めばいいな」と判断する。
profiles/ とか patterns/ は今回関係ないからスキップ。
Progressive Disclosure（段階的開示）と呼ばれる設計で、「最初は目次だけ見せて、必要なところだ
け開く」という考え方です。
図書館で言えば、フロアガイド → 棚の分類表 → 本の目次 → 該当ページ、と辿る流れ。
これを毎日の運用で回しています。 77本以上の記事をこの仕組みで書いてきて、コンパクションで口
調が崩れることはほぼなくなりました。
何をやるか？       → operations/（日次ログ・計画）
どう自動化するか？ → agents/（自動化の設計）
voice.md        → 書き味・トーン定義
profiles/       → 性格診断結果
patterns/       → 行動パターン・トリガールール
bio-variants.md → プロフィール文

コンパクションに強いCLAUDE.mdの書き方——3
つのルール
【画像: 3つのルールを表す図。ルール1「目次だけ書く」: CLAUDE.mdが薄くてスッキリ。ルール2
「詳細は別の部屋に」: 別ファイルに詳しい内容が入っている。ルール3「道順を書く」: 矢印付きのル
ーティング表。キャラが「これならコンパクションされても大丈夫！」と笑顔 (05-three-rules.png)】
ここまでの話をルール化すると、3つ。
ルール1: CLAUDE.mdは「参照パス」と「絶対ルール」だけにする
目安は300行以内。
「絶対ルール」は前回の記事で紹介した「壁」の層。 「このファイルは消すな」「この操作はするな」
みたいな、破られたら取り返しがつかないもの。
それ以外は、参照パスだけ書く。
ルール2: 詳細は別ファイルに置いて、必要なときだけ読ませる
「口調のルール」「フォルダ構造の説明」「手順書」は全部別ファイル。
CLAUDE.mdから切り出せるものは、どんどん切り出す。
判断基準はシンプルで。 「これ、毎回読む必要ある？」と自問する。
毎回必要 → CLAUDE.mdに残す。 特定の作業のときだけ → 別ファイルに移す。
ルール3: 「いつ・何を読むか」のルーティングルールを書く
これが一番大事かもしれない。
記事を書く → writing-style.md を読む
英語学習   → knowledge/english/ を見る
画像生成   → generate-image の手順に従う

<!-- page 8 -->

ファイルを分けただけだと、AIが「どのファイルを読めばいいか」わからない。
だから「○○のとき → △△を読む」というルーティングを明示する。
こうしておけば、AIは作業内容を見て、自動的に必要なファイルだけ読みに行く。
コンパクションが起きても、この「道順」は短いから残る。
道順さえ残っていれば、詳細はいつでも読み直せる。
本の内容を全部暗記する必要はない。 本棚のどこにあるかだけ覚えておけばいい。
まとめ
【画像: Before/After図。左（Before）: 巨大なCLAUDE.mdから情報が溢れ出て、キャラが「また忘れ
た！」と頭を抱えている。右（After）: スリムなCLAUDE.mdから矢印が伸びて各ファイルを指し示
し、キャラが「必要なときだけ取りに行く」と落ち着いた顔 (06-before-after.png)】
CLAUDE.mdに全部書く → 長くなる → コンパクションで消える → 設定が吹っ飛ぶ。
これが「記憶喪失」の正体でした。
解決策はシンプル。
「全部書く」をやめて「道順だけ書く」に変える。
1. CLAUDE.mdは300行以内。絶対ルールと参照パスだけ
2. 詳細は別ファイルに切り出す
3. 「いつ・何を読むか」のルーティングを明示する
| 作業 | 参照先 |
| 記事を書く | content/ → writing-style.md |
| 英語の練習 | knowledge/english/SPEAKING_STRATEGY.md |
| デイリーログ | operations/logs/daily/ |

してたのは、AIじゃなくて私の設計だったっていう。
まぁ、世の中そんなもんかなと。
