自分は今、Claude Codeのインスタンスを11体同時に動かして、 RPGのギルドパーティとして毎日運 用している。 ギルドマスター（司令塔）、勇者（実行指揮官）、賢者（記事専任）、 残り8体がGLM-5の 実行部隊。 モデルもOpus 4.6とGLM-5を混在させて、コストと品質を両立させてる。

この運用をやってみて分かったのは、 「サブに何を読ませるか」で、チーム全体の生産性が劇的に変 わるということ。

今回はOpenClaw、Claude Code、自作ギルドシステムの3つを比較して、 「どう設計すれば、サブ が"使える子分"になるのか」を解説する。

比較する3つのアーキテクチャ

3つのアーキテクチャ概要：OpenClaw・Claude Code・ギルドシステム

今回比べるのはこの3つ。

1. OpenClaw ─ githubで100kスター（お気に入り）がつくほどの怪物OSS 起動するたびに7つのファ イルを読み込んで「人格」を形成する。 サブエージェントにはそのうち2つしか渡さない、という割 り切り設計。

2. Claude Code ─ Anthropic純正のCLIツール サブエージェント（Task tool）と、新機能のAgent Teamsの2段構え。 それぞれコンテキストの渡し方がまったく違う。

3. 自作ギルドシステム ─ RPGパーティ型マルチエージェント Claude Codeの独立したAIを11個たばね て並列起動させてる将軍システムを改造した自前のマルチエージェントシステム。 それぞれに専用の ロールファイル（人格）を注入する力技構成。要するにパワープレイしてる。

OpenClaw ─ 7つのファイルが「人格」になる

OpenClawの7ファイル構成：メイン7つ vs サブ2つだけ

OpenClawのコンテキスト設計はめちゃくちゃ分かりやすそう。

起動時に読み込まれるファイルはこの7つ（+1）。

全部テキストファイル。 データベースもAPIもない。ただのMarkdown。

ここが面白いんだけど、SOUL.mdは「設定」じゃなくて「哲学」を書く場所。 「丁寧に話せ」じゃな くて「なぜ丁寧に話すのか」を書く。

で、記憶（MEMORY.md）は毎ターン注入されるから、 ファイルが肥大化するとトークン消費がエグ いことになる。 /context listで確認できるけど、油断するとすぐコンテキストが溢れる。

OpenClawのサブエージェント ─ 潔い割り切り

ここがこの記事の核心。

OpenClawのサブエージェントが受け取るのは、①と③だけ。

① AGENTS.md → 全員共通のルール ② SOUL.md → 人格、性格、哲学 ③ TOOLS.md → 使えるツール一覧 ④ IDENTITY.md → 自己紹介（名前・アバター等） ⑤ USER.md → ユーザーの情報 ⑥ HEARTBEAT.md → 定期チェック項目 ⑦ MEMORY.md → 過去の記憶 （⑧ BOOTSTRAP.md → 新規ワークスペースのみ）

メインエージェント → ①②③④⑤⑥⑦ 全部読む サブエージェント → ①③ だけ

つまりサブは： ❌ SOUL.md（人格）なし

人格も記憶もユーザー情報もない。 ルール（AGENTS.md）とツール（TOOLS.md）だけ渡されて、 「この作業やれ」と言われて動く。

人間で例えると、派遣社員がマニュアルだけ渡されて現場に送り込まれる感じ。

しかもサブエージェントは別のサブを生成できない（ネスト禁止）。 使い終わったら60分でアーカイ ブされる。 完全に使い捨ての作業マシンとして設計されてる。

辛口で言うと、これは「サブが使えない」じゃなくて「使えなくしてる」設計。

ただし、この割り切りにはちゃんとメリットがある。 サブのコンテキストが小さい = トークン消費が 少ない = 安い。

重い作業はサブに安いモデルを割り当てて回せるから、 メインはOpus 4.6、サブはHaikuみたいな使 い分けも簡単にできる。 「バカだけど安い」を戦略的に使う設計。これは潔い。

❌ IDENTITY.md（自己紹介）なし ❌ USER.md（ユーザー情報）なし ❌ HEARTBEAT.md（定期チェック）なし ❌ MEMORY.md（記憶）なし
