スキルの「隠れた上限」を知ってるか？

2%バジェット制約の図解 — 63個中21個が非表示になる仕組み

Claude Codeのスキル（/xxx で呼び出すカスタムコマンド）には、公式ドキュメントに明記されてい ない制約があります。

それが 2%バジェット。

これは何かというと、スキルの一覧情報（名前・説明文など）がシステムプロンプト（Claude Codeに 最初から組み込まれているAIの基本設定）全体の2%を超えると、超えた分のスキルが 非表示化 され る仕組みです。

非表示 = AIが存在を認識できない。

つまり /my-skill と打っても「そのスキルは知りません」と返ってくる。作ったのに。登録してるの に。

具体的な数字で言うと、スキルの読み込みに使える上限は約16,000文字。ただしこれは英語基準の数 字で、日本語は1文字あたりの消費が大きいので、実質的な上限はもっと少なくなります。スキルの名 前と説明文の合計がこれを超えると、優先度の低いスキルから順に消えていきます。

実際にどれくらいで溢れるのか

GitHub Issue #13099（Claude Codeの公式バグ報告・要望掲示板に投稿されたレポート）で報告され た実測データが生々しい。

63個のスキルを登録したユーザーが確認したところ、33%（21個）が非表示になっていた。3分の1が 消えてるって、なかなかの衝撃ですよね。

自分の環境でも調べてみました。

1.2%なのでギリギリ2%の枠内に収まってる。

でもこの数字、description（説明文）を意識的に短くしてるから出せてるんです。

descriptionというのは、スキルごとに書く「このスキルは何をするか」の一行説明。AIはこの説明文 を読んで「どのスキルを使うべきか」を判断しています。

つまり、この説明文が長いと1スキルあたりのトークン消費量が格段に増える。スキル数が少なくても 簡単に2%を超えます。

今すぐ確認する方法

Claude Codeで /context コマンドを実行するだけ。

これでスキルがどれだけバジェットを消費しているか確認できます。1.5%以下なら安全圏。それ以上 なら整理が必要です。

ちなみに、環境変数 SLASH_COMMAND_TOOL_CHAR_BUDGET で上限を上書きもできます。

ただしこれは「枠を広げてるだけ」で根本解決ではない。情報量が増えれば、AIの処理能力に別の負 荷がかかります。

Boris Cherny（Claude Code開発者本人）

Claude Codeの開発者自身が、自分のセットアップについてこう語っています。

「最良のセットアップはあなたの邪魔をしない」

設定の微調整に時間を使うより、検証ループ（コードを書いて → テストして → 直す）に投資すべき だと。

そして自分のセットアップは「驚くほどバニラ」だそうです。バニラというのは、アイスクリームの 一番シンプルな味がバニラであることから「ほぼ素の状態」を意味するIT用語。

開発者本人がバニラ。これ、結構衝撃的じゃないですか。

Shrivu Shankar

Claude Codeのパワーユーザーとして知られるShrivu Shankarのカスタムコマンドは、たった 2個だ け。

/catchup（プロジェクトの状況を把握する）と /pr（プルリクエストを作る）。

それだけ？と思いますよね。でも彼の主張はこう。

「大量のカスタムコマンドを作ること自体が、システムの改善が必要なサインだ」

つまり、スキルを量産してる時点で設計を見直すべきタイミングだと。耳が痛い...。

Andrej Karpathy

AI業界の重鎮Karpathyは、LLM（大規模言語モデル）の使い方についてこう指摘しています。

「LLMは過剰に複雑化する。『1000行ではなく100行で書けないか？』を自問さ せるルールを入れろ」

これはコードの話ですが、スキル設計にも当てはまります。10個のスキルでやってることを、2個に集 約できないか。そういう発想が必要。

彼が最も強力と評価しているのは、Plan Mode（計画モード）。スキルを増やすより、AIに考えさせる 時間を与えるほうが効果的だという話です。

Simon Willison

Pythonコミュニティで著名なSimon Willisonは、スキルの利点をMCPサーバー（外部ツール連携の仕 組み）と比較して語っています。

「スキルはMCPより軽量で柔軟。MCPはGitHubだけで46,000トークン消費する」

46,000トークンって、文庫本で言うと50ページ分くらい。それがGitHub連携だけで消えるのは痛い。

彼の提案は「ドメイン（用途）ごとにモジュール化すべき」。全部入りの巨大スキルではなく、小さく て目的が明確なスキルに分けようということです。
