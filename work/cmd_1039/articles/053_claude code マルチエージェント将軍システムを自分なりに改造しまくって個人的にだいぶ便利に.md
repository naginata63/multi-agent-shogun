将軍システムからギルドシステムへの改造

世界観の改造 — 将軍からギルドへ

まず最初にやったのが、名前の変更です。

戦国時代の軍制も悪くなかったんですけど、RPGのギルド運営の方がしっくり来たんですよね。

対応はこんな感じ。

【将軍システム】→【ギルドシステム】

将軍（1体） → ギルドマスター（戦略参謀） 家老（1体） → 勇者（自律型司令塔） 足軽（4体） → パーティメンバー（10人） 剣士、武闘家、魔法使い、僧侶、 踊り子、盗賊、商人、狩人、賢者、遊び人

で、名前だけじゃなくて口調も設定してます。

Maid-Agent-Controllerもエージェントにキャラクターペルソナを付けてて、「これ、めっちゃいいな」 と思ったんですよね。

実際やってみたら、かなり効く。

ギルマス: 「ふむ、よかろう。勇者に依頼を出そう」

勇者: 「了解だ！kenshi、mahou、出撃だ！」

賢者: 「ええ、お任せなさい。記事の品質は保証するわよ」

10体以上のエージェントが同時に動いてると、ログが大量に流れるんですよね。

でも口調が違うから、誰の発言か一瞬でわかる。

実装は --append-system-prompt でロールファイルを読み込ませるだけ。

名前と口調、たったこれだけでログの可読性がめちゃくちゃ上がります。

地味だけど、一番効果があった改造かもしれない。

構成の改造 — 2チーム制から1チーム大所帯へ

前回は2チーム制でした。

claude --append-system-prompt "$(cat brave-role.md)"

これを1チーム制に変えました。

なぜ戻したかというと、チーム間の連携が難しかったから。

チームAの結果をチームBが使いたい時、将軍を経由する必要があって、そのオーバーヘッドが結構大 きかったんですよね。

それよりも、1人の優秀な司令塔（Opus勇者）が10人を直接指揮する方がシンプルで速い。

あとtmuxのレイアウトも変えました。

さらに、背景色でロールを識別してます。

【前回】 将軍（1体） ├ チームA: 家老A + 足軽4体 └ チームB: 家老B + 足軽4体

【今回】 ギルドマスター（1体） └ 勇者（1体） └ パーティメンバー（10体）

【前回】pane方式 → 1画面を6分割。各エージェントの表示が狭すぎた

【今回】window方式 → 各エージェントが独立したフルスクリーンwindow → tmuxのタブで切り替える

ギルマス: 金色 勇者: 緑色 賢者: ピンク パーティ: 赤色

tmuxのタブバーを見るだけで、今どのエージェントが何をしてるか一目でわかる。

これも地味に便利。

通信基盤の改造 — YAMLからMCPへ

前回はYAMLファイルでエージェント間通信をしてました。

正直、これが一番つらかった。

特にsend-keysは本当にヤバくて。

tmux send-keysの本質は「ターミナルにキーボード入力を偽装する」こと。

相手が忙しい時に送ると、普通にロストするんですよね。

しかも送った側は「届いたかどうか」がわからない。Fire and Forget。

ということで、MCPサーバーを自作しました。

MCP（Model Context Protocol）は、Claude Codeに外部ツールを追加できる仕組みです。

これを使って、タスク管理の専用サーバーを作った。

【前回の課題】 1. ファイルの読み書きで競合が起きる 2. ポーリング（定期チェック）でトークンが溶ける 3. send-keysが不安定（送ったのに届いてない問題）

ポイントはイベント駆動になったこと。

前回は「定期的にファイルを見に行く」ポーリング方式で、これがトークンを溶かしてた。

今は「通知が来たら動く」だから、無駄なAPI呼び出しがゼロ。

あと、Webダッシュボードも作りました。

ブラウザで http://localhost:3100/dashboard を開くと、全エージェントの進捗がリアルタイムで見え る。

「対応待ち」セクションには、人間の判断が必要なタスクが自動で表示される。

タスクの状態がYAMLファイルじゃなくてMCPサーバーに一元管理されてるから、競合も起きない。

これが一番大きな改造でしたね。

YAML直接操作からMCP通信への移行

指揮系統の改造 — 誰が誰に指示するか

基本の指揮系統はこう。

【今回の仕組み】 ギルマス: create_command("記事を3本書け") ↓ MCPサーバーが自動でイベント生成 勇者: poll_events() → 新しい指令を取得 ↓ タスク分解 勇者: create_task("kenshi", "リサーチしろ") ↓ MCPサーバーが自動でイベント生成 kenshi: poll_events() → タスクを取得して実行
