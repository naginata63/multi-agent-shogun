Before：Claude Code 1体運用の限界

誤解しないでほしいんですけど、Claude Code単体でも十分強いです。

自分もしばらくは1体だけでガシガシ使ってました。 コード書いてもらって、レビューしてもらって、 修正してもらって。 それだけでも生産性はかなり上がってたんですよね。

でも限界が見えてきた。。。というか、だんだん超速スピードに慣れてきて、「もっと精度高く早くでき るようになりたい」って思うよになりました（超欲張り）

まず、コンテキストの混濁問題。

1つのAIで、いろいろなタスクをやらせると、 情報が混ざってきて精度が落ちる。要するにAIも頭がこ んがらがる訳です。

「さっきの修正忘れてない？」 「あれ？あっちのタスクでさっきデキてたことがデキなくなってる！」 ってツッコミが増えてくる。

次に、待ち時間。

複雑なタスクだと、1つの処理が終わるのを待ってから次に進む。

自分がボトルネックじゃなくて、エージェントの処理待ちがボトルネック。 これがけっこうストレス なんですよね〜。長い時はコーヒー淹れたり、散歩する時間があるくらい待つ。これ本当です。

そして一番つらかったのが、コンテキストの枯渇。 長いやり取りを続けてると、途中でコンテキストウィンドウが限界に達する。人間で言うところの 「忘れる」ってやつですね。

そうなると最初からやり直し。 「ここまで積み上げた文脈が全部消えるの？」っていう絶望感。

まーた同じこと言わなきゃっていう頻度がねぇ〜結構あるんです。

導入のタイミング

Agent Teamが公式に発表されるちょっと前に、Shogunシステム（勝手に命名しましたm(_ _)m ）を 導入しました。

構成はさっき言った通りこんな感じ。

将軍（1体）：全体の指揮・方針判断

家老（1体）：タスクの分解・進捗管理・品質チェック

足軽（4体）：並列でタスクを実行する実働部隊

tmuxという機能をclaude codeに追加して、6エージェントが同時に動きます。 1つの画面に6つのAI指 示用のエリアがが並んでる状態になります。

イメージとしては↓こんな感じ。

将軍が「このプロジェクト、こう進めるぞ」と方針を出す。 家老がそれを「タスクA〜D」に分解す る。 足軽4体がそれぞれ1タスクずつ引き受けて、並列で作業を開始する。

人間がやることは、将軍に最初の指示を出すだけ。 あとは軍勢が勝手に動いてくれます。

コスト戦略：GLMで軍勢をお得に使い倒す

考える将軍はOpus、その他は超安のGLM

「6体も動かしたらコストやばくない？」

って思いますよね。自分も最初はそう思いました。動かしてみたらパフォーマンスはめっちゃいいけ どスグ制限きちゃってリセット待ち。

なので僕は考えました…！

将軍だけトークンコストの高いOpus4.6を使う。 一緒に方針考えたり戦略考えたり。とにかく「考える」という知能の高さが必要だから。

その他は判断力も求められるけど「思考」のレベル感は下がっても問題ない。ということで家老と足 軽は、めっちゃくちゃ安い「GLM-4.7」に切り替えてます。

え？Claude Code使ってるんじゃないの？ Claude ってopus・sonnet・haikuという3種類のAIモデルしか使えないでしょ？？ なにGLM-4.7って？？？

そう思ったアナタ。非常に勘が鋭いですね。

実はClaude Codeの設定をイジると、Claude 以外のLLM（AIの頭脳）を使うことができます。その別 の頭脳が「GLM-4.7」というやつなんですね。

とは言え、、、GLMのコスパ、正直ヤバいです。

年間約5,000円。コーディングに関してはOpusのちょっとしたChatGPT5.2より上って自負してるみた いです。

ぶっちゃけ体感的にはSonnet 4.5くらいかなぁというレベル。それでも結構良いですよね。それが年 間5,000円ですよ。

しかもClaude Pro Planの3倍のトークン容量があって、 5時間でリセットされる。実質使いたい放題。

ということは、将軍（Opus）以外の5体を、 ほぼコストを気にせず全力投入できるわけです。

言い忘れてましたが、僕はClaudeのMAX（$100）プランに入っています。1日のうち、ほとんどをAI を使いながら作業している人間からすると、shogunシステムのチーム体制をとるとバカスカリミット に到達してしまいます。

なので、この「GLM-4.7」を積んだ部下たちが必要になります。

「年間5000円払ってお得に軍勢を使い倒して敵（タスク）を倒しにいける」

この構成にたどり着いたとき、 「あ、これコスパ的にも現実的だ」って確信しました。

最大のメリット：コンテキストが濁らない

脳内グチャグチャ＝出力精度だだ下がり

Shogunシステムの一番のメリットはここだと思ってます。

1体に全部やらせると、コンテキストに色んな情報が混ざりますよね。 いろんな複雑なタスクが１つ の頭の中で同居してる状態。

そうなると、どうしても精度が落ちる。どんなに頭の良いAIでも 「あれ、さっきの変数名なんだっけ」 みたいなブレが出てくる。 指示を出し直す手間も増えて、結局時間がかかる。

でもShogunシステムのようにチーム体制だと、各スタッフが1タスクに専念して処理できるようにな ります。だからちょっと頭が悪くなっても問題ない。

足軽AはタスクA。足軽BはタスクB。 足軽Cは・・・。足軽Dは・・・。と１個ずつ渡して確実に実行 してもらう。

それぞれのAIの頭の中（コンテキストウィンドウ）が整理されている（クリーン）だからタスク処理 結果の精度も良くなる。 「前の話忘れてない？」が起きずらい。

しかも4タスクが並列で走るので、体感の速度も劇的に上がりました。

1体でやってたときの4倍速…… いや、精度向上で手戻りが減った分も含めると5倍以上ですね。

「5倍って盛ってない？」って思われるかもしれませんけど、 並列処理 x 精度向上 x 手戻り削減の掛け 算なので、むしろ控えめかなと。

ヘッドレス（何してるか見えない）より「見える化」が 好き

マルチエージェント（チーム体制）の運用方法として、 裏で自動実行する「ヘッドレスモード」もあ ります。

全部バックグラウンドで動かして、終わったら結果だけ受け取る。 効率だけ考えるなら、こっちのほ うがいいかもしれません。

でも自分は、tmuxでリアルタイムに見える方が好きなんですよね。

「今、足軽Bがどのファイルをいじってるのか」 「家老がタスクをどう分解したのか」 「将軍がどう いう方針判断を下したのか」

こういうのがリアルタイムで見えると、介入すべきタイミングがわかる。 「あ、足軽Cの方向性がズ レてるな」って気づいたら、 すぐに家老経由で軌道修正できるんです。

あと正直な話、複数の画面で 将軍・家老・足軽が同時に動いてる画は、テンション上がります。それ だけで人間の私の作業効率があがる。不思議。

「自分だけの開発チームが動いてる」って感覚。 これは実際に見てみないとわからない良さがありま すね〜。これをつまみにグビっと一杯イケる。。。マジで。

注意点もある

万能ではないので、注意点も正直に書いておきます。

セットアップは簡単ではない。 なんでもそうですが初めてはよくわからないこともあり「ん？どうやってチームメンバーの動いてい る様を見れるんだ？」とか。使いこなすまでに多少時間はかかります。

ただ、一度組んでしまって使い方に慣れてしまえば高速回転の始まり。

足軽同士のファイル競合に注意。 2体の足軽が同じファイルを同時に編集すると、当然コンフリクトが起きました。でも一回指摘したら 即なおりました。タスク分割の段階で、担当するファイルが被らないように設計する。 ここは家老の 腕の見せどころですね。

調子に乗って足軽増やしたら家老が切腹 もっとメンバー増やそうぜ！って将軍に無理言ったら、将軍も「いいね！」ってノってきてくれたの はいいのですが、肝心の中間管理職がフリーズしました。足軽が4名なのはちゃんとした理由があっ た。

でも一度動き出すと、もう1体運用には戻れなくなります。 そのくらい体験が変わるので。

まとめ

Agent Teamが公式に出た今、マルチエージェントはスタンダードになっていくはず。

個人でも「チーム開発」ができる時代が来てます。 1人なのに、6人分の開発力が手に入る。しかも24 時間文句を言わずに動く。

Shogunシステムは、その先取りみたいなものでした。 公式のAgent Teamと比較してどうか、という のは今後検証していきますね。
