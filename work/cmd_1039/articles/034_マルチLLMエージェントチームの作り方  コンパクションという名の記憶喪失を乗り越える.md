それぞれの特徴をざっくり紹介すると、こんな感じ。

Codex CLI（OpenAI製）: --full-auto で自動実行。AGENTS.md で指示を永続化

Gemini CLI（Google製）: --yolo で自動実行。GEMINI.md で指示を永続化

3つともMCPに対応してるので、タスク管理を統一すれば同じチームに入れられる...はず。

そう思ってました。

最初の壁 ── 「おまえ誰だっけ？」問題

コンパクションでAIエージェントが記憶を失うイメージ図

AIエージェントをチームで動かすには、各メンバーに「パーティロール」を渡す必要があります。

役割、口調、ワークフロー。

「おまえは盗賊だ。タスクはMCPで受け取れ。完了したら報告しろ。」みたいなやつ。

Claude Codeの場合、--append-system-prompt でシステムプロンプト層に注入できる。

これが最強で、どれだけ会話が長くなっても消えない。

じゃあCodexとGeminiは？

...ここで壁にぶつかりました。

CodexにもGeminiにも --append-system-prompt に相当するオプションがない。

「じゃあ初回メッセージとして渡せばいいか」と思ったんですが、これがダメだった。

理由はコンパクション。

コンパクションって何？って話ですよね。

LLMの会話が長くなると、コンテキストウィンドウがいっぱいになります。

そうすると、古い会話を圧縮・削除して容量を空ける処理が走る。

これがコンパクション。

つまり、初回メッセージとしてロールを渡しても、会話が長くなった時点で「古い会話」として消さ れちゃう。

エージェントが「自分が誰か」を忘れる。

しかも口調だけの問題じゃないんですよ。

ワークフロー全体を忘れます。

タスクの受け取り方（get_my_task）を忘れる

完了報告の書き方（update_status）を忘れる

禁止事項（他人のファイルに触るな等）を忘れる

コンパクション復旧手順自体を忘れるので、自力回復が不可能

最後のが一番キツい。

「忘れた時はこうしろ」っていう指示自体が消えるので、詰む。

整理するとこうなります。

またか。

解決策を探す旅 ── CLIのソースコードまで読んだ

CLIのソースコードを読んで解決策を探す調査プロセス

発想はシンプルでした。

「コンパクションで消えない場所に書けばいい。」

Claude Codeには CLAUDE.md がある。

プロジェクトのルートに置いておけば、システムプロンプトとして常に読み込まれる。

じゃあCodexやGeminiにも「CLAUDE.md相当」があるんじゃないか？

ということで調べました。

Codex CLIの場合

GitHub上のソースコードを読み込んでいくと、~/.codex/AGENTS.md というファイルを発見。

これがCLAUDE.md相当で、developerロールメッセージとして注入される仕組みでした。

CLI | ロール注入先 | コンパクションで消える？ Claude Code | システムプロンプト層 | 消えない（最強） Codex CLI | 初回メッセージ | 消える（致命的） Gemini CLI | 初回メッセージ | 消える（致命的）

さらに嬉しいことに、experimental_compact_prompt_file という設定がある。

コンパクションが発生した時に「何を保持するか」を制御できるファイルを指定できる。

つまり、AGENTS.mdでロールを注入 + autocompact.mdで保持指示。

二重防御が組めます。

Gemini CLIの場合

こちらもソースコードを読みました。

~/.gemini/GEMINI.md というファイルがあって、これが systemInstruction 層に注入される。

で、ここが肝心なんですけど、コンパクションの実装（chatCompressionService.js）を追ってみると...

コンパクション後に startChat() で新しいチャットが再作成される。

そのとき getCoreSystemPrompt() でGEMINI.mdが再注入される仕組みになってた。

つまり、コンパクションで消えるのは古い会話履歴だけ。

GEMINI.mdは絶対に消えない。

これだ。
