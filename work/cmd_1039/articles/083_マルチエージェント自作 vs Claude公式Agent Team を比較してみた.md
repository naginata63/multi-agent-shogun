そもそもShogunシステムって何？

将軍→家老→足軽という戦国時代の軍制を、 AIエージェントの統率に応用したシステムです。

通信はYAMLファイルベース。 tmux上で6つのペインが同時に動いてる状態ですね。

特徴は「手動オーケストレーション」。 人間が将軍に指示を出したら、あとは軍勢が勝手に動く。

ただし、セットアップは全部手作業。 tmuxのペイン作って、YAMLのキューを用意して、 各エージェ ントに役割（システムプロンプト）を与える。

要は「フルカスタム」なわけです。

Claude公式のAgent Teamって何？

Claude Code v2.1.30から実装された、公式のマルチエージェント機能です。

使い方は超シンプル。

将軍（1体）: 全体の指揮・方針決定 ↓ 家老（1体）: タスク分解・進捗管理 ↓ 足軽（4体）: 並列でタスクを実行

claude --agent-team

たったこれだけ。

Claude Codeが自動的に複数のエージェントを立ち上げて、 タスクを分担して並列処理してくれま す。

セットアップゼロ。 YAMLもtmuxも不要。何も考えなくていい。

ただし、全エージェントがClaude（SonnetかOpus）。 後述しますが、これがコスト面で重要なポイン トになります。

他の選択肢もある：AutoGen、CrewAI、 LangGraph

マルチエージェントのアプローチは他にもあります。

AutoGen（Microsoft Research）

Microsoft製のマルチエージェントフレームワーク。 Python製で、複数のエージェントに役割を与えて 会話させる形式。

エージェント同士の会話ログが残るので、 「なぜその判断をしたか」が後から追いやすい。

ただし、Pythonのコードをガッツリ書く必要があります。 プログラミング経験がないと厳しいかも。

CrewAI

タスクベースでエージェントを配置する考え方。 「このタスクにはこのエージェント」を明示的に定 義する感じ。

わかりやすいんですけど、こちらもコーディングが必要。 しかもそこそこボイラープレートが多い印 象。

LangGraph

LangChainのエコシステムの一部。 グラフ構造でエージェントのフローを定義する。

強力なんですけど、学習コストが高い。 「とりあえず動かしてみる」のハードルはやや高めです。

比較軸を5つ設定してみた

以下の5つの軸で、各アプローチを比較します。

1. コスト: 運用にかかる金銭的負担

2. 制御性: どこまで細かく制御できるか

3. 柔軟性: LLMの選択肢、カスタマイズ性

4. セットアップ複雑度: 導入の手間

5. 信頼性: クラッシュ時の回復、安定性

それぞれ見ていきます。

1. コスト比較

Shogun: 格安運用が可能

僕の構成はこうです。

将軍（1体）: Claude Opus 4.6（Max Plan）

家老・足軽（5体）: GLM-4.7（Z.AI経由、年間5,000円）

実質コストは年間5,000円 + Max Planの月額。

GLM-4.7がやばくて。 Sonnet 4.5よりちょっといいレベルのLLMが、 年間5,000円で実質使い放題な んですよね。

だから、判断力が必要な将軍だけOpusにして、 作業をこなす家老・足軽はGLMに切り替えてます。

コスパ、ぶっちぎりでいい。

Agent Team: 全員Claudeなのでコストかさむ

Agent Teamは全エージェントがClaude。

仮に5体のチームを作ったとすると、 全員がProプランのトークン制限を共有する形になります。

並列処理してる間にトークンがガンガン減る。 月額制限に引っかかりやすいんですよね。

「無限にタスクを投げ続ける」みたいな使い方は厳しい。

AutoGen / CrewAI / LangGraph: LLM選択肢が広い

これらのフレームワークは、OpenAI、Anthropic、ローカルLLM、 何でも組み合わせられます。

「重いタスクはGPT-4、軽いタスクはGPT-3.5」とか、 「安いタスクはローカルLlama」みたいな使い 分けも可能。

ただし、その設定を自分で書く必要がある。 コード書けないと詰む。

2. 制御性：どこまで細かく制御できるか

Shogun: フルコントロール

YAMLファイルベースの通信なので、 「誰が何をどう判断したか」が全部見えます。

こういうYAMLが家老に渡って、家老がタスクに分解する。 その分解されたタスクも全部YAMLで残 る。

途中でおかしな判断をしてたら、 YAMLを直接編集して軌道修正できます。

Agent Team: ブラックボックス

Agent Teamは完全にClaude Code任せ。

内部でどういう判断がされてるかは見えません。 途中で介入もできない（と思う。少なくとも公式ド キュメントには書いてない）。

「自動運転モード」みたいなもんですね。 楽だけど、制御できない。

AutoGen / CrewAI / LangGraph: コードで制御

これらはコードで制御ロジックを書くので、 理論上は完全にコントロール可能。

command: cmd_id: cmd_003 objective: | 記事を2本書く。 記事1は○○について。記事2は△△について。 constraints: - 画像は8枚以上 - 書き味ルールに厳密に従う status: assigned

ただし、そのためにはフレームワークを理解して、 Pythonを書いて、デバッグして……という手間が かかります。

3. 柔軟性：LLMの選択肢

Shogun: 複数LLMを混在可能

僕の構成だと、将軍だけOpus、他はGLM。 こういう「ハイブリッド構成」ができます。

しかも、Claude以外のLLMも使える。 Kimi Code、Codex、Copilotにも対応してます。

「予算に合わせて最適なLLMを選ぶ」が可能。

Agent Team: Claude縛り

Agent TeamはClaude縛り。 OpenAIとか、GLMとか、他のLLMは使えません。

まあ、Claude Code使ってる時点でClaude好きなわけで、 そこまでデメリットには感じないかもです が。

AutoGen / CrewAI / LangGraph: 最も柔軟

これらのフレームワークはLLM選択肢が最も広い。

OpenAI、Anthropic、Google、Cohere、HuggingFace、 ローカルのOllamaまで、なんでも組み込めま す。

ただし繰り返しになりますが、 その設定を書く技術力が必要。

4. セットアップの複雑度

Shogun: そこそこ手間

tmuxセッションを作って、YAMLキューを用意して、 各エージェントに役割（システムプロンプト） を設定する。

初回セットアップに半日くらいかかりました。

ただし、一度作ったら使い回せます。 起動スクリプトを叩くだけで6体が立ち上がる。

【画像: setup-process-timeline.png】

Agent Team: セットアップゼロ

これがAgent Teamの最大の強み。

これだけ。 3秒で起動。何も考えなくていい。

初心者でも即使える。

AutoGen / CrewAI / LangGraph: 学習コスト高い

これらはフレームワークのドキュメントを読んで、 Pythonコードを書いて、依存関係をインストール して……。

初回セットアップに数日かかることもザラ。

エンジニアじゃないとキツいですね。

bash bin/shogun-start.sh

claude --agent-team

5. 信頼性・復旧性

Shogun: 手動復旧が必要

エージェントがクラッシュしたとき、 手動で復旧する必要があります。

家老が無応答になったら、tmuxペインを確認して、 /clear でコンテキストリセットして再起動。

tmuxの知識がないと詰む。

ただし、ハンドオーバー機能を作って、 クラッシュ前の状態をYAMLに保存する仕組みは入れてます。

完全自動復旧ではないけど、 「どこまでやってたか」は引き継げる。

Agent Team: 自動復旧（多分）

公式のAgent Teamは、内部でエラーハンドリングしてるはず。

エージェントが落ちても、Claude Codeが自動で立て直してくれる…… と思いますが、正直まだ検証 できてないです。

少なくとも、Shogunみたいに手動介入する必要はない。

AutoGen / CrewAI / LangGraph: 自分で実装

エラーハンドリング、リトライロジック、 全部自分で書く必要があります。

一番大変だけど、一番柔軟。

実際に使ってみた感想

tmux send-keysの不安定さ（詳細）

tmux send-keysの本質は「ターミナルにキーボード入力を偽装する」こと。 ゴーストが相手のキーボ ードを勝手に叩いているようなもの。

具体的な問題：

1. 配信確認ができない（Fire and Forget）: 送ったあと、相手が受け取ったか・処理したかは分から

ない。sleepしてcapture-paneで目視確認するしかない

2. タイミング問題: 相手がツール実行中→入力を受け付けない。テキスト出力中→混ざってゴミにな

る。コンパクション中→そもそも応答不能

3. 双方向でも同じ問題: 家老→将軍に報告をsend-keysで送る場合も、将軍側の状態が制御できないの

で同じ不安定さが発生

4. メッセージとEnterを別コマンドで送る必要がある: 1回で送ると動かない

Agent Teamsとの違い:

Agent Teamsのメールボックスは「キュー」。相手が忙しくても溜めておいて、準備ができたら読 む

send-keysにはこのバッファがない

イベント駆動 vs ポーリング問題

tmuxには通知の仕組みがない。状態を知りたければポーリングするしかない

ポーリング = トークンが溶ける → だから将軍システムではポーリング禁止（F003）

代わりに「人間（上様）がポーリングの代わり」になっている。進捗確認は人間が聞いた時だけ

Agent Teamsはメンバーの完了・アイドルが自動通知。リーダーは待っていればメッセージが来る

コスト制御の詳細

Agent Teamsでもモデル指定（Sonnet等）は可能だが、Anthropic API内の選択肢に限られる。GLMや Geminiへの振り分けは不可能。

初心者→上級者の成長パス（記事の構成案）

セッション永続性の実態

正直な結論: 今の使い方では永続性の恩恵をほぼ受けていない

大半のタスクは1セッション内で完了（記事執筆、画像生成、調査 = 数十分〜数時間）

ハンドオーバー機能は作ったが、実際にはほぼ使われていない（起動時に「157時間前に更新」と 出る始末）

家老コンパクション復旧の仕組みもあるが、コンパクション時点でタスクは終わりかけか、最初か らやり直した方が早いことが多い

つまり永続性は「保険」として設計したが、「保険金を受け取った場面がない」

【Agent Teams】全員Anthropic API課金 リーダー: Claude (Opus/Sonnet) メンバー×4: Claude (Sonnet) → Cコンパイラ事例で$20,000

【将軍システム】将軍だけAnthropicに課金、他はGLM 将軍: Opus 4.6 家老×2 + 足軽×8: GLM-4.7（Z.AI API、格安） → 10人中9人が格安LLM。桁違いのコスト削減

初心者: Agent Teams（環境変数1つで即開始） ↓ 慣れてきた、もっとカスタマイズしたい 中級者: メンバー間通信、ダッシュボード、レポート体制がほしくなる ↓ 上級者: tmux + 独自アーキテクチャ（将軍システムのような）

永続性が本当に効くはずの場面（仮説・未検証）：

数日〜数週間かかる大規模開発

途中経過が複雑でゼロから再構築するとコストが大きい場合

夜間にエージェントを走らせ続けて朝結果を見る運用

→ ただし、これらはまだ試していない。本当の価値は実際にやってみないと分からない。

夜間自律運転の壁

「夜間に走らせて朝結果を見たい」を実現するための壁：

実質的な壁は1と2だけ。

コンパクション復旧は実はそこそこ自動化されている

最初「復旧が自動化されていない」と言ったが、これは不正確だった。

Claude Code自体の仕組み：

コンパクションは自動で起きる（コンテキストが長くなると自動圧縮）

圧縮後もエージェントは止まらない。そのまま動き続ける

CLAUDE.mdは毎回再読み込みされるので基本ルールは保持

TaskCreate/TaskUpdateのタスクリストもコンパクション後に残る

将軍システムの仕組み：

家老のロールファイルにコンパクション復旧手順が書いてある

タスクYAML、ダッシュボードはディスク上のファイルなので消えない

コンパクション後、家老はロールファイルを再読して、YAMLから状態を復元

コンパクション後に失われるもの：

1. トークンリミット → Claudeでは数時間が限界。GLMなら長時間可能 2. 人間の不在 → タスク完了後の次指示、エラー対応が無人 3. コンパクション → △ そこそこ自動復旧できる（致命的ではない）

会話の細かいニュアンス（要約されるので）

「なぜこの判断をしたか」の文脈

途中の試行錯誤の記憶

コンパクション後に残るもの：

CLAUDE.md（基本ルール）

タスクYAML（何をやるべきか）

ダッシュボード（全体の進捗）

ロールファイル（自分は何者か）

→ 完全復旧ではないが致命的ではない。明確なタスクをこなす足軽レベルなら問題なし。

残る本質的な壁

GLMを使えばトークンリミットは消える。コンパクションもそこそこ復旧できる。 残るのは「人間が いない時に次の指示を誰が出すか」——これだけ。

これが解決されれば、夜間自律運転が現実になる。Agent Teamsのリーダー機能はここに一つの答え を出しているが、Agent Teamsは逆にトークンリミットの壁がある。どちらも一長一短。
