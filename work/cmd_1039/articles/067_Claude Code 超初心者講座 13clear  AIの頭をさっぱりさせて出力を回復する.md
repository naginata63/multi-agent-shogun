コンパクションの仕組み図
まず前提として、僕は
/clear
をほとんど使いません。
Claude Codeには自動コンパクションという機能があって、会話が長くなると勝手に圧縮してくれま
す。 手動で
/compact
を打つこともできますが、基本は自動で回る。
普段の作業はこれで十分です。 会話が長くなりすぎてAIの調子が悪くなる、みたいなことはあまり起
きない。
じゃあ
/clear
はいつ使うのか。 一番わかりやすいのが、計画 → 実装の切り替えポイントです。
計画のあと、リセットするか否か
計画後にコンテキストを残すか/clearするかの判断図
Claude Codeでちょっと大きめのタスクを頼むと、まず計画を立ててくれます。 「こういう方針でや
ります」とプランを見せてくれて、OKを出すと実装に入る。

<!-- page 3 -->

このとき、選択肢が出ることがある。
そのまま実装する（/clearしない）
計画の議論がまるごとコンテキストに残る
「なぜその方針にしたか」の経緯もAIが覚えている
ただし、計画中のやり取り（ボツ案、迷い、修正指示）もノイズとして残る
コンテキストの枠を計画の会話が食うので、実装に使える余裕が減る
/clearしてから実装する
会話がまっさらになる
AIは計画ファイルだけを読んで、ゼロから実装を始める
コンテキストの枠を最大限実装に使える
ただし、計画ファイルに書かれていないニュアンスは消える
どう使い分けるか
判断基準はシンプルです。
そのまま行く方がいい場面:
計画がシンプルで、すぐ実装に入れるとき
計画中に「ここはこうしてね」と細かいニュアンスを伝えたとき
変更するファイルが少ないとき
/clearした方がいい場面:
計画の議論が長引いたとき（ボツ案や方針転換が多かった）
実装の規模が大きくて、コンテキストの余裕が欲しいとき
計画が明確にファイルに書き出されているとき
要は、計画中の会話に「残す価値のある情報」がどれくらいあるかで決まる。

<!-- page 4 -->

計画ファイルに全部書いてあるなら、会話を残す意味はない。リセットして、きれいな状態で実装に
入った方がいい。 逆に「口頭で伝えたニュアンス」が多いなら、そのまま続けた方がAIの精度は高
い。
もう一つの出番：AIの思い込みをリセットしたいとき
AIの誤診を/clearでリセットする流れ
計画→実装の切り替え以外にも、
/clear
が効く場面があります。 AIが間違った方向にハマって、修正しても戻ってこないとき。
たとえばこういうケース。
「このページの表示が崩れてるから直して」と頼んだら、AIが「見た目のレイアウトの問題ですね」
と判断して修正を始めた。 でも実際はレイアウトじゃなくて、データの取り方がおかしかっただけ。
「いや、見た目じゃなくてデータの方だと思う」と伝えても、AIの頭にはもう「レイアウト問題」と
いうフレームができている。 データ側を見てくれるけど、なんとなくレイアウト寄りの解釈が残る。
修正案もどこかズレてる。
これ、会話が長いほど起きやすい。 AIは過去のやり取り全部を踏まえて判断するので、最初の誤診が
後の判断をずっと引っ張る。
こういうときは「違う、こうだ」と説得するより、
/clear
して最初から頼み直す方が早い。
「このページの表示が崩れてる。データ取得のロジックを確認して」
一発目の指示を正確にすれば、余計な思い込みなしでまっすぐ動いてくれます。

まとめ
普段のコンテキスト管理は自動コンパクションに任せればいい
/clear
が効くのは、計画 → 実装の切り替えポイント
計画の議論が長くてノイズが多いなら
/clear
計画がシンプルで口頭ニュアンスが大事ならそのまま
AIが間違った方向にハマったら、説得するより
/clear
してやり直す
