将軍システムって何なのか

簡単に言うと、Claude Codeの複数エージェントを戦国時代の軍制で指揮する仕組みです。

構成はこう。

将軍：全体の指揮。人間からの指示を受けて、家老に委任する

家老：タスク分解と進捗管理。将軍の指示を実行可能なタスクに分解して、足軽に配分する

足軽：実働部隊。家老から受けたタスクを並列で実行する

人間がやることは、将軍に最初の指示を出すだけ。

あとは軍勢が勝手に動いてくれる。

なぜ「将軍」なのか

名前の由来は、まあ戦国時代の軍制なんですけど、理由は明確です。

階層構造が必要だったから。

マルチエージェントって、フラットに並べるだけだと指揮が取れないんですよね。

4体のAIが好き勝手に動いたら、ファイルの競合が起きるし、タスクが重複するし、誰も進捗を把握で きない。

だから「指揮官（将軍）→マネージャー（家老）→実働部隊（足軽）」という階層を作った。

役割が明確だから、混乱しないんです。

将軍（1体） └ 家老（1体） └ 足軽（4体）

通信プロトコルはYAMLファイル

エージェント間の通信は、YAMLファイルベースで動いてます。

実績：SubTracker MVPを45分で完成させた

このシステムを使って、実際にMVPを作りました。

SubTrackerという、サブスク管理アプリです。

NetflixとかSpotifyとか、複数のサブスクの支払い予定を一元管理できるやつ。

技術スタック：

Next.js 15 (App Router)

Supabase (認証 + DB)

Stripe (課金)

Tailwind CSS + shadcn/ui

これを、4体の足軽が並列で作った。

タスク分解はこんな感じ。

足軽1: Next.jsプロジェクト作成 + Tailwind + shadcn/ui セットアップ

足軽2: Supabase認証（login/signup ページ）

足軽3: Stripe統合（pricing ページ + checkout API）

足軽4: ダッシュボードページ + README + 最終動作確認

依存関係があるから、足軽1が完了してから足軽2・3・4が動き出す。

でも、足軽2と3は並列で動ける。

結果、45分でMVP完成。

1体でやってたら、どう考えても3〜4時間かかってたと思う。

もう1つの実績：2記事 + 16枚の画像を並列生成

記事執筆でも使いました。

以前、「LLM切り替えの仕組み」と「ハンドオーバー機能」という2つの記事を同時に作ったんですけ ど、これも将軍システムで回しました。

タスク分解：

足軽1: 記事1執筆（3000字）

足軽2: 記事1の画像7枚生成

足軽3: 記事2執筆（3400字）

足軽4: 記事2の画像8枚生成

全部並列で動いたから、2記事 + 16枚の画像が1時間かからず完成。

1体で順番にやってたら、半日かかってたでしょうね。

しかも、記事執筆と画像生成のコンテキストが混ざらないから、精度が落ちない。

これが一番大きいメリットだと思ってます。

コンテキストが濁らない = 精度が上がる

将軍システムの最大のメリットは、コンテキストが混ざらないこと。

1体のAIに全部やらせると、フロントとバックエンドとインフラの情報が混ざるんですよね。

そうなると、「あれ、さっきの変数名なんだっけ」みたいなブレが出てくる。

でも将軍システムだと、各足軽が1タスクに専念してる。

足軽A：フロントエンド専任

足軽B：バックエンド専任

足軽C：API設計専任

足軽D：テスト専任

それぞれのコンテキストがクリーンだから、出力の精度が全然違う。

「前の話忘れてない？」が起きない。

これが、並列化のスピードアップに加えて、精度向上でさらに手戻りが減るという掛け算になるんで す。

だから5倍速って言ってるのは、盛ってるわけじゃないんですよね。
