noteの投稿、自動化したくない？

sasuu｜非エンジニアのAI活用術 2026年2月7日 14:47

前回の記事で、Claude Codeのサブエージェントを使った記事生成の仕組みを作りました。

「記事書いて」と言うだけで、自分の書き味の記事が出てくる。 画像も自動生成される。 下書きファイルとして保存もされる。

ここまでは完成してます。

でも、最後の「noteに投稿する」だけが手動なんですよね。

下書きファイルを開いて、noteのエディタにコピペして、画像を1枚ずつ挿入して。 これが地味にダルい。

ということで、ここも自動化しよう、と。

結論から言うと、この「ラストワンマイル」が一番キツかったです。

非公式APIを調べてみた

まず最初に考えたのはAPIでの投稿です。

ただ、note.comには公式のAPIがありません。 外部サービスとの連携用APIは存在しないんですよね。

でも、ブラウザのDevToolsでネットワークを見ると、内部的にはAPIを叩いてます。 `https://note.com/api/v1/` 系のエンドポイントが見える。

これを使えないか調べてみました。

できたこと

認証: メールアドレスとパスワードでログインしてCookieを取得

下書き作成: `POST /api/v1/text_notes` で新規下書きが作れる

テキスト投稿: 記事の本文テキストを下書きに書き込める

ここまでは順調でした。

できなかったこと

画像の挿入: エディタ内の特定の位置に画像を差し込むAPIが見つからない

リッチテキスト操作: noteのエディタはProseMirrorベースで、内部のJSON構造がかなり複雑

テキストだけならAPIでいけます。 でも画像が入った瞬間、詰む。

ProseMirrorのドキュメント構造をJSON形式でAPIに投げる必要があるんですが、画像ノードの構造が エディタの内部仕様に依存してます。 外から組み立てるのが現実的じゃなかったんですよね。

全部Playwrightでやることにした

APIでは画像挿入が無理。

じゃあブラウザ自動操作でやるしかない。 Playwright（ブラウザ自動操作ツール）に全振りすることにしました。

Claude CodeにはPlaywright MCPというブラウザ操作ツールが使えます。 これでnoteのエディタを直接操作する作戦です。

テキスト貼り付けは成功

記事のテキストはわりとすんなりいきました。

1. Markdownの記事をクリップボードにコピー（`pbcopy`）

2. noteのエディタにフォーカス

3. `Cmd+V` で貼り付け

noteのエディタはMarkdownの貼り付けに対応してるので、見出しやリストもちゃんと反映されます。

ここまでは「お、いけるやん」という感じでした。

問題はここからです。

ProseMirrorとの戦い

noteのリッチテキストエディタはProseMirrorというライブラリで作られてます。

ProseMirrorは高機能なエディタフレームワークなんですが、外部からの自動操作を一切想定してない んですよ。

これが地獄の始まりでした。

Endキーが段落の末尾に行かない

画像を挿入するには、まず「この段落の後ろに画像を入れたい」という位置にカーソルを置く必要が あります。

普通に考えたら、段落をクリックして `End` キーを押せば段落の末尾に行きますよね。

行かないんです。

ProseMirrorの `End` キーは「段落の末尾」じゃなくて「視覚的な行の末尾」に移動します。

たとえば、こういう段落があるとします。

問題があれば自動で修正してから保存される。人間がチェックする手間がぐっ と減ります。

これがエディタ上で2行に折り返されてるとき、1行目の末尾で `End` を押すと「減り」の後ろにカー ソルが行く。 「ます。」の後ろじゃなくて。

ここで `Enter` を押すと、段落が「減り」と「ます。」に分割されて、画像がその間に入ってしまう。

Enterキーで段落が作られない

じゃあ段落の末尾に正確にカーソルを置いて `Enter` を押せばいいかと思うじゃないですか。

段落が作られないんです。

DOMのSelection APIでカーソル位置を設定しても、ProseMirrorの内部状態と同期しない。 `Enter` を押しても、テキストノードが分割されるだけで、新しい段落ブロックが生成されない。

何度やっても同じ結果でした。

DOM Selection APIをProseMirrorが無視する

「じゃあJavaScriptで直接カーソルを設定すれば？」と思いますよね。

こういうコードでカーソルを段落末尾に配置する。

DOMレベルでは確かにカーソルが移動してます。 でもProseMirrorはこの変更を認識しない。

ProseMirrorは独自の内部ステート管理を持っていて、DOMの変更を直接反映しないんです。

ProseMirrorのviewにアクセスできない

ProseMirrorを正しく操作するには `view` オブジェクト経由で `dispatch` する必要があります。

でもnote.comのエディタでは、この `view` オブジェクトがクロージャの中に閉じ込められてます。

DOMから辿っても、React Fiberから辿っても、`window` グローバルを探しても見つからない。 正規のAPIにアクセスする方法が存在しませんでした。

唯一の成功パターン

数時間の試行錯誤の末、1つだけ動くパターンを見つけました。

これだけが、ProseMirrorに正しく認識される段落作成方法でした。

ただし、これにも制約があります。 見出し（heading）の直前では使えない。

const range = document.createRange(); range.setStart(lastTextNode, lastTextNode.textContent.length); window.getSelection().removeAllRanges(); window.getSelection().addRange(range);

// pmViewDescにはこれだけ {parent, children, dom, contentDOM, dirty, node, outerDeco, innerDeco, nodeDOM} // viewプロパティが無い

triple-click（段落全体を選択）→ ArrowRight（選択を末尾に畳む）→ Enter（新段落作成）

見出しの前に新しい段落を挿入しようとすると、見出しのテキストが分割されたり、カーソルが意図 しない場所に飛んだりします。

画像10枚の結果

ヘッダー画像1枚 + 本文中の画像9枚、合計10枚を挿入した結果がこちらです。

10枚中、ノーミスで入ったのは5枚。4枚はテキスト分割で手動修正が必要。1枚はそもそも挿入でき なかった。

about:blank事故

最後の10枚目を挿入しようとしていたとき。

画像の位置がずれたので `Cmd+Z`（undo）を押しました。

するとブラウザのページが `about:blank` に遷移。

エディタごと消えた。

再度アクセスしようとしたら、セッションのCookieも無効になっていて、ログインページにリダイレ クトされました。

数時間の作業がここで強制終了。

まぁ下書き保存は自動で走っていたので、9枚目までの状態は残ってましたが。 精神的ダメージはでかかったですね、正直。

自動化してよかったところ

記事生成（サブエージェント）: 「書いて」で記事が出てくる。これは最高

画像生成（Gemini連携）: 記事に合った画像が自動で出てくる。これも最高

下書きファイル保存: draftsフォルダに自動保存。管理が楽

テキスト貼り付け（Playwright）: クリップボード経由でMarkdown貼り付け。これは成功

自動化しないほうがよかったところ

noteエディタへの画像挿入: 手動5分 vs 自動で数時間 + $45相当のトークン

「最後の10%」の罠

全体の90%は自動化できてました。 記事生成、画像生成、ファイル保存、テキスト貼り付けまで。

残りの10%（画像の位置指定と挿入）に、全体の90%以上のコストがかかった。

これ、ソフトウェア開発でも似たパターンありますよね。 「90%完成したけど、残り10%に同じ時間かかる」ってやつ。

90%の自動化で止めておく勇気

「全自動」は魅力的です。 「あと少しで完全自動化」と思うと、つい突き進みたくなる。

でも、コストに見合わない自動化は自動化じゃなくて趣味です。

今の自分の結論はこうなりました。

記事生成 → 自動（サブエージェント）

画像生成 → 自動（Gemini連携）

ファイル保存 → 自動

テキスト貼り付け → 自動（Playwright）

画像挿入 → 手動（5分で終わる）

90%の自動化で十分。 残り10%は手でやったほうが速いし安い。
