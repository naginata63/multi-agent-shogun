# QCチェック基準テンプレート

家老がQCタスクYAMLを作成する際、成果物タイプに応じてこのテンプレートからチェック項目をコピーせよ。
軍師/将軍がQCを実施する際もこの基準に従う。

## 絶対ルール（全タイプ共通）

1. **実ファイルを必ず読め** — grepカウント・数値報告だけでPASS/FAIL判定するな
2. **冒頭・中盤・終盤の3箇所** を目視確認せよ（冒頭だけ見て「OK」は禁止）
3. **「問題なし」ではなく証跡報告** — 「Xを確認しYだった」と具体的に書け
4. **タスクYAMLの前提情報を鵜呑みにするな** — 足軽の報告値と実ファイルが一致するか自分で検証せよ
5. **所要時間の目安を守れ** — 3成果物のQCを14分で終わらせるな。実ファイル読みだけで最低20分かかる
6. **フォールバック＝異常** — ログに「フォールバック」「fallback」「warn」が1件でもあれば、それは「動いているが正しくない」状態。PASSにするな。症状の消失（特定エラーが出ない）ではなく、根本原因の解消を確認せよ（2026-03-15 cmd_705教訓: smile_r2フォールバック0件でPASSしたが、surprise_r3→r2の同表情フォールバックが残っていた）
7. **acceptance_criteriaが症状で定義されている場合、根本原因まで遡って確認せよ** — 基準を満たしていても、上流に問題が残っていればNG
8. **QC開始前にclaude-memで過去知見を検索せよ** — `bash scripts/cmem_search.sh "<対象のキーワード>"` を実行し、過去の類似QCでの指摘・失敗・ベストプラクティスを確認してからチェック項目に入れ。出力形式・curl例・詳細は `shared_context/cmem_quickstart.md` 参照

## タイプ別チェック項目

### A. SRTファイル（字幕）

#### A-1. 構造チェック
- [ ] 冒頭30エントリを読み、SRT形式（index/timestamp/text）が正しいか
- [ ] 中盤30エントリを読み、内容が動画の中盤に対応しているか
- [ ] 終盤30エントリを読み、エンディング付近の内容か

#### A-2. タイムスタンプ検証
- [ ] 時系列逆行チェック（逆行0件であること）
```bash
python3 -c "
import re
with open('TARGET.srt') as f:
    times = re.findall(r'(\d{2}):(\d{2}):(\d{2}),(\d{3})', f.read())
starts = [int(h)*3600+int(m)*60+int(s)+int(ms)/1000 for h,m,s,ms in times[::2]]
jumps = sum(1 for i in range(1,len(starts)) if starts[i] < starts[i-1] - 1.0)
print(f'逆行: {jumps}件')
print(f'エントリ数: {len(starts)}')
"
```

- [ ] 時間分布（5分バケット）— 特定区間に集中していないこと
```bash
awk '/-->/{split($0,a,"-->"); gsub(/[[:space:]]/,"",a[1]); split(a[1],s,/[:,]/); min=int((s[1]*3600+s[2]*60+s[3])/300); bucket[min]++} END{for(i=0;i<=9;i++) printf "%d-%dmin: %d entries\n", i*5, (i+1)*5, bucket[i]+0}' TARGET.srt
```

- [ ] 尺分布 — 200ms未満の超短尺エントリが10%以下であること
```bash
awk '/-->/{split($0,a,"-->"); gsub(/[[:space:]]/,"",a[1]); gsub(/[[:space:]]/,"",a[2]); split(a[1],s,/[:,]/); split(a[2],e,/[:,]/); st=s[1]*3600+s[2]*60+s[3]+s[4]/1000; et=e[1]*3600+e[2]*60+e[3]+e[4]/1000; d=et-st; if(d<0.2)short++; total++} END{print "<200ms:",short+0,"/"total,"("short/total*100"%)"}' TARGET.srt
```

- [ ] フォールバック率 — ,000（秒精度）の割合が基準以下であること
```bash
grep -oP ',\d{3}' TARGET.srt | sort | uniq -c | sort -rn | head -10
```

#### A-3. コンテンツ検証
- [ ] 話者ラベルが正しい形式か（[ドズル]: 等。Speaker_1等の生IDが残っていないこと）
- [ ] 動画の既知のシーン（タイトルコール、エンディング等）が対応する時間帯にあるか
- [ ] 大きな欠落がないか（20分間エントリゼロ等）

---

### B. YAMLデータ出力（候補リスト等）

#### B-1. 構造チェック
- [ ] YAMLとして正しくパースできるか
- [ ] 全文読み — 全候補の内容を目視確認（件数だけ数えるな）

#### B-2. 候補品質
- [ ] 全候補の尺が指定範囲内か（ショート: 15-60秒、ハイライト: 指定による）
- [ ] タイムスタンプが動画長の範囲内か
- [ ] 動画全体をカバーしているか（前半だけ・後半だけに偏っていないか）
- [ ] メンバー名がマッピング済みか（dozle, bon, qnly, orafu, oo_men, nekooji のいずれか）
- [ ] 各候補の「理由」が具体的か（「面白いシーン」等の汎用文でないこと）
- [ ] viral_score等のスコアに根拠があるか（全部同じスコアでないこと）

#### B-3. 欠落チェック
- [ ] 必須フィールドが全候補にあるか（id, start, end, title, speakers等）
- [ ] HL候補が0件の場合: バグかコンテンツ不足か判断し、バグなら即FAIL

---

### C. スクリプト/コード

#### C-1. 動作確認
- [ ] 指定のテストコマンドを実行して正常終了するか
- [ ] 出力ファイルが期待通りの形式・内容か
- [ ] エラーケース（ファイルなし、API失敗等）で適切なエラーメッセージが出るか

#### C-2. コード品質
- [ ] 新規関数/変更箇所を読み、ロジックが正しいか
- [ ] ハードコードされた値がないか（パス、APIキー、モデル名等）
- [ ] 既存機能を壊していないか（変更前後の動作を比較）

---

### D. 動画/画像

- [ ] ffmpegで冒頭フレームを抽出して目視確認
- [ ] 解像度・アスペクト比が仕様通りか
- [ ] 音声トラックが存在し再生可能か
- [ ] テキスト表示（字幕・テロップ）が正しいか

---

## 判定基準

| 判定 | 条件 |
|------|------|
| PASS | 全チェック項目をクリア |
| CONDITIONAL PASS | 軽微な問題のみ（問題点と「修正不要」の理由を明記） |
| FAIL | 1つでも重大な問題あり（NG箇所と修正指示を具体的に記載） |

**重大問題の例:**
- データの大幅な欠落（20分間ゼロ等）
- タイムスタンプの時系列崩壊
- 必須候補が0件（バグ起因）
- 97%以上のエントリが異常値

## QCレポート記述フォーマット

```
### [成果物名] — [PASS/CONDITIONAL PASS/FAIL]

確認内容:
- A-1構造: 冒頭Entry1-30読了。[具体的所見]
- A-2逆行: 0件 ✅ / X件 ❌
- A-2分布: [5分バケットの結果コピペ]
- A-2短尺: <200ms X件/Y件 (Z%)
- A-3話者: [確認結果]

問題点:
- [具体的な問題。エントリ番号やタイムスタンプを明記]

修正指示（FAILの場合）:
- [何をどう直せば良いか]
```

## 使い方

家老がQCタスクYAMLを書く際:
1. このテンプレートを参照指示に含める: `shared_context/qc_template.md を参照してQCせよ`
2. 成果物タイプに応じたセクション（A/B/C/D）を指定
3. 検証コマンドのTARGET.srtを実際のファイルパスに置換
4. プロジェクト固有の追加チェック項目があれば追記

## 動画タスクQC追加項目

動画系cmd（視点切替MIX・4画面MIX・ハイライト等）のQCでは、上記タイプ別チェックに加え以下を**必須実行**せよ。
ffprobe/API確認だけでQC PASSを出してはならない（cmd_597/cmd_1464教訓）。

### E-1. 視覚検証（必須）
- [ ] `mpv --speed=2.0` でフル視聴（38分動画なら4-5分）
- [ ] seg境界の transition（wipeleft等）が正しく適用されているか視認
- [ ] 視点切替パターンが鉄則3（最初/最後 基準視点固定 + 中間バリエーション）に準拠しているか視認
- [ ] 右上テロップ規格（鉄則4: `<ボス名>(<戦闘番号>)` 形式）が正しいか視認
- [ ] 1戦目短seg（≤200秒）の場合、シンプルパターン（MEN→4画面→MEN）が適用されているか確認

### E-2. sync_record検証
- [ ] `sync_record.yaml` が存在し、multi_view_sync.md Step 7フォーマットに準拠しているか
- [ ] `visual_verification.result=pass` が記録されているか
- [ ] 各segの `view_switches` が実際のMIX結果と一致するか（冒頭・中盤・終盤3箇所サンプリング確認）

### E-3. サンプル動画検証
- [ ] `verify_60s_*.mp4` サンプル動画を実再生し、音声echo無し・視点切替正常を確認
- [ ] 解像度・アスペクト比が仕様通りか

### E-4. 総合判定（動画系）
- [ ] 上記E-1〜E-3の全項目をクリアでのみPASS判定
- [ ] ffprobe/APIチェックのみでPASSは**禁止**（cmd_1464 QC形骸化の再発防止）
- [ ] 視覚検証をスキップした場合は必ず理由を明記し、CONDITIONAL PASS扱いとする

### E-5. master/telop 二段確認（cmd_1486 恒久ルール）
- [ ] master.mp4 (テロップなし) が存在し、成果物ディレクトリに保管されていること
- [ ] with_telop.mp4 が master.mp4 + drawtext 1passで生成されていること (ffprobe で filter_complex 痕跡確認)
- [ ] master.mp4 と with_telop.mp4 の動画長が一致していること (`ffprobe -v error -show_entries format=duration`)
- [ ] master.mp4 にテロップが入っていないこと (`ffmpeg -ss 5 -i master.mp4 -frames:v 1` で目視確認)
